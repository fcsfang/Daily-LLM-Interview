### 题目 1：在 Agent 长短期记忆设计中，当新信息与旧记忆冲突时，你是如何进行冲突仲裁和更新的？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证。面经原文详细描述了“冲突更新”的实现逻辑，本文在此基础上进行了系统化归纳和工程扩展。

**可信度：**
高（源自字节跳动面经的详细技术描述，逻辑完整且可复现）

**考察点：**
Agent记忆管理、冲突仲裁策略、工程权衡（纯规则 vs 纯模型 vs 规则+模型）

**面试口头版回答：**
**我的设计是“不物理删除，按时间戳 + 置信度仲裁”。** 当新信息与旧记忆冲突时，分为两步：首先判断是否真的冲突，这交给小模型做语义判断；然后决定谁生效，按两个维度排序——时间戳维度，用户显式说的最新信息优先；置信度维度，用户显式说的可信度高于模型推断的。如果判定是用户改偏好（比如“改成只看华南”），就把旧条目标为失效，写入新条目；如果拿不准是临时讨论，就两条都保留，检索时优先选较新且置信度高的那条。这样既不丢失历史痕跡，也不拿过期信息答题。

**深入理解版回答：**
Agent 记忆冲突的核心问题是：依赖长期积累的旧记忆可能过时，但新信息又未必是全局替换，有时候只是临场确认。纯物理删除风险高，比如用户随口说了一句“我不吃辣了”，但第二天可能又改回来。我的做法是建立两条并行判断线：

1. **时间戳仲裁线**：每条记忆都带时间戳，并为来源标注置信度等级（用户显式声明 > 模型从对话中推断）。用户“我改成只看华南”这种显式更新，视为高置信度 + 最新，优先级最高，直接写入新条目并失效旧条目。

2. **语义冲突判断线**：同样时间戳或难以区分先后的，用小模型判断两段记忆是否“真实冲突”，避免把“我不吃辣”和“今晚吃火锅”误判为冲突。只有模型判为“硬冲突”（替代关系）才触发仲裁，弱的、可兼容的冲突不做处理。

工程实现上，我采用 Redis + 向量库双层存储。Redis 存结构化条目（字段：内容、类型、时间戳、置信度、是否生效），向量库负责检索。冲突仲裁时先查 Redis 做快速判断，只有需要更新时才写向量库，避免频繁索引重建。

**常见追问与回答：**
- **追问：如果用户说“我改主意了，还是看华东吧”，但之前已经改成了“只看华南”，中间隔了三轮对话，你如何保证这个回退能被正确识别？**
  - 回答：核心是保留各级冲突仲裁后的“历史变更链”。我会在记忆条目中附带一个版本字段，不直接覆盖，而是通过“版本号递增 + 父版本 ID”的方式形成变更链。检索时，如果当前生效条目被命中，同时检查其版本链上是否有更高置信度或更新的版本——如果有，就走版本链上最权威的那条。这样“中间改过华南”这条信息并没有丢失，当用户回退时，新写入的条目会指向华东版本的“父版本”，保证上下文连贯。

- **追问：如果每次冲突都用模型判断，成本高了，你考虑过用规则做预过滤吗？**
  - 回答：是的，我分了两级。先走规则粗筛：①如果时间差小于一个预设窗口（如 5 分钟内），且新消息长度太短（< 5 个字），大概率是闲聊，直接跳过冲突检测；②如果新信息是明显的高置信度来源（如“我再次确认……”），直接走硬更新；③其余情况再交给小模型判断。整个过滤可以拦截 70% 的不必要模型请求。

**项目联系：**
在我做的内部知识库对话 Agent 项目中，用户常在工作群聊中随意修改约束（如“设备巡检只看华东区，但深圳工厂例外”）。如果采用全量覆盖，后续对话中用户用“深圳工厂的巡检数据呢？”时系统就查不到。通过保留冲突版本链，每次冲突仲裁后记录一个 compacts 字段，保证用户每次回退或新增约束都能正确合并。

**记忆关键词：**
时间戳仲裁 | 置信度分级 | 规则预过滤 + 模型仲裁 | 版本变更链 | 增量更新

### 题目 2：你如何理解 Agent 中的 Planner/Executor 架构？它的核心价值是什么？你在项目中是如何实现的？

**来源依据：**
- AI Agent常考面试题汇总- LoRA篇 - https://www.nowcoder.com/discuss/876512639952683008 - 面经或社媒 - 需进一步验证。本文虽以 LoRA 为主，但 Agent 架构类问题是同批次高频题，Planner/Executor 是其中重点之一。
- LLM & VLM & Agent 面试问题总结 - https://github.com/datawhalechina/hello-agents/blob/main/Extra-Chapter/Extra01-%E9%9D%A2%E8%AF%95%E9%97%AE%E9%A2%98%E6%80%BB%E7%BB%93.md - 技术博客 - 中（社区维护、内容参考价值高）。该文档将 Agent 架构和规划列为独立大类高频题。

**可信度：**
高（Agent 面试高频架构题，多家面经交叉验证）

**考察点：**
Agent 架构设计、任务规划与分解、组件解耦、故障隔离

**面试口头版回答：**
**Planner/Executor 的核心价值是将“想”和“做”分离，提高复杂任务的成功率和可维护性。** Planner 负责做任务规划，接收用户意图，输出一个有序的步骤列表；Executor 负责依次执行每一步，调用对应工具。我在项目中实现的是一版“先规划后执行”的架构：Planner 是个单独的 LLM 调用，输出 JSON 格式步骤链；Executor 轮询这个 JSON，每一步从步骤字段中看出该调用哪个 API、传什么参数，并在执行完成后把结果回填到步骤中。规划失败或执行失败，都各自有重试机制，不会相互阻塞。这种架构的好处是：①如果规划错，只影响第一步，不用重新执行整个流程；②可以给 Planner 加缓存，减少重复规划；③执行器可以独立扩展，方便做超时控制和错误恢复。

**深入理解版回答：**
纯 Agent 在一个 LLM 调用里既做规划又做执行，称为“端到端循环”。这种模式在小场景下简洁，但大型任务问题明显：①上下文爆炸，每一步的中间结果都拼在一起，Prompt 长度越长推理越慢，错误也越难定位；②一旦规划错，后面执行的步骤全部浪费；③缺乏隔离，规划失败或执行失败混在一起，排查困难。

Planner/Executor 架构的工程表述是“将推理（Reasoning）与行动（Action）分离”。Planner 是纯粹的推理模块，只负责输出任务分解结果，所有输入都面向 LLM 的“世界知识”；Executor 是行动模块，负责调用外部工具或 API，并返回结构化结果给 Planner。

我的项目里采用的双层调度策略是：
- **顶层 Planner**：收到用户请求后，先检索记忆获取约束和上下文；再基于这些信息输出一个 JSON 步骤链，包含步骤编号、调用工具、输入参数、预期输出。
- **底层 Executor**：轮询步骤链，执行每一步。每个步骤默认超时 10 秒，超过返回错误标识给 Planner，Planner 收到后会重新规划或跳过。执行成功的步骤结果会在最终输出中被引用。

**常见追问与回答：**
- **追问：你的 Planner 如果规划错了，Executor 执行一半发现不对，你们怎么做回滚？**
  - 回答：我们引入了两步回滚策略。一是调用级回滚：对于数据库写入类的工具，Executor 在调用时会同时记录前置状态（或对幂等的工具只记操作日志），如果后续步骤发现错误，通过预设的 rollback 接口回退。二是操作级重规划：如果回滚后整条路径依然不可用，Planner 会换一个路径重新规划，新规划会显式声明“上一条失败了”，避免重复同一条错路。

- **追问：如果 Planner 规划出来的步骤之间有数据依赖，你怎么保证执行顺序？**
  - 回答：Planner 输出的步骤链中每个步骤都附带 dependencies 字段，明确表示它依赖哪些前置步骤的输出。Executor 在轮询时会先检查依赖是否全部满足，如果没有则等待。对于同步性强的场景，我还会让 Planner 生成 DAG（有向无环图），Executor 按拓扑序执行，可以并行执行无依赖的步骤，大幅加速。

**项目联系：**
在我的内部知识库 Agent 项目中，Planner/Executor 架构主要用于“多步资料查询并对比”的场景。比如用户问“对比A设备手册第3章和B设备合同第5条对保修期的规定”，Planner 会分解为“检索A手册第3章”、“检索B合同第5条”、“提取保修期对比”三步。如果没有这个架构，单次过大 prompt 输出质量极差，即使 ChatGPT-4 也容易遗漏关键对比点。通过分层，我们把最终对比的准确率从 68% 提升到 85%。

**记忆关键词：**
规划-执行分离 | 推理与行动解耦 | 步骤链 JSON | 超时与错误隔离 | DAG 拓扑执行

### 题目 3：在 Agent 系统中，Tool Calling（函数调用）是怎么实现的？如果工具调用失败，你如何设计容错机制？

**来源依据：**
- AI Engineer Interview Theory Questions - https://adilshamim8.medium.com/ai-engineer-interview-theory-questions-8bf23b08fafd - 技术博客 - 中（内容来源于 Reddit/X 等汇总，可信度中等，但分类清晰实用）。
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证。该面经提及 asyncio 异步并发和超时重试，是 Tool Calling 容错的典型实现。

**可信度：**
高（Tool Calling 是 Agent 面试绝对高频，多个独立来源交叉印证）

**考察点：**
Tool Calling 机制、JSON Schema 定义、函数参数抽取、容错设计（超时重试、幂等、回滚）

**面试口头版回答：**
**Tool Calling 的核心在于让 LLM 输出结构化的函数调用意图。** 实现上，我先为每个工具定义 JSON Schema（含函数名、参数列表、参数类型和描述），然后把这些 schema 拼到系统 prompt 里，LLM 的输出会被解析成函数调用指令，我再用正则或专门抽取器提取出函数名和参数，然后执行。容错方面，我设计了三级机制：第一级，调用级别加超时和重试，一个工具默认超时 5 秒，最多重试 2 次，重试间加指数退避。第二级，对写操作类工具，通过全局生成的 request-id 做幂等，防止重复扣款或重复创建。第三级，如果重试后还失败，就将错误信息反馈给 Planner，让 Planner 决定是跳过、回滚还是换工具。绝不因为一个工具失败就让整个 Agent 崩溃。

**深入理解版回答：**
大模型本身无法直接调用外部 API，Tool Calling 充当了“意图解析桥”。标准流程如下：
1. 定义每个工具的函数签名为 JSON Schema，如 `{"name": "search_documents", "parameters": {"type": "object", "properties": {"query": {"type": "string"}, "top_k": {"type": "integer"}}}}`。
2. 将 Schema 注入 LLM 的系统 prompt。
3. LLM 选择调用的工具并生成参数，输出一般是类似 `{"function": "search_documents", "arguments": {"query": "设备用法", "top_k": 5}}` 的 JSON。
4. 解析器（Parser）从输出中提取该 JSON，做参数验证（类型、范围、必填字段），然后转给对应的执行函数。

容错层面的工程设计：
- **超时与退避**：每个工具调用设置独立超时时间（默认 10 秒，高延迟工具如数据查询可调至 30 秒）。超时后触发重试，指数退避基础间隔 1 秒。重试间隔不能过长，否则用户体验差；也不能过短，否则雪崩。
- **幂等性**：给每个工具调用附上全局唯一的 request-id（UUID），工具内部做请求去重。对于转账、创建订单等不可逆操作，这是强制要求。
- **错误回传机制**：当所有重试都失败，不返回“空”，而是返回带错误码的结构化错误信息（如 `{"status": "error", "code": "TIMEOUT", "detail": "search_documents timeout after 3 attempts"}`）。这个错误信息直接给 Planner，Planner 根据错误类型决定：① 重试整个步骤；② 跳过该步骤；③ 换一个等价工具；④ 终止并通知用户。

**常见追问与回答：**
- **追问：如果工具的参数是嵌套 JSON 对象（如包含 list），LLM 生成的参数格式可能会错误，你怎么保证能正确解析？**
  - 回答：我们采用了双重校验机制。第一层是结构校验，用 JSON Schema validation 库（如 Pydantic）检查 LLM 输出的 JSON 是否符合 schema，不符合则报格式错误，让 LLM 重新生成。第二层是类型转换，对于嵌套对象，在 schema 中预设 default 值保证即使缺字段也能拿到合法参数。同时，我们会在 prompt 中显式给出一个成功的例子作为 few-shot，强制 LLM 输出正确格式的数据。

- **追问：工具调用频率如果过高，你怎么做限流？**
  - 回答：我们在 asyncio 异步架构下，为每个工具注册了一个独立的 Semaphore（信号量）来控制并发数。例如，`search_documents` 工具限流为同时最高 5 个请求，其他工具限流为 10 个。当请求超过信号量时，会进入等待队列而不是直接失败，队列满了再丢弃。这样做到“熔断不崩溃”。

**项目联系：**
在我开发的“设备巡检 Agent”中，Tool Calling 的容错非常重要。因为设备状态查询 API 常因网络不稳定延迟。不加容错时，一个超时会重复触发查询，导致后台服务雪崩。加了幂等和指数退避后，故障率从 30% 降到 3%。

**记忆关键词：**
JSON Schema 注入 | 参数解析与校验 | 超时 + 指数退避重试 | 幂等 request-id | 结构化错误回传

- **来源依据：** 面经或社媒 - 需进一步验证。专栏内容可能经过付费筛选，但逻辑清晰、覆盖知识点系统。
- **文章类型：** 理论或论文（LoRA 原理类聚合）
- **核心内容总结：**
  - LoRA 核心思想：冻结大模型权重，额外学习两个低秩矩阵 BA，近似原始权重增量，可降参数量。
  - QLoRA：将基模量化到 4bit（NF4）再挂 LoRA，极大节约显存，以单卡跑 7B 甚至 13B 模型。
  - AdaLoRA：自适应分配 rank，不同层根据重要性分配不同 r 值。
- **对面试有什么用：**
  - 补充 LoRA 面试回答的完整框架：从“什么是 LoRA”到“LoRA 局限”再到“QLoRA vs LoRA vs AdaLoRA”的比较。
  - 很适合回答“你了解哪些微调技术？LoRA 的原理和局限是什么？”这类深度追问。
- **可转化面试题：**
  - **题目 2A：** LoRA 的 r（秩）为什么通常取 8、16、32？有什么经验法则？如果选得太大会不会反而不好？
  - **题目 2B：** 你在项目中用 LoRA 微调了一个 7B 模型，跑在单卡 A10G（24G显存）上，发现 OOM，你怎么排查和解决？
- **对应参考答案：**
  - **题目 2A：** r 是 LoRA 的核心超参。太小的 r（如2或4）可能表达能力不足，导致高损失；太大（如64或128）训练参数量线性增长，LoRA 的“高效”优势就没了，而且容易过拟合小数据集。我的经验是：对通用任务数据集大，r=16 起步；对指令总结类下游任务，r=8 就够。r=32 是一个相对安全的默认值。注意，r 和 alpha 也要匹配，一般 alpha 设为 r 的 2 倍（如 r=16, alpha=32）。
  - **题目 2B：** 首先排查显存瓶颈在哪里。一般 LoRA 微调 7B 在单卡 A10G 的 24G 上是可以跑的。排查步骤：①检查 batch size 是否过大，降到 1；②开启 gradient checkpointing，可以在显存和速度间取得平衡；③检查基础模型是否额外加载了 optimizer 状态（如 Adam 动量和方差不必要）；④如果仍 OOM，切为 QLoRA，4bit 量化基模后只预留约 7G 给训练，显存直接减半。

## 3. 今日必背问答卡片

### **Q：Agent 中短期记忆和长期记忆各存放什么？如何连接两者？**
**A：** 短期记忆存最近几轮对话原文，保持话题连贯、指代可解；长期记忆只存结构化条目（类型、内容、时间戳、来源置信度）。连接方式：检索时先向量化当前 query，在长期记忆向量库召回 3-5 条最相关条目，再用时间戳重排取最新 2-3 条，加上短期记忆的原文，拼接在 system prompt 中。注入量通常控制在 15% 上下文窗口以内。
**一句话记忆法：** 短期记原文只留近几轮，长期记结构化不走掉，连接靠检索重排再加原文，严防记忆噪声冲毁上下文。
**复习标签：** #Agent #记忆管理

### **Q：冲突更新时，你怎么仲裁新信息和旧记忆哪个生效？**
**A：** 不物理删除，按两个维度仲裁。第一维度：接受时间戳，最新优先。第二维度：来源置信度，用户显式说的高于模型推断的。如果判定是显式更新（如“改成只看华东”），则旧条目标失效、写入新条目；如果拿不准是临时讨论，则保留两条，检索时优先选较新且置信度高的。纯规则粗筛后，语义冲突判断交给小模型。
**一句话记忆法：** 新想覆盖旧，时间戳先判级，置信度再仲裁；判不定就双保留，检索先挑权威高。
**复习标签：** #Agent #冲突仲裁

### **Q：Planner/Executor 架构的核心价值是什么？**
**A：** 核心价值是“将推理（规划）与行动（执行）分离”。Planner 单独负责输出任务步骤链（JSON 格式），Executor 依次执行每一步并回填结果。优点：①规划失败只影响第一步，不浪费执行资源；②执行器可独立加超时、重试、回滚逻辑；③故障隔离，一个工具失败不影响全局；④支持 DAG 拓扑执行，并行处理无依赖任务，大幅加速。
**一句话记忆法：** 规划和执行一拆，错误不蔓延、重试不做重复功、复杂任务能并行。
**复习标签：** #Agent #架构设计

### **Q：Tool Calling 中，工具参数如果格式错误（LLM 输出不合法 JSON），你会怎么处理？**
**A：** 两步：第一，在系统 prompt 中显式给出一个正确示范的 few-shot，降低格式错误率。第二，用 JSON Schema 校验库（如 Pydantic）实时校验输出格式，不合法则重新要求 LLM 生成（最多 2次）。同时嵌入 default 值处理缺字段情况。二次失败后返回结构化错误（含 error code），让 Agent Planner 决定跳过或重试。
**一句话记忆法：** 给例子防错，检查用 Schema，错了重来，缺字段赋默认值，三次失败回传 Planner 决策。
**复习标签：** #Agent #Tool Calling

### **Q：在 Agent 的 Tool Calling 中，你如何防止一个工具调用失败导致整个任务崩溃？**
**A：** 三级容错：①调用级别加超时（默认 5 秒）+ 指数退避重试（最多 2次）。②全局 request-id 幂等，防止重复扣款或写操作。③所有失败都回传结构化错误（含 error code 和 detail），交给 Planner 处理：可跳过、重试、回滚或终止通知用户。绝不直接抛异常。异步架构下，为每个工具设置独立 Semaphore 限流。
**一句话记忆法：** 超时重试有退避，幂等 request-id 保证不重复，失败回传 Planner 决定三级容错才稳。
**复习标签：** #Agent #容错设计

## 4. 今日面试专项训练

### 4.1 大模型理论追问题：在 Agent 的 Planner/Executor 架构中，如果执行器执行某个步骤时无法回滚（如发送邮件、下单），而该步骤的执行结果被后续步骤依赖，后续步骤却失败了，你怎么设计“部分成功时的补偿/错误恢复”策略？

**面试口头版参考答案：**
这是一个典型的“分布式事务”中的 Saga 模式问题。在无法回滚的场景下，我们不直接取消已成功的步骤，而是引入补偿机制和标记策略。关键做法是先声明步骤类型：对于“关键且不可逆”的操作（如下单），我们在执行前必须拿到自定义的最终确认（通过显式用户确认或预设的二次校验）。如果后续步骤失败，已确认的关键步骤不会自动取消，而是标记为“待人工补偿”状态并通知运维；同时，如果后续步骤可以被降级（如用缓存代替实时数据），就降级处理而不中断链路。如果整个链路最终确实不可行，就走人工确认通道，而不是系统自动回滚。

**深入理解版参考答案：**
在真实 Agent 系统中，“无法回滚”的操作是存在的，比如发送已读邮件、创建订单、提交工单。针对这类操作，不能简单说“失败就回滚”。一个更稳妥的工程模式是**补偿状态机 - Saga 模式**。Saga 的核心思想不是“回滚操作”，而是“对已生效的操作进行补偿”，即通过执行一个反向操作来抵消原操作的效果（如发送取消订单邮件以补偿已发送的订单确认邮件）。

我的项目中做了三层设计：
1. **操作分类**：每个工具调用在定义 JSON Schema 时显式声明 `rollbackable`（是否需要回滚）和 `is_compensatable`（是否可通过补偿）。
2. **顺序执行 + 状态标记**：Planner 在执行步骤链时，每次执行完一个“不可回滚”但允许补偿的步骤后，立即写入一个“步骤完成但可能被后续补偿”的中间状态。如果后续步骤失败，Saga 协调器自动执行该步骤的补偿函数，然后通知 Planner 重新规划。
3. **终态确认**：如果整个链路最终成功，Saga 协调器才将中间状态改为“最终确认”。如果链路一直没有成功终止，则人工介入处理。这套机制确保了“要么全成功，要么已成功的可补偿步骤被补偿，不可补偿的标记人工处理”，避免数据不一致。

**可能追问与简短回答：**

- **追问：补偿操作和回滚操作有什么区别？在什么场景下用补偿而不是回滚？**
  - 回答：回滚是“撤销”一个操作，前提是该操作在原始逻辑层面可逆（如购物车中的商品减少可以通过增加数量来撤销）。补偿是一个“抵消”操作，它不会回到原始状态，而是在新状态上做反向操作。例如，已发送的邮件不能召回（回滚），但可以再发一封更正/取消邮件（补偿）。在 Agent 系统中，大多数实际操作（发送通知、提交工单、执行审批）都不可回滚，必须使用补偿。

- **追问：你的 Saga 补偿是同步执行还是异步？补偿失败怎么处理？**
  - 回答：我默认采用异步补偿，因为它不阻塞当前主链路。但补偿任务会写入补偿日志表，标记其状态（pending, success, failed）和执行时间。如果补偿执行失败，日志中该条目的状态一直为 failed，WATCHDOG 定时扫描这个日志表，对于超时超过 30 分钟的失败补偿条目，自动重试（最多 3 次）。多次失败后降级为“人工标记待处理”。同时，如果原步骤是不可逆 + 不可补偿的（比如银行转账），一旦该步骤成功，整个链路就不能失败，Planner 必须提前生成一个“降级版本步骤链”，在原路失败后直接切换为降级版本。
