### 题目 1：请详细描述你如何为一个对话 Agent 设计长短期记忆系统？特别是记忆的提取、压缩和冲突更新是如何实现的？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证

**可信度：**
中（来自面经搬运，有具体项目背景和技术细节，比较可信，但需注意是二手资料。）

**考察点：**
Agent 记忆系统设计、状态管理、信息压缩、冲突解决策略、工程实现细节。

**面试口头版回答：**
“我一般把记忆拆成短期和长期两层。短期记忆存最近几轮对话原文，保证对话连贯和指代还原；长期记忆只存结构化条目，比如用户偏好、硬约束、关键事实。提取时，每轮结束用一个 prompt 把值得长期记的信息抽成 `{类型，内容，时间戳}` 条目，再过滤掉寒暄。压缩方面，定期对同主题条目做摘要合并，比如用户三次提到口味偏好，合并成一条，避免无限膨胀。冲突更新最难处理：我用‘时间戳 + 来源置信度’仲裁。如果用户显式说‘改成只看华南’，就按新条目覆盖旧条目并标失效；如果是模型推断的补充信息，可能两条都留，检索时优先取最新的。总之，规则和模型结合，减少纯规则覆盖不到的语义冲突。”

**深入理解版回答：**
设计一个生产级 Agent 的记忆系统，核心是平衡信息量、检索效率和准确性。
1.  **分层设计**：短期记忆（Short-term Memory）存储最近 3-5 轮对话原文，用于处理指代消解和上下文连贯性。长期记忆（Long-term Memory）存储经过提炼的关键信息，不存原文，避免上下文窗口被占满。
2.  **三个核心动作**：
    *   **提取（Extraction）**：每一轮对话结束后，用一个轻量级 Prompt 或小模型，从对话中抽取结构化信息。字段可以定义成 `{type: "user_profile" | "constraint" | "fact" | "todo", content: string, source_round: int, timestamp: float}`。之后还需过滤：长度太短、纯寒暄的内容直接丢弃。
    *   **压缩（Compression）**：长期记忆不能无限膨胀。我的做法是：对同一 `type` 且有重叠 `content` 的条目，定期（例如每隔 10 轮或每天）进行摘要合并，保留最新值。但要注意，对于“硬约束”（如只看华东区），我会打上特殊标签，不参与合并，避免被平均掉。
    *   **冲突更新（Conflict Update）**：当用户修改或撤回之前的指令时产生冲突。我用“时间戳 + 来源置信度”仲裁。
        *   **硬规则**：时间戳更新且来源为“用户显式指令” > 旧条目。
        *   **语义判断**：如果新的指令是模糊修改（如“算了，不用那么严格”），则交给模型判断是否属于冲突，以及是临时需求还是永久更改。如果是临时需求，我会在旧条目上加一个临时权重，而非直接删除。
3.  **检索与注入**：检索长期记忆时，先做向量语义检索，再按时近性做加权重排序，通常只注入 3-5 条最相关的进上下文。硬约束是例外，只要未失效就每轮都带。
4.  **常见坑**：最容易被追问的是“如何保证记忆不被过度压缩，丢失关键信息”。我的应对是：对“硬约束”和“待办”等关键类型做白名单保护，不参与通用压缩流程。
5.  **工程优化**：可以基于 LangGraph 实现，将每个动作封装成一个 Node，用 State 管理记忆的版本和状态流转。错误恢复时，可以设计一个“记忆回滚”节点，当检测到冲突更新出错时，回退到上一份可靠的记忆状态。

**常见追问与回答：**
*   **追问 1：为什么记忆用向量库存，而不是直接拼进每轮 prompt？**
    *   **回答**：早期方案试过全量拼入，但很快遇到 Token 超限和 Lost in the Middle 问题。向量库可以按相关性检索，控制注入量，并且支持增删改，灵活度更高。但需要维护一套索引，增多了运维成本。
*   **追问 2：记忆条目的‘类型’超过 10 种怎么办？需要维护一个 Schema 吗？**
    *   **回答**：是的，当类型变多后，我会定义一个 JSON Schema 来约束输出，确保下游解析的一致性。同时，可以使用小模型（如 Qwen-1.8B）来做分类抽取，成本低且稳定。

**项目联系：**
在我的项目中，我使用 LangGraph 构建了一个客服 Agent。记忆模块就是通过 State 来管理的。我在 State 里定义了两个字段：`short_term_buffer` 和 `long_term_memory_db`。每次对话结束后，LLM Node 负责执行“提取”和“冲突更新”，然后通过 Graph 的 State 更新机制写入。当用户提问时，一个专门的 “Memory Retrieval Node” 从向量库中检索，并拼入 Prompt。这样做的好处是：可以方便地做 A/B 测试，比如关闭记忆模块后看用户满意度是否下降。

**记忆关键词：**
#Agent记忆 #冲突更新 #状态管理 #记忆压缩 #工具调用

### 题目 2：当 Agent 调用一个外部工具（Tool Calling）并返回错误（如超时、返回空结果、格式错误）时，你是如何设计错误恢复和兜底策略的？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 大模型面试题 | 小林面试笔记 - https://xiaolinnote.com/ai - 面经或社媒 - 需进一步验证

**可信度：**
中（源自两份面经，都是实操问题，可信度尚可。）

**考察点：**
Tool Calling 可靠性、Agent 容错、重试与降级策略、错误分类、异常处理流程。

**面试口头版回答：**
“这个问题在工程上非常关键。我设计了一套重试-降级-恢复策略。首先，我会对错误进行分类：`timeout`、`parse_error`、`empty_result`。对于 `timeout`，我会设置指数退避重试，最多 3 次。对于 `parse_error`，我会把错误信息反馈给模型，让模型修正输出的参数格式。对于 `empty_result`，不直接重试，而是触发一个‘搜索优化节点’，尝试改写 query 再查。如果都失败，就走降级兜底，例如给用户返回‘该功能暂时不可用，请稍后再试’或者‘未找到相关信息’。为了避免无限循环，我会上全局的 Token 超时限制和最大迭代步数。最重要的是，我会把失败的经验和恢复动作记录到 Agent 的长期记忆中，这样下次遇到类似问题时，模型可以直接参考成功过的模式，提升成功率。”

**深入理解版回答：**
在 Agent 系统中，工具调用失败的恢复能力是衡量系统健壮性的关键。我的设计方案是一个三阶段流程：

1.  **阶段一：错误分类与轻量恢复**
    *   **分类：** 将错误明确分类。*   **Timeout：** 网络瞬间波动，最可能通过重试恢复。
        *   **Parse Error：** 模型输出的 JSON 格式错误。
        *   **Empty/No Result：** 工具执行成功但无有用数据。
        *   **Tool Response Error：** 工具内部逻辑错误。
    *   **轻量恢复：**
        *   **Timeout & Parsing Error：** 采用指数退避（1s, 2s, 4s）重试，并返回具体错误信息给 LLM 让其修正。例如：“工具调用 `get_weather` 因超时失败，是否再次尝试？”或者“你输出的参数 `city` 应该是字符串而非数字，请修正。”
        *   **Empty Result：** 不重试。触发“Query Rewrite” Node，让模型尝试换一种问法。例如，从“如何提高转化率”改为“提升转化率的方法有哪些”。

2.  **阶段二：全局抑制与深度兜底**
    *   如果轻量恢复失败（例如重试 3 次仍超时），进入此阶段。
    *   **降级（Degradation）：** 如果 Agent 依赖该工具完成核心任务，则必须有一个安全的降级路径。例如，一个订餐 Agent，如果餐具查询工具失败，回答：“无法获取餐具数量信息，我会为您下单标准配置的餐具。”
    *   **终止：** 对每个 Agent 任务设置最大迭代步数（如 10 步）和全局 Token 消耗限制，防止 Agent 陷入重试死循环，花光所有预算。

3.  **阶段三：经验复盘与自适应学习**
    *   将这次失败的经验、分类、恢复策略执行结果，都记录到 Agent 的长期记忆中。格式如：`{ task: '获取天气', failed_attempt: { error_type: 'timeout', strategy: 'retry_3_times', result: 'success' } }`。
    *   当下次遇到类似查询时，Agent 可以学习到“该工具容易超时，可以提前告知用户”或者“应该使用 Query Rewrite 策略”。
    *   **工程落地：** 在 LangGraph 中，这可以抽象为一个 `RetryNode` 或 `ErrorHandlingEdge`，通过条件边进行流程控制。

**常见追问与回答：**
*   **追问 1：怎么判断是网络错误还是工具本身逻辑错误？**
    *   **回答：** 在工具调用时，我会封装一个统一的 API 调用库。对于 HTTP 请求，通过 HTTP 状态码和返回的 `error_code` 来区分。对于本地工具，通过标准化的返回体（如 `{status: "success/error", data: ..., error_msg: ...}`）。如果是 `5xx` 一般是网络或服务问题；如果是 `4xx` 且带业务错误码，一般是输入数据问题。
*   **追问 2：如果兜底策略也失败了，你怎么保证系统不挂掉？**
    *   **回答：** 我会在架构上采用隔离和熔断机制。每个工具调用都封装在独立的协程（asyncio）中，并设置超时。如果某个工具连续失败超过阈值（例如 5 次/分钟），网关层或 Agent 调度器会熔断该工具 60 秒，避免流量继续涌入。

**项目联系：**
我在一个文档问答 Agent 项目中实践了这个策略。该 Agent 需要调用业务搜索 API。我们线上监控发现 `parse_error` 占比很高，原因是模型输出的 JSON 格式不规范。于是我在 Prompt 中加入了严格的 Few-shot 示例，并要求输出前先进行一次 JSON 格式校验。如果校验失败，系统会捕获异常并返回“输出格式不合法，请严格遵守 `{` 开头的 JSON 格式”，重试次数上限设为 2 次。上线后，`parse_error` 从 15% 下降到 2%。

### 题目 3：你如何理解 Agent 的 Planner/Executor 架构？它和 ReAct 范式有什么区别？在什么场景下你会选择 Planner/Executor？

**来源依据：**

**可信度：**
中（来自面试题网站和个人项目总结，概念定义清晰，但没有直接引用官方文档。）

**考察点：**
Agent 设计范式、任务分解、Plan-and-Execute、ReAct、架构选型、适用场景。

**面试口头版回答：**
“在我看来，Planner/Executor 是一种‘先规划，后执行’的架构，特别适合可预测的、多步骤的复杂任务。Planner 负责把用户的目标拆解成一个 DAG 或线性列表，Executor 负责一步步执行。这和有思考-行动-观察循环的 ReAct 不同。ReAct 是每轮动态决策，灵活但不可预测；Planner/Executor 是静态规划后执行，结果更稳定可控。我一般在任务结构明确、执行步骤确定性强、对执行路径有严格要求的场景下选 Planner/Executor，比如一个标准的企业入职流程处理 Agent。而 ReAct 更适用于需要探索性、开放性回答的场景，比如写代码或创意写作。一个简单的区分是：Planner/Executor 做‘工程实现’，ReAct 做‘模型推理’。”

**深入理解版回答：**
理解 Planner/Executor 的关键在于区分“规划”和“执行”的粒度以及“动态”与“静态”的关系。
1.  **架构本质**：
    *   **ReAct（Reason + Act）**：这是一种动态推理循环。每一步，模型先思考（Reason）下一步该做什么，然后执行（Act），再观察（Observe）结果，思考下一步。这是**每步动态决策**，灵活性高，但 Token 消耗大，且可能出现“计划赶不上变化”乱跑的问题。
    *   **Planner/Executor**：这是一个明确的**规划->执行**两阶段。**Planner** 接收任务，输出一个结构化的、不可变的计划（通常是一个 DAG 或步骤列表）。然后 **Executor** 严格按计划执行，它本身没有决策能力，只是一个执行引擎。
2.  **核心区别**：
    *   **决策时机**：ReAct 在每一步决策；Planner/Executor 在第一步把所有决策做完。
    *   **可观测性**：Planner/Executor 的执行路径是确定的，便于记录和回溯；ReAct 的下几步取决于中间观察结果，难以预测。
    *   **容错能力**：ReAct 比较灵活，观察结果不对可以立刻调整；Planner/Executor 如果一个步骤失败，会导致整个计划中断，需要额外的错误恢复机制。
3.  **工程实践与选型**：
    *   **选 Planner/Executor**：适合那些**步骤已知、逻辑明确、对过程有要求**的场景。例如：企业客户报修流程（第一步：获取工单号，第二步：查询客户信息，第三步：派单，每一步是固定的）。此时 Planner/Executor 更稳定、更可解释。
    *   **选 ReAct**：适合**需求开放、步骤未知、需要探索**的场景。例如：写一个数据分析报告，Agent 需要先思考用什么库，写代码，如果报错再改。ReAct 的灵活性在这里是优势。
4.  **实现细节**：
    *   **Planner**：通常用 Prompt 指导 LLM 输出一个 Markdown 列表或 JSON 数组。需要考虑 Planner 本身的 Token 消耗和规划准确性。
    *   **Executor**：可以用状态机实现，确保每一步的执行结果存入 State。如果某一步失败，需要设计错误恢复 Edge，要么重试，要么通知 Planner 重新规划。
5.  **常见坑**：如果任务本身高度不确定，Planner/Executor 会因为执行路径不可预测而频繁失败，此时 ReAct 更合适。反之，如果任务确定，ReAct 会因为不必要的思考循环浪费 Token 和时间。

**常见追问与回答：**
*   **追问 1：你是如何在 Planner 阶段保证计划的可行性？**
    *   **回答：** 我会给 Planner 提供一份详细的工具清单，包括工具名称、输入输出、用途和限制。Planner 的输出必须是这个清单的子集。此外，我会在规划 Prompt 中明确让其避免生成过多步骤（比如最多 5 步），并提示它考虑依赖关系。
*   **追问 2：如果 Executor 在执行中报错，你如何让 Planner 重新规划？**
    *   **回答：** 我设计了一个 `ReplanEdge`。当 Executor 失败且无法通过简单的重试恢复时，它会停止执行，并将失败信息连同当前状态一并回传给 Planner。Planner 会分析失败原因，输出一个新的计划。这种做法需要在 LangGraph 中设置条件 Edge 来判断是否重路由。

**项目联系：**
我在做一个“酒店投诉处理 Agent”时采用了 Planner/Executor 架构。流程是固定的：1）获取投诉类型；2）查询住客信息；3）查询历史投诉记录；4）推荐补偿方案。我使用 LangGraph 的 State 管理每一步的状态。Planner Node 输出一个固定的 DAG，Executor Node 遍历整个 DAG。当查询住客信息失败时，System 自动执行一个 `FallbackEdge`，降级为“无法获取住客信息，建议人工介入”。这比 ReAct 更冷静，不会出现模型瞎猜然后瞎回答的情况。

## 3. 今日必背问答卡片

### 卡片 1
- **Q:** Agent 长期记忆的“冲突更新”具体是如何用规则和模型结合的？
- **A:** 硬规则是“新时间戳 + 用户显式指令”优先，直接覆盖旧条目。语义判断（如模糊修改、临时需求）交给模型处理，模型判断后可能有两种结果：废弃旧条目并写入新条目，或保留旧条目但加临时权重。关键是**用户显式指令 > 模型推断**，时间戳作为最终仲裁依据。这样既不会丢失历史，也不会被过期信息误导。
- **一句话记忆法：** “显式指令是铁律，模型推断是软路，时间戳是法官。”
- **复习标签：** #Agent #记忆 #冲突更新

### 卡片 2
- **Q:** 如何设计 Agent 的 Tool Calling 错误恢复机制？
- **A:** 首先对错误分类：超时、格式错误、空结果。超时和格式错误用指数退避重试，并反馈错误信息给模型修正。空结果用 Query Rewrite。重试失败后走降级兜底（如告诉用户功能暂不可用）。最后设置全局 Token 和步数限制防止死循环。将失败经验存入长期记忆，供下次参考。
- **一句话记忆法：** “先分类，后重试，再降级，记经验。”
- **复习标签：** #Agent #工具调用 #错误恢复

### 卡片 3
- **Q:** Planner/Executor 和 ReAct 的核心区别是什么？
- **A:** Planner/Executor 是“先规划后执行”，计划一步做完不可变，执行步骤确定，适合步骤已知、逻辑固定的任务（如企业流程）。ReAct 是“边想边做”，每步动态决策，适合开放、未知、需要探索的任务。Planner 更稳定可控，ReAct 更灵活易变。
- **一句话记忆法：** “Planner 画图给 Executor 照着做，ReAct 是摸着石头过河。”
- **复习标签：** #Agent #Planner #ReAct

### 卡片 4
- **Q:** 在 Agent 中检索长期记忆时，如何平衡“相关性”和“时近性”？
- **A:** 先根据相关性（向量距离）召回 Top-K 条目，再用时近性进行加权重排，同样相关的条目优先取最新的。注入上下文时，通常只取 3-5 条最相关的。但硬约束（如只看华东）是例外，只要未失效就每轮都必须带上。
- **一句话记忆法：** “先找对的，再找新的，只塞最精的。”
- **复习标签：** #Agent #记忆 #检索

### 卡片 5
- **Q:** 如何量化评估 Agent 的记忆模块优化效果？
- **A:** 构建一个专属的多轮对话评测集，标好“理想答案”。固定其他变量，对比“带记忆 + Query 改写”和“纯单轮”的答案正确率。每次调整记忆策略（压缩算法、冲突仲裁），都在该评测集上跑，看正确率变化。确保优化方向可量化、可验证。
- **一句话记忆法：** “没有评测集的记忆优化，都是自我感动。”
- **复习标签：** #Agent #评测 #记忆

### 4.1 大模型理论追问题：为什么使用“时间戳 + 置信度”仲裁比纯规则或纯模型更好？

**问题：**
在 Agent 长期记忆的冲突更新中，你提到使用“时间戳 + 置信度”仲裁。请从工程原理和模型能力的角度，解释为什么这种混合策略比纯时间戳规则或纯模型推理更好？

**面试口头版参考答案：**
“纯时间戳规则虽然简单，但遇到‘用户说算了，不看了’这种模糊修改时，它不知道这是‘临时取消’还是‘永久放弃’，容易误删。纯模型推理虽然能理解语义，但成本高、不稳定，而且对模型的 Prompt 敏感，很难保证每次都说对。‘时间戳+置信度’是两者的平衡：用时间戳做硬规则，直接淘汰过期信息；用置信度做软判断，决定是否保留。这样既保证了核心逻辑的确定性，又利用模型处理了边界模糊的场景，是一个成本可控、效果稳定的工程方案。”

**深入理解版参考答案：**
从工程原理和模型能力角度看：
1.  **纯时间戳规则的局限**：
    *   **逻辑缺陷**：它假设用户的最新指令总是最正确的。这适用于“意见修改”，但当用户做出“临时变更”时，它无法识别。例如，用户说“今天不看华东数据”，按规则这个约束失效了。但明天用户又可能问起华东的数据。纯规则无法区分“永久”还是“临时”。
    *   **实现成本**：规则简单，但需要定义非常多复杂的“例外”子规则，维护成本高。

2.  **纯模型推理的局限**：
    *   **模型幻觉**：LLM 可能误解“临时变更”为“永久放弃”，导致误删。
    *   **高延迟和高成本**：每次冲突都调大模型，不仅慢，还烧钱。
    *   **Prompt 脆弱性**：改变 Prompt 可能导致冲突处理逻辑大变，不稳定。

3.  **混合策略的工程优势**：
    *   **效率与成本的平衡**：**时间戳**是个 `O(1)` 的简单比较，可以作为第一层快速仲裁。只有遇到**语义模糊**的情况（如时间戳新但内容是模糊修改），才触发模型推理。这极大降低了模型调用量。
    *   **确定性 + 灵活性**：**时间戳 + 置信度**中的“置信度”由模型动态评估（如用户显式说过的置信度 0.9，模型推断的置信度 0.6）。仲裁逻辑是：`if (新条目时间戳 > 旧条目时间戳) and (新条目置信度 > 阈值) then 覆盖 else 保留两者`。这既有规则的确定性，又有模型的灵活性。
    *   **可解释性**：仲裁过程可记录，如“因为新条目置信度（0.95）高于阈值（0.8）且时间戳更新，所以覆盖旧条目”。方便调试和回滚。

**可能追问与简短回答：**
*   **追问 1：这个“置信度”阈值怎么定？**
    *   **回答**：先统计线上数据，人工标注一批“覆盖”和“保留”案例，找到最佳阈值。通常会设一个安全区间，比如 0.7-0.9，并在评测集上通过 Grid Search 找最优值。
*   **追问 2：如果用户后来又说“上次说错了，还是按老规矩来”，你的系统怎么处理？**
    *   **回答**：这是一次新的冲突。用户说“还是按老规矩”，这是一个高置信度的显式指令。系统会将其作为新的记忆条目写入。仲裁时，会将其与之前“不看了”的条目比较，由于新的“按老规矩”时间更新、且置信度更高，系统会强制覆盖旧的“不看”条目并标记为失效。
