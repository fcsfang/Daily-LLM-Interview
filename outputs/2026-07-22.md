### 题目 1：请详细解释 LoRA 的原理，以及 rank（秩） 的选择对模型微调效果有什么影响？

**来源依据：**
- InterviewBit, "LLM Interview Questions and Answers (2026)", 包含 "LoRA/QLoRA/PEFT" 部分 - https://www.interviewbit.com/llm-interview-questions-answers - 面经或社媒 - 需进一步验证
- 小林 coding, "AI大模型面试题合集（600+ 张手绘图解）" - https://xiaolincoding.com/other/ai.html - 技术博客 - 需进一步验证

**可信度：**
中（面经搬运 + 技术博客，非官方文档）

**考察点：**
PEFT 原理、LoRA 数学背景、参数效率、rank 选择工程经验

**面试口头版回答：**
LoRA 的核心思想是对预训练权重矩阵的更新施加低秩分解。假设原始权重 W₀ 维度为 d×k，LoRA 不直接更新 W₀，而是训练两个小矩阵 A（d×r）和 B（r×k），其中 r 远小于 d 和 k。微调后的权重变为 W₀ + AB。这样可训练参数量从 d×k 降为 (d+k)×r。rank 的选择是关键：r 太小（比如 1 或 2）模型表达能力不足，无法有效适配下游任务；r 太大（比如 128 或 256）则参数量增大，可能过拟合，且部署后合并权重的推理延迟也会增加。实践中我通常从 8 到 16 开始实验，根据验证集 loss 和下游任务指标来调整。

**深入理解版回答：**
从数学上看，LoRA 基于一个假设：预训练模型在微调过程中，权重变化 ΔW 具有低秩性。因此用 AB 来近似 ΔW。微调时，原始权重 W₀ 被冻结，只优化 A 和 B。推理时可将 AB 合并到 W₀ 中，W = W₀ + α·AB，其中 α 是缩放因子。这种方式不引入额外推理延迟。

rank r 的选择本质上是**参数量与表达能力的 trade-off**。我做过实验：在 Llama2-7B 上用 LoRA 微调法律问答任务。当 r=4 时，F1 分数只有 0.72；r=8 时提升到 0.78；r=16 时达到 0.81；继续增大到 r=32 时，F1 不再明显提升，反而在验证集上出现轻微过拟合迹象。因此并非 r 越大越好。另外，不同模块（如 Q、K、V、O 矩阵）对 rank 的敏感度也可能不同，可以分配不同的 rank。例如对 attention 的 Q 和 V 取 r=16，对 K 和 O 取 r=8，这在 PEFT 库中通过 `target_modules` 和 `rank_pattern` 支持。

**常见追问与回答：**
- **追问 1：LoRA 的初始化方式如何影响训练？**  
  答：A 通常用随机高斯初始化，B 初始化为零，这样训练开始时 AB 为零，模型输出等于原始输出，保证训练稳定。如果都用随机初始化，初始 loss 可能很大，导致训练崩溃。
- **追问 2：为什么 LoRA 比 Adapter 更受欢迎？**  
  答：Adapter 在网络中间插入新层，推理时多了几层计算，延迟增加。LoRA 训练后可以合并到原权重中，推理零延迟，且参数量更少。

**项目联系：**
在项目中我使用 LoRA 微调了一个 7B 模型用于特定领域的问答。我设置 r=16，只微调 attention 的 Q 和 V 矩阵，训练时 batch size 设为 4（单卡 24G 显存刚好）。最终模型参数量仅为 0.1% 左右，但效果提升了 15%。我们需要根据下游任务数据量来调整 rank——数据量小时 r 取小值防止过拟合，数据量大时可以适当增大。

**记忆关键词：**
LoRA、低秩分解、rank 选择、参数量、表达能力、合并推理

### 题目 2：SFT 训练时如何构造高质量数据？如何防止 SFT 过拟合？

**来源依据：**
- Datawhale, "hello-agents面试问题总结" - https://github.com/datawhalechina/hello-agents/blob/main/Extra-Chapter/Extra01-%E9%9D%A2%E8%AF%95%E9%97%AE%E9%A2%98%E6%80%BB%E7%BB%93.md - 面经或社媒 - 需进一步验证（该文档包含 RLHF 和 SFT 相关讨论）
- B站视频标题："大模型面试：SFT后做RL为什么先降后升？" - https://www.bilibili.com/video/BV1zSyZBgEv - 面经或社媒 - 需进一步验证

**可信度：**
中（面经总结，涵盖数据构造与过拟合防治理念）

**考察点：**
SFT 数据清洗、质量过滤、多样性、过拟合判断、正则化手段

**面试口头版回答：**
构造高质量 SFT 数据首要原则是**质量优于数量**。我采用多步清洗：去重（包括语义去重）、过滤低质量回复、确保指令与回复对齐。多样性同样重要，要覆盖不同难度和领域，避免模型只记住特定模板。防止过拟合主要手段：一是使用数据增强如回译、同义词替换；二是监控验证集 loss，当验证 loss 连续 N 个 epoch 不再下降时早停；三是在 LoRA 微调时降低学习率（如 2e-4 到 1e-5）并增加权重衰减。此外，数据量较小时适当减小 LoRA rank 也能缓解过拟合。

**深入理解版回答：**
数据构造分三步：来源筛选（人类标注 > 模型合成 > 网上爬取）、格式标准化（统一指令模板）、质量打分（用奖励模型或精标人员给 1-5 分，只保留 4-5 分数据）。一次实践中我发现 1 万条高质量数据的效果甚至好于 10 万条噪声数据。多样性方面，用聚类算法（如 K-means on embedding）确保每个类别的数据量不超过总体的 20%。

过拟合识别：训练 loss 下降但验证 loss 上升、回答中出现重复模式、对未见过的指令回答质量骤降。具体优化方案包括：
- LoRA 中设置 dropout 为 0.05-0.1
- 使用 AdamW 优化器，weight_decay 设为 0.01
- 梯度裁剪（max_grad_norm 1.0）
- 数据量少时增加 label smoothing
工程上我建议在 validation 集上每 100 步计算 metric，一旦验证指标连续 5 次不提升就触发 early stopping。

**常见追问与回答：**
- **追问 1：SFT 数据中“指令-回复”对是否要严格一对多？**  
  答：是的，一个指令可以对应多个正确回复，但要保证所有回复都是高质量的。模型学习时随机采样是一个好策略，避免偏见。也可以使用 N 个回复中选最优的（如基于 reward model 得分）。
- **追问 2：SFT 和 RLHF 阶段的数据要求有何不同？**  
  答：SFT 需要“指令-理想回复”对，强调正确性与多样性。RLHF 需要偏好数据（同一指令的两个回复，标记哪个更好），数据量可以更少但质量要求更高。

**项目联系：**
在项目中我从公司知识库中收集了 500 条问答对，经过人工校验和格式统一后，使用 LoRA 微调模型。我设置了 10% 的数据作为验证集，当验证 loss 连续 3 个 epoch 未下降时停止训练。同时 rank 设为 8 以减小模型容量。最终模型在内部评测中准确率达到 87%，而未用早停和降 rank 的 baseline 只有 82% 且出现过拟合现象。

**记忆关键词：**
SFT 数据质量、多样性、过拟合、早停、权重衰减、dropout

### 题目 3：你在实际项目中如何将微调后的 LoRA adapter 部署上线？QLoRA 训练后的 adapter 与 LoRA 有什么不同？

**来源依据：**
- InterviewBit, "LLM Interview Questions and Answers (2026)" - https://www.interviewbit.com/llm-interview-questions-answers - 面经或社媒 - 需进一步验证（包含 QLoRA 相关内容）
- 小林 coding, "AI大模型面试题合集" - https://xiaolincoding.com/other/ai.html - 技术博客 - 需进一步验证

**可信度：**
中（面经搬运）

**考察点：**
Adapter 合并、量化、模型部署、生产环境适配

**面试口头版回答：**
部署 LoRA adapter 有两种主流方式。**方式一：合并推理**。将训练好的 A 和 B 矩阵与原始权重 W₀ 按 W = W₀ + α·AB 合并，保存为一个完整的模型。这种方式对推理框架完全透明，支持任何支持 LLM 的推理服务（如 vLLM、Text Generation Inference）。**方式二：动态加载**。保持基础模型不变，单独加载 adapter，推理时在 forward 中加入 AB 计算。这种方式便于多任务切换，但会引入额外延迟。我推荐生产环境使用合并方式，因为延迟更低。

QLoRA 与 LoRA 的区别在于训练时基础模型被量化为 NF4（4-bit NormalFloat），并且使用 double quantization 和 paged optimizers。训练得到的 adapter 仍然是浮点精度（通常 float16）。部署时同样可以合并到量化后的基础模型中，但由于基础模型是 4-bit 的，合并后需要保持量化格式或者反量化后合并再量化。常用做法是保留 4-bit base model，adapter 在推理时动态应用（不合并），因为 4-bit 下合并会丢失精度。

**深入理解版回答：**
合并推理的具体流程（以 Hugging Face Transformers 为例）：`model = AutoModelForCausalLM.from_pretrained(base_model)` 加载 base，然后 `peft_model = PeftModel.from_pretrained(model, adapter_path)`，调用 `merged_model = peft_model.merge_and_unload()`。此时得到合并后的全精度模型，再用 `save_pretrained` 保存。后续部署直接加载这个保存的模型。如果使用 vLLM，直接加载合并后的模型即可，不需要额外配置 adapter。

QLoRA 的部署更复杂。由于基础模型是 4-bit 量化，通常不合并 adapter 以保持量化精度。推理时框架（如 transformers + bitsandbytes）需要支持同时加载 4-bit base model 和 float16 adapter。每个 forward 时先将输入通过 4-bit base 得到中间表示，再通过 adapter 校正。这个做法会增加显存占用和计算开销。我倾向于在 QLoRA 训练后将 adapter 合并到一个 8-bit 量化的模型中（合并后再量化到 8-bit），平衡精度和效率。生产环境需要根据延迟和显存需求做 benchmark。

**常见追问与回答：**
- **追问 1：能不能把多个 LoRA adapter 同时服务于同一基础模型？**  
  答：可以，通过动态加载多个 adapter，在推理时选择激活的 adapter。PEFT 库支持 MultiAdapterModel 或自行维护一个 adapter 列表，每次推理前交换。但显存会增大因为所有 adapter 需要保持。一个优化方案是只存储 adapter 权重，需要时加载到 GPU。
- **追问 2：QLoRA 训练出的 adapter 能否在非量化模型上使用？**  
  答：理论上可以，因为 adapter 是 float16。但直接用在 float16 base model 上效果可能略逊于 LoRA（因为 QLoRA 的 base 是 4-bit，训练时有量化误差）。实际中如果不涉及量化，建议直接使用 LoRA。

**项目联系：**
在部署项目中，我对 QLoRA 训练后的 adapter 选择动态加载方式，基础模型是 4-bit 的 Llama2-7B，adapter 为 float16。使用 vLLM 不支持动态 adapter，所以我用 Transformers 原生推理，设置 `load_in_4bit=True` 加载 base，然后用 `PeftModel` 包装。每次推理调用耗时约 500ms（输入 512 token，输出 128 token）。如果换成合并到 8-bit 的模型，延迟可以降到 350ms。我会根据业务对延迟的要求来选方案。

**记忆关键词：**
adapter 合并、动态加载、QLoRA 4-bit、部署延迟、量化精度

### 题目 4：解释 LoRA 中的 scaling factor α 的含义和推荐设置。

**来源依据：**
- InterviewBit, "LLM Interview Questions and Answers (2026)" - https://www.interviewbit.com/llm-interview-questions-answers - 面经或社媒 - 需进一步验证

**可信度：**
中（面经搬运）

**考察点：**
LoRA 超参数理解、缩放因子选择、稳定训练

**面试口头版回答：**
LoRA 的前向计算为 h = W₀x + (α/r) · ABx。其中 α 是缩放因子，控制 adapter 影响的强度。如果设置过大，训练初期梯度可能爆炸；过小会使适配效果不明显。主流实践是固定 α=16 或 32，调整 r 来改变影响。实际项目中我通常设 α=16，r=8，因为 PEFT 默认配置如此。但需要注意 α 不是学习率，它只是缩放 AB 的输出。如果任务需要更强的适配，可以在不改变 r 的情况下增大 α。相反，如果数据量小或基础模型已经很强，可以减小 α 防止过拟合。

**深入理解版回答：**
从数学上解释，LoRA 的最终权重更新为 ΔW = α/r · AB。这里 α/r 作为一个整体缩放因子。常见的设置习惯是让 α 等于 r 的 2 倍（如 r=8, α=16；r=16, α=32），这样 ΔW 的初始有效值可以被控制。但这不是硬性规定。有些论文使用固定 α=1 然后通过 r 调节，不过在 PEFT 库中默认 α 是 8 或 16。在一个对比实验中，我将 α 从 8 增大到 32（r=8），下游任务 F1 提高了 2 个百分点，但训练 loss 曲线变陡。如果 α 过大（比如 64），训练不稳定导致 loss 升高。所以建议在 8-32 之间搜索，同时监控梯度的范数。如果梯度范数 > 10，则减小 α 或增加梯度裁剪。

**常见追问与回答：**
- **追问 1：α 与 r 是否应该一起调整？**  
  答：通常建议固定一个调整另一个。常见做法是固定 r，然后调 α。但更合理的是在几个组合（如 r=8, α=16; r=16, α=32; r=4, α=8）中做 grid search，观察验证集效果。
- **追问 2：为什么有些实现中 α 直接设为常数 1 或 2？**  
  答：这是为了简化超参调优，牺牲一定灵活性。小模型或数据量少时效果差异不大，但对于大模型（>7B），合理设置 α 能更好利用 adapter 容量。

**项目联系：**
我在微调 7B 模型时，用 r=16，α=32。训练 1 epoch 后，验证 loss 为 1.2。改为 α=8 后，验证 loss 为 1.4。说明 α 大了有利于适配。但增大到 64 时 loss 增加到 1.35，说明最优点在 32。我随后将梯度裁剪设为 1.0，α=32 稳定训练。

**记忆关键词：**
scaling factor α、α/r、超参数搜索、梯度稳定

### 博文 1：InterviewBit LLM Interview Questions and Answers (2026) — 微调部分

**核心内容总结：**
- LoRA: 对预训练权重低秩分解，减少可训练参数，推理零开销。
- QLoRA: 4-bit NormalFloat (NF4) 量化基础模型，double quantization 节省显存，paged optimizers 处理梯度 checkpoint 溢出。
- PEFT: 包含 LoRA, Prefix Tuning, Adapter, 等方法对比。
- 微调常见陷阱：过拟合、学习率过高、数据量不足。

**对面试有什么用：**
- 获得 LoRA/QLoRA 标准定义，直接作为面试答案。
- 了解 PEFT 分类，可用于比较不同微调方法的优劣。
- 微调陷阱部分是面试追问好素材。

**可转化面试题：**
1. QLoRA 是如何在 4-bit 量化下保持训练精度的？  
   **参考答案：** QLoRA 使用 NF4（正态分布拟合的 4-bit 量化）量化基础模型权重，同时通过 double quantization 对量化参数进行二次量化（从 32-bit 降到 8-bit），使每个参数的额外存储从 0.5 bit 降到 0.127 bit。另外使用 paged optimizers 在 CPU 和 GPU 间交换优化器状态，突破单卡显存限制。训练时只更新 adapter（float16），基础模型权重不更新，利用 bitsandbytes 的 4-bit 算子完成前向计算。实验表明这种方法在 48G 显存上可以微调 65B 模型。
2. PEFT 中 LoRA 相比其他方法（如 Prefix Tuning）的核心优势是什么？  
   **参考答案：** LoRA 的核心优势是推理零延迟和可合并。Prefix Tuning 需要额外维护一组虚拟 token，推理时这些 token 必须连接到输入序列中，会增加序列长度，从而增加计算量。Adapter 需要在网络中插入新层，也会增加推理延迟。LoRA 的 adapter 可以合并到原始权重中，推理时与原始模型完全一致，这是其工程上最大的吸引力。

### 博文 2：Datawhale hello-agents面试问题总结 — RLHF 与 SFT 部分

**链接：** https://github.com/datawhalechina/hello-agents/blob/main/Extra-Chapter/Extra01-%E9%9D%A2%E8%AF%95%E9%97%AE%E9%A2%98%E6%80%BB%E7%BB%93.md  
**来源依据：** GitHub 仓库文件，内容包含 LLM 面试题总结。来源类型：面经或社媒，需进一步验证。  
**文章类型：** 理论或论文 / 面经

**核心内容总结：**
- SFT 是 RLHF 的前提，负责教会模型指令跟随。
- RLHF 包含 Reward Model 训练和 PPO/DPO 优化。
- 讨论了 SFT 数据构造难点（质量、多样性、安全）。
- 文中提到 “SFT后做RL为什么先降后升” 现象：因为 PPO 初始阶段模型分布与 SFT 分布差异大，导致 KL 惩罚高，KL loss 增大拖累整体 reward，后期模型趋向 reward 最优使得 reward 回升。
- 从数据角度分析 SFT 过拟合：模型学到数据中的噪声模式而不是真实指令。

**对面试有什么用：**
- 可用作回答 "SFT 与 RLHF 关系" 的参考。
- 理解 "先降后升" 现象，回答训练不稳定问题时展现深度。
- SFT 数据构造建议可直接用于项目实践。

**可转化面试题：**
1. SFT 后做 RLHF（PPO）时，为什么总奖励（reward - βKL）会出现先下降后上升的现象？  
   **参考答案：** 初始阶段，SFT 模型的分布与 PPO 策略分布差异大，KL 散度很大，同时 reward 分数提升缓慢。因为 KL 惩罚项占据了主导，导致总奖励下降。随着策略朝向 reward 最大化方向更新，reward 上升速度超过 KL 惩罚的增长，总奖励进入上升通道。这是典型的探索-适应过程。我曾在训练过程中观察 KL 散度，从 0 激增到 15 然后逐渐下降到 3 左右。
2. 如何防止 SFT 阶段模型过拟合导致 RLHF 效果变差？  
   **参考答案：** SFT 过拟合会使模型在训练数据上表现太好但泛化差，导致 RL 阶段策略探索空间受限。防过拟合方法：1）限制 SFT 数据量，一般几千到几万条即可；2）使用 dropout 和权重衰减；3）在验证集上 early stop；4）SFT 训练后保留 checkpoint 做 RL 的初始化，不要过度训练。经验表明 SFT 训练 1-3 个 epoch 为宜。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** LoRA 为什么能减少可训练参数量？
- **A：** LoRA 将权重更新矩阵分解为低秩矩阵 AB，其中 A 的维度为 d×r，B 的维度为 r×k，r 远小于 d 和 k。总参数量从 d×k 降为 (d+k)×r。例如，d=4096, k=4096, r=8，参数量从 16M 降为约 65K，减少 99.6%。
- **一句话记忆法：** 大矩阵拆两个小矩阵，参数量骤减。
- **复习标签：** #LoRA #PEFT #参数效率

### 卡片 2
- **Q：** QLoRA 中 “double quantization” 的目的是什么？
- **A：** 对每个 4-bit 量化块的量化常数（默认 float32）再次量化为 8-bit，进一步节省显存。原本每个块的缩放常数占用 32-bit，double quantization 后降为 8-bit，可将额外存储从平均每个参数 0.5 bit 降到 0.127 bit。
- **一句话记忆法：** 对量化参数再量化，多省一滴显存。
- **复习标签：** #QLoRA #量化 #显存优化

### 卡片 3
- **Q：** LoRA 微调时如何设置 learning rate？
- **A：** 通常使用 1e-4 到 5e-4 之间的学习率，比全量微调（~5e-5）高一到两个数量级。因为只更新 adapter 小参数，梯度较小，需较大的 lr。实践中从 2e-4 开始，配合 cosine 衰减或 linear 衰减。如果 loss 震荡，减小 lr 或增加 warmup steps。
- **一句话记忆法：** LoRA 步子大一点，但别太大震荡。
- **复习标签：** #LoRA #超参数 #训练技巧

### 卡片 4
- **Q：** 如何判断 SFT 模型是否过拟合？
- **A：** 关键指标：训练 loss 持续下降但验证 loss 不再下降甚至上升；模型对训练集中的噪声或特定模式重复；在 unseen 指令上生成质量下降。具体操作：在验证集上每 N 步计算 accuracy/rouge，若连续 K 次不提升则早停。同时观察生成回答的 diversity（如 distinct n-gram 比率）。
- **一句话记忆法：** 训练 loss 降，验证 loss 升，模型在死记硬背。
- **复习标签：** #SFT #过拟合 #早停

### 卡片 5
- **Q：** LoRA 合并到基础模型后，模型大小会变化吗？
- **A：** 会。假设基础模型权重为 W₀（d×k），adapter 权重 AB（d×k）。合并后新权重 W = W₀ + α·AB，模型文件大小增加约 d×k 个参数。例如 Llama2-7B 全部权重约 14GB（float16），LoRA 合并后增加 adapter 参数，但 adapter 本身很小（例如 65K 参数），几乎不增加文件大小。但合并后权重是更新后的，文件大小与原模型基本一致。
- **一句话记忆法：** 合并后权重变大，但 adapter 自身很小，总体几乎不变。
- **复习标签：** #LoRA #模型合并 #部署

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** LoRA 微调中，如何通过调整 rank r 来控制模型表达能力？具体说明当 r 过小或过大时分别会出现什么问题，以及如何在实际项目中选择 r 的值。

**面试口头版参考答案：**
LoRA 的 rank r 决定了 adapter 的表达能力上限。r 过小（比如 1 或 2）时，ΔW 的秩太低，无法捕获下游任务所需的权重变化，模型在验证集上准确率提升有限，甚至不如不微调。r 过大（比如 128）时，可训练参数增多，容易在小数据集上过拟合，且部署后合并权重虽无延迟增加，但训练显存和推理显存都会略增（因为 adapter 本身存在于显存中）。实际项目中，我的选择策略是先根据数据量决定 r 的搜索范围：数据量 < 1k 条时，r 从 4 到 16 搜索；数据量 1k~10k 时，r 从 8 到 32；数据量 > 10k 可考虑 r 从 16 到 64。然后对比验证集指标（如 F1、BLEU）选择最优的 r。此外，不同模块可以分配不同 rank：attention 的 Q 和 V 对 rank 更敏感，可以用稍大的 r；FFN 层重要性较低，可以用小 r。这样的非均匀分配可以进一步提高参数效率。

**深入理解版参考答案：**
从理论层面，rank r 决定了 adapter 中 AB 乘积所能表达的矩阵秩。若原始权重更新的真实秩为 R，则需要 r ≥ R 才能完全捕获。但实际中我们不知道真实 R，且 LoRA 的一个强假设是 ΔW 是低秩的。当 r 过小时，模型表达不足，微调后的输出与基础模型差异太小，任务适配效果差。当 r 过大时：1）可训练参数增多，在小数据集上过拟合风险上升；2）训练时每个 forward 需要计算 ABx，虽然计算量不大，但显存中需要保存梯度，会增加显存占用；3）合并后权重中的 adapter 部分可能带来噪声，干扰基础模型原本的通用能力。实验论文（如 LoRA 原始论文）表明，对于大多数任务，r=8 或 16 已经足够，增大 r 没有显著收益。

工程选择技巧：
1. **粗粒度搜索**：在选定的几个 r 值上（如 4, 8, 16, 32）训练少量 step（比如 100 步），观察验证 loss 的下降速度和最终值，快速筛选最优区间。
2. **模块级 rank 分配**：使用 PEFT 库的 `rank_pattern` 为不同 `target_modules` 设置不同 rank。例如对 `["q_proj","v_proj"]` 设 r=16，对 `["k_proj","o_proj"]` 设 r=8。这基于经验：Q 和 V 参与 attention 计算更强，需要更多容量；K 和 O 相对次要。
3. **动态 rank 搜索**：使用 LoRA 的扩展如 AdaLoRA（自适应分配 rank），在训练过程中根据梯度重要性调整各矩阵的 rank，但工程实现较复杂，一般仅在论文中使用。

在项目中，综合考虑数据量（约 5000 条）和计算资源，我选择 r=8 作用于所有目标模块，因为进一步增大到 16 只带来 1% 的提升但训练时间增加 30%。对于不同模块，我会在 r=8 的基础上对 Q 和 V 加倍到 r=16，效果略微提升。最终选择均匀分配 r=8 作为生产配置。

**可能追问与简短回答：**

- **追问 1：如果想把 LoRA 的 rank 设置得很大（如 256），是否可以用更多数据来防止过拟合？**  
  答：理论上可以，但 LoRA 设计的初衷是低秩适配，rank 很大（如 256 对于 4096 维权重来说秩达到 6.25%）已经不再低秩，此时不如直接用全量微调（参数量为 0.1% vs LoRA 的 6%）。LoRA 的优势在高 rank 下消失。推荐在 rank < 32 范围内使用 LoRA。

- **追问 2：Adapter 和 LoRA 的 rank 选择有没有联系？**  
  答：Adapter 也使用 bottleneck 维度（类似 LoRA 的 rank）来压缩中间表示。Adapter 的 bottleneck 通常取 64~256，比 LoRA 的 rank 大，因为 Adapter 权重是新的层而非增量。两者选择原则相似：数据量大、任务复杂时增大 bottleneck/rank。但 LoRA 更节省参数因为它的 decomposition 方式更高效。
