### 题目 1：在 RAG 系统中，如何确定 chunk_size？你做过哪些实验或有哪些经验方法？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- RAG Interview Questions and Answers (2026) - https://www.interviewbit.com/llm-interview-questions-answers - 技术博客 - 中（综合性面试题库）
- 大模型RAG面试题详解 - https://blog.csdn.net/star_nwe/article/details/153679906 - 技术博客 - 需进一步验证

**可信度：** 中（面经和博客，但内容合理，可参考）

**考察点：** RAG 预处理、chunk 策略、工程实验方法

**面试口头版回答：**  
chunk_size 通常根据文档类型和模型上下文窗口来定，没有固定数字。我常用的方法是：先基于文档语种和内容结构做一个初始切分，比如中文按 512 tokens、英文按 256 tokens，然后做一个 A/B 测试。具体地，我会固定检索和生成的其他参数，只调 chunk_size，用一份人工标注的评测集看 Top-5 召回率（Recall@5）和最终答案的 Faithfulness。实验发现：对知识型问答（如产品手册），256-512 tokens 效果最好；而长文档说明类，512-768 tokens 更合适。关键还要看 overlap（重叠），我一般设 10%-20% 的 overlap，防止关键信息刚好被切在边界上。

**深入理解版回答：**  
chunk_size 直接影响两个环节：召回相关性（Recall）和生成质量（上下文完整性）。如果 chunk_size 太小（比如 <128 tokens），每个块语义不全，召回容易丢失细粒度信息，且生成时缺乏上下文，容易跑偏；如果太大（>1024 tokens），一个块可能包含多个主题，召回会返回很多噪声，并且浪费 token 配额，还会加剧“Lost in the Middle”现象。  
我推荐的做法是：  
1. **基于文本结构切分**：先按文档段落（header）、句子边界做 semantic chunking，而不是简单按 token 数硬切。比如用 LangChain 的 RecursiveCharacterTextSplitter，以段落、句子、子句为层次。  
2. **实验流程**：构建 200-500 条人工标注的 query-chunk 对应关系，测试不同 chunk_size 下的 Recall@K 和 MRR（Mean Reciprocal Rank）。同时记录生成阶段模型的 Faithfulness 得分（比如用 RAGAS 的 answer_relevancy 或 faithfulness）。  
3. **常见坑**：中文切分时模型 tokenizer 的边界问题——中文字符的 token 数通常比英文字符多，所以要按实际 token 数而非字符数。另外，多模态文档（含表格、代码）需要单独处理，chunk_size 要相应调整并保留结构信息。  
**优化方向**：可以引入自适应 chunk，先做意图分类，如果是事实性查询就用小块，如果是综述性查询就用大块甚至整段。

**常见追问与回答：**  
- **追问1：chunk_size 和 overlap 怎么配合调参？**  
  Overlap 主要防止边界断裂。我通常固定 chunk_size 后，遍历 overlap 比例为 0%, 10%, 20%，看 Recall 的变化。通常 10% 就够了，超过 30% 会引入过多冗余，降低检索效率。  
- **追问2：如果知识库文档长短不一，怎么统一 chunk_size？**  
  我会按文档类型分别设 chunk_size：短文档（<512 tokens）不切分，直接作为 chunk；长文档按策略切。每个 chunk 要记录来源文档 ID，方便后去重和引用。

**项目联系：**  
在我们团队的企业内部知识问答项目中，我用过一个“最小有效 chunk”的思路：先做一次小规模实验，对不同文档类型（产品手册、合同条款、FAQ）分别找最优 chunk_size，然后写入配置文件。上线后通过 A/B 实验对比用户满意度，发现产品手册用 256 tokens、合同条款用 512 tokens 效果最好，最终固定了这两种切分策略。

**记忆关键词：** chunk_size, semantic chunking, 边界 overlap, Recall@K, Faithfulness, A/B 实验

### 题目 2：RAG 知识库的增量更新（upsert）在工程上如何实现？如何处理删除残留和版本一致性问题？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证（其中提到了“增量索引:chunk 粒度 upsert / 删除残留 / 版本一致性”）
- 2026 大模型面试题 | Agent 面试题 | RAG 面试题 | AI 应用开发面试指南 - https://javaguide.cn/ai/interview-questions/ai-interview-guide.html - 技术博客 - 需进一步验证

**可信度：** 中（面经细节+通用知识）

**考察点：** RAG 知识库维护、向量数据库更新、工程可靠性

**面试口头版回答：**  
增量更新一般通过向量数据库的 upsert API 实现，但工程上要考虑三点。第一是 chunk 粒度：文档更新后，不是全量重写，而是找到该文档关联的所有 chunk，逐个替换。我通常为每个 chunk 加一个 “doc_id + chunk_index” 的复合 ID，upsert 时指定相同 ID 就会覆盖。第二是删除残留：如果文档被删除，需要同步删除其所有 chunk，否则检索会返回已失效的信息。我会在删除时先查一次该 doc_id 下的所有 chunk ID，然后批量 delete。第三是版本一致性：如果同时有全量重建和增量写入，必须保证原子性。我常用双 buffer 模式：一个库是当前服务库，另一个是影子库；新数据先写入影子库，更新完成后切换到影子库，旧库再清空。这样可以避免更新过程中的不一致。

**深入理解版回答：**  
增量更新是生产 RAG 系统最容易被忽视的坑之一。简单用 update_all 成本高且影响线上服务。更好的架构是：  
1. **元数据标记**：每个 chunk 存储 metadata 字段，包括 `doc_id`, `version`, `last_updated`。  
2. **增量流水线**：使用消息队列（如 Kafka）监听源文档变更事件。每次变更事件触发：解析文档 → 切分 → 计算 embedding → 对每个 chunk 执行 upsert（用 doc_id+chunk_index 做唯一键）。  
3. **删除优化**：采用软删除 + 定期清理。元数据中加 `deleted_flag`，检索时过滤掉 `deleted = true` 的 chunk。每过一定周期（如每晚）清理一次硬删除的向量记录，以减少存储碎片。  
4. **版本冲突解决**：如果同一文档先后收到两个更新事件，用时间戳机制：只保留 timestamp 最新的那条。向量数据库本身不具备事务性，所以要用外部协调，比如通过 Redis 缓存最新版本号判断。  
5. **全量重建兜底**：定期（如每月）做一次全量重建，确保数据的一致性。全量重建时用影子库做切换，不影响线上。

**常见追问与回答：**  
- **追问1：如果向量数据库不支持 upsert（只支持增删），怎么办？**  
  那就先 delete 对应 doc_id 的所有 chunk，再 insert 新 chunk。必须保证原子性，可以用代码事务（先删后增，如果失败则回滚到快照）。  
- **追问2：存量知识库很大，每次全量重建太慢，怎么优化？**  
  可以采用分库分表策略，按文档类别或时间区间分多个向量集合（collection），全量重建时只重建变动频繁的集合。

**项目联系：**  
在我们的智能客服知识库中，每天都有新政策文档入库。我设计了一个增量更新模块：变动文档会触发一个 Lambda 函数，将文档切分后 upsert 到 Milvus。为了保证一致性，每个 chunk 记录 `policy_version`，检索时只取版本号不超过某个阈值的 chunk。同时每天凌晨 3 点做一次全量重建，重建时切换到影子集群，用户无感知。

**记忆关键词：** 增量更新, upsert, 删除残留, 版本一致性, 软删除, 影子库

### 题目 3：在 RAG 中，即使检索到了正确的上下文，模型仍然可能生成与上下文矛盾的答案。你如何诊断和修复这种 faithfulness 问题？

**来源依据：**
- Top Interview Questions on RAG for Data Science and AI Engineer Roles - https://buildml.substack.com/p/top-interview-questions-on-rag-for - 技术博客 - 需进一步验证（其中提到“how you reduce hallucinations even when the correct context is retrieved”）
- RAG Interview: 40 Questions to Go from Beginner to Advanced - https://www.analyticsvidhya.com/blog/2026/02/rag-interview-questions-and-answers - 技术博客 - 中（其中包含 evaluation 和 hallucination 内容）
- 大模型面试题 - kopoo - 博客园 - https://www.cnblogs.com/wusier/p/19084900 - 面经或社媒 - 需进一步验证（已讲过 SELF-RAG，但可换角度）

**可信度：** 中（博客和面经，但在资料中有直接提及）

**考察点：** 幻觉控制、faithfulness 评估、Bad Case 分析、Prompt 优化

**面试口头版回答：**  
这种情况属于隐式幻觉（即 retrieval 没问题，但 generation 错了）。我的诊断流程是：先收集一批 Bad Case，人工标注“检索块是否包含答案”和“模型回答是否与检索块矛盾”。然后统计模块化归因。常见原因有三个：一是 Prompt 约束不够，比如没有明确要求“仅根据检索内容回答”；二是模型本身对复杂逻辑推理不强，会在多个信息中混淆；三是检索出的 Top-K 中有冲突信息，模型选择了错误的那一个。修复手段依次是：优化 Prompt 加入“If the context does not contain the answer, say you don’t know”；对生成做结构化约束（比如要求输出必须有引用根据的 chunk ID）；对检出的冲突信息做重排序（如 Cross-Encoder rerank 剔除低质量 chunk）；最后，如果还不改善，可以用 SELF-RAG 的思路，让模型自己判断上下文是否足够。

**深入理解版回答：**  
Faithfulness 问题本质是 generation 阶段没有严格受限于 retrieval 结果。理论上，即使 recall=1，如果 generator 违背 context，答案仍然是错误的。工程上可以从三个层面干预：  
1. **输入层**：优化 Prompt 设计。标准 Prompt 如 “Based ONLY on the following documents, answer the question. If the documents do not provide enough information, say "I cannot find the answer". Do not add external knowledge.” 还可以在 few-shot 中加入正面和反面例子。  
2. **模型层**：使用更小但更服从指令的模型（如 Qwen-7B），或者微调一个专门的信令模型（instruction-following），要求它输出时必须引用 chunk 序号。  
3. **输出层**：后处理校验。对生成文本做 NLI（自然语言推理）——用一个小模型检查回答是否与检索块 entail，如果 entail 失败则降权重试甚至拒答。业界常用工具如 RAGAS 的 faithfulness 指标、TrueTeacher 等。  
**实际案例**：在一个医疗知识问答项目中，我们发现模型会把“可能引起头痛”写成“一定引起头痛”，即使检索块写的是“possibility”。解决办法是：在生成 Prompt 中加入“注意程度词不要改变”，同时在后端用正则检测程度词变化并标记告警。  
**量化归因**：对于 Bad Case，我建立一条流水线：记录每一步的输入输出（query、retrieved chunks、generated answer）。然后自动计算 faithfulness_score（用 NLI）。分数低于阈值的自动进入人工复审队列。每周一次分析饼图，看是检索问题（context 不含答案）还是生成问题（context 含答案但回答错误）。这样可以针对性优化。

**常见追问与回答：**  
- **追问1：NLI 模型的准确率够高吗？会不会有误杀？**  
  目前开源 NLI 模型（如 DeBERTa-v3）在 entailment 分类上准确率约 90%+，但也不能完全信任。我把它作为初筛，对“不肯定”的结果标记为“可疑”，送入人工抽检。同时可调整阈值，平衡召回和精度。  
- **追问2：如果模型总是输出“I cannot find the answer”怎么办？**  
  那可能是 Prompt 过度保守。可以回调 Prompt 中“不足则拒答”的权重，或增加 few-shot 中成功回答的例子。同时确保检索的真实 recall 够高，否则拒答多是因为真的没检索到。

**项目联系：**  
在实习中，我封了一个自动化排障脚本：输入一个 Bad Case，它会自动对比 retrieved chunks 和 answer，给出 faithfulness 得分，并高亮疑似矛盾的词。然后根据得分建议是改 Prompt 还是调 rerank。这个脚本在团队内被广泛复用。

**记忆关键词：** faithfulness, NLI, 后处理校验, 隐式幻觉, Prompt 约束, Bad Case 回归

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** RAG 中为什么需要 Rerank？双塔 embedding 检索已经够好了吗？  
- **A：** 双塔 embedding 检索速度快，但缺乏 query 和 chunk 之间的交互，只能做浅层语义相似度，无法精细判别。Cross-Encoder reranker 能同时将 query 和 chunk 送入模型，做交互式注意力计算，排序更准。一般先用双塔检索粗筛，再用 reranker 重排 Top-20 → Top-3，提升生成质量。  
- **一句话记忆法：** Embedding 像快筛，Reranker 像细查。  
- **复习标签：** #RAG #Rerank #Cross-Encoder

### 卡片 2
- **Q：** 如何保证 RAG 回答中的引用是可追溯的？  
- **A：** 两种做法：①Prompt 强制模型在回答中输出来源 chunk ID 或文档标题。②生成后做后处理校验：从答案中抽取出 ID，检查是否存在于元数据缓存中，不存在则降权或拒答。还可以在答案中附带超链接，让用户点击查看原始内容。  
- **一句话记忆法：** 引用像身份证，校验像查户口。  
- **复习标签：** #RAG #引用 #幻觉控制

### 卡片 3
- **Q：** 什么是 RAG 中的“Lost in the Middle”问题？如何缓解？  
- **A：** 指 LLM 对长上下文中间部分的信息容易遗忘或忽视。缓解方法：①将检索到的最重要 chunk 放在 prompt 开头或结尾；②压缩上下文，只保留 top-N 个 chunk；③使用 sliding window 或 hierarchical retrieval 先摘要。  
- **一句话记忆法：** 中间信息是盲区，重要内容放两端。  
- **复习标签：** #RAG #LostInTheMiddle #上下文

### 卡片 4
- **Q：** RAG 评测中，Faithfulness 和 Answer Relevancy 有什么区别？  
- **A：** Faithfulness 衡量生成的回答是否忠实于检索到的上下文（即是否矛盾或编造）。Answer Relevancy 衡量回答是否针对用户问题，不跑题。两者是正交的：回答可能相关但充满幻觉，也可能忠实但答非所问。通常用 LLM-as-Judge 或 NLI 模型分别打分。  
- **一句话记忆法：** Faithfulness 是“说的有依据”，Relevancy 是“答的对问题”。  
- **复习标签：** #RAG #评测 #Faithfulness #AnswerRelevancy

### 卡片 5
- **Q：** 增量更新 RAG 知识库时，如何避免“删除残留”问题？  
- **A：** 使用软删除：为每个 chunk 设置 `deleted_flag`，检索时过滤。定期执行硬删除（如每周清理一次）。同时记录元数据中的版本号和更新时间，保证同一文档更新时旧 chunk 被标记。  
- **一句话记忆法：** 先标记再清理，别留僵尸 chunk。

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**题目：** 请解释 Cross-Encoder 和 Bi-Encoder（双塔）在 RAG 中的本质区别。为什么 Cross-Encoder 的排序质量更高？它的主要缺点是什么？如何权衡？

**面试口头版参考答案：**  
Cross-Encoder 把 query 和 candidate document 拼接成一句话，让同一个 Transformer 模型做双向注意力，能捕捉 query 与 document 之间丰富的交互信息（如精确匹配、语义对齐、逻辑推理），所以排序更准确。Bi-Encoder 则把 query 和 document 各自独立编码成向量，靠余弦相似度计算，无法看到交互，但好处是 document 向量可以预计算、缓存、用近似最近邻（ANN）快速检索。因此工程上典型的做法是：Bi-Encoder 做第一轮粗筛（Top-k 大一点），Cross-Encoder 做第二轮细排（Top-N 小一点），在精度和速度间取得平衡。

**深入理解版参考答案：**  
- **结构差异**：Bi-Encoder 是双塔，query 和 document 不交互；Cross-Encoder 是单塔，输入为 `[CLS] query [SEP] document [EOS]`，计算所有 token 之间的注意力，输出一个分类 logits 表示相关性。  
- **为什么 Cross-Encoder 更好**：因为交互注意力可以捕获词汇重叠、语义蕴涵、否定词影响等细微关系。例如，query：“该药物不能与哪种食物同服？” document：“该药物可与苹果同服，但不能与西柚同服。” Bi-Encoder 的向量可能认为“与苹果同服”和“与西柚同服”相似度相近，而 Cross-Encoder 能明确否定关系。  
- **缺点**：推理成本高。对于每个 query-document pair，Cross-Encoder 都需要一次完整前向计算，而 Bi-Encoder 的 document 向量可预存，只需对 query 做一次编码。假设 Top-20 文档需要 rerank，Cross-Encoder 需要 20 次推理（batch 处理可合并），Bi-Encoder 只需要 1 次 query 推理和一次 ANN 搜索。  
- **权衡策略**：①使用轻量级 Cross-Encoder（如 MiniLM），精度仍高于双塔但速度提升；②控制 rerank 数量，通常只对 Top-20 或 Top-50 做；③对长文档只取头部部分做 rerank 以减少 token 数。  
- **实际效果**：在 MS MARCO passage ranking 上，Cross-Encoder 的 MRR@10 通常比 Bi-Encoder 高 5-10 个点，但在 latency 上慢 1-2 个数量级。

**可能追问与回答：**  
- **追问1：能否用知识蒸馏将 Cross-Encoder 的排序能力转移到 Bi-Encoder？**  
  可以。训练一个教师 Cross-Encoder，用它的 soft labels（相关性分数）作为 Target，微调一个学生 Bi-Encoder。学生模型在推理时保持双塔速度，但排序质量接近教师。实践中很多检索模型（如 ColBERT 等）采用了类似思想。  
- **追问2：如果选择 Cross-Encoder，模型的输入长度有限（如 512 tokens），chunk 太长怎么办？**  
  可以对长 document 做 sliding window 切分成多个片段，每个片段和 query 一起过 Cross-Encoder，取所有片段中最高分作为该文档的得分。或者只截取 document 的首部 512 tokens（大多数情况关键信息在前部）。
