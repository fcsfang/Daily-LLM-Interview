### 题目 1：设计一个企业内部智能客服 Agent 系统

**来源依据：**
- 腾讯云技术社区文章：*这几道高频RAG面试题你都会了吗* - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 可信度高，其中提到了“RAG作为Agent的工具”架构
- 小林面试笔记网站：*大模型面试题（Agent 专题）* - https://xiaolinnote.com/ai - 面经或社媒 - 需进一步验证，但结构完整可作为参考

**可信度：** 高

**考察点：** 系统设计能力、跨模块整合（Agent + RAG + Tool Calling）、项目讲述、选型决策

**面试口头版回答：**
我设计的企业内部智能客服Agent系统，核心是“大脑+工具”架构。大脑使用ReAct推理模式，负责意图识别和任务拆解；工具包里包含RAG知识库、HR查询API、工单系统等。RAG专门处理FAQ和规章制度查询，通过稠密+稀疏混合检索、rerank精排，最后让模型引用原文回答。Agent在每步推理后会检查工具返回结果，遇到空结果会触发“退而求其次”策略（比如尝试同义词改写查询）。系统通过LangGraph编排多步状态机，保证中间状态持久化。我们还会在线上埋点日志，包括每个步骤的耗时、调用次数、用户满意度反馈，用于后期优化。

**深入理解版回答：**
设计企业智能客服Agent，首先需要明确三个核心能力：知识查询、任务执行、对话管理。我的方案分为以下模块：

1. **意图识别与规划层**：采用ReAct循环，每次先“思考”当前需要做什么，再“行动”调用对应工具。为了避免模型在复杂任务中“原地打转”，我会限制最大推理步数（例如5步），并在每步后对计划的合理性做评分（可以用一个小LLM或规则检查）。
2. **工具层**：包括RAG知识库、内部API、计算器等。RAG部分采用**语义分块**（利用LLM判断句子是否完整再切分），使用BGE-large作为Embedding模型，向量库用Milvus（支持10亿级向量）。检索时混合BM25关键词召回，得到Top 200后由BGE-Reranker精排到Top 10，保证喂给LLM的上下文高相关。当Agent调用工具超时或返回错误时，设计重试机制（最多2次，每次指数退避），失败后上报错误类型并引导用户转人工。
3. **记忆管理**：短期记忆通过上下文窗口（message list）保留最近3轮对话；长期记忆使用向量DB存储用户画像（通过每次对话后由LLM提取摘要并embedding）。记忆更新策略是：如果新摘要与已有记忆的余弦相似度>0.85，视为冲突，利用时间戳仲裁——保留最新的内容，同时保留原始内容作为候选。
4. **可观测性与评测**：每个请求生成trace_id，记录Agent每一步的思考内容、工具调用参数和结果。离线评测用RAGAS框架计算Faithfulness、Answer Relevancy；在线评测关注用户点赞率、挂起率、首次响应时间。当整体满意度低于阈值时，自动拉取对应trace日志，分析是哪个环节出了问题（比如频繁触发退而求其次，说明检索召回不足）。

优点：高内聚、低耦合，便于单独优化每个模块。局限：增加一处外部依赖后，整体延迟可能增加（RAG检索+rerank可能要多几百ms），需要异步编排来缓解。

**常见追问与回答：**
- **追问1：你如何处理多个用户同时访问时的性能？**  
  答：Agent的LLM部分使用部署好的推理服务，支持动态batching；RAG向量库用Milvus的分布式部署；工具调用是HTTP请求，会做连接池和限流。整体无状态，可以水平扩展副本数。
- **追问2：如果Agent连续两次工具调用返回同样的错误，你怎么处理？**  
  答：在重试逻辑中加入“上下文敏感”的检测：如果上次同一工具已报错，本次不重复调用，直接向用户解释“该功能暂时不可用”，并推荐替代方案。
- **追问3：如何保证Agent不会泄露敏感信息？**  
  答：在输入输出层加正则过滤和LLM自校验提示词；RAG文档本身已做脱敏；Agent回答前会调用一个安全检查工具，如果检测到敏感词，拒绝回答并上报。

**项目联系：**
在我的个人项目中，我用LangGraph实现了一个简化版客服Agent，包含一个RAG工具和一个天气查询API。RAG部分我手动构建了200份公司制度文档（来自公开资料）。评测时我对比了不加rerank和加rerank后的Faithfulness分数（0.72→0.89）。你可以在GitHub上找到这个项目，代码结构清晰，面试时可以直接展示。

**记忆关键词：** #Agent #ReAct #RAG #工具调用 #系统设计 #记忆管理

### 题目 2：当RAG系统回答质量下降，如何系统排查？

**来源依据：**
- 腾讯云技术社区文章：*这几道高频RAG面试题你都会了吗* - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 可信度高，其中“内容缺失问题”的排查三步法
- LinkedIn帖子（Bilal Momin）：*AI Engineer Interview: LLM Applications and System Design* - https://www.linkedin.com/posts/bilal-momin_generativeai-llm-langgraph-activity-7353048174648840192-C0CR - 面经或社媒 - 需进一步验证，但提到如何减少幻觉的方法

**可信度：** 高（结合已有的可观测性知识进行综合提问）

**考察点：** 问题排查能力、RAG链路理解、日志分析、闭环思维

**面试口头版回答：**
我会从三个层面系统排查。第一层：检查 **检索质量**。我看到用户反馈说“回答错误”，先查trace日志中召回的前K个文档的相关性。如果召回结果与问题完全不相关，说明问题在向量检索或chunking。此时我会对比BM25和稠密检索的召回率差距，可能原因有：Embedding模型不够好、文档分块破坏了语义。第二层：检查 **生成质量**。如果召回文档相关，但LLM仍然编造，说明模型没有遵循“只基于给定文档回答”的指令。我会在prompt中强化约束，并让模型输出引用索引。第三层：上线 **自动化评测**，比如用RAGAS计算每一批请求的Faithfulness分数，设定阈值告警，这样不用等用户反馈就能主动发现问题。

**深入理解版回答：**
系统排查需要建立从“用户反馈”到“根因”的链路，我总结为“五步闭环法”：

1. **采集反馈**：在对话界面上 “👍/👎” 按钮，并在👎时主动问一句“是答案不对还是找不到答案？”同时记录trace_id。
2. **拉trace**：根据trace_id获取该请求的全链路数据，包括原始query、chunk ID、召回文档、rerank后的文档、LLM回答、token用量、耗时。
3. **定位阶段**：先看 **检索阶段** 是否出问题。计算Hit Ratio（Top K中是否有正确答案的chunk），如果Hit Ratio低，则问题大概率在“查不准”。接着看Rerank后文档是否相关，如果不相关，可能是rerank模型误判。再看 **生成阶段**，如果召回相关但回答不相关，计算Faithfulness（回答是否忠实于文档）。常见低Faithfulness的情况：LLM未引用或引用了不相关的段落。
4. **根因分析**：根据定位结果，分三类处理：
   - **检索召回不足**：优化chunk策略（语义切块）、混合检索、Query改写。
   - **rerank误判**：更换更强大的reranker（如BGE-Reranker-v2）或调整阈值。
   - **生成幻觉**：强化prompt约束、要求输出chunk ID引用、加验证步骤（LLM自检）。
5. **验证修复**：在离线数据集上回测，确保修复后指标不倒退，然后灰度上线，继续监控在线满意度。

这个闭环持续迭代，每两周审查一次整体指标趋势。

**常见追问与回答：**
- **追问1：如何自动判定“召回结果相关”？**  
  答：构建标注数据集，用人工标注每个query的相关文档作为ground truth，然后计算Hit Rate / MRR。线上可以用LLM as Judge打分，相关性评分≥0.7算相关。
- **追问2：如果用户反馈“回答正确但不够详细”，你的排查思路有什么不同？**  
  答：那就是“Retrieval粒度”问题，可能chunk太小，缺乏上下文。我会检查召回文档是否只包含了片段而没有完整逻辑。优化方向：增加chunk重叠行数、使用parent-child chunk（大chunk检索，小chunk生成）。

**项目联系：**
在RAG项目中，我实现了自动打点trace log，包括query、recall docs、score、LLM output。有一次发现问答系统的Faithfulness从0.93跌到0.68，查trace发现召回了一篇无关pdf（因为chunk中包含部分无效字符导致embedding偏差）。修复chunk清洗规则后指标回升。这个排查过程我整理成了博客，面试时可以分享。

**记忆关键词：** #RAG #排查 #可观测性 #Faithfulness #闭环 #chunking

### 题目 3：对比微调（Fine-tuning）与 RAG 的适用场景，并给出选型决策框架

**来源依据：**
- 腾讯云技术社区文章：*这几道高频RAG面试题你都会了吗* - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 可信度高，其中第一道题详细对比了微调和RAG
- 小林面试笔记 - https://xiaolinnote.com/ai - 面经或社媒 - 需进一步验证，但提及RAG与微调的优势说明

**可信度：** 高

**考察点：** 两种技术本质理解、选型决策、项目落地权衡

**面试口头版回答：**
微调相当于让模型死记硬背，把知识固化到参数里，适合需求稳定、知识不变且需要高度自定义输出格式的场景（如特定风格的对话）。RAG相当于开卷考试，知识外挂在向量库，适合高频更新、需要引用的场景（如内部知识库、客服）。我的选型决策框架是：先判断 **知识更新频率**，如果每周都有变化，必须用RAG；如果半年不变且需要模型具备特殊风格，用LoRA微调。然后是 **数据可用性**：如果只有少量QA对，RAG即可；如果有大量高质量平行语料，可以考虑用RAG+微调互补。在实践中，我通常优先RAG，如果检索后回答质量不足，再在RAG输出上微调一个结果摘要模型。

**深入理解版回答：**
微调和RAG不是对立关系，而是各有优劣。微调（尤其是LoRA）可以改变模型的行为和知识，但成本高、更新麻烦；RAG则天然支持知识动态更新，但依赖检索质量。

我的选型决策框架是一个简单的二叉树：

1. **是否需要引用原文或严格合规？** → 是 → 必须RAG（法律、医疗、金融合规场景）
2. **业务知识更新周期 < 1个月？** → 是 → RAG（优先）
3. **模型输出格式需要高度一致（比如按固定JSON Schema输出）？** → 是 → 结合RAG + 微调（先RAG拿到信息，再用微调模型规范化格式）
4. **有大量专有数据且不需要频繁变？** → 是 → 微调（适合内部客服话术模板）
5. **其他情况** → 先RAG上线，如果效果不理想再补微调

在成本方面，微调需要标注数据、GPU训练时间、模型版本管理；RAG需要维护向量库、定时更新。两者结合的例子：先用RAG检索出最相关的10个文档片段，然后让一个轻量微调模型对这些片段做摘要和重写，既能保证知识实时性，又能控制输出风格。

常见坑：以为微调能解决所有问题，但实际上模型会“遗忘”原有能力（灾难性遗忘），而RAG如果检索不到，模型会胡编。因此最佳实践是RAG-First，微调作为补充。

**常见追问与回答：**
- **追问1：如果业务同时需要多语言支持，你会怎么选？**  
  答：RAG更灵活，只要文档有多语言版本即可；微调需要额外多语种数据，成本高。但微调可以让模型学会更好的跨语言对齐，如果有充足资源可以两者结合。
- **追问2：RAG+微调的具体结合方式是什么？分享一个方案。**  
  答：我在项目中采用“RAG检索 → 大模型生成 → 微调模型rerank”的方式：先用RAG + LLM生成候选答案列表，然后用一个微调的小模型（比如BERT）排序最佳答案。另一种方案是对生成结果做后处理微调，比如用LoRA微调一个“结果格式化”模型。

**项目联系：**
在RAG项目中，我对比了纯RAG、纯微调（LoRA）、和RAG+微调（微调用于规范输出）的效果。纯RAG的Faithfulness 0.88，但输出格式乱；加上微调后格式正确率达95%。我写了一个实验报告，对比三个设置的指标，可以用在项目介绍中。

**记忆关键词：** #微调 #RAG #选型 #决策框架 #LoRA #RAG-First

### 题目 4：解释 MoE（Mixture of Experts）架构的核心思想及其在 LLM 中的优势与挑战

**来源依据：**
- B站视频：*[大模型面试] 主流LLM为何选用MoE架构? MoE相较Dense的核心优点? LLM不可能三角* - https://www.bilibili.com/video/BV1t9CrBzENj - 面经或社媒 - 需进一步验证（仅凭标题推断内容）
- 小林面试笔记网站：*大模型工程面试题* - https://xiaolinnote.com/ai - 面经或社媒 - 需进一步验证，其中包含关于Scaling Law、Transformer等内容

**可信度：** 中（无直接答案，但主题常见）

**考察点：** Transformer架构深入理解、模型部署权衡、计算资源优化

**面试口头版回答：**
MoE核心思想是把一个大模型拆成多个“专家”子网络，外加一个路由门控网络。每次推理时，门控网络决定激活哪几个专家（通常是Top-2），其他专家不计算。这样在计算量不变的情况下，模型参数量可以大幅增加，因为每次只激活部分专家。优势是：用更少的FLOPs获得更大的模型容量，训练速度快、推理成本相对低。挑战在于：负载均衡——如果路由不平衡，某些专家过热，某些闲置；通信瓶颈——专家分布在多卡上，需要跨卡通信；内存占用依然大（因为需要加载所有专家参数）。主流方案有专家并行、动态路由调度等。

**深入理解版回答：**
MoE（Mixture of Experts）最早在2017年被提出用于翻译，后来被用于LLM（如Mixtral 8x7B、DeepSeek-V2等）。其结构：每一层FFN替换为由N个专家网络和一个门控网络组成。输入token经过门控，得到一个概率分布，然后选择Top-K个专家做前向传播，最后加权求和。这样，同样的总FLOPs下，参数量可以扩大到K倍以上。

**优势：**
- 在相同推理成本下，模型性能更好（因为容量大，专家可以 specialize 不同子知识）。
- 训练时可以通过更多的专家提升模型能力，但开销只增加线性。
- 适合在多语言、多任务场景中，不同任务激活不同专家。

**挑战：**
- **负载均衡**：如果路由偏向少数专家，会导致专家利用率不均衡，出现“死专家”。常用辅助损失（如Z-loss）和动态调整。
- **通信开销**：专家需要分布在多GPU上，token需要广播给所有专家所在设备，然后收集结果。这引入all-to-all通信，延迟较大。
- **内存占用**：所有专家参数必须加载到内存（虽然推理只激活少数，但加载是全部）。工程上可以用专家并行（Expert Parallelism）来分摊。
- **推理优化**：传统batch推理中，不同token可能激活不同专家，导致计算不规整。需要特殊优化如Tutel、DeepSpeed-MoE。

**常见追问与回答：**
- **追问1：MoE和dense模型相比，在部署上哪个更贵？**  
  答：从GPU内存看，MoE需要更多显存（因为参数多），但推理的计算量可能更少（因为只激活2个专家）。综合成本要看token量、批量大小和专家分布；在token密集型场景，MoE可能更省算力。
- **追问2：Load Balancing Loss怎么设计？**  
  答：通常对门控输出的概率分布计算均匀分布的KL散度，或者加一个辅助损失：鼓励每个专家处理相近数量的token。比如DeepSeek-V2使用“偏差”系数来动态调整路由概率。

**项目联系：**
在模型部署项目中，我测试过用vLLM部署MoE模型（Mixtral）。发现其token生成吞吐量在批量1时比7B dense模型快30%，但批量增大后优势减弱，因为专家并行通信开销增加。我记录了不同batch size下的延迟对比，可以用于面试中展示对部署优化的理解。

**记忆关键词：** #MoE #混合专家 #路由 #负载均衡 #专家并行 #推理优化

### 文章一：这几道高频RAG面试题你都会了吗

- **链接**：https://cloud.tencent.com/developer/article/2661823
- **来源依据**：腾讯云技术社区 - 技术博客 - 可信度高
- **文章类型**：工程实践
- **核心内容总结**：
  - 微调 vs RAG：核心区别是参数记忆 vs 非参数记忆，RAG知识动态更新成本低。
  - RAG效果评估分层：事实准确性（Hit Rate/MRR）、引用精确度（Faithfulness）、ROUGE指标，并首推RAGAS框架自动化评估。
  - 内容缺失问题排查三步法：切块策略→混合检索→重排序。
  - RAG作为Agent工具：Agent大脑负责意图识别，RAG负责知识查询；还有Agent引导检索、RAG记忆历史对话等进阶玩法。

### 卡片 2
- **Q：** 你如何理解RAG中的“内容缺失”问题？如何从指标上发现它？
- **A：** 内容缺失指用户问题涉及的知识在文档中没有对应片段，导致召回结果为0或很低。指标上观察Hit Rate（Top 5中是否包含相关文档），如果Hit Rate低于80%，大概率就是内容缺失。另一个指标是“Answer with No Support”，即模型回答里不引用任何来源的比例。
- **一句话记忆法：** “查不到”就是系统说“我不知道”的根本原因，看Hit Rate就行。
- **复习标签：** #RAG #检索 #Hit Rate

### 卡片 3
- **Q：** 什么是LLM的“Lost in the Middle”现象？工程上如何缓解？
- **A：** 模型在长上下文中，对中间位置信息的关注和利用能力下降，更依赖开头和结尾。工程缓解方法：1）将最相关文档放在开头和结尾；2）缩短上下文长度，只保留高相关片段；3）使用位置编码优化（如ALiBi）或Memory机制。
- **一句话记忆法：** 模型“只看头和尾，中间是空气”，所以类似面试时“第一印象”和“最后总结”最重要。
- **复习标签：** #LLM #长上下文 #注意力

### 卡片 4
- **Q：** 你如何进行Query Rewrite（查询改写）？举一个具体例子。
- **A：** 当用户输入“它怎么设置？”时，需要结合历史对话将“它”解析为具体对象。我通常用一个LLM做Query Rewriter，输入是历史对话+当前query，输出是改写后的完整query。例如用户上轮说“我想配置邮箱”，本轮说“怎么设置？”→ 改写为“如何配置邮箱的自动转发规则？”。
- **一句话记忆法：** 像搜索引擎自动补全 + 消歧义，让检索更精准。
- **复习标签：** #RAG #Query重写 #语义解析

### 卡片 5
- **Q：** 在MoE模型中，如何解决路由负载不均衡问题？
- **A：** 常用两种方法：1）添加负载均衡辅助损失（如对每个专家的分数施加KL散度惩罚）；2）动态调整门控网络的偏置（bias），使热门专家分数降低，冷门专家升高。在训练时还可采用“专家容量”限制+丢弃机制。

## 4. 今日面试专项训练

### 4.1 大模型理论追问题：MoE路由机制与负载均衡的博弈

**问题：** 请深入解释MoE中路由机制（Routing）的原理，以及在训练和推理中分别存在哪些负载均衡挑战？你会如何设计一个既高效又公平的路由策略？

**面试口头版参考答案：**
MoE路由由一个可学习的门控网络（Gating Network）实现，输入token被映射到每个专家上的分数，然后选取Top-2专家执行前向计算。训练时面临的挑战是“专家崩溃”——少量专家承担大部分任务。我采用的策略是在损失函数中加入Load Balancing Loss，比如对专家接受的token数做均匀分布惩罚。推理时，因为专家参数是分布在不同GPU上的，可能产生通信热点。我会在路由决策时优先选择与token所在GPU距离最近的专家（基于拓扑），以减少跨卡通信；同时可以在推理前用启发式预分配来平滑负载。

**深入理解版参考答案：**
路由机制的本质是离散选择问题。门控网络输出一个与专家数等长的实数向量，经过softmax得到概率分布，再用Top-2（或Top-1）选择。路由的不可微性（离散选择不可导）通过使用Gumbel-Softmax或多任务损失来绕过。

**训练时负载均衡挑战：**
- 专家退化：某个专家由于随机初始化优势，持续获得高分数，其他专家几乎不被选中。
- 解决方法：在损失函数中增加Load Balancing Loss，常见的有三种形式：
  1. **重要度平衡**：计算每个专家的使用率熵，鼓励均匀分布。
  2. **令牌平衡**：对每个专家的token计数的平方和做惩罚。
  3. **哈希路由**：使用固定的哈希函数代替可学习路由（如Switch Transformer中）。
- 另外，可以引入“专家容量”限制：每个专家最多处理C*token数（C>1），超出的token被丢弃（不做任何专家处理）或随机分给其他专家。这可以防止个别专家过载。

**推理时负载均衡挑战：**
- 推理时专家参数分布在不同GPU上，每个token需要将自己通过all-to-all通信广播给所有专家所在GPU，然后收集结果。如果某GPU上专家的负载过高，会成为瓶颈。
- 缓解措施：
  1. **动态容量缩放**：根据在线负载实时调整各专家的容量上限。
  2. **拓扑感知路由**：在路由决策时，优先选择与token同一GPU上的专家（如果有），避免通信。
  3. **Kube+MoE**：结合Kubernetes调度，将专家pod分布在节点上，并用消息队列平滑请求。

**2 个可能追问与简短回答：**
- **追问1：为什么训练时不能用固定的负载均衡而必须用辅助损失？**  
  答：因为路由函数不可微，固定分配无法端到端学习。辅助损失是唯一让门控网络感知负载不均的方式，同时不会破坏主损失梯度。
- **追问2：推理时如果某专家因容量限制被跳过，那些token怎么办？**  
  答：通常直接丢弃，不经过任何专家，路由信号的加权输出中该专家贡献为0。虽然会损失一点性能，但保证了系统不会因个别专家过载而整体延迟飙升。高质量的训练可以容忍少量token被跳过。

**记忆关键词：** #MoE #路由 #负载均衡 #辅助损失 #专家容量 #通信优化
