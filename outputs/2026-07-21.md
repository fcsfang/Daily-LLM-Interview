### 题目 1：设计一个 Agent 的 tool calling 流程，如何确保 LLM 输出的动作格式正确、且能处理工具调用失败？
**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 2026 大模型面试题 | Agent 面试题 | RAG 面试题 | AI 应用开发面试指南（含答案与图解） | JavaGuide - https://javaguide.cn/ai/interview-questions/ai-interview-guide.html - 技术博客 - 需进一步验证

**可信度：**
中（面经总结，但问题经典且被多次提及）

**考察点：**
tool calling 流程设计、结构化生成格式、重试与回退策略、工具执行错误处理

**面试口头版回答：**
我会把 tool calling 分成三步：格式约束、严格执行、错误恢复。第一步，在系统提示里给出清晰的动作定义和 JSON Schema，并强制模型用 structured output（比如 OpenAI 的 response_format 或 Outlines）来保证输出可解析。第二步，代理框架拿到解析后的 action 后，调用对应工具，捕获异常。第三步，如果工具调用失败（超时、返回错误码），我不会直接报错，而是把错误信息写回上下文，让模型决定是重试、换参数还是换工具。同时我还会设置最大重试次数，防止无限循环。在项目中，我用 LangGraph 的节点状态来维护当前步数，如果重试超过 3 次就触发降级策略——比如直接返回“暂时无法处理”或转人工。

**深入理解版回答：**
一个生产级 tool calling 流程必须处理两个核心问题：格式可靠性和运行错误。
格式可靠性方面，纯靠 prompt 约束是不够的，LLM 可能输出不合法 JSON、参数拼写错误、或者调用了不存在的工具名。工程上我会采用三层防御：第一层，在 prompt 中提供 tools 的 OpenAPI 描述，并给示例；第二层，利用模型的 structured output 能力（如 OpenAI Function Calling / json_mode 或本地部署时使用 Outlines、lm-format-enforcer）直接将输出约束为合法 JSON；第三层，在代码层做 schema 验证（pydantic），如果验证失败，不中断流程，而是将解析异常信息格式化为一条“system message”重新发给模型，让它修正。我做过 A/B 实验，加这层验证修正后，工具调用成功率从 87% 提升到 96%。
运行错误方面，工具本身可能不可靠（网络超时、权限不足、返回的数据为空）。我会为每个工具注册一个 error handler，将异常信息标准化后注入上下文。然后 LLM 根据错误信息判断下一步：如果是临时性错误（超时），重试；如果是参数错误，修正参数；如果是不可恢复错误（比如搜索 API 无权限），则向用户给出解释并提供替代方案。同时我还会监控整体工具调用次数，当单次任务中调用次数超过预设阈值（比如 10 次）时，主动结束循环并输出摘要，避免 token 浪费和延迟过高。这套机制在我负责的客服 Agent 项目中，将任务完成率从 72% 提升到了 88%。

**常见追问与回答：**
- **追问：如果模型在修正后依然输出错误格式，你会怎么做？**
  答：我会在第 N 次重试后（比如 3 次），放弃工具调用，直接让模型用已有信息生成一个保守的回答，或者标记“工具暂时不可用”并转接人工。关键是不要让用户面对一个死循环。
- **追问：如何防止模型在工具调用成功后继续无意义地调用工具？**
  答：我会在系统提示中明确“任务目标完成后必须回复用户并停止调用”，同时在代码中判断“最终答案是否已包含在上下文”，如果是，则强制跳出循环。

**项目联系：**
在之前的智能客服 Agent 项目中，我设计了这套 tool calling 流程。具体实现了工具注册区（每个工具带错误码定义）、重试次数配置、以及一个 Callback 类来记录每次调用的延迟和结果。这个设计让我在面试中能一步步讲清楚“如何从工程上保障可靠性”。

**记忆关键词：**
结构化输出、Schema 验证、重试降级、错误回注、最大步数限制

### 题目 2：Agent 的 Memory 如何设计？请从短期记忆、长期记忆、冲突更新三个角度展开。
**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- AI-Agent 面试题汇总 - 大模型篇_牛客网 - https://m.nowcoder.com/discuss/860538803759386624 - 面经或社媒 - 需进一步验证

**可信度：**
高（字节面经详细描述了该项目级设计，且牛客网汇总了常见 Memory 问题）

**考察点：**
Memory 分层架构、信息提取与压缩、冲突检测与仲裁、注入量控制

**面试口头版回答：**
我将 Memory 分成短期和长期两层。短期记忆保存最近几轮对话原文，保持对话连贯和指代还原；长期记忆只存结构化条目，字段包括类型、内容、来源轮次、时间戳。每条目的类型分为用户画像、硬约束、关键事实、待办等。提取阶段，每轮结束我用轻量 prompt 从对话中抽取值得长期保存的信息，并通过规则过滤掉寒暄和一次性内容。压缩阶段，对长期记忆中同一主题的条目定期做摘要合并，比如用户三次提到口味偏好，合并成一条最新画像。冲突更新方面，当新信息与旧记忆矛盾时，我不直接物理删除，而是根据时间戳和来源置信度判断：用户显式说的优先于模型推断的，新的优先于旧的。拿不准时，两条都保留，检索时按置信度排序。检索注入时，我只取 3-5 条最相关的长期记忆条目，加上最近 2-3 轮短期记忆原文，避免上下文膨胀导致“lost in the middle”。

**深入理解版回答：**
设计 Agent 的 Memory 本质是解决“记什么、怎么压、冲突了听谁的”三个问题。
记什么：不是每句话都存。每轮结束后我用一个轻量抽取 prompt（约 200 token）提取关键信息。我定义了一个 schema：type (profile/constraint/fact/todo)，content, source_round, timestamp。然后通过规则过滤掉长度小于 5 个词、包含纯寒暄词（如“你好”“谢谢”）的条目。剩下的再经过一个小模型（如 GPT-3.5-turbo）打“长期价值”标签，阈值以上的才入库。
怎么压：短期记忆保留原始文本，长期记忆只保留结构化条目。我对长期记忆做定期（比如每 10 轮或每新增 5 条）的摘要合并：对同一 type 且 content 语义相近的条目，合并成一条最新的值。但硬约束（如“只看华东区”）不参与合并，它们被独立标记，每轮都强制携带。
冲突了听谁的：我维护一个冲突仲裁器。当新条目与旧条目的 content 矛盾时（通过 embedding 相似度 + 语义判断），我检查两者的时间戳和来源标记。来源优先级：用户显式陈述 > 用户确认的推断 > 模型自发推断。时间戳方面，如果新条目时间更新且来源优先级更高，则旧条目标记为“失效”（不删除，保留历史），新条目激活。如果来源优先级相同但时间相反，则两条都保留，检索时按置信度排序。这种设计在我的项目中避免了“用户改偏好”后旧信息污染答案的问题。
注入量控制：我严格限制上下文中记忆的数量。短期记忆保留最近 3 轮完整对话，长期记忆只取 top-5 条（按 query 相关性召回后，再用时近性加权重排）。硬约束例外，每轮都带。测试表明，超过 7 条长期记忆后，答案 Faithfulness 开始下降。

**常见追问与回答：**
- **追问：记忆向量化后，你会怎么平衡相关性和时效性？**
  答：我用混合排序：先用 embedding 算相关性分数，然后乘以一个时间衰减系数（如 exp(-Δt/τ)），再相加。τ 设为 24 小时，这样同一个话题最近几小时的记录优先。
- **追问：如果用户在一次对话中频繁改变约束怎么办？**
  答：我会记录每次变更，保留最后两次。如果两次变更间隔很短（3 句内），且语义冲突，我倾向于信任最后一句，但会在最终回复中加一句“您是否最终确认使用 XX 规则？”，让用户确认。

**项目联系：**
我在“企业智能问答 Agent”中实现了这套 Memory 系统。我们利用 LangChain 的 BaseMemory 抽象，自定义了 LongTermMemory 模块，内部集成向量存储 (Chroma) 和仲裁逻辑。离线评测集显示，带有冲突仲裁的版本比纯时序覆盖版本，多轮问答准确率提高了 14%。

**记忆关键词：**
短期记忆、长期记忆、结构化提取、摘要合并、冲突仲裁、时间戳+置信度、注入量控制

### 题目 3：如何设计 Agent 的 Planner/Executor 架构？如何处理规划失败或执行异常？
**来源依据：**
- 2026 大模型面试题 | Agent 面试题 | RAG 面试题 | AI 应用开发面试指南（含答案与图解） | JavaGuide - https://javaguide.cn/ai/interview-questions/ai-interview-guide.html - 技术博客 - 需进一步验证
- hello-agents/Extra-Chapter/Extra01-面试问题总结.md - https://github.com/datawhalechina/hello-agents/blob/main/Extra-Chapter/Extra01-%E9%9D%A2%E8%AF%95%E9%97%AE%E9%A2%98%E6%80%BB%E7%BB%93.md - 技术博客 - 需进一步验证

**可信度：**
中（综合技术社区整理）

**考察点：**
Planner/Executor 分离设计、状态管理、局部重规划、最大步数限制、错误恢复

**面试口头版回答：**
我会将一个复杂任务分解为 Planner 和 Executor 两个组件。Planner 负责生成多步行动计划（比如 ReAct 中的 Thought-Action 链），它定义每一步的目标和预期结果。Executor 负责逐步执行计划：每步调用工具，并把结果写回状态。如果执行中出错，我不直接终止，而是把错误信息送回 Planner，让 Planner 重新调整后续步骤。为了应对无限循环，我会设置最大步数（比如 20 步）和最大重试次数。同时我会保存执行过程中的状态快照，当失败回退时，可以从上一个成功状态继续。在实际项目中，我用 LangGraph 实现了一个带 StateGraph 的 Planner-Executor，每一步后都检查是否达到目标，如果连续 3 步没有进展则触发局部重规划或直接向用户请求澄清。

**深入理解版回答：**
Planner/Executor 架构的核心是将复杂任务分解为可控子任务，并隔离决策和执行的职责。
具体设计：Planner 模块接收用户目标和当前状态，生成一个 JSON 列表，每个元素包含 step_id、action（工具名/子任务描述）、parameters、expected_outcome。Executor 模块依次执行：调用对应工具，并将返回值保存到全局状态 dict 中。执行后的状态会被送回 Planner，Planner 判断是否继续、修正或终止。
容错设计至关重要：
1. 局部重规划：当某一步执行结果与预期差异较大（比如搜索 API 返回空结果）时，Planner 不重新生成全部计划，而是仅修改后续步骤，保留已完成的步骤。这避免了从头计算的成本。
2. 状态快照：每完成一步，我就将当前状态（包括已完成步骤、中间结果、记忆缓存）序列化并存储。如果之后某一步失败且无法修复，我可以回滚到最近可用的快照，而不是完全从头开始。
3. 最大步数与超时：我会设置一个全局步数上限（例如 30），以及单步超时（例如 30s）。超过后，Agent 输出部分结果或向用户说“无法在有效时间内完成，请重新描述”。
4. 人类反馈接入：遇到无法自动决策的情况（如工具返回的信息不足以继续），Planner 可以生成一个提问给用户，待用户回复后再继续。这需要定义好暂停和恢复机制。
我参与的一个自动化报表 Agent 项目中，Planner/Executor 架构让我们将多步骤数据获取任务的完成率从 60% 提高到 85%，主要归功于局部重规划和状态快照。

**常见追问与回答：**
- **追问：如果 Planner 生成的计划本身就是错误的，怎么办？**
  答：我会在执行前增加一个 Plan 校验节点，让 LLM 自己检查计划是否合理、是否有循环依赖。另外，执行过程中如果发现连续 N 步没有实际进展（比如每步都调用了同一个工具但无新信息），则触发强制重规划。
- **追问：状态快照会占用很多内存吗？**
  答：我会控制快照粒度。只保存关键上下文（用户目标、已完成步骤摘要、当前 step 的输入输出），而不是所有原始对话。通常一个快照在几百 token 内，可以接受。

**项目联系：**
在我的“多工具数据分析 Agent”中，我用 LangGraph 实现了 Planner/Executor。Planner 基于 ReAct prompt，Executor 是一个简单的循环。我用状态字典维护所有中间变量，并将每一步结果存入长期记忆。当 SQL 执行出错时，Planner 自动修正 SQL 命令，最多重试 3 次。

**记忆关键词：**
Planner/Executor、局部重规划、状态快照、最大步数、超时控制、人类反馈兜底

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** Agent 中如何设计短期记忆和长期记忆的注入策略？
- **A：** 短期记忆保留最近 3 轮对话原文，保证指代还原。长期记忆只存结构化条目（画像、硬约束、关键事实），通过向量检索取 top-5，并按时近性加权重排。硬约束每轮强制携带。注入总量通常不超过上下文窗口的 30%。
- **一句话记忆法：** 短期 3 轮保连贯，长期 5 条压关键，硬约束每轮带。
- **复习标签：** #Agent #Memory

### 卡片 2
- **Q：** 如何保证 Agent 的 tool calling 输出格式正确？
- **A：** 三层保障：第一层 prompt 给出 Schema 和示例；第二层使用模型的结构化输出（如 Function Calling 或 Outlines）约束生成；第三层代码层用 pydantic 验证，解析失败时重试或回退。
- **一句话记忆法：** 提示结构 + 强制格式 + 验证重试，三层锁死工具调用。
- **复习标签：** #Agent #tool calling

### 卡片 3
- **Q：** Agent 的状态管理如何实现？错误恢复有哪些手段？
- **A：** 使用全局状态字典保存用户目标、已执行步骤、中间结果。每步执行后序列化快照。错误恢复包括：回滚到最近快照、局部重规划（修改后续步骤）、自动重试（设置重试次数和退避策略）、超过阈值后向用户请求澄清。
- **一句话记忆法：** 状态快照+局部重规划+重试，三招搞定 Agent 错误恢复。
- **复习标签：** #Agent #状态管理 #错误恢复

### 卡片 4
- **Q：** Planner/Executor 中如何避免无限循环？
- **A：** 设置最大步数（如 20），每步记录是否带来新信息。如果连续 3 步无进展（工具调用结果无变化或重复之前动作），则触发强制终止或局部重规划。也可用 LLM 自检：让模型评估当前进度并决定是否继续。
- **一句话记忆法：** 最大步数 + 无进展检测 + 自检，三驾马车防死循环。
- **复习标签：** #Agent #Planner #安全

### 卡片 5
- **Q：** Agent 记忆冲突更新时，如何处理置信度判断？
- **A：** 按来源优先级：用户显式说 > 用户确认 > 模型推断。时间戳：新的优先。如果冲突且置信度相同，两条都保留，检索时让 LLM 结合当前 query 自行判断或向用户确认。
- **一句话记忆法：** 显式 > 确认 > 推断，时间新优先，同级不删留。
- **复习标签：** #Agent #Memory #冲突仲裁

## 4. 今日面试专项训练

### 4.1 大模型理论追问题：Agent 的规划能力——ReAct 与 Tree-of-Thoughts 的区别及适用场景

- **问题：** 请比较 ReAct 和 Tree-of-Thoughts（ToT）两种规划框架，说明它们各自的核心思想、优缺点以及适用场景。

- **面试口头版参考答案：**
  ReAct 是一种交替生成推理轨迹和动作的框架：模型每步先“思考”（Thought），再“行动”（Action），然后观察结果。它的优点是简洁、效果稳定，适合大多数单路径决策任务。Tree-of-Thoughts 则是在推理过程中显式地探索多条可能的思维路径，在每个步骤考虑多个分支，并通过评估函数选择最有希望的一条继续。它的优点是更适合复杂、多步推理（如数学问题、创意写作），但计算成本高，因为需要同时生成和评估多个分支。在实际工程中，我会优先使用 ReAct，只有任务经常需要回溯或需要对比多种方案时才考虑 ToT。

- **深入理解版参考答案：**
  ReAct（Reasoning + Acting）将思维链（CoT）与工具调用结合。模型在每一步输出一个“thought”来解释当前状态，然后通过“action”调用工具，工具返回结果后作为观察（Observation），再进入下一步。所有步骤都在同一个对话上下文中串联。优点是实现简单，prompt 只需定义好格式，模型自动遵循。缺点是它是线性展开的，如果某一步决策错误，后续都会受到影响，而且没有显式的分支探索。适合工具使用明确的场景，如问答、数据查询。
  Tree-of-Thoughts（ToT）受人类“树状思考”启发，在每一步生成多个可能的下一步“thought”，并为每个分支打分，然后选择评分最高的分支继续探索。如果某条路径遇到死胡同，可以回溯到上一个分岔点选择另一条路。ToT 通常需要 LLM 作为评估器（想想每个分支的问题是否合理、能否解决问题）。优点是可以避免过早承诺，适合需要多步推理和探索的任务，如规划旅行、解决谜题。缺点是 token 消耗巨大（每个分支都需要完整生成和评估），且评估器的质量直接影响效果。
  实际选型：工程上我会根据任务的确定性来决定。如果目标明确（如“查询公司 Q3 财报”），用 ReAct 一步到位。如果任务开放性高（如“设计一个新产品方案”），我会先用 ReAct 框架收集初步信息，当发现信息不足以决策时，再切换到 ToT 生成多个方案并评估。另外，LangGraph 等工具支持混合使用：主循环用 ReAct，遇到复杂节点时调用子图实现 ToT。

- **可能追问与回答：**
  - **追问1：如何在工程上降低 ToT 的高成本？**
    答：可以限制分支数（比如最多 3 个分支），只在首次决策或关键节点使用 ToT，其余步骤用贪心或 ReAct。还可以用更小的模型作为评估器，或者只评估部分分支。
  - **追问2：如果模型在 ReAct 中产生了错误的 Thought，你怎么发现并纠正？**
    答：我会在每一步执行 action 后，对比 expected outcome 和 actual outcome。如果差距大，说明前面 Thought 可能错了。此时可以注入一条“反思”步骤，让模型重新解释差距并修正后续计划。本质上就是 ReAct 的自适应变种。
