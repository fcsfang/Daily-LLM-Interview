### 题目 1：ReAct、Plan-and-Execute、Reflection 三种 Agent 设计范式的核心区别是什么？实际项目中该如何选型？

**可信度：** 中

**考察点：** Agent 设计范式、推理模式、工程选型

**面试口头版回答：**
“这三种范式解决的核心问题不同。ReAct 是边思考边行动的循环模式，典型流程是 Thought → Action → Observation → Thought，适合需要即时反馈的单步任务，比如客服问答，每一步都可以根据环境观察调整下一步。Plan-and-Execute 是先规划再执行的两阶段模式，先把任务拆解成子步骤，生成一个执行计划，然后按顺序执行。适合复杂多步任务，比如订机票，需要先查航班、再比价、最后下单。Reflection 则是在执行后增加了一个回顾反思步骤，让 Agent 检查自己的输出是否合理，适合对结果质量要求高的场景，比如代码生成后检查语法错误。我的选型原则是：任务简单直接选 ReAct；任务复杂度高、步骤可控选 Plan-and-Execute；对结果有准确度要求，比如金融风控报告生成，我会加上 Reflection 机制。”

**深入理解版回答：**
这三种范式本质上是对 Agent 执行过程中“思考”和“行动”之间关系的不同设计。

**ReAct（Reasoning + Acting）模式**：这是最基础的 Agent 模式，由 Yao et al. 在 2022 年提出。核心是交替执行推理和行动。每次 Agent 先思考当前状态（Thought），决定下一步要做什么（Action），执行工具调用得到反馈（Observation），然后基于反馈进入下一轮思考。这种模式最大的优势是灵活，能根据实时反馈调整策略。但缺点是可能陷入死循环，比如工具调用一直失败，Agent 会不停重试。工程上我通常会设置最大迭代轮次（如 10 轮），超时则强制结束并返回当前结果。

**Plan-and-Execute 模式**：先由 LLM 生成一个完整任务计划（Plan），比如去旅游的规划是“查机票 → 查酒店 → 查天气 → 生成行程表”，然后由执行器按顺序执行每个子任务。这种模式的好处是可解释性强，用户能看到规划；执行过程稳定，子任务之间依赖关系明确。但缺点是当环境变化时，原计划可能失效。比如查机票时发现目标日期已无航班，后续的酒店和天气查询就失去意义。工程上我会引入“执行中重新规划”机制，当某个子任务失败或返回异常时，触发 LLM 重新生成后续计划。

**Reflection 模式**：在执行结果生成后，增加一个自我审查步骤。比如 Agent 生成了代码后，再让 LLM 检查代码是否有 bug、是否遵循了最佳实践。这种模式适合高质量要求的场景。但代价是增加了推理轮次和 Token 消耗。工程上我会结合自动测试，比如对生成的代码先跑单测再输出。

**选型建议**：
- 客服问答系统：ReAct，因为问题简单，需要快速响应
- 电商购物助手：Plan-and-Execute，因为流程固定（选品 → 加购 → 支付）
- 代码生成工具：ReAct + Reflection，先生成代码，再自动检查
- 金融报告生成：Plan-and-Execute + Reflection，先规划章节，生成后再检查事实性

**常见追问与回答：**
**追问1：ReAct 模式中出现死循环怎么办？**
答：设置最大迭代轮次（比如 10 步），超时后触发 fallback 策略，比如返回“当前无法完成请求，请简化指令”。还可以引入观察去重，如果连续三轮观察结果相同，说明 Agent 在兜圈子，直接终止。

**追问2：你怎么评估 Plan-and-Execute 的规划质量？**
答：离线评测可以用规划完成率，即执行完所有子任务后是否达到了预期目标。在线评测用用户满意度，比如用户是否修改了执行结果。同时可以监控“重新规划”的频率，该指标过高说明初始规划质量差，需要优化 LLM 的规划 Prompt。

**项目联系：**
我的个人项目是设计一个智能客服 Agent。最初用的是纯 ReAct 模式，结果发现用户问“我的订单为什么还没到”这种复杂问题时，Agent 需要反复查物流状态、查仓库地址、查客服排班，经常超时。后来我改成 Plan-and-Execute：先让 LLM 规划“查订单状态 → 查物流轨迹 → 查客服处理时间表 → 汇总生成回复”，然后顺序执行，效果好了很多，超时率从 30% 降到了 5%。

**可信度：** 中

**考察点：** Agent 架构、记忆管理、工程实践

**面试口头版回答：**
“Agent 的记忆分两大块：短期记忆和长期记忆。短期记忆其实就是 LLM 的上下文窗口，保存当前对话轮次中的交互记录，比如用户的这轮提问、工具调用结果。但因为窗口大小有限，如果对话很长就需要压缩或截断。长期记忆则持久化存储用户的历史偏好、关键事实，比如用户常用的收货地址，一般会存入向量数据库或传统数据库。在实际开发中，我遇到的主要问题是记忆膨胀。比如用户连续问 20 个问题，短期记忆可能爆掉。我的做法是引入 Memory Summarization，对历史对话定期做摘要，把原始的 10 轮对话压缩成 200 字总结，再放入上下文。同时用 RAG 检索长期记忆，只在需要时把相关的用户历史偏好取出来。这样就解决了长对话的上下文管理问题。”

**深入理解版回答：**
Agent 记忆机制的设计直接决定了系统能否处理复杂、长时间的任务。核心记忆架构通常包括三个层级：

**第1层 - 短期记忆（Working Memory）：** 本质是 LLM 的上下文窗口。存储当前任务相关的信息，包括用户输入、Agent 的推理轨迹（Thought-Action-Observation 循环）、中间结果。在工程实现上，我通常用一个滑动窗口来管理，比如保留最近 10 轮对话。当对话超过窗口时，我采用“摘要压缩”策略：用 LLM 对最旧的 5 轮对话生成 200 字总结，替代对话原文放入上下文。这种方式的优势是保留关键上下文，而不是粗暴截断。

**第2层 - 长期记忆（Episodic/Semantic Memory）：** 跨会话持久化存储。我把它分两类：一类是“情景记忆”，比如用户上次对话中提到的“我下周要去上海出差”；另一类是“语义记忆”，比如用户固定的偏好“回复不要太啰嗦，尽量简洁”。工程上，我使用向量数据库（如 Milvus）存储情景记忆，检索时根据当前查询的 Embedding 召回相关历史片段。语义记忆则用键值对数据库（如 Redis），以用户 ID 为 key，存储结构化的用户画像。

**第3层 - 记忆压缩与遗忘：** 这是工程落地的关键。记忆如果不管理，存储成本和推理成本都会爆炸。我使用以下方法：1）**时间衰减**：超过 7 天的记忆自动归档到冷存储；2）**重要性评分**：让 LLM 对每段记忆打 1-5 分，低于 3 分的优先压缩。3）**去重**：同一用户反复提及的相同信息（如邮箱地址），只保留最新版本。

**常见坑点：**
1. **记忆不冲突**：短期记忆（LLM 上下文）和长期记忆（数据库）内容可能矛盾，比如用户今天说“不再用这个地址了”，但长期记忆还保留着旧地址。我加了冲突检测，当检索到长期记忆中的地址时，会让 Agent 先向用户确认是否仍有效。
2. **记忆检索噪声**：查询的 Embedding 可能召回大量无关记忆，干扰 LLM。我设定了相关性阈值，低于 0.7 分的记忆不放入上下文，同时用 Rerank 排序。

**常见追问与回答：**
**追问1：短期记忆爆了怎么办？除了摘要压缩还有没有其他方法？**
答：还可以用“关键信息提炼”，不是压缩整轮对话，而是只提取当前对话中的实体和关系。比如用户问“帮我查一下北京的天气，然后推荐酒店”，我只提炼“北京、天气、酒店推荐”三个关键信息放入上下文，比摘要更节省 Token。

**追问2：长期记忆的错误累积怎么处理？**
答：我会定期对长期记忆做一致性校验。比如每 7 天用 LLM 扫描一遍用户记忆，如果发现矛盾信息，就让 Agent 主动提问“我发现您之前想用地址A，但上次您好像提到了地址B，请问哪个是当前有效的？”并提供选项让用户确认。

**项目联系：**
在我做的个人知识助手 Agent 中，短期记忆用 Redis 存储对话历史，设置 TTL 为 30 分钟；长期记忆用 MySQL 存结构化信息（用户感兴趣的话题标签）和向量数据库存语义片段。每次新对话开始时，会先检索长期记忆中的用户画像，然后拼接成系统 Prompt 的前缀。我测试过，加入记忆模块后，用户对“它记得我喜欢什么”的反馈特别好，对话完成率提升了 40%。

**可信度：** 低（来源为面经和需验证的博客）

**考察点：** LoRA 微调、Function Calling、错误修复、Agent 可靠性

**面试口头版回答：**
“模型输出错误的函数名，本质上是一个生成质量问题。我的处理分三个层面：第一，在微调阶段优化训练数据。构造 SFT 数据时，不仅保留正确的 Action 调用，还要加入典型的 Bad Case，比如用户问‘帮我查股价’，训练数据中的 Action 必须是 `get_stock_price` 而不是 `check_price`。如果模型还是输出了错误的函数名，第二层在工程上做格式校验，用正则或 Schema 验证工具调用的 JSON 结构，如果不符合规范，就触发重试，让模型重新生成。第三层是在 Agent 执行层面做容错。如果重试后仍然失败，我就启动 fallback 策略，比如将错误信息作为 Observation 反馈给模型，并提示‘只能使用以下可用工具：工具A、工具B、工具C’，让模型基于反馈修正。如果还是不行，最后的兜底是转接人工或返回默认答案。”

**深入理解版回答：**
这个问题触及了 LoRA 微调在 Agent 场景下的核心挑战：微调后的模型可能丢失或混淆了 Function Calling 的知识，导致输出了不存在或拼写错误的函数名。

**根本原因分析：**
1. **训练数据不充分**：LoRA 微调只更新低秩矩阵，如果训练数据中 Action 调用的分布不均衡，某些不常用的函数可能没有被充分学习。
2. **过拟合导致泛化下降**：LoRA 在小数据集上容易过拟合，模型可能记住了训练集中的几种调用模式，但在新的、相似的查询上会出错。
3. **参数设置不当**：rank 和 alpha 如果设置不当，会导致微调对原始模型的改变过大或过小，影响工具调用能力。

**工程解决方案（三层防线）：**

**第一层：数据增强（事前预防）**
- 在训练数据中，对所有可用的工具函数名进行“同义词扩增”。比如 `get_weather` 扩增为 `fetch_weather`、`query_weather`，让模型知道这些变体都是不合规的。
- 加入“负样本数据”：故意在输入中提供错误的工具调用示例，让模型学会拒绝错误调用。比如输入“帮我用 `check_weather()` 查天气”，答案应是“这个工具不存在，请使用 `get_weather()`”。
- 使用“工具列表注入”：在训练数据的 system prompt 中明确列出所有可用工具及其精确描述，让模型学会正确映射。

**第二层：工程校验（事中拦截）**
- 使用 JSON Schema 验证：模型输出的 Function Calling 必须是合法的 JSON 结构，包含 `name` 和 `arguments` 字段。
- 对 `name` 字段做白名单匹配：如果不在已注册的函数列表中，直接 reject 并返回错误。
- 对 `arguments` 做类型校验：比如参数类型是 integer，模型输出却是 "Hello"，就报错。

**第三层：Agent 容错（事后恢复）**
- 重试机制：一般最多重试 2 次，每次重试时在 Prompt 中增加更明确的约束，比如“你只能从以下列表中选择函数名：函数A、函数B”。
- 动态函数注入：如果重试失败，我可以动态地在下一轮 Prompt 中注入更多示例，比如“用户查询天气的正确用法是：`get_weather(location="北京")`”。
- fallback 策略：如果所有重试都失败，就向用户返回“当前无法处理该请求，请重试或联系客服”。

**LoRA 特有的优化：**
- 可以尝试在 LoRA 微调中冻结 70% 的底层，只微调后面 30% 的层，因为这些层更与工具调用相关。
- 训练时加入“格式损失”：对模型输出的 JSON 格式进行正则约束，让模型学会只输出合法 JSON。

**常见追问与回答：**
**追问1：如果模型输出了合法的函数名，但参数是错的，比如调用了 `get_weather(city="北京123")`，怎么处理？**
答：这属于参数校验范畴。我会在函数调用前做参数预置检查。比如对 `city` 参数，用正则校验只能包含中文字符；对数字参数，校验是否在有效范围内。校验失败后，同样触发重试机制，并在 Prompt 中提示具体错误信息。

**追问2：重试次数太多会不会让用户体验变差？**
答：确实会。我一般控制在 2 次以内，同时采用“渐进式超时”：第一次重试正常响应，第二次重试时启用快速模式，限制 max_tokens 为 50，只让模型生成函数名，不生成冗长的思考过程。如果 2 次后仍失败，立即 fallback。

**项目联系：**
我在做智能客服 Agent 时，最初用 LoRA 微调了一个 7B 模型，发现它对“查询账户余额”这个操作，有时输出 `check_balance()`，有时输出 `query_balance()`，甚至 `get_account_balance()`。后来我按照上面的方法，在训练数据中加入了大量负样本，并在工程层加了白名单校验和 2 次重试。上线后，工具调用准确率从 82% 提升到了 96%。

**题目 1：在工程实践中，为什么有时候选择「手搓」Agent，而不是直接用成熟框架（如 LangChain、CrewAI）？**

**面试口头版回答：**
“我觉得主要看场景和需求。框架的好处是开箱即用，快速搭建原型，比如用 LangChain 的 AgentExecutor 几行代码就能跑一个 ReAct Agent。但在生产中我经常选择手搓，原因有三个。第一是控制力，框架的 Prompt 是黑盒的，很难 Debug。我手搓时可以精确控制每一轮 Prompt 的结构，包括 system prompt 的格式、工具描述的组织方式。第二是性能，框架里有很多无关的抽象层，比如 LangChain 的 Chain 和 Callback，在低延迟场景下这些额外的函数调用开销无法接受。第三是容错，框架的错误处理往往是通用的，比如超时就返回空，但我需要自定义的 fallback 策略，比如重试时动态注入错误信息。所以我的原则是：原型用框架验证想法，线上用手搓优化性能和可维护性。”

**题目 2：复杂任务怎么做的任务拆分？为什么要拆分？效果如何提升？**

**面试口头版回答：**
“任务拆分的核心原因是 LLM 的单次推理能力是有限的，直接丢给它一个复杂任务，如写一篇包含调研、分析、结论的长报告，模型很容易遗漏细节或逻辑混乱。我的拆分方法分两步：第一步，让 LLM 分析任务，输出一个任务依赖图，比如‘查资料 → 数据整理 → 生成结论’。第二步，按依赖关系逐一执行，每个子任务有独立的上下文和工具集合。效果提升方面，我测试过，对一个 20 步的复杂任务，拆分后完成率从 50% 提升到了 85%。具体优化点是：为每个子任务设置明确的目标和输出格式，比如子任务是‘查资料’，输出必须是 JSON 格式的摘要列表。这样做让模型的目标更清晰，减少了来回推理的 Token 消耗。”

**可转化面试题与参考答案：**

**题目 1：Agent+RAG 系统中，Agent 如何决定何时调用 RAG？**

**面试口头版回答：**
“这是一个很关键的问题。简单粗暴的方式是：用户每次提问都强制调用 RAG，但这样成本高且不灵活。我的策略是让 Agent 自己决策。分三步走：首先，在 Prompt 里给 Agent 一个明确规则，比如‘如果你认为回答需要外部知识或最新信息，就调用 RAG 工具；如果是闲聊或常识问答，可以直接回答’。其次，给 Agent 提供 RAG 检索结果的元数据，比如检索结果的相关性分数，如果所有检索结果的分均低于阈值（如 0.6），Agent 可以选择‘我不确定，建议用户澄清问题’。最后，如果 Agent 对自己的回答置信度低（比如在输出中加入了‘可能’、‘大概是’这些词汇），就触发二次验证，即调用 RAG 或让用户确认。这样既控制了成本，又保证准确性。”

**题目 2：Agent+RAG 系统中如何处理检索失败的情况？**

**面试口头版回答：**
“检索失败分两种情况：一种是检索不到任何结果，比如用户问的知识库中根本没有；另一种是检索结果不相关。我的处理方式是分层兜底。第一层，根据检索结果数量，如果为 0，就触发查询重写：让 LLM 将原问题改写为更宽泛的表述再检索一次。第二层，如果仍然为 0，就用模型自身知识（假设模型有相关知识）直接生成答案，但要加上‘这是我根据已有知识回答的，建议您查阅官方资料’的免责声明。第三层，如果模型自身也不确定，就返回‘抱歉，我无法找到相关信息，请换个问法试试’。同时，所有的检索失败都会被记录到监控日志，用来优化知识库的建设，比如发现某类问题频繁检索失败，就考虑补充相应的文档。”

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** Agent 中的 Memory 模块，你是如何做短期记忆和长期记忆的区分的？各自用什么存储？
- **A：** 短期记忆（Working Memory）处理当前对话轮次，本质是 LLM 上下文窗口，我用 Redis 缓存，TTL 设为 30 分钟，对话过长时用 LLM 生成摘要压缩。长期记忆（Episodic Memory）跨会话持久存储用户历史，我用向量数据库（如 Milvus）存语义片段，Redis 存结构化偏好（键值对）。检索时先查长期记忆，取相关性 Top-3 放入上下文，并设 0.7 为相关性阈值过滤噪声。
- **一句话记忆法：** “短期记对话，长期存偏好；缓存用 Redis，向量进 Milvus。”
- **复习标签：** #Agent #Memory #工程实践

### 卡片 2
- **Q：** Agent 工具调用失败时，你如何实现错误恢复？
- **A：** 我采用分层兜底策略。第一层校验：对模型输出的 JSON 做格式和函数名白名单校验，失败则重试（最多 2 次）。第二层容错：重试后仍失败，将错误信息作为 Observation 返回，并在 Prompt 中动态注入可用工具列表。第三层 fallback：返回“无法处理该请求”或转人工。同时记录失败的调用日志用于调优。
- **一句话记忆法：** “校验 -> 重试 -> 动态注入 -> 转人工，层层兜底不丢脸。”
- **复习标签：** #Agent #错误恢复 #工具调用

### 卡片 3
- **Q：** AdaLoRA 与普通 LoRA 有什么区别？在 Agent 场景下优势是什么？
- **A：** 普通 LoRA 为所有层分配相同的 rank，但不同层对任务的重要性不同。AdaLoRA 通过引入 SVD 分解，在训练中自适应地为更重要的层分配更高的 rank，不重要的层分配更低的 rank。在 Agent 场景下，Adapter 需要同时适配推理（工具调用相关的层）和生成（语言建模相关的层），重要性不同。AdaLoRA 能自动优化参数分配，比普通 LoRA 提高 5-10% 的工具调用准确率。缺点是训练计算量略高。
- **一句话记忆法：** “普通 LoRA 一刀切，AdaLoRA 看菜下饭。”
- **复习标签：** #LoRA #AdaLoRA #Agent #参数高效微调

### 卡片 4
- **Q：** 如何评估一个 Agent 系统的效果？给出至少三个关键指标。
- **A：** 我设计三个核心指标：1）**任务完成率**：用户在多少比例的问询中一次性得到正确解答。2）**工具调用成功率**：模型输出的 Function Calling 中，语法正确且函数名在白名单中的比例。3）**平均失败修复成本**：工具调用失败后，重试到成功需要的平均额外轮次和 Token 数。此外，我还会监控“人类介入率”，即需要人工打 tag 的对话占比。这三个维度分别衡量了 Agent 的完成度、可靠性和效率。
- **一句话记忆法：** “完成率、调用成功、修复成本，三驾马车拉 Agent。”
- **复习标签：** #Agent #评测 #指标

### 卡片 5
- **Q：** 在什么情况下你会选择“手搓”Agent 而不是直接用 LangChain 框架？
- **A：** 三种情况我会手搓：第一，**对性能有极致要求**，比如延迟必须小于 200ms，框架的抽象层和回调会增加 10-30ms 的 overhead。第二，**需要精细控制 Prompt**，框架的 Prompt 模板是黑盒，无法逐行 Debug。我手搓时可以精确控制每轮 Prompt 的格式、思维链的触发时机。第三，**需要定制化容错**，框架的错误处理是通用的，但我可能需要自定义重试策略、动态注入、转人工等逻辑。所以我的原则是：原型用框架快速验证，线上正式跑用手搓优化。
- **一句话记忆法：** “原型用框架，线上用手搓；性能控制靠手艺，框架只管快。”
- **复习标签：** #Agent #手搓vs框架 #工程选型

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 在 Agent 的反思机制（Reflection）中，如何确保反思本身的正确性？因为反思也是 LLM 生成的，如果反思错了，岂不是错上加错？（提示：从 LLM-as-Judge 的局限性和 Agent 的自我修复能力两个角度展开）

**面试口头版参考答案：**
“这个问题问得很好，确实存在反思出错的风险。我的应对方案是分层验证。首先，我让反思 Agent 只做‘模式化检查’，比如检查生成的 JSON 是否符合 Schema、引用是否来自检索结果、是否包含矛盾信息。不做复杂的逻辑推理，因为推理容易出错。其次，我把反思结果做正反对比，把反思结论作为 Observation 输入回主 Agent，让主 Agent 判断是否采纳这个反思，相当于加了一层仲裁。最后，如果反思本身不通过（比如反思 Agent 输出格式错误），就跳过反思步骤，直接输出当前结果。所以反思不是强制的，是一个可选的、有监督的质量门禁。”

**深入理解版参考答案：**
反思机制的核心是“让模型检查自己的输出”，但这个检查本身也可能出错。问题的根因在于 LLM-as-Judge 存在偏差：位置偏差（偏好开头和结尾的内容）、自我放大偏差（倾向于认同自己生成的内容）、以及长度偏差（偏好更长的回复）。这些偏差会让反思 Agent 做出错误的评判。

我设计的反思机制是分层、分角色的：

**第一层：格式与约束校验**
- 用正则和 JSON Schema 做自动化检查，比如模型输出的引用是否在检索结果中、Action 的函数名是否在白名单中。这部分不用 LLM，避免偏差。
- 如果格式校验不通过，直接否决输出并触发重试。

**第二层：模式化反思**
- 让反思 Agent 只做三类检查：1）是否存在自相矛盾（比如先说“用户住在北京”，后说“用户上海的地址”）。2）输出是否与检索结果冲突（比如检索结果是“明天晴”，模型却说“明天有雨”）。3）是否输出不安全的语言。
- 这三类检查都有明确的规则，反思 Agent 只是“模板匹配”，而不是自由推理。

**第三层：仲裁与反馈回路**
- 如果反思 Agent 发现冲突，它不会直接修改输出，而是输出一条 Observation，比如“存在冲突：用户地址在 A 和 B 之间矛盾，请确认”。
- 主 Agent 收到这个 Observation 后，自行决定是否修改输出，或向用户提问确认。
- 如果反思 Agent 自己的输出不符合格式规范（比如输出乱码），主 Agent 会忽略反思结果，保持原输出。

**局限性与改进方向：**
- 这种方法仍然无法检查深层的逻辑错误，比如模型在推理中偷偷改了前提假设，格式校验发现不了。
- 我正在尝试引入“多角色反思”，即用 Agent A 和 Agent B 互为反思者，交叉校验。如果 A 认为 B 的反思错了，会产生第三轮对话来仲裁，虽然增加了 Token 消耗，但能提升准确度。

**可能追问与简短回答：**

**追问 1：反思机制会增加多少延迟和成本？怎么权衡？**
答：实测每轮反思增加约 0.5-1 秒延迟和 200-300 Token 成本。我的权衡策略是：只在关键输出（如金融报告、医疗建议）时启用反思；对低风险的对话（如天气查询）关闭反思，用格式校验兜底。

**追问 2：如果反思 Agent 发现错误，但在重试后生成的输出仍然是错的，怎么办？**
答：设置最大反思次数，一般是 2 次。如果两次反思后 LLM 都输出错误答案，就触发人工接管（在金融场景）或返回默认答案并记录 Bad Case（在客服场景）。同时记录反思日志，用于后续的模型微调和 Prompt 优化。
