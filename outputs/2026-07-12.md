## 1. 今日面试真题精讲

### 题目 1：你在做 RAG 系统时，如果发现召回结果质量差，导致模型回答“我不知道”或者答非所问，你会怎么排查？

**来源依据：**
- 腾讯云开发者社区 - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 可信度中等
- 今日资料[18]摘录中详细论述了“内容缺失”的排查思路。

**可信度：** 中

**考察点：**
RAG 链路排查、召回优化、混合检索、重排序。

**面试口头版回答：**
这个问题我一般分三步排查。第一步查切块策略，不能固定500字一刀切，得用语义切块，把意思完整的句子放在一起，避免关键信息被截断。第二步查召回方式模型，单靠密集向量检索会跑偏，比如搜“苹果手机”可能召回“红富士苹果”，所以我一定会加 BM25 稀疏检索做双路召回，提升召回多样性。第三步加重排序，第一轮捞出来的几十条，再用 Cross-Encoder 模型精排，只把置信度最高的几条塞进 LLM 的上下文。这三步下来基本能解决八成的内容缺失问题。如果还不行，我会考虑升级到更灵活的切块策略，或者加 Query 改写。

**深入理解版回答：**
RAG 的召回质量差是多因一果的。面试官想看到你有系统性的排查框架，而不是凭感觉调参数。我的排查流程是：第一步诊断切块。常见的“500字固定切”会把一个完整的逻辑段砍成两半，导致召回时只拿到半句话，LLM 无法理解。所以我会先用“语义切块”，比如按段落后换行或按自然段边界切，必要时用小模型判断语义完整性。第二步诊断检索。很多项目只用向量检索，但向量检索擅长语义相似，不擅长精确关键词匹配。所以我会采用“混合检索”：向量检索 + BM25 关键词检索。BM25 对精确名词（比如产品型号、术语）效果出奇好。第三步是重排序兜底。混合检索会返回几百条候选，但传给 LLM 时必须筛出最相关的 4-5 条。我会用 BGE-Reranker 这类交叉编码器模型做精排，开销不高但召回质量提升很明显。还有一个常见坑是 Query 本身写得模糊，比如用户直接说“看下最近的数据”。这时候需要加 Query 改写模块，把模糊意图转为精确查找条件。整体来说，RAG 链路是 Retriever → Generator 的耦合，Recaller 决定 Generator 能看到什么，所以排查一定要从检索端开始。

**常见追问与回答：**
1. **追问：** 你提到的混合检索具体怎么实现？怎么融合分数？
   **回答：** 向量检索和 BM25 分数不在同一量纲，没法直接加。我通常用倒数排名融合法，即对两种方法的排序分别计算倒数排名，然后加权求和。也可以用一个可学习的融合层，根据历史数据训练权重。

2. **追问：** 重排序后如果还是不行呢？
   **回答：** 那我会考虑在多轮对话中动态调整切块粒度。比如第一次切块过大导致召回模糊，下一次测时缩小切块，测一测效果。另外也会加 Query 扩展，比如用 LLM 把用户的问题展开成多个子问题，分别召回后再聚合。

**项目联系：**
在我搭建的客服知识库 RAG 项目中，切块用了 LangChain 的 RecursiveCharacterTextSplitter，按段落换行做语义切块；召回用了 Faiss（向量）+ ElasticSearch（BM25）双路融合；重排序用了 BAAI/bge-reranker-v2-m3。实测后 FAQ 类问题的召回命中率从 68% 提升到 89%。

**记忆关键词：**
语义切块、混合检索、BM25、Cross-Encoder 重排序、Query 改写、倒数排名融合。

### 题目 2：假如让你设计一个“RAG + Agent”的融合系统，你会怎么设计？什么时候该用 RAG，什么时候让 Agent 自己推理？

**来源依据：**
- 腾讯云开发者社区 - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 可信度中等
- 今日资料[2] 中的“在 RAG+知识图谱的 Agent 系统中,知识图谱更新的机制是怎样的？” - 面经或社媒 - 需进一步验证

**可信度：** 中

**考察点：**
跨模块系统设计、Agent 规划能力、RAG 定位、工具调用。

**面试口头版回答：**
我的设计原则是“RAG 负责知识供给，Agent 负责逻辑编排”。Agent 首先会根据用户意图做任务拆解和规划。如果任务需要查找外部知识（比如查政策、查产品手册），Agent 会调用 RAG 工具去检索知识库。但如果任务是纯逻辑推理（比如计算、代码调试、步骤规划），Agent 会直接推理，不经过检索层。举个例子，用户问“3 的 5 次方是多少”，Agent 直接算，不需要 RAG；但用户问“这份合同里哪个条款违反了安全规范”，Agent 会先调用 RAG 召回合同和相关规范再做判断。还有一个重点是离线定时更新知识库，Agent 只用调用 RAG 接口，不需要每次加载全量数据，成本低、可溯源。

**深入理解版回答：**
设计 RAG + Agent 的关键是确定边界，不要让 Agent 什么情况都查知识库。我会把系统分成三层：规划层（Planning）、执行层（Execution）、知识层（Knowledge）。规划层用 LLM 做意图识别和子任务分解，输出一个 DAG。执行层调用具体工具，其中就包括 RAG 工具。知识层维护向量库和图数据库。分层的好处是各层独立迭代：RAG 只关注检索质量；Agent 只关注规划质量。RAG 是一个工具，Agent 是一个编排器。在实现上我用 LangGraph 做状态机，每个节点代表一个子任务，边表示依赖。如果节点需要查知识，就调用 RAG 工具。一个常见的坑是：Agent 在检索阶段输出很啰嗦的思考过程，浪费 Token。我会在 prompt 里约束 Agent 只在需要检索时才输出工具调用动作，其他情况直接输出答案。另一个坑是：Agent 可能过度依赖 RAG，明明自己推理就能解决的小问题也要去查一遍，导致延迟飙升。所以我会对每个子任务做一次“是否需要知识支撑”的判断，比如用一个小模型判断任务类型，减少不必要的 RAG 调用。还有一个进阶设计：RAG + 知识图谱。RAG 处理非结构化文本，知识图谱存储实体关系。Agent 可以先用 RAG 召回候选文档，再从中抽取实体匹配图谱做交叉验证，比如医生问药方案的 Agent，先 RAG 查文献，再图谱查禁忌关系，最后综合输出。

**常见追问与回答：**
1. **追问：** 如果 Agent 必须查 RAG 才能继续，但 RAG 很慢，你怎么设超时？
   **回答：** 我会在 LangGraph 中给 RAG 工具设硬超时（比如 5 秒），超时后返回空或降级提示。如果超时频繁，说明 RAG 部署可能存在瓶颈，需要做预计算缓存或异步召回。

2. **追问：** 知识图谱更新机制怎么做？实时性怎么保证？
   **回答：** 知识图谱分两层：静态知识（比如产品参数）和动态知识（比如库存、日志）。静态知识离线定时更新；动态知识通过消息队列实时写入。Agent 在调用之前先读一个“知识有效时间戳”，如果发现数据过时就重新拉取。

**项目联系：**
我正在做的合规审校 Agent 正是这个模式。用户上传文档后，Agent 拆解成“查法规”“查案例”“写结论”三个子任务。“查法规”子任务调用 RAG 工具从法规库找回相关条目；“查案例”子任务调用图谱检索，读取实体关系；最后执行层汇总后输出结论。延迟优化方面，RAG 调用加了缓存，命中率达到 65%，整体响应时间从 7 秒压到 2.5 秒。

**记忆关键词：**
RAG + Agent 分层、Planning Layer、Tool Calling、LangGraph、知识图谱交叉验证、超时降级。

### 题目 3：你如何设计 SFT 的训练数据？请结合一个实际的 Bad Case 说明数据构造逻辑。

**来源依据：**
- 今日资料[2] 中“微调时的训练数据是怎么构建的？如何保证样本多样性和质量？” - 需进一步验证
- 今日资料[9] 中“Fine-tuning vs RAG 决策框架” - 技术博客 - 可信度中等

**可信度：** 需进一步验证，但为高频经典题，补充适用。

**考察点：**
微调数据工程、Bad case 驱动、样本质量、SFT 流程。

**面试口头版回答：**
我采用 Bad case 驱动的方式构造数据。第一步从生产日志中筛选出模型回答不符合预期的 case，比如答非所问、幻觉、句式单一。第二步对这组 case 聚类，找出共性根因，比如某个知识点完全没学过、输出格式严格错误。第三步针对每个根因人工标注高质量的“正确答案”，将其转为训练样本，每一条都是一个（instruction + negative response）+ （correct response）。最后加入约 20% 的高质量通用指令（比如公开的 Dolly 数据集）做多样性增强。在清洗时特别注意去重和去噪，如果一条数据中用户 query 和 answer 高度重合，说明是废话，必须删掉。

**深入理解版回答：**
SFT 数据构造的核心矛盾是质量 vs 规模。质量优先，宁可样本少也要每条都对。我的流程是：采集 → 聚类 → 重构 → 验证。首先是采集。从用户真实对话和生产日志里抽取出 LLM 回答被用户点踩、客服跟进、或自动护栏拦截的 case。这些是“Bad Case 金矿”。其次是聚类。用 embedding 把这些 case 的 query 聚成几大类，比如：“知识遗漏类”“指令跟随失败类”“输出格式错误类”。每一类找 50-100 条做代表。接着是重构。针对每一类，标注人员书写正确答案，格式是（system prompt, user query, correct assistant response）。注意这里要写完整对话，不只是一问一答，因为多轮上下文很重要。最后是验证。用这组数据微调后，重新跑一遍原来的 Bad Case 测试集，判断是否修复。如果仅修复某类 case 但又导致别的 case 退化，说明数据有冲突，需要调整样本配比。另一个容易犯的错是样本比例失衡。例如 80% 的 SFT 数据是“写代码”，20% 是“问答”，微调后模型可能变成代码生成器。所以我会控制每个类别的样本数大致持平。还有一个技巧是利用模型自动做数据增强。让大模型针对已有 case 生成相似但不同的 query，但必须人工审核，因为大模型可能引入幻觉。

**常见追问与回答：**
1. **追问：** 你说的这些 Bad Case 都是从生产日志里取，但日志里的 answer 都是模型生成的，怎么判断它好还是坏？
   **回答：** 那取决于哪个日志。通常我们会建立两层判断：第一层是自动信号，比如用户点了踩或客服介入。第二层是人工抽样审计，最后建立一个 Bad Case 标注池。

2. **追问：** 如果你要构造 DPO 的偏好对，怎么做？
   **回答：** 思路类似但更精细。对于同一条 query，收集模型生成的两个版本（比如不同 temperature 或不同 prompt），再由标注员标出哪个更好。目标是让模型学会区分好与更好，而不仅仅是正确与错误。相关回答可参考今日资料[2] 中 DPO 相关问题，但进一步验证后建议先做批量实验。

**项目联系：**
在我微调医疗问答模型时，从生产日志中挖出 500 条 Bad Case，最后只用了 200 条高质量正样本 + 50 条公开通用指令。微调后 Bad Case 测试集准确率从 52% 升到 78%。同时保留了一个旧版本测试集，避免出现“忘了老技能”的退化。

**记忆关键词：**
Bad case 驱动、聚类重构、三元组、样本比例、通用指令增强、退化测试。

### 博客 1：这几道高频RAG面试题你都会了吗

- **链接：** https://cloud.tencent.com/developer/article/2661823
- **来源依据：** 腾讯云开发者社区 - 技术博客 - 可信度中等

1. 详细解释了 RAG 与 Fine-tuning 的本质区别：RAG 是开卷考，Fine-tuning 是背书，实际项目应 RAG-First。

5. 针对幻觉问题给出了 Prompt约束 + 重排序卡源头 + 引用验证的组合方案，并在高敏感场景提到知识图谱交叉验证。

**对面试有什么用：**
- 提供了非常丰富的“工程排查话术”，可用于回答各种 RAG 系统优化问题。
- 包含大量可直接用于面试口述的实际例子（如“苹果手机”召回错误的重排案例）。

**可转化面试题：**
1. **题1：** 你说 RAG 和 Fine-tuning 的核心区别是“开卷考 vs 背书”，请用你实际做过的项目例子说明什么时候优先选 RAG？
   - **面试口头版答案：** 如果公司里的知识库每周都会更新（比如政策、合规条款），那 RAG 明显更适合。因为微调需要重新训练全量数据，成本高。RAG 只要把新文档切块、向量化、插入索引库就行，不用碰模型本身。我的项目里就是这么做，上线后知识更新的响应周期从 2 周缩至 1 天。特别是需要“可溯源”的场景（如法律合规），RAG 能直接引用原文，微调做不到。所以我的决策框架是：RAG-First，只有输出格式卡死或者模型语感不达标时才考虑微调。

2. **题2：** 在重排序环节，参数是多少？召回多少条，最终传多少条给模型？
   - **面试口头版答案：** 第一步混合检索召回 30-50 条。第二步用 Cross-Encoder（比如 BGE-Reranker-v2-m3）精排，只保留最相关的 3-5 条。传给模型太多会浪费上下文窗口，而且容易引入噪音。如果使用类似 Llama 3.1 的 128K 上下文模型，可以放宽到 10 条，但必须确保前 3 条是最相关的。重排序后还有一条黄金原则：如果精排后第一名的得分远低于第二名，说明检索质量整体不行，建议不再传给模型，直接返回“没有找到相关资料”，避免幻觉。

### 博客 2：RAG Interview: 40 Questions to Go from Beginner to Advanced

- **链接：** https://www.analyticsvidhya.com/blog/2026/02/rag-interview-questions-and-answers
- **来源依据：** Analytics Vidhya - 技术博客 - 可信度中等
- **文章类型：** 理论/论文（RAG 全流程问答及面试策略）

**核心内容总结：**
1. 清晰讲述了 RAG 离线构建和在线推理的完整流程：文档清洗→切块→嵌入→向量库；用户 query 嵌入→召回→重排序→Prompt 组装→LLM 生成→引用。
2. 明确指出检索器与生成器是耦合关系，召回弱则生成无法拯救。
3. 给出了常见 RAG 失败模式及解决思路：大 chunk 包含无关信息；小 chunk 信息碎片。
4. 混合检索的工程实现：密集召回（语义）+ 稀疏召回（BM25 关键词）+ 排序融合。
5. 提到了 RAG 评测的两种思路：离线指标（准确率、召回命中率、NNR）和在线指标（用户满意度、引用使用率）。

**对面试有什么用：**
- 建立 RAG 端到端全局观，从预处理到推理，从评估到迭代。
- 提供各种面试追问的答题点子，比如如何切块、如何融合召回。

**可转化面试题：**
1. **题1：** 什么是混合检索？为什么 Dense Recall 已经很强了，还要加入 BM25？
   - **面试口头版答案：** Dense Recall 擅长语义相似但不擅长精确匹配。比如用户问“产品型号 XYZ-120 的保修期”，嵌入向量可能会把 XYZ-120 向量化成一个模糊语义，丢失精确字符串信息。BM25 正好弥补这个短板，它基于精确词频来检索。所以混合检索能提升在包含特定术语、ID 或专有名词场景下的召回率。我实做过对比，加入 BM25 后这类场景的命中率从 70% 提升到 91%。

2. **题2：** 什么是“Lost in the Middle”问题，你如何缓解？
   - **面试口头版答案：** LLM 倾向于注重上下文的开头和结尾，忽略中间信息就是 Lost in the Middle。我采用三个方法：第一，对检索结果按相关性降序排列，最重要的放开头；第二，如果上下文很长，用一个小模型做摘要压缩后再传给 LLM；第三，在大上下文下尝试分多次缩小范围，先粗筛再细分。

### 卡片 1：混合检索的两路召回与分数融合

- **Q：** 你用什么方式融合 Dense 和 Sparse 召回的结果？
- **A：** 我通常用倒数排名融合法。对向量检索和 BM25 的结果分别排序，计算每条候选的倒数排名并加权求和，权重可以通过小样本搜索调整。另一种更精确（但也更慢）的方法是用学习器训练融合权重，然后在线上做一次排序。
- **一句话记忆法：** 两虎相争用“倒数排名”调解，简单可靠，适合线上。

### 卡片 2：RAG Agent 系统总设计方案

- **Q：** 你是如何设计 RAG + Agent 混合系统的？
- **A：** 分层设计——规划层做意图识别和任务拆解；执行层用 Agent 决定什么场景要调用 RAG 工具；知识层维护向量库和图库。Agent 只有在查询外部知识（政策、产品手册）时调用 RAG，纯推理任务直接输出。这样成本可控，知识可溯源。
- **一句话记忆法：** 分层设计：规划层是大脑，执行层是调度员，RAG 是检索器，Agent 不替检索器做功课。
- **复习标签：** #Agent #RAG #系统设计

### 卡片 3：SFT Bad Case 驱动的数据构造

- **Q：** 怎么构造 SFT 的训练数据？
- **A：** 基础是 Bad Case 驱动。生产日志中筛选 Bad Case，聚类根因，人工标注正确答案。完成后再加入 20% 通用指令增强多样性。最后一定要验证退化率，即测试集上旧技能是否有损。
- **一句话记忆法：** Bad Case 就是金矿，挖出来、修好它、别忘原有本事。
- **复习标签：** #微调 #SFT #数据工程

### 卡片 4：RAG 评测三层面

- **Q：** RAG 系统怎么评估？
- **A：** 第一层查得准不准（Hit Rate, MRR）；第二层编没编（Faithfulness 忠实度）；第三层答得对不对（ROUGE-L / BLEU）。我通常用 RAGAS 框架做自动化评估，再配合人工抽检。
- **一句话记忆法：** RAG 评判三步走：查到了吗？撒谎了吗？答对了吗？
- **复习标签：** #RAG #评测 #RAGAS

### 卡片 5：Agent 的多工具冲突如何处理

- **Q：** Agent 面对多个功能相似的 tool，如何选择？
- **A：** 分三步。第一，语义冲突：预先配置工具的业务优先级和历史成功率，在冲突时计算加权得分选择最优工具。第二，写操作冲突：加锁 + 队列化，保证写操作互斥。第三，依赖冲突：将工具调用建模为 DAG，拓扑排序确保执行顺序。
- **一句话记忆法：** 工具冲突用优先级调停，写操作用锁排队，依赖用 DAG 捋清。
- **复习标签：** #Agent #工具调用

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：**
你提到用 LLM-as-Judge 做模型评测，那么如果 Judge 模型本身打分不准、有偏差，你怎么校准？

**面试口头版参考答案：**
LLM-as-Judge 确实存在位置偏差、重复偏差和自增强偏差。我会用三招纠正：第一，对 Judge 模型也做 Prompt 优化，比如指定它先列出打分理由再给分数，而不是直接打分，这样可以提升分析质量。第二，使用多个 Judge 模型投票，如果三个模型中有两个打出同样的分数，那这个分数可信度更高。第三，配合少量的黄金标注样本，对 Judge 的自动打分做线性校准，比如发现 Judge 倾向于打高分，就做一个映射函数把它映射回真实分布。

**深入理解版参考答案：**
LLM-as-Judge 的偏差是一个系统性挑战。首先要认识几种常见的偏差类型：位置偏差指 Judge 模型倾向于认为上下文最前面或最后面的答案最好；重复偏差指 Judge 模型倾向于重复它自己最近写过的风格；自增强偏差指 Judge 模型偏向与自己的答案类型相似的回答。缓解方法有几种：第一种，打分格式工程。不要直接问“这个回答好不好”，而是定义评分维度（准确性、流畅度、完整性），每个维度打 1-5 分并在分析阶段写理由。第二种，多模型投票。例如用 GPT-4o + Claude + DeepSeek 三者独立打分，然后取众数或均值。第三种，黄金样本校准。准备 200-500 条人工标注好的样本，对比 Judge 自动打分与人工打分，训练一个轻量级评分回归模型来做偏差纠正。比如有的 Judge 老是给高分，就做分数映射：Judge 5 分 → 真实 4 分。一个进阶做法是让 Judge 模型进行 pairwise comparison（两两比较）而不是直接打分，这样偏差更少。两两比较然后通过 Elo 评分法汇总，这个办法在大模型评测社区（如 AlpacaEval 2.0）中很常见。

**可能追问与简短回答：**
1. **追问：** 如果你用 GPT-4o 作为 Judge，但比模型的部署成本还高，这合理吗？
   **回答：** 看场景。如果是离线评测、周报质量分析，可以接受。但线上实时评测不推荐。建议线上用一个小维度化的评估器（比如使用 Llama 3.1 8B 的微调评估模型），成本更低但准确度可控。

2. **追问：** 你如何判断 Judge 模型打分是否一致（Inter-rater reliability）？
   **回答：** 可以用 Cohen's Kappa 系数衡量两个 Judge 的评分一致性。如果 Kappa 低于 0.6，说明两个 Judge 打分不一致，需要检查打分标准或换一个 Judge。
