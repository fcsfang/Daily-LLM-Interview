## 1. 今日面试真题精讲

### 题目 1：Agent 系统的 Prompt 是怎么设计和迭代的？有没有做过 Prompt 自动优化？当用户提出不完整请求时，如何补全用户意图？
**来源依据：**
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 个人项目资料 - 需进一步验证

**可信度：** 中（个人整理的面试案例集，但问题本身是经典高频题）

**考察点：** Prompt 工程、意图补全、迭代方法、LLM-as-judge 辅助优化

**面试口头版回答：**
我的 Prompt 迭代遵循三步法。第一步，从业务上梳理场景，写出包含角色、目标、约束、输出格式的原始 prompt。第二步，用小批量真实或构造 query 测试，人工标注失败 case，分类比如“输出不遵循格式”、“幻觉”、“漏掉约束”。然后根据根因针对性修改 prompt 措辞或增加 few-shot。第三步，当 prompt 变复杂后，我会用 LLM-as-judge 做一个自动评估 pipeline：用强模型对输出打标，生成负例分数，然后在验证集上对比两个 prompt 版本的效果。对于不完整请求，我采用“反问 + 上下文推理”策略：先让 Agent 根据对话历史推测缺失信息，如果不确定，就主动反问用户。

**深入理解版回答：**
Prompt 设计不是一个一次性的写作文案，而是一个持续优化闭环。初始阶段，我通常会用角色定义 + 任务描述 + 约束 + 输出格式这些模板。但实际跑起来会发现很多问题，比如指令冲突、格式解析困难、模型忽略约束等。
迭代时，我维护一个“bad case 池”。例如用户问“那它呢”，Agent 不理解指代。我会在 prompt 里显式加一条“如果用户问题中有指代词（它、他、这个），先结合对话历史解析出完整意图”。对于更复杂的 case，会添加 few-shot 示例。当 prompt 达几百 token 后，手动修改效果越来越差，所以我引入了自动优化：用 LLM-as-judge 给每个输出打分（如结构合规、事实正确、无冗余），然后自动对比两个 prompt 版本在验证集上的平均分，选出优胜版本。这相当于用自动化评测替代手动评估。
对于意图补全，我具体用过两种策略：一是“软补全”，让模型在生成回答前先输出一段“用户意图还原”，再基于此回答；二是“硬补全”，用独立分类模型判断意图完整性，若不完整则触发澄清反问。生产环境中我倾向软补全，因为更灵活。

**常见追问与回答：**
1. **你怎么保证自动优化不会让 prompt 过拟合验证集？**
   我会在多个不同的验证集上评测，同时引入多样性惩罚，比如定期用人工检查改变度较大的 prompt。
2. **意图补全时，如何避免反问太多让用户觉得烦？**
   我会设一个置信度阈值 – 只有模型对补全内容的置信度低于 0.6 时才反问，并且最多允许连续反问两次。
3. **你用 LLM-as-judge 评估 prompt 时，怎么防止 judge 模型自身的偏差？**
   我会用多个弱模型投票，或者让 judge 模型做 chain-of-thought 后打分，并定期采样人工校准。

**项目联系：**
在我的对话 Agent 项目中，我维护了一个 500+ 条的 bad case 库，每次修改 prompt 后都用自动化 pipeline 跑一遍，对比 LLM-as-judge 分数变化。这让我可以量化 prompt 改进效果，而不只是靠感觉。

**记忆关键词：**
Prompt 迭代、Bad case 驱动、LLM-as-judge、意图补全、软硬策略

### 题目 2：多轮对话上下文状态管理是如何做的？如何在高并发场景下保证一致性？
**来源依据：**
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 个人项目资料 - 需进一步验证

**可信度：** 中

**考察点：** 状态管理、并发安全、隔离性、LangGraph 状态机

**面试口头版回答：**
我是用 LangGraph 的状态管理机制来管理的。LangGraph 把每轮对话的状态定义为一个字典，包含 messages、agent_state（当前所在节点）、context 等字段。每个用户会话对应一个 StatefulGraph 实例，状态存储在外部缓存里（比如 Redis）。高并发下隔离通过“会话 ID + 线程绑定”实现：每个用户请求带一个 session_id，系统将这个 session 的状态引用绑定到当前线程的局部变量，避免共享。写状态时用乐观锁（版本号），读状态时从 Redis 一次性取出，不用多次 IO。这样既保证了隔离，又避免了竞态。

**深入理解版回答：**
多轮对话状态管理的一个核心矛盾是：既要状态持久化，又要高效读写，还要防止不同用户的状态混乱。我先讲状态结构。我用一个全局字典，键是 session_id，值是一个 Pydantic 模型，字段包括 chat_history（消息列表）、agent_memory（系统记忆）、current_step（工作流当前节点）、metadata（时间戳等）。LangGraph 本身维护一个图执行器，每个节点执行后更新状态，这些更新是原子的。
在高并发下，我遇到的最大坑是“脏写” – 两个请求同时到达同一 session，写入覆盖。解决方案：第一，前端保证同一个 session 的请求串行（比如每次只允许一个 in-flight 请求）。如果场景不允许串行（如多个 Agent 协作更新），我会用 Redis 分布式锁，锁的粒度是 session_id。第二，写操作使用条件更新，对比版本号，版本不匹配则重试或拒绝。第三，读操作不做缓存，每次从 Redis 全量读取状态，因为状态大小通常很小（几百 KB），多次读成本高但可以接受。LangGraph 恰好支持 checkpointing，它的设计就是每次节点执行后保存一次 checkpoint，并支持快照回滚，这对我来说是现成的一致性保证。

**常见追问与回答：**
1. **如果状态特别大（比如几 MB），每次读写 Redis 会不会太慢？**
   会。我会对大状态做分片存储 – 比如把 chat_history 单独存，只把关键字段存主状态。读时延迟在 10ms 以内，可以接受。
2. **LangGraph 的状态保存和恢复怎么做的？**
   它的原理是每次节点执行前序列化当前状态到 checkpoint（默认内存/可配置 Redis），异常回滚时 restore 上一个 checkpoint。
3. **你如何保证多服务实例间状态同步？**
   状态存储统一在 Redis，所有实例指向同一数据源。实例无状态，只有状态存储有状态。

**项目联系：**
我的 Agent 项目用 LangGraph 编排，最开始状态存在内存，导致重启后丢数据。后来我改为 Redis 后端，并加了版本号冲突检测，生产环境每天 10 万+ 会话，没再出现状态覆盖问题。

**记忆关键词：**
状态管理、LangGraph、Redis 锁、乐观锁、版本号、隔离性

### 题目 3：在 LLM 应用系统中，你如何设计安全护拦（Guardrails）来保证输出合规？请从输入过滤、输出过滤、工具调用权限三个维度展开。
**来源依据：**
- YouTube - https://www.youtube.com/watch?v=TMyXH0OwSRI - 视频摘要 - 需进一步验证

**可信度：** 低（视频摘要未确认详细内容，但概念是经典）

**考察点：** 安全护拦设计、输入输出过滤、权限控制、工程落地

**面试口头版回答：**
我会从三条防线设计。第一层是输入过滤：在用户请求进入 LLM 之前，检查是否有攻击意图（如 prompt injection），可以用关键词黑名单 + LLM 分类器做双重检查。第二层是输出过滤：在模型生成结果后，用正则或 NER 模型检测是否有敏感信息泄漏，同时用另一个 LLM 做安全校验，过滤不符合合规的内容。第三层是工具调用权限：每次 Agent 调用外部工具前，检查该 session 是否有对应工具的权限，权限配置存在 RBAC 表中，按角色分配。例如普通用户不能调用“删除数据库”工具。出口处还会做速率限制和审计日志。

**深入理解版回答：**
安全护拦必须分层，不能依赖单一策略。我讲一个完整的系统设计。请求进来，先经过一个“输入过滤器”，它包含两个子模块：
1. 基于规则的快速拦截：匹配危险关键词、SQL/XSS 模式。
2. 基于模型的深度检测：用一个小型的 LLM 或序列分类模型判断是否是 prompt injection，准确率约 95%。
如果通过，进 LLM 生成。生成完成后，进入“输出过滤器”：
1. 检测 PII（个人信息）：用正则或 spaCy NER 识别姓名、电话、地址等，自动脱敏或拒绝输出。
2. 内容安全：用内容审核模型（开源如 Perspective API 或自训练分类器）过滤暴力、涉政等违规内容。
3. 事实性检查（可选）：用知识库交叉验证，检测幻觉。
第三层是工具调用权限。我的 Agent 工具集分公开工具（搜索、计算）和受限工具（写 DB、发邮件）。每个 session 从 Redis 缓存该用户的“工具集权限列表”，工具调用时先查权限，无权限则返回错误并记录日志。权限变更通过异步消息更新 Redis，保证实时性。最后，所有护拦的动作都写入可观测性日志（如 Elasticsearch），用于事后审计和优化规则。

**常见追问与回答：**
1. **规则和模型双层过滤会不会增加延迟？**
   会，但输入过滤器平均耗时 <30ms（用 fastText 或小型 ONNX 模型），输出过滤器约 50ms。整体 <100ms，可接受。如果延迟敏感，可以降低模型的深度（如只保留规则）。
2. **用户权限变更后，如何保证旧 token 不继续使用？**
   每次请求强制从权限表中读取最新权限，不依赖 token 中缓存的权限。token 只做认证，不做授权缓存。
3. **如何平衡安全与用户体验，比如误拦截太多？**
   我会设计一个“置信度阈值”可调，并加入白名单机制。同时收集被拦截的请求，定期人工 review，降低误拦率。

**项目联系：**
在我做的企业问答 Agent 中，为了防止员工通过对话获取敏感合同信息，我实现了三层护拦：输入禁止问“密码”、“签署人”等关键词；输出脱敏；工具调用限制只有经理以上角色可查“薪资”字段。上线后安全事件为 0。

**记忆关键词：**
Guardrails、输入过滤、输出过滤、权限控制、RBAC、审计日志

  1. **问题1：Agent 的长期记忆系统中，如何实现记忆的“冲突更新”？举一个具体的例子。**
     **面试口头版回答：**
     冲突更新指新信息与旧记忆矛盾时的处理策略。我的做法是不直接物理删除，而是按“时间戳 + 来源置信度”裁决。例如用户先说“我只看华东区的数据”，后来又说“改看华南区”，时间戳更新，且是用户显式说的（高置信度），那我就把旧条目标记为失效，写入新条目。如果是模型从对话推断出的偏好（低置信度），我可能两条都留，检索时优先取较新、置信度高的那条。这样既不丢失历史，也不会用过期信息答题。
     (源自面经中“冲突更新”部分)
  2. **问题2：RAG 系统中，如何实现增量索引（chunk 粒度 upsert）？需要处理哪些工程难点？**
     **面试口头版回答：**
     增量索引的核心是在不重建全部索引的情况下，增加、更新或删除单个 chunk。我用向量数据库（如 Milvus 或 Qdrant）的 upsert 支持。难点有三个：一是删除残留，当原文档修改后旧 chunk 必须彻底删掉，否则会召回过期信息；二是版本一致性，同一文档不同 chunk 之间需要版本号标记，避免新旧混用；三是并发控制，多个 chunk 同时 upsert 时要保证原子性。我通过为每个文档和 chunk 分配更新时间戳，并在查询时过滤掉旧版本来解决。
     (源自面经中“增量索引”部分)

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** Prompt 自动优化常用什么方法？如何防止过拟合验证集？
- **A：** 常用 LLM-as-judge 评估框架：用强模型对输出打分，自动调整 prompt 措辞。防过拟合采用多验证集混合评测+正则化（如限制修改幅度），以及人工抽样校验。
- **一句话记忆法：** 自动调 Prompt 像 AutoML，只是 evaluation 用 LLM 而非指标。
- **复习标签：** #Prompt工程 #自动优化

### 卡片 2
- **Q：** 用 LangGraph 管理多轮对话状态，相比手写状态机有哪些优势？
- **A：** LangGraph 提供声明式状态定义、自动 checkpoint 和故障恢复、节点执行原子化、内置循环支持，以及 Redis 等后端持久化。手写状态机需要自己实现序列化、并发控制和回滚，开发成本高。
- **一句话记忆法：** LangGraph 是带超能力的状态机，自带走查+暂停键。
- **复习标签：** #Agent #LangGraph #状态管理

### 卡片 3
- **Q：** LLM 应用中，输入过滤有哪些常用技术？列举两个。
- **A：** 一是关键词黑名单匹配（如 SQL 注入、敏感词），速度快但覆盖不全。二是基于 LLM 的分类器，如用一个小模型判断自然语言是否是攻击 prompt（提示注入），召回高但增加延迟。实际中两者结合：先快速规则过滤，再模型深度检测。
- **一句话记忆法：** 输入过滤 = 规则筛子（快） + 模型放大镜（准）。
- **复习标签：** #安全护拦 #输入过滤

### 卡片 4
- **Q：** 如何设计 LLM 应用的可观测性？至少说出三个关键指标。
- **A：** 可观测性分三部分：日志（structured format，记录每次请求的输入输出、延迟、错误）、指标（Token 消耗速率、TP99 延迟、Bad Case 占比）、追踪（每个环节耗时，如检索、生成、rerank）。使用 OpenTelemetry 统一上报。
- **一句话记忆法：** 可观测性 = 日志记案底 + 指标看体检 + 追踪做CT。
- **复习标签：** #可观测性 #日志 #监控

### 卡片 5
- **Q：** 在多用户高并发下，如何保证 Agent 工作流的状态隔离？
- **A：** 核心方案：每个用户绑定唯一 session_id，状态存储在外部存储（如 Redis）中，请求处理由无状态服务实例完成。写操作使用乐观锁（版本号）防止覆盖。LangGraph 的 checkpointing 支持自动版本管理，可以在此基础上加 Redis 锁。
- **一句话记忆法：** 状态隔离 = 唯一钥匙（session_id） + 保险柜（Redis） + 涂改笔（版本锁）。

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 什么是 LLM 应用中的“可观测性”？你会如何设计一个 LLM 应用的可观测性体系，使其能够实时发现 Bad Case 并辅助 Prompt 迭代？

**面试口头版参考答案：**
可观测性是指通过日志、指标、追踪三个维度来理解系统内部状态。对于 LLM 应用，我设计三层可观测：第一层，每次请求都写 structured log，包括输入、输出、检索结果、延迟、Token 数、是否调用工具等。第二层，定义核心指标：生成质量（用 LLM-as-judge 分数）、检索命中率、用户反馈率（比如踩/赞）。第三层，用 OpenTelemetry 实现分布式追踪，看到每一步的耗时和错误。在线上，通过告警规则自动捕获 Bad Case：比如当 LLM-as-judge 分数低于阈值或用户主动点踩时，自动把该次请求的日志和追踪信息存入 Bad Case 库。这些 Bad Case 可以直接用来驱动 Prompt 迭代，比如分析出高频模式，然后针对性修改 prompt。

**深入理解版参考答案：**
可观测性（Observability）在 AI 系统中比传统软件更重要，因为 LLM 输出非确定，需要通过内部信号推断状态。我的设计分三个数据源：
1. **日志（Logs）**：采用结构化日志（JSON），字段包括 request_id、session_id、时间戳、模型名称、输入输出、检索的文档片段、各环节耗时（检索、rerank、生成）、错误码、是否命中缓存、Token 使用量等。使用日志聚合平台（如 Elasticsearch + Kibana）做全文搜索。
2. **指标（Metrics）**：通过 Prometheus 采集。关键指标有：TP50/TP99 延迟（按环节拆分）、Token 消耗率（输入 vs 输出）、错误率（按错误类型分）、LLM-as-judge 质量分数分布、用户满意度评分（点赞/踩）分布。设阈值告警，比如质量分数均值下降 5% 即告警。
3. **追踪（Traces）**：用 OpenTelemetry 给每个请求加 span，记录调用链路：网关 → 检索 → 重排序 → LLM 生成 → 输出过滤。这样可以一眼看出瓶颈在哪。

发现 Bad Case 的流程：当用户点“踩”或 LLM-as-judge 分数低于 3/5 时，触发自动采样，把该请求的日志、追踪、检索片段打包到 Bad Case 数据库（如 MongoDB）。每周对这些案例做聚类分析，自动提取高频错误模式（如“模型忽略约束”、“检索召回不相关”）。这些模式直接指导 Prompt 迭代：比如发现很多 case 是模型没有遵循“只输出 JSON”约束，那么就在 prompt 中增加一个更严格的 JSON schema 示例。

此体系与测试集相结合，每次 Prompt 修改后，先在 Bad Case 库上跑回归自动化评测，确保旧 case 不退化，New 分数上涨。

**常见追问与回答：**
1. **如果 LLM-as-judge 本身有偏差，导致 Bad Case 抓取不准怎么办？**
   我会设置一个“人工抽样复核”流程：每周随机抽取 200 条被标记为 Bad Case 和 200 条未被标记的样本，人工标注，从而校正 judge 模型的权重或阈值。同时使用多个 judge 模型投票。
2. **指标告警中，质量分数下降多少算严重？如何权衡误报和漏报？**
   我通常用三天滚动平均值与七日前均值对比，下降超过 10% 或标准差超过 2 倍就告警。初期间阈值宽松些（避免频繁告警），随着系统稳定再收紧。也会对比其他指标（如延迟是否正常），排除非模型问题导致的分数下降。
