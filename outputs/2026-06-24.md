### 题目 1：LoRA 微调的原理是什么？秩 r 的选择会对模型表现产生什么影响？

**来源依据：**  
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - GitHub 技术博客 - 可信度中（社区维护）  
- AI大模型八股面试题 - https://www.cnblogs.com/xiaolincoding/p/20534325 - 技术博客 - 可信度中  
- 需进一步验证：部分内容来自个人整合。

**可信度：** 中

**考察点：** PEFT 原理、秩 r 选择、过拟合、表达能力。

**面试口头版回答：**  
LoRA 的核心是在预训练模型的权重矩阵旁插入一个低秩分解矩阵，训练时只更新这个低秩矩阵，冻结原始权重。具体来说，对每个线性层，我们用两个小矩阵 A 和 B 的乘积来近似权重更新量 ΔW，其中 A 是 d×r，B 是 r×d，r 远小于 d。秩 r 决定了更新量的表达能力：r 太小（比如 r=1）可能无法捕捉任务特定的模式；r 太大（比如 r=64）会引入更多可训练参数量，增加过拟合风险，同时抵消 LoRA 节省显存和加速的优势。实践中，对于大多数通用微调任务，r 取 8 或 16 是一个比较稳妥的起点，然后根据验证集困惑度或下游任务指标做网格搜索。

**深入理解版回答：**  
LoRA（Low-Rank Adaptation）是一种参数高效微调方法。其数学原理为：对于预训练权重矩阵 W₀ ∈ R^{d×k}，冻结 W₀，训练时引入两个低秩矩阵 B ∈ R^{d×r} 和 A ∈ R^{r×k}，则前向传播变为 h = W₀x + BAx。初始时 B 为零矩阵，A 用随机高斯初始化，这样一开始微调不影响原有输出。训练时只优化 A/B 的参数，更新后的权重等效为 W₀ + ΔW。

秩 r 的选择直接影响模型的表达能力和泛化能力。从线性代数角度看，低秩矩阵的秩上限限制了它能捕捉到的“独立方向”数量。若 r 过小（如 r=1），ΔW 退化为秩 1 矩阵，只能拟合极简单的变换，对于需要改变大量权重的任务（如领域适配）往往欠拟合。若 r 过大（如 r=64 或 128），可训练参数量接近全量微调的 10% 甚至更多，显存占用和训练时间接近全量微调，且容易在有限的数据集上过拟合。常见做法是在不同层采用不同的 r：注意力层的 Q/K/V/O 通常需要较大 r（如 16），因为它们是语义理解的核心；FFN 层可以适当减小 r（如 4-8）。此外，LoRA 论文实验表明，即使 r=1 在某些任务上也能取得不错效果，但为了稳定性，建议从 8 开始试。α（scaling factor）通常设为 2 倍 r，即 α=2r，这样初始化时 BAx 的 scale 与原始 W₀x 大致匹配。

工程实践中，LoRA 的另一个优点是灵活部署：训练完成后只需保存 A/B 两个小矩阵（通常几 MB 到几十 MB），可以在推理时动态加载到基座模型上，无需存储完整副本。当需要微调多个下游任务时，可以保留基座模型的一个副本，根据请求动态切换不同的 LoRA adapter，显著节省存储和显存。这也是 Ray Serve、vLLM 等推理框架支持多 LoRA 切换的底层原理。

**常见追问与回答：**  
- **追问：LoRA 训练时，Transformer 中哪些参数被更新？**  
  答：只有我们手动插入的低秩矩阵 A 和 B 被更新。原始预训练权重（包括所有层的权重矩阵）都是冻结的，不参与梯度计算。  
- **追问：LoRA 和 Adapter、Prefix Tuning 有什么异同？**  
  答：LoRA 在权重层面插入低秩矩阵，不改变模型结构；Adapter 在 Transformer 层的子层间插入小型神经网络；Prefix Tuning 在输入序列前拼接可学习的虚拟 token。三者都是参数高效微调，但 LoRA 的推理延迟增加最小（可以合并权重到原始 W₀），而 Adapter 和 Prefix Tuning 都需要额外的计算。  
- **追问：为什么 LoRA 默认只更新注意力层的参数，不更新 FFN 层？**  
  答：早期论文实验发现注意力层的权重对下游任务影响更大，但后续实践表明 FFN 层也值得微调。现在主流做法是对所有线性层都应用 LoRA，但不同层可以设置不同的 r。

**项目联系：**  
在我的项目中，我使用 LoRA 对 LLaMA-7B 进行中文客服领域微调。初始 r=8 在意图识别准确率上比 r=4 提升 2 个百分点，但 r=16 时验证 loss 开始回升，说明已经过拟合。我最终采用了分层不同的 r：注意力层 r=16，FFN 层 r=8。同时保留多个领域的 LoRA weights，在部署时通过 vLLM 的 LoRA 切换功能实现低成本多业务线接入，每次切换只需加载约 5MB 的权重，大约 50ms。

**记忆关键词：**  
LoRA 低秩分解、秩 r 选择、可训练参数量、过拟合、α scaling、多 adapter 切换

### 题目 2：SFT 训练数据是怎么构建的？如何保证样本多样性和质量？SFT 的 loss 如何只计算回答部分？

**来源依据：**  
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - GitHub 技术博客 - 可信度中  
- AI大模型八股面试题 - https://www.cnblogs.com/xiaolincoding/p/20534325 - 技术博客 - 可信度中  
- 需进一步验证：具体 loss 实现细节来自常见开源代码实践（如 LLaMA-Factory）。

**可信度：** 中

**考察点：** SFT 数据构造、数据清洗、loss masking、指令微调质量。

**面试口头版回答：**  
SFT 数据构造一般分三步：第一，从高质量来源（人工标注、GPT-4 蒸馏、业务日志）收集指令-回答对；第二，对数据进行清洗和多样性筛选，通常采用去重、基于 embedding 聚类的多样性采样、按难度分层等策略；第三，构造训练格式，比如用 ChatML 模板。关于 loss 只计算回答部分，关键是用一个 `labels` mask 将 prompt 部分的 loss 置为 -100。在 Hugging Face Trainer 中，我们只需要在 `labels` 中将 prompt 对应的 token id 设为 -100，loss 计算函数（如 CrossEntropyLoss）默认会忽略 -100 的位置。这样模型只会学习如何生成正确的回答，而不关心 prompt 的重建。

**深入理解版回答：**  
SFT（Supervised Fine-Tuning）是让大模型学会遵循指令和生成高质量回答的关键步骤。数据构造必须同时关注源头和清洗流程：

1. **数据来源**：常见有四种。人工标注（成本高但质量最可控）、从现有模型蒸馏（如用 GPT-4 生成高质量回答后用于训练小模型）、用户行为日志清洗（需要严格的隐私和噪声处理）、开源数据集（如 ShareGPT、Dolly、Alpaca，需注意中文领域覆盖）。
2. **质量保证**：需要多轮清洗。包括长度过滤（过短或过长的回答通常质量差）、敏感内容检测、去重（MinHash 或 Embedding 相似度去重）、对抗噪声过滤（如模板化回复、重复样本）。
3. **多样性保证**：防止模型只掌握少数领域的回答。常用方法：基于文本 Embedding 对样本聚类，再从每个簇中按比例采样的“聚类采样”；或者按任务类型（分类、推理、生成、总结）分层采样。也可以人工设计任务分类标签，确保覆盖全部目标能力。
4. **Loss 只计算回答部分的实现**：在标准自回归训练中，输入是 `input_ids = [prompt_tokens + response_tokens]`，`labels` 通常与 `input_ids` 相同，但我们把 prompt 部分对应的 labels 全部设为 -100。以 LLaMA 的 tokenizer 为例，在构造数据时，先 tokenize prompt 和 response，令 `labels = tokenized_input_ids.copy()`，然后将 `labels[:len(prompt_tokens)] = [-100] * len(prompt_tokens)`。这样模型在 prompt 位置不产生 loss，只在 response 位置学习。注意：如果 prompt 中包含 <|assistant|> 等特殊 token，通常也需要包含在 loss 计算中，以保证模型学会生成特殊格式。

**常见追问与回答：**  
- **追问：SFT 数据中，一个样本是多轮对话还是单轮？怎么构造多轮对话的 labels？**  
  答：支持多轮。可将整个对话历史拼成一个字符串，每个 assistant 的回复标签为计算 loss，user 消息和 system prompt 全部 mask 掉。工程上，我们逐轮给标签，每一轮 assistant 的 tokens 对应的 labels 保留，其他置 -100。  
- **追问：你们项目数据规模多大？SFT 数据是如何清洗和构建的？”**  
  答：我们从中英文混合的业务对话日志中收集了 10 万条高质量 SFT 数据。清洗逻辑包括：排除 token 长度 <50 或 >2048 的样本；用语言模型对回答做事实一致性评分，剔除得分低的。我们用 LLaMA-Factory 框架训练，loss 只计算回答部分的实现如上所述。  
- **追问：微调后如何评估？**  
  答：我们构建了领域特定评测集，包括意图分类、实体提取、生成质量（BLEU/Rouge + 人工抽检）。单侧对比发现，SFT 后在任务上的准确率提升约 15%。

**项目联系：**  
我曾在构建内部知识库问答 SFT 数据时，发现直接用了原始对话日志，模型有时会学到错误的客服回复模式。于是我们引入了基于 GPT-4 的质量打分（满分 10），过滤掉 5 分以下的样本，再用聚类采样保证意图类型均衡。训练时使用 `data_collator` 中对 labels 做 mask，效果显著提升。

**记忆关键词：**  
SFT 数据清洗、聚类采样、标签 mask、-100 ignore、ChatML 格式

### 题目 3：全量 SFT 和 LoRA SFT 的 lr 设置有什么技巧？

**来源依据：**  
- 2026大厂大模型/Agent面试题 - https://gitcode.csdn.net/6a236d7b662f9a54cb7a2bf1.html - 面经或社媒 - 可信度中（需进一步验证）  
- 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 可信度低（社区转载，需进一步验证）

**可信度：** 中（多份面经提及，未见官方文档）

**考察点：** 微调超参、学习率缩放、优化器选择。

**面试口头版回答：**  
全量 SFT 和 LoRA SFT 的学习率差异很大。全量微调通常会使用较小的学习率，比如 1e-5 到 5e-5，因为要小幅度调整所有预训练权重，防止灾难性遗忘。而 LoRA 微调的学习率可以更大，通常在 1e-4 到 5e-4 量级，因为 LoRA 只更新几个小矩阵，而且初始值为零，需要较大步长来快速学习新任务。经验上，LoRA 的 lr 可以比全量微调大 5~10 倍，同时配合适当的 warmup 和余弦退火调度。对于 AdamW 优化器，如果使用 LoRA，有时也需要将权重衰减调小（如 0.01 降至 0.001），避免小参数被过早衰减到零。

**深入理解版回答：**  
学习率是微调中最敏感的超参数之一。全量 SFT（Full Fine-Tuning）由于更新全部参数，如果 lr 过大会破坏预训练学到的通用特征，导致训崩或过拟合；过小则收敛慢且可能陷入局部最优。常见范围：1e-5 ~ 5e-5，常用 2e-5 为 baseline。优化器一般用 AdamW，权重衰减 0.01~0.1。

LoRA SFT 的学习率选择与全量微调不同，主要有几个原因：

- **参数规模差异**：全量微调更新几亿到几十亿参数，LoRA 只更新几百万到几千万参数，每一步的参数变化幅度更大。
- **初始化不同**：LoRA 的 B 初始为零矩阵，A 初始为随机高斯，所以初始梯度为零，需要较大的 lr 让参数快速离开零点。
- **表达空间受限**：低秩矩阵的表达能力有限，需要适度大的步长才能充分学习任务特征。

实践中，LoRA 的 lr 通常取全量微调的 5~10 倍。例如全量用 2e-5，LoRA 用 1e-4 或 2e-4。但也不能过大，否则会导致 loss 震荡。我见过的最佳实践是：先全量微调少量步数得到 lr baseline，再对 LoRA 在 1e-5 ~ 1e-3 范围内做对数网格搜索。同时，LoRA 的 α（scaling factor）也会影响有效学习率，α 越大，等效 lr 越大。通常 α = 2r，如果 r 小，可适当增大 α 来 “放大” 更新量。

此外，全量微调时通常需要更长的 warmup 步数（总步数 10%），而 LoRA 可以适当减少 warmup（5%）。学习率调度方面，余弦退火对两者都适用。

**常见追问与回答：**  
- **追问：LoRA SFT 中，如果 lr 和全量一样大会怎样？**  
  答：大概率 loss 会很快就降到一个较低值，但随后震荡或过拟合，尤其是训练数据较少时。因为 LoRA 参数少，同样 lr 导致相对变化更大，缺乏全量微调的“缓冲”。  
- **追问：有没有使用过不同层不同学习率？**  
  答：有。可以让比较关键的注意力层用更大的 lr，而 embedding 和 head 层用较小的 lr，但一般 LoRA 只需要统一 lr，因为各层参数规模相似。

**项目联系：**  
我在微调 Qwen2-7B 时，全量微调使用 lr=2e-5，LoRA 使用 lr=1e-4，都配合 200 步 warmup、余弦退火。对比实验显示，LoRA 收敛更快，且最终在领域数据集上的 BLEU 得分与全量微调相差小于 1%，但训练时间只需 1/3。通过调整 lr，我找到了在过拟合和欠拟合之间的平衡点。

**记忆关键词：**  
全量 lr 1e-5 ~ 5e-5、LoRA lr 1e-4 ~ 5e-4、缩放因子 5~10 倍、warmup 长 vs 短

### 题目 4：LLM 微调时选择冻结哪些层？依据是什么？

**来源依据：**  
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - GitHub 技术博客 - 可信度中  
- 2026大厂大模型/Agent面试题 - https://gitcode.csdn.net/6a236d7b662f9a54cb7a2bf1.html - 面经或社媒 - 可信度中（需进一步验证）

**可信度：** 中

**考察点：** 层冻结策略、灾难性遗忘、迁移学习。

**面试口头版回答：**  
选择冻结哪些层主要取决于资源约束和任务特性。最常用的策略是全部不冻结（全量微调），但显存和训练时间成本高。如果资源受限，常见做法是：冻结前 k% 层，只微调靠近输出的层（如最后几层 Transformer 和 LM head），因为这些层包含了更多的任务特定知识。另一个策略是只微调 attention 层（或通过 LoRA 微调所有层），而冻结 FFN 层和 embedding 层。依据是：底层编码通用语言特征（句法、词法），顶层编码语义和任务信息；因此冻结底层、微调顶层可以在节省显存的同时保证任务性能。论文研究表明，全量微调上层 25% 的层通常能达到接近全量微调的效果。LoRA 实际上就是一种“微调所有层但只更新少量参数”的智能方式，避免了手动冻结的不灵活。

**深入理解版回答：**  
在资源有限时（如单卡训练），手动冻结部分层是常见的做法。选择依据可以从以下几个维度考虑：

1. **模型架构层次**：Transformer 底层更关注通用语法、词性等，顶层更关注语义和任务。因此保留底层、微调顶层可以更好地保持通用能力。
2. **任务类型**：如果是文本分类等简单任务，可能只需微调 LM head 和最后几层就能达到不错效果；如果是生成任务，需要微调更多层以确保输出流畅。
3. **数据量**：数据量很少时，冻结大部分层可以防止过拟合；数据量充足时，可以全量微调以获得最佳效果。
4. **显存限制**：冻结层可以减少优化器状态所需显存，因为被冻结的层不产生梯度。

实际上，现代实践中手动冻结越来越少见，因为 LoRA 这种 PEFT 方法几乎零开销地实现了“微调所有层但参数极少”：它自动在每个线性层旁插入小矩阵，所有层都参与训练但只更新极少参数，不需要人工决定哪些层冻结。而且 LoRA 可以动态调整每个层的秩（即参数规模），相比手工冻结更灵活。

如果一定要手动冻结，常见工程做法：例如训练 LLaMA-7B，冻结前 20 层，只微调最后 12 层，可节省约 40% 的优化器显存，但往往性能略低于 LoRA。

**常见追问与回答：**  
- **追问：冻结底层后，模型会不会丢失部分通用能力？**  
  答：一般不会，因为底层权重不变，通用特征（如句法分析）依然可用。顶层权重微调后适应新任务。但若新任务需要底层特征（如情感分类依赖于语法结构），也可以考虑调高冻结比例但保持底层某些层可训练。  
- **追问：你知道哪些模型训练实践会特殊处理 embedding 层吗？**  
  答：embedding 层通常不参与冻结，或者单独设置较小学习率。因为 embedding 层参数量大（vocab_size × hidden_size），更新它可能引入大量噪声。有的框架会关闭 embedding 层的 weight decay。

**项目联系：**  
我在使用 Qwen2-72B 时，由于单卡显存限制，曾尝试手动冻结前 40 层，只微调后 20 层和 LM head。结果在下游摘要任务上 ROUGE-L 下降 0.03，但训练速度提升了 1.5 倍。后来改用 LoRA，在同样显存下微调所有层，ROUGE-L 反超全量微调 0.01，说明 LoRA 确实更优。

**记忆关键词：**  
冻结层策略、底层通用/顶层任务、LoRA 替代手工冻结、embedding 更新

### 题目 5：如何为 LLM 设计多 LoRA 低成本部署和快速切换？

**来源依据：**  
- Multi-LoRA deployment — Ray 2.55.1 - https://docs.ray.io/en/latest/serve/llm/user-guides/multi-lora.html - 官方文档 - 可信度高  
- 需进一步验证：实际实现细节与 vLLM 的 LoRA 功能可能有差异。

**可信度：** 高

**考察点：** LoRA 部署、adapter 切换、缓存与路由、推理延迟优化。

**面试口头版回答：**  
多 LoRA 部署的核心目标是节省显存和加速切换。常用的工程方案有两种：一是合并权重到基座模型；二是推理框架原生支持动态加载。第一种方案每次切换前将基座权重与 LoRA 权重合并，得到新的完整权重加载到 GPU，切换成本高。第二种方案更优，例如 vLLM 或 Ray Serve LLM 支持在推理时动态替换前向传播中的 LoRA 矩阵。以 Ray Serve LLM 为例，它维护一个 LRU 缓存，记录每个 replica 上已加载的 adapter。当请求到来时，首先检查是否有 replica 已加载该 adapter 且不超载，如果有则路由到该 replica；如果没有，则路由到最空闲的 replica 并加载该 adapter。缓存淘汰策略是 LRU，通过 `max_num_adapters_per_replica` 控制最大并发适配器数量。这种方案下，adapter 切换延迟仅约几十毫秒（加载小矩阵入显存），且每个 replica 可以同时服务多个 adapter。

**深入理解版回答：**  
在生产环境中，通常需要为不同客户/业务线提供个性化模型，例如不同客服场景、不同语言版本。如果为每个业务单独部署一个完整模型，成本极高。多 LoRA 部署（Multi-LoRA Serving）应运而生。

**方案一：权重合并（Weight Merging）**  
将 LoRA 的权重合并到基座模型：`W_new = W_base + α * BA`，得到一个完整的模型权重文件。这种方案简单，但每次切换需要重新加载整个模型权重到 GPU（耗时数十秒到数分钟），且磁盘存储多个完整模型副本。只适合离线批量推理，不适合在线实时。

**方案二：推理框架原生支持动态 LoRA（推荐）**  
vLLM 和 Ray Serve LLM 都支持多 LoRA。原理是在基座模型前向传播的线性层计算中，动态将低秩矩阵 BA 加到基座权重上。具体实现：将 LoRA 权重预先存在 CPU 内存或 SSD 中，当一个请求指定要使用某 adapter 时，框架将其权重加载到 GPU 显存（通常只需几 MB 到几十 MB），然后在前向传播时执行 `output = W_base * input + scale * B * (A * input)`。解码过程中，KV cache 是共享的，无需为每个 adapter 保存独立副本（但注意 attention 层中的 LoRA 权重仍需要区分）。

**Ray Serve LLM 的路由策略**（摘自官方文档）：
1. 检查是否有 replica 已加载该 adapter 且没有过载，直接路由。
2. 否则，找已加载 adapter 但负载最低的 replica，在其上加载 adapter。
3. 如果所有 replica 都没有加载，按默认路由策略（如 Power of Two Choices）选择一个 replica 并加载。
4. LRU 缓存管理：每个 replica 的 `max_num_adapters_per_replica` 控制最大缓存数，超过时淘汰最久未使用的 adapter。

**切换延迟优化**：通常 LoRA 权重仅为几 MB，加载到显存耗时约 10-50ms，远小于模型加载（几秒到几十秒）。进一步优化可通过预加载热门 adapter、使用更快的存储（如 NVMe SSD 或直接内存映射）。

**常见追问与回答：**  
- **追问：多个 LoRA 共用一个基座模型时，显存占用如何？**  
  答：基座模型占用大部分显存，所有 adapter 共享该基座。每个 adapter 的权重很小（几 MB 到几十 MB），所以显存几乎不随 adapter 数量线性增长，主要受 `max_num_adapters_per_replica` 控制，以及 KV cache 等。  
- **追问：如果业务线有 1000 个 adapter，如何确保高效切换？**  
  答：可对 adapter 进行分层：热门 adapter 常驻缓存，冷门 adapter 按需加载。使用分布式路由层，将请求均匀分配到多个 replica，每个 replica 维护自己的 LRU 缓存。还可以基于请求频率动态调整每个 adapter 的缓存优先级。

**项目联系：**  
我在实习项目中负责优化多业务线部署，使用 vLLM + LoRA Serving。最初每个业务各自部署一个完整模型，需 4 张 A100。通过合并到一个基座模型 + 多 LoRA 方式，只需 1 张 A100 即可同时服务 20 个业务，每个 adapter 切换延迟约 30ms，满足线上实时性要求。

**记忆关键词：**  
多 LoRA 部署、权重合并 vs 动态加载、LRU 缓存、adapter 路由、显存共享

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** LoRA 训练时，Transformer 中哪些参数被更新？  
- **A：** 只有我们插入的低秩矩阵 A 和 B 被更新。原始预训练权重（所有层的权重矩阵）被冻结，不参与梯度计算。  
- **一句话记忆法：** “LoRA 不碰原装参数，只练两个小分身（A 和 B）。”  
- **复习标签：** #LoRA #PEFT

### 卡片 2
- **Q：** SFT 训练中如何实现只计算回答部分的 loss？  
- **A：** 在 `labels` 中将 prompt 对应的 token id 设为 -100。Hugging Face 的 CrossEntropyLoss 默认忽略 -100，从而实现只对 answer 部分计算梯度。  
- **一句话记忆法：** “Labels 里塞 -100，模型只学怎么答，不学怎么问。”  
- **复习标签：** #SFT #训练

### 卡片 3
- **Q：** LoRA 的秩 r 选择依据是什么？  
- **A：** r 太小（如 1）表达能力不足，容易欠拟合；r 太大（如 64）参数量大且容易过拟合。实践先试 8 或 16，根据验证 loss 微调。不同层可设置不同 r：注意力层大、FFN 层小。  
- **一句话记忆法：** “r=8 是甜点，大了过拟合，小了欠拟合；注意力层多吃，FFN 层少吃。”  
- **复习标签：** #LoRA #秩选择

### 卡片 4
- **Q：** 全量 SFT 和 LoRA SFT 的学习率设置技巧？  
- **A：** 全量 SFT 用 1e-5 ~ 5e-5，LoRA SFT 用 1e-4 ~ 5e-4。LoRA 的 lr 可比全量大 5~10 倍，因为参数少、初始为零，需要大步长快速学习。  
- **一句话记忆法：** “全量学步似散步（1e-5），LoRA 小跑进公园（1e-4）。”  
- **复习标签：** #LoRA #SFT #学习率

### 卡片 5
- **Q：** 如何实现多 LoRA 的低成本部署和快速切换？  
- **A：** 使用推理框架（如 vLLM、Ray Serve）的多 LoRA 支持：动态加载小权重（几 MB）到显存，不合并到基座模型。路由层优先路由到已加载该 adapter 的 replica，缓存淘汰用 LRU。切换延迟约几十毫秒。  
- **一句话记忆法：** “基座不动，LoRA 小跑；路由找熟人，LRU 清冷宫。”  
- **复习标签：** #LoRA #模型部署 #推理优化

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** LoRA 的秩 r 选择与模型过拟合之间存在怎样的关系？请从参数规模、表达能力和训练数据的角度分析。

**面试口头版参考答案：**  
LoRA 的秩 r 直接决定了可训练参数的数量和表达能力。参数越多，模型拟合训练数据的能力越强，但若数据量不足，就容易过拟合。具体来说：r 越大，低秩矩阵 ΔW 的秩越高，能捕捉更多方向的变化，但同时也引入了更多可训练参数。假设基座模型隐藏层维度是 d，r 对应的可训练参数是 2×d×r。当 r=8 时，参数量约为 2×4096×8=65,536；当 r=64 时，参数量约 524,288，相当于原始权重的 1.6%（假设 d=4096），但仍然明显。在训练数据只有几千条时，r=64 很容易学到噪声；而 r=8 相当于一个更强的正则化，有助于泛化。实践中，我通常先用 r=8 训练，观察验证集 loss 是否持续下降；如果不下降，再尝试增大 r。另外，通过调大 α（如 α=2r）也能在固定 r 下放大更新量，等价于更大的学习率。

**深入理解版参考答案：**  
本问题需要从统计学习中的“模型容量”角度理解。LoRA 的秩 r 控制着低秩矩阵的容量。一个秩为 r 的矩阵的“自由度”约为 (d+k)r - r²（d 和 k 为输入和输出维度）。当 r 翻倍时，可训练参数量近似线性增长。从这个意义上说，LoRA 可以视为一种隐式正则化：低秩约束本质上限制了权重更新矩阵的秩，相当于在参数空间中施加了结构性的稀疏性约束。  
- **当 r 过小（例如 r=1）**：更新矩阵秩为 1，只能表示一个方向的线性变换。对于需要大量调整权重的任务（如全领域迁移），表达能力不足，导致欠拟合，验证集 loss 居高不下。  
- **当 r 过大（例如 r=64 或 128）**：可训练参数接近全量微调的 10% 甚至更多，模型容易对训练集中的噪声和模式过拟合，尤其是当训练数据量不足（比如 <10K 条）。此时验证集 loss 可能先下降后上升，出现典型的过拟合曲线。  
- **数据量因素**：假设训练样本数为 N，每个样本提供的信息量有限。参数越多，每个参数分到的样本越少，过拟合风险越高。经验规则是：若 N 小于 10K，r 建议不超过 16；若 N 大于 50K，可尝试 r=32 或 64。  
- **层间差异化**：注意力层（Q/K/V/O）的任务通常是调整语义对齐，需要更多自由度，可以容忍稍大 r；FFN 层主要存储事实知识，调整量较小，可以用更小的 r。  
- **α 与 r 的配合**：α 是 scaling factor，通常设为 r 的 2 倍，但也可以独立调节。增大 α 等效于放大更新量的 scale，但不增加参数，这提供了一种在固定 r 下调节有效学习率的手段。若验证集 loss 下降缓慢，可尝试增大 α 而非增大 r，以避免增加过拟合风险。  
- **实验验证**：我曾在 LLaMA-7B 上对同一个领域数据集（约 15K 条）分别做 r=4,8,16,32 的 SFT，发现 r=8 时验证 loss 最低，r=32 时 loss 小幅上升。同时，模型在 unseen 问题上的准确率也是 r=8 最优。

**可能追问与简短回答：**  
- **追问：LoRA 的秩 r 和全量微调的参数更新相比，为什么能避免过拟合？**  
  答：LoRA 通过低秩约束限制了权重更新的自由度，等效于一种正则化；而全量微调更新所有参数，自由度远大于 LoRA，在数据不足时更容易陷入假相关。  
- **追问：如果训练数据极大（百万级），是否应该使用更大的 r？**  
  答：理论上可以，但过大的 r 会失去 LoRA 节省参数的优势。百万级数据下，可以考虑全量微调或更大 r 的 LoRA，但实践中 r=64 通常已足够，因为模型权重本身已有很强的通用性。建议优先尝试全量微调，若资源受限则使用 LoRA 且 r 不宜超过 64。
