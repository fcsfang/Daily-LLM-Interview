**可信度：** 中（多来源交叉，但无官方文档直接支撑）

**考察点：** RAG 离线评测、指标设计、工程落地

**面试口头版回答：**  
评估 RAG 系统必须从检索和生成两个阶段分别量化。检索阶段我主要看 Hit Rate 和 MRR，Hit Rate 是 top-k 中是否包含正确答案，MRR 看正确答案在排序中的平均位置。生成阶段最核心的是 Faithfulness（忠实度），衡量回答是否严格基于检索到的文档，没有编造；其次是 Answer Relevance，看回答是否回答了用户问题。工程上我会用 RAGAS 框架来自动化计算这些指标，它提供了标准评分 prompt 和评估流程，每天在离线评测集上跑一次，把结果录入监控面板。对于生产环境，还需要在线统计用户反馈率、bad case 数量等。

**深入理解版回答：**  
RAG 系统评估的核心在于拆解管道与指标对应。第一阶段是检索质量评估，通常需要标注好的查询-答案对数据集。常用指标包括：  
- **Hit Rate (Recall@k)**：top-k 检索结果中是否包含相关文档。分子是包含相关文档的查询数，分母是总查询数。  
- **MRR (Mean Reciprocal Rank)**：第一个相关文档排位的倒数，平均所有查询。反映排序效果。  
- **NDCG@k**：考虑位置折扣累积增益，带相关性等级评分（如 0/1 或更多级），更适合排序任务。  
第二阶段是生成质量评估，分两点：  
- **Faithfulness（忠实度）**：通过 LLM-as-judge 判断回答每个声明是否在给定上下文中可得到支持。常用方法是让评判模型逐句判断是否被上下文支撑，计算支撑比例。  
- **Answer Relevance**：回答与问题的相关程度，同样用大模型打分，或计算回答语义嵌入与问题嵌入的余弦相似度（但不够准确）。  
- 另外可辅以 ROUGE-L、BLEU 等词覆盖指标，但语义等价场景下效果有限。  
工程落地时，推荐使用开源 RAGAS 框架。它内置了上述指标的评估 prompt，并提供了统一的评估 pipeline。我们需要提前构造一个有标准答案的评测集（约 200-500 条），离线运行后输出各指标分数。同时要监控评估一致性：可以用两份不同模型中任一个做评判，比较结果差异。若差异大，需要人工复审或使用更强模型（如 GPT-4）做裁决。还需要注意评估的成本：用 LLM-as-judge 也会消耗 token，建议先在小批量上验证 prompt 有效性再全量运行。

**常见追问与回答：**  
- **问：Hit Rate 和 MRR 的分子分母具体怎么定义？**  
  答：假设有 N 个查询，每个查询有 M 个相关文档。Hit Rate@k = (包含至少一个相关文档在前 k 的查询数) / N。MRR = (1/N) * Σ (1/第一个相关文档的排位)。  
- **问：Faithfulness 用 LLM 评估，怎么保证评判模型的准确性？**  
  答：我们一般用 GPT-4 或 Qwen2.5-72B 作为评判模型，并事先人工标注 50 条样本做校准，若评判模型与人工一致率低于 80%，则改进评判 prompt 或更换模型。还需要做自我一致性检验：重复评估同一批数据看结果稳定性。  
- **问：如果检索和生成评分都好，但用户仍不满意怎么办？**  
  答：说明离线指标与在线体验有 gap。需要分析 bad case，比如用户对格式不满意、缺少引用、涉及主观判断等。我还会加一个“用户隐式反馈”指标：用户是否继续提问、是否点击引用、是否在对话中表达负面情绪。

**项目联系：**  
在我参与的知识库问答项目中，我们基于 RAGAS 搭建了完整离线评测 pipeline。每周用新增的 100 条用户真实问题标注期望答案，跑一次全量评测。发现有一次 Hit Rate 从 0.85 骤降到 0.72，排查后是因为重新分块时使用了不同的 chunk 策略，导致许多长文档被截断丢失了关键短语。回滚旧策略并加 overlap 后指标恢复。

### 题目 2：你在实际项目中如何利用 LLM-as-judge 自动评估大模型输出的质量？请详细描述你的评估流程、prompt 设计与一致性校验策略。

**来源依据：**  
- 字节跳动面经中多次出现“用于评测的另一个大模型，如何构造问题、分析文字并进行对比？这个大模型的正确率以及模型自我一致性怎么确定？” – 2026大厂大模型/Agent面试题（AtomGit）  
  https://gitcode.csdn.net/6a236d7b662f9a54cb7a2bf1.html – 面经或社媒 – 需进一步验证  
- "这几道高频RAG面试题你都会了吗" – 腾讯云开发者社区  
  https://cloud.tencent.com/developer/article/2661823 – 技术博客 – 可信度中  
- 根据上述资料延伸，需进一步验证。

**可信度：** 中（面经直接提及，但缺乏官方文档佐证）

**考察点：** LLM-as-judge 设计、prompt 工程、评估一致性

**面试口头版回答：**  
我用 LLM-as-judge 自动评估模型输出时，会挑选一个更强的模型（如 GPT-4 或 Qwen2.5-72B）作为裁判。评估流程分三步：第一，定义评估维度，比如事实准确性、完整性、格式规范性；第二，为每个维度写打分 prompt，要求裁判先给出分析再打分，同时提供少数几个参考例子（few-shot）；第三，对同一批数据重复评估两次，检查裁判自身的一致性。如果一致性低，说明 prompt 歧义大，需要调整。工程上我还会控制输出格式为 JSON，方便后续统计。对于公平性问题，我会避免裁判模型与测试模型同源，降低偏见。

**深入理解版回答：**  
LLM-as-judge 是当前离线评测的趋势，但需要精心设计才能稳定可靠。整体流程：  
1. **评测集构建**：从业务场景中采集 500-1000 条输入，人工标注期望回答（golden answer）或仅需答案可以（用于评估事实性）。每条输入应附带上下文文档（如果是 RAG 场景）。  
2. **评判维度定义**：通常包括：  
   - **Correctness**：答案是否正确（需要 golden answer）。  
   - **Faithfulness**：答案是否基于给定上下文。  
   - **Completeness**：是否覆盖问题所有方面。  
   - **Style/Format**：是否符合输出格式要求。  
3. **评判 prompt 设计**：  
   ```
   你是一个公正的评估助理。下面是一个用户问题、参考上下文以及模型生成的答案。
   请从以下维度打分（1-5分）：事实准确性、忠实度、完整性。
   在给出分数前，先逐条列出你的分析依据。
   输出格式：{"analysis": "...", "scores": {"accuracy": 4, "faithfulness": 5, "completeness": 3}}
   用户问题：{question}
   上下文：{context}
   模型答案：{answer}
   ```
   同时建议提供 1-2 个 few-shot 示例，让模型理解评分标准。  
4. **一致性校验**：  
   - **同一裁判复跑**：将同一批数据对调顺序或稍改 prompt 后重复评估，计算分数间 Pearson 相关系数，应 >0.8。  
   - **不同裁判对比**：用两个不同模型（如 GPT-4 与 Qwen2.5-72B）分别评分，看一致性。若差异大，需要人工做 tie-breaking。  
   - **校准**：随机抽取 50 条结果由人类专家评价，比较人类与裁判的一致率（完美一致率或允许 1 分误差的一致率）。若低于 70%，需要优化 prompt 或更换裁判模型。  
5. **工程化**：将评估流程封装成异步任务，使用缓存避免重复调用同一 prompt 不同数据点。对批量评估结果生成可视化报表，如各维度分数分布、bad case 列表。

**常见追问与回答：**  
- **问：如果裁判模型本身就比测试模型差，评分可信吗？**  
  答：所以必须选更强或同级别的模型，且经过校准。也可以尝试用多个模型投票，但会增加成本。  
- **问：如何处理裁判对长文本的偏见（比如更喜欢长的答案）？**  
  答：在 prompt 中明确禁止“长度偏好”，并在 few-shot 中展示短而准的答案得高分的情况。还可以在评估前对答案做脱敏处理（脱去长度信息）。  
- **问：能不能用开源小模型做裁判？**  
  答：可以，但需要仔细校准。例如用 Qwen2.5-7B 做裁判时，我们发现在事实性维度上其一致率比 GPT-4 低约 15%。如果资源有限，可以先在小规模验证，确认可接受再放大。

**项目联系：**  
在项目中我们有一个“智能客服评测”模块，每天新部署的模型版本都会自动用 LLM-as-judge 进行回归测试。有一次发现新版模型事实得分从 4.6 降到 4.1，排查发现是因为升级后模型对某些常见问题的回答不再引用文档，而是凭记忆作答。我们立即回滚并改进了 RAG 提示中的强制引用指令。

**记忆关键词：**

### 题目 3：在评估 Agent 时，你如何设计测试用例来衡量其工具调用准确性、计划合理性和恢复能力？

**来源依据：**  
- 字节跳动-Agent开发实习生面试题中出现大量 Agent 评估相关追问，如“模型是怎么选择调用哪个工具的？”“限制工具调用怎么做？”“有没有遇到agent不适配其他手机机型的情况？怎么处理的？” – 2026大厂大模型/Agent面试题（AtomGit）  
  https://gitcode.csdn.net/6a236d7b662f9a54cb7a2bf1.html – 面经或社媒 – 需进一步验证  
- 根据上述资料延伸，需进一步验证。

**可信度：** 中（面经来源，但问题本身具有普遍性）

**考察点：** Agent 评估、测试用例设计、错误恢复

**面试口头版回答：**  
评估 Agent 比评估纯 RAG 复杂，因为它涉及多步推理和工具调用。我通常把 Agent 评估拆成三个维度：工具调用准确性、计划合理性、错误恢复能力。工具调用准确性就检视 Agent 是否在正确的时机选择了正确的 tool，且参数格式正确。计划合理性看它分解任务时步骤是否合理、是否有冗余或逻辑跳跃。错误恢复能力模拟工具超时、返回空结果等场景，看 Agent 能否优雅降级。测试用例我会设计单元测试（单步 tool 选择）、集成测试（完整流程）和压力测试（并发、复杂）。对每个 case 我用人工标注期望轨迹，再用自动化脚本对比实际轨迹与期望轨迹的匹配度。

**深入理解版回答：**  
Agent 评估需要构建“轨迹”（trajectory）级别的测试集。具体做法：  
1. **定义评估指标**：  
   - **工具调用准确性**：针对每个步骤，检查调用的工具名称、参数格式、返回值处理是否正确。可以计算工具选择准确率（Tool Accuracy）和参数正确率（Parameter Correctness）。  
   - **计划合理性**：步骤顺序是否合理、是否有不必要的步骤、是否过早终止或陷入循环。可以用步骤级编辑距离（Step-Level Edit Distance）来衡量。  
   - **错误恢复能力**：注入工具超时（模拟 5 秒返回异常）、空结果、恶意返回值。看 Agent 是否重试、更换工具、或合理告知用户。  
2. **测试用例设计**：  
   - **单元测试**：给定一个明确需要调用某个工具的场景，如“查询天气”，期望 Agent 调用 `get_weather(city)`。可用 python 脚本自动注入断言。  
   - **集成测试**：设计多步任务，如“预定出差行程”：先查航班、再查酒店、然后计算总费用。期望轨迹是顺序调用对应工具。  
   - **异常注入**：通过 mock 工具层，让工具随机返回错误，检查 Agent 行为。  
3. **评估框架**：可以使用 LangSmith 或自定义脚本。每条测试案例包含：输入、期望轨迹（步骤列表）、可接受替代轨迹集（允许多种合理路径）。评估结果用轨迹匹配率（Trajectory F1）和成功率（Task Success Rate）报告。  
4. **人工抽检**：自动化指标不能覆盖所有质量细节，如回答的友好性、合规性。每周随机抽 30 条轨迹人工标注，与自动评分对比校准。

**常见追问与回答：**  
- **问：如果 Agent 用不同的计划达到了同样好的结果，怎么处理？**  
  答：我们允许多条优秀轨迹，定义“轨迹等价类”，收录在期望轨迹列表中。匹配时只要落在等价类内就算正确。  
- **问：怎么设计压力测试？**  
  答：并发发送同一任务，检查 Agent 是否状态混肴或上下文串号。也可以发送超长历史记录，看 agent 是否丢失信息或超时。  
- **问：如何处理 Agent 不调用工具浪费时间的情况？**  
  答：在评估指标中加入“步骤效率”，比如最少步骤任务的期望步骤数，超过则扣分。

**项目联系：**  
在开发一个客服 Agent 时，我们设计了 200 个测试用例，覆盖了常见的工具调用和异常场景。后来发现工具参数错误率达到 12%，通过改进 prompt 中工具描述格式并加入 few-shot 示例，错误率降到 3%。

### 题目 4：请描述一个你在 RAG 系统中跟踪和处理 bad case 的完整流程，从发现到定位根因再到修复验证。

**来源依据：**  
- "这几道高频RAG面试题你都会了吗" 中专门提到“内容缺失”问题的排查套路（切块策略→召回方式→精排兜底）  
  https://cloud.tencent.com/developer/article/2661823 – 技术博客 – 可信度中  
- 结合 RAG 系统常见问题总结，需进一步验证。

**可信度：** 中（技术博客提供实操思路）

**考察点：** bad case 分析、系统化排查、工程落地

**面试口头版回答：**  
我处理 bad case 的流程是“发现-定位-修复-验证”四步。第一步发现：通过用户反馈、日志搜索和离线评测中的低分样本收集 bad case。第二步定位：我按层排查——先看检索阶段，数据集里是否包含正确答案？Hit Rate 多少？如果不包含，可能是切块或 embedding 问题。如果包含但模型答错，再看生成阶段：上下文是否被正确传入？prompt 是否让模型忠实于上下文？第三步修复：根据根因采取对应措施，比如优化切块策略、加混合检索、改 prompt。第四步验证：用新 bad case 测试修复效果，同时回归整个评测集，确保没有引入新问题。

**深入理解版回答：**  
一个系统的 bad case 分析 pipeline 应该具有工程化和可复用性：  
- **发现阶段**：  
  - 在线监控面板标记用户反馈“回答不满意”的条目。  
  - 离线评测每隔一段时间会产生大量低分 case，自动归入 bad case 池。  
  - 定期从日志中随机采样 5% 的问答对进行人工审查。  
- **定位阶段**：  
  1. **检索溯源**：从日志中提取检索到的 chunk ID 列表。检查正确答案是否在其中（需要先构建 golden chunk 标注）。  
  2. **上下文质量**：看传入 prompt 的 top-k chunk 是否包含明显噪音？是否有重复内容？  
  3. **模型行为**：看模型是否忽略了上下文中的正确答案（常见于长篇上下文时“lost in the middle”）；或者模型是否正确引用了上下文。  
  4. **工具调用**：对于 Agent 增强的 RAG，还要检查 tool 调用是否出错、参数错误等。  
  常用工具：我们可以将每个 bad case 的完整请求-响应细节记录到一个 JSON 文件中，包括用户 query、检索结果、最终 prompt（含 context chunk）、模型输出。这样可以通过写脚本批量分析，例如统计哪些 chunk 被检索但没被使用。  
- **修复阶段**：  
  - 若检索丢失：调整 chunk 策略（语义切块或重叠百分数）、换 embedding 模型、添加混合检索（BM25+向量）。  
  - 若模型忽略上下文：重排序加强（将最相关 chunk 放最前）、prompt 中强调“必须基于下面内容回答”。  
  - 若模型不忠实：使用更具体的约束 prompt，或加入 citation 要求。  
- **验证阶段**：对修复后的系统，先跑之前收集的 bad case 确认不再复现，然后用完整评测集做回归。通常还需要人工抽检 20-30 条，确保修复没有引入新问题。

**常见追问与回答：**  
- **问：如果 bug 是偶发的，比如只有某类问题出现，怎么定位？**  
  答：我会对 bad case 聚类，按 query 类型（如日期、实体）分组，看是否某一类集中出问题。然后针对该类构造更多测试用例复现。  
- **问：如果修复后某些 case 变好了，另一些原来的好 case 变差了，怎么处理？**  
  答：说明修复引入 trade-off。需要结合业务优先级，在发布前做 A/B 测试，对比新旧版本在整体指标上的差异，如果只是少量 case 变差且影响不大，可以接受。或者考虑用模型路由：根据不同问题类型采用不同处理策略。  
- **问：有没有用过自动化的 bad case 分析工具？**  
  答：LangSmith 提供了 trace 浏览器和排序功能。我们自己也写了一个简单的工具：用 streamlit 展示 bad case 详情，支持按指标过滤，链接到检索 chunk 原文。

**项目联系：**  
之前业务反馈“产品价格咨询”回答经常出现错误。我通过 bad case 分析发现：原因是产品价格文档中价格是表格形式，默认 chunck 切块把表格切碎，导致检索不到完整价格。修复方法：使用表格感知切块（保留表格整体 chunk），并添加价格 FAQ 问答对到知识库。修复后该场景准确率从 72% 提升到 96%。

### 题目 5：你如何设计一个针对模型幻觉（Hallucination）的离线评测方案？请详细说明数据集构建、评测指标和阈值设置。

**来源依据：**  
- "美团面试：LLM大模型存在哪些问题？RAG 优化有哪些方法？" 中提到幻觉的定义和缓解 – 博客园  
  https://www.cnblogs.com/crazymakercircle/p/18841145 – 技术博客 – 可信度中  
- 微软等公司关于 hallucination eval 的公开资料，需进一步验证。

**可信度：** 中（博客内容，但概念通用）

**考察点：** 幻觉评测、数据集构建、指标设计

**面试口头版回答：**  
要离线评测幻觉，我首先构造一个包含挑战性样本的评测集：包括需要精确事实的查询、需要多步推理的查询、以及故意让模型超出上下文才能答的查询（诱骗幻觉）。评测指标我采用“支持率”（Faithfulness score）：用 LLM-as-judge 逐个判定每个声明是否在上下文中可被支持。公式是：支持率 = 被支持的声明数 / 总声明数。如果模型生成的答案没有任何声明可以从上下文中找到，说明产生了无根据幻觉。还会看“矛盾率”（Contradiction rate）：是否生成与上下文冲突的声明。阈值方面，如果支持率低于 0.8，我会标记为严重幻觉，需要回滚模型或调整 prompt。

**深入理解版回答：**  
幻觉离线评测需要定量的 framework。具体做法：  
1. **构建评测集**：  
   - 选取 300-500 个 pairs，包含：  
     a) 事实性查询（需要精确数字、人名等）。  
     b) 上下文故意留白（如“关于XX，文档中没有提到任何信息”），看模型是否强行回答。  
     c) 上下文包含模糊或矛盾信息，看模型是否能在矛盾中做出合理解释。  
   - 每个 pair 需要人工标注预期的忠实评分（或直接标注是否 hallucinate）。  
2. **评测指标**：  
   - **Faithfulness 分数**：利用 LLM-as-judge 将答案拆解为原子声明，然后逐个检查是否可被上下文支持。计算方法：支持声明数 / 总声明数。  
   - **Hallucination 率**：完全无支持的答案比例（即支持声明数为 0 的答案数 / 总数）。  
   - **Contradiction 率**：与上下文明显矛盾的声明个数 / 总声明数。  
   - **Intrinsic Hallucination（内部幻觉）**：答案包含事实但不来自上下文（但上下文可能有其他相关内容）。  
3. **阈值设置**：  
   - 根据业务容忍度，例如金融服务中，需要 Faithfulness > 0.95，Hallucination 率 < 2%，否则不能上线。  
   - 还可以设立“严重幻觉”阈值：生成声明中有 3 个以上无支持且涉及关键实体，直接告警。  
4. **评测自动化**：  
   - 使用开源工具如 `LangChain's Hallucination Grades` 或 `RAGAS Faithfulness`。  
   - 定期（每次模型更新、prompt 修改）运行。  
   - 结果用看板展示趋势，结合历史数据做报警。

**常见追问与回答：**  
- **问：LLM-as-judge 本身也会产生幻觉，怎么避免误判？**  
  答：我们会对评判模型做校准：人工标注 50 条 case 的忠实度，与 LLM 评分对比。如果一致率低于 75%，则优化评判 prompt。还可以要求评判模型输出每步分析，便于人工审计。  
- **问：如果模型因为上下文不足而合理回答“不知道”，这算幻觉吗？**  
  答：不算，这是合理的。所以评测时我们要区分“无根据断言”和“承认不知道”。在声明提取时，排除模型表达不确定性的句子。  
- **问：如何处理模型生成的答案中有部分支持、部分幻觉的情况？**  
  答：需要子声明级别的评估。例如一个答案有两条声明，一条支持，一条无支持，则认定该答案有局部幻觉。可以计算“部分幻觉率”辅助分析。

**项目联系：**  
在金融问答项目中，我们上线前需要提交一个“幻觉风险评估报告”，我搭建的评测 pipeline 自动产出 Faithfulness 分数和 Hallucination 率。在一次 prompt 调整后，Hallucination 率从 1.5% 升到 4.2%，我们立即回滚，并发现原因是新 prompt 减少了“如果不确定请说不知道”的约束。

**记忆关键词：**  
#幻觉评测 #Faithfulness #Hallucination率 #声明分解 #校准

### 卡片 1
- **Q：** RAG 离线评测中，Hit Rate@k 和 MRR 分别怎么计算？  
- **A：** Hit Rate@k = 前 k 个结果中包含至少一个相关文档的查询数 / 总查询数。MRR = 所有查询第一个相关文档排位的倒数之和 / 总查询数。前者关注“是否召回”，后者关注“排位质量”。  
- **一句话记忆法：** Hit 问“有没有”，MRR 问“排第几”。  
- **复习标签：** #RAG #离线评测 #指标

### 卡片 2
- **Q：** 如何用 LLM-as-judge 评估 Faithfulness（忠实度）？  
- **A：** 将模型答案拆分为原子声明，让评判模型逐一判断每个声明是否被给定上下文支持，支持比例即为忠实度分数。评判 prompt 需包含任务描述、few-shot 示例和严格格式要求。  
- **一句话记忆法：** 先拆声明，再问裁判“上下文支持不支持”。  
- **复习标签：** #LLMasJudge #评测 #忠实度

### 卡片 3
- **Q：** bad case 分析的“三层排查”是哪三层？  
- **A：** 第一层检索：检查正确答案是否在检索结果中（Hit Rate）。第二层上下文：检查传入 prompt 的 chunk 是否有噪音或位置不佳（lost in the middle）。第三层生成：检查模型是否忽略上下文、是否强迫自己回答。根据问题出在哪一层采取对应修复。  
- **一句话记忆法：** 先查查没查对，再查给没给准，最后查说没说错。  
- **复习标签：** #bad_case #RAG #根因分析 （新角度）

### 卡片 4
- **Q：** Agent 评估中的轨迹匹配（Trajectory Matching）是什么意思？  
- **A：** 将 Agent 实际执行过程的工具调用序列（步骤列表）与期望轨迹进行比较。允许一定容忍度：只要落在“可接受轨迹集”内就算正确。常用指标为轨迹 F1 和任务成功率（Task Success Rate）。  
- **一句话记忆法：** 看 Agent 走的路线对不对，而非只看终点。  
- **复习标签：** #Agent评估 #轨迹 #任务成功率 （新角度）

### 卡片 5
- **Q：** 为什么离线评测指标好但线上用户不满意？  
- **A：** 离线指标（如 ROUGE、Faithfulness）侧重内容正确性，忽略了交互体验，如格式、响应速度、用户期望管理等。需要结合在线指标实现闭环：如用户满意度调查、对话长度、重查率等。对低位 bad case 比对分数更具洞察。  
- **一句话记忆法：** 考试考得好不代表客户喜欢，还需要看客户心情和体验。  
- **复习标签：** #评测 #线上vs线下 #用户满意度 （新角度）

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 当使用 LLM-as-judge 进行自动化评测时，如何解决裁判模型本身的“顽固化”（stereotype）或“偏见偏好”问题？请从 prompt 设计、模型选择和结果验证三个角度回答。

**面试口头版参考答案：**  
我会从三个维度缓解偏见。首先，在 prompt 中明确要求只基于内容评分，避免对语言风格、长度、结构产生偏好；可以加入反例说明：例如“一篇虽然短但准确的答案应得高分，而长而跑题的答案应给低分”。其次，选择裁判模型时尽量选择与测试模型不同系列（如用 GPT-4 评估 Qwen，避免两者同源），并定期用人工标注样本校准裁判模型。最后，对裁判结果做双重验证：跑两次相同的评估计算自我一致性，或在有争议的 case 上用第二个裁判模型做比较。

**深入理解版参考答案：**  
LLM-as-judge 的偏见问题是一个重要且尚未完全解决的课题。常见的偏见包括：  
- **长度偏见**：模型倾向于给长答案更高分，即使内容质量不高。  
- **自夸偏见**：裁判模型更认可与自己训练数据风格一致的答案（如 GPT-4 更偏向格式化的回答）。  
- **顺序偏见**：若在 prompt 中先看到某答案后看到标准答案，裁判可能受影响。  

解决方案：  
1. **Prompt 设计**：  
   - 使用“匿名化”输入：把模型名字或来源去掉。  
   - 在评分标准中明确“长度不重要，准确性最重要”。  
   - 使用 Chain-of-Thought 评分，让模型先分析再打分，减少直觉性偏见。  
   - 平衡答案顺序：对于 pairwise comparison 的情况，可以随机交换顺序取两次评分的平均。  
2. **模型选择**：  
   - 使用大参数模型（如 GPT-4、Claude3.5）做裁判，偏见相对较小。  
   - 避免使用同一公司模型评估自身（如用 Llama3 评估 Llama2 可能偏袒同系列）。  
   - 如果资源允许，用多个裁判模型投票（“三人成虎”策略）。  
3. **结果验证**：  
   - 定期做人工抽检，计算裁判与人类的一致率。如果发现某个维度（如风格）的偏差高，则调整 prompt 或剔除该维度。  
   - 对裁判的评分做统计分析：比如计算评分与答案长度的相关系数，若大于 0.5，说明有长偏，需纠正。  
   - 运行对抗性测试：故意输入一些极短但正确的答案，看裁判打分是否合理。  

**可能追问与回答：**  
- **问：如果发现裁判模型有系统性偏差，你如何纠正？**  
  答：首先尝试在 prompt 中加入更详细的“禁忌”说明，如“请避免因为答案长度影响评分”。其次可以引入一个偏差修正模型：训练一个轻量级回归器，以裁判原始分数和答案长度等特征为输入，输出矫正后的分数。最后，如果问题严重，只能更换裁判模型或回归人工评估。  
- **问：有没有开源工具可以帮助检测和缓解这些偏见？**  
  答：有一些研究提出过的框架如 `SelfCheckGPT` 和 `LLM-Eval`，但我不确定成熟度。我会建议使用 `lm-evaluation-harness` 的基础方法，自己构建校准数据集。具体来说，我准备 100 条人类评分好的 case，再让裁判打分，对比分布，如果偏向明显就需要调整 prompt。  

**记忆关键词：**  
#LLMasJudge #偏见 #prompt设计 #校准 #对抗性测试
