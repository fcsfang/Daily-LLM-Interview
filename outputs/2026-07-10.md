## 1. 今日面试真题精讲

### 题目 1：SGLang 的 RadixAttention 和 vLLM 的 PagedAttention 有什么区别？为什么 SGLang 能实现更高效的前缀复用？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 转载内容，需进一步验证
- 大模型面试题 | 小林面试笔记 - https://xiaolinnote.com/ai - 技术博客 - 整理型内容，需进一步验证

**可信度：** 中（面经搬运 + 整理型博客）

**考察点：** 推理框架、前缀缓存、显存管理、batch 调度

**面试口头版回答：**

PagedAttention 和 RadixAttention 都是为了解决推理时显存浪费和前缀复用问题。PagedAttention 的思路是把 KV cache 切成固定大小的“页”，像操作系统虚拟内存一样按需分配，减少了显存碎片，但前缀复用时每次还是要从第一页开始计算索引。RadixAttention 的核心是把 KV cache 组织成一颗 Trie 树（前缀树）。想象一个场景：用户和大模型对话，问题开头都是“帮我分析一下我们公司上一季度的销售数据，包括华东、华南……”，不同用户的前缀部分完全相同。RadixAttention 能够自动识别并共享这部分公共前缀的 KV cache，避免重复计算。简单说：PagedAttention 解决了“显存不够存所有请求的 KV cache”的问题，RadixAttention 在之上又解决了“相同前缀重复计算”的问题，所以 SGLang 在处理多轮对话和批量推理时延迟更低。

**深入理解版回答：**

首先，我们要理解为什么需要这两种技术。推理时，每生成一个新 token，模型都需要把之前所有 token 的 K 和 V 值存下来（这就是 KV cache）。如果并发请求多，KV cache 会吃掉大量显存。vLLM 的 PagedAttention 借鉴了操作系统的分页思路。它不提前为一个请求分配最大的显存（比如 max_seq_len * hidden_size * 2 * num_layers），而是按需分配固定大小的块（比如每个块存放 16 个 token 的 KV 值）。多个请求的块可以物理上不连续，逻辑上通过块表连接。这样做的好处是：显存浪费从“预先分配但没用到的部分”降低到了“块内最后一个 token 可能用不完”的程度。但 PagedAttention 有一个限制：它认为每个请求的 KV cache 是独立的。如果两个请求共享相同的前缀（比如系统提示词、多轮对话历史的开头），PagedAttention 会各自存一份，导致重复计算和显存浪费。

SGLang 的 RadixAttention 针对这个问题做了优化。它用一颗 Trie 树来组织所有请求的 KV cache。Trie 树的每个节点代表一个 token 序列的前缀。当新的请求到来时，SGLang 会先在 Trie 树上匹配最长公共前缀，然后只对差异部分（后缀）重新计算 KV cache。匹配到的前缀直接复用。这种设计对 Chat 场景非常有效：每个用户请求的开头都是同样的系统提示词和几轮历史对话，RadixAttention 可以让所有用户共享这部分，显存占用和计算量大幅降低。代价是 Trie 树的维护和查找需要额外开销，对于完全没有前缀共享的独立请求（例如随机 query），SGLang 的额外开销会带来一些性能损失。

在面试中，可以进一步补充：两者都能配合 continuous batching 使用，但 RadixAttention 在前缀共享场景下的吞吐量提升更明显。如果你要在面试中展示深度，可以说出：vLLM 的 PagedAttention 更适合独立请求的 Open Domain QA 场景；SGLang 的 RadixAttention 更适合多轮对话、Agent 系统中有共享 System Prompt 的场景。

**常见追问与回答：**

追问 1：如果用户请求的前缀完全不重复，SGLang 的 RadixAttention 会比 vLLM 差吗？
回答：理论上会差一些，因为维护 Trie 树和匹配需要额外的时间开销。但实际测试中，因为大部分生产场景都有系统提示词，所以 prefix 命中率很高。即使没有命中，SGLang 也能退化为常规的 KV cache 管理，性能差距通常在 5% 以内。

追问 2：RadixAttention 的 Trie 树会不会无限增长？内存怎么释放？
回答：会。通常做法是 LRU 淘汰：当显存不足时，淘汰最近最少使用的 Trie 节点（也就是对应的 KV cache 块）。SGLang 内部有一个引用计数机制，当一个节点没有被任何活跃请求引用时，就可以被标记为可淘汰。

追问 3：这两种技术对推理延迟的影响有多大？能给出一个数量级吗？
回答：这取决于场景。在纯独立请求场景下，两者延迟差异不大，主要看 batch 大小。但在多轮对话中（比如 8 个用户，20 轮对话），RadixAttention 可以把 prefix 部分的计算开销归零，延迟通常会降低 30%-60%，显存节省 20%-40%。建议面试时说“具体要看 benchmark，但 prefix 越长优势越明显”。

**项目联系：**

在做 Agent 项目时，我实现了一个多轮对话的 Agent 系统。每个 Agent 实例在调用 LLM 时，prompt 里包括系统提示词 + 四轮历史对话。如果直接用普通的推理框架，每个请求都要从头计算 KV cache。经调研后，我了解到 SGLang 的 RadixAttention 非常适合这个场景。尽管我没有在生产环境切换 SGLang，但在我的个人项目中，我模拟了一个简单的 KV cache 共享策略：在内存中维护一个 LRU 字典，key 是 prompt 的前 2000 token 的 hash，value 是对应的 KV cache。新请求来临时，先计算 hash，如果命中就直接拼接上后续的 KV cache 计算。虽然比起 RadixAttention 粗粒度很多，但也让我的模型推理延迟降低了约 20%。面试中我会说“这让我更深入理解了前缀复用的工程实现”。

**记忆关键词：**
PagedAttention, RadixAttention, KV cache, Trie 树, 前缀共享, 显存碎片

### 题目 2：你如何为一个模型做量化？量化到 4bit 或者 8bit 后，对推理的延迟和显存分别有什么影响？如果模型量化后精度下降，你会怎么排查和恢复？

**来源依据：**
- 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 字节大模型Rag算法面试题汇总 - https://www.nowcoder.com/feed/main/detail/848a3edd6f1c404b8565318231b9332d - 面经或社媒 - 需进一步验证
- 大模型面试题 | 小林面试笔记 - https://xiaolinnote.com/ai - 技术博客 - 需进一步验证

**可信度：** 中

**考察点：** 量化原理、部署优化、精度回退、工程排查思路

**面试口头版回答：**

模型量化就是把模型的权重和激活值从高精度（比如 FP16）映射到低精度（INT8、INT4），来减少显存占用并加速计算。如果要做量化，我会先用 GPTQ 或者 AWQ 来做权重量化。GPTQ 是一种训练后量化，它不需要重新训练模型，而是在一个小的校准数据集上，通过最小化量化前后的输出差异来确定每个权重矩阵的最优量化参数。如果量化为 4 bit，模型显存占用会降低到原来的 1/4 左右，显存带宽不够导致的延迟瓶颈也会显著改善。具体来说，推理 7B 模型用 FP16 大概需要 14GB 显存，INT4 只需要 4GB 左右。延迟方面，对生成瓶颈（显存带宽瓶颈）的任务，量化后 token 生成速度可以提升 2-4 倍。当然，代价是精度损失。如果量化后精度下降，我的排查思路是：先回归校准数据集，看是“全局精度下降”还是“个别样本变差”；再检查哪个层量化误差最大，尝试对该层保留 FP16；最后，如果依然不够，就换成 8bit 量化或者用 QAT（量化感知训练）微调一下。

**深入理解版回答：**

量化分为权重量化（Weight-only Quantization）和权重+激活量化（Weight-Activation Quantization）。对于大模型推理，当前最成熟的是权重量化（W8A16 或 W4A16），就是说权重被量化到 8bit 或 4bit，但激活值（计算过程中的中间量）保持 FP16。这样显存减少效果明显（4bit 权重占用只有 FP16 的 1/4），而且因为计算时权重需要从显存取到计算单元，权重变小后 IO 瓶颈也会缓解。

常用的量化方法包括：
1. **GPTQ (GPT Quantization)**：在少量校准数据上优化量化参数，逐层对权重矩阵做量化。它把量化问题建模成一个逐层的二阶优化问题，效果通常优于简单的 Round-to-Nearest 量化。适合对质量要求高的场景。
2. **AWQ (Activation-aware Weight Quantization)**：观察到一小部分“重要”的权重通道对模型输出影响大，所以对不同的通道做不同的量化缩放（保留重要通道的精度）。AWQ 实现简单，不需要二阶优化，效果和 GPTQ 接近。
3. **NF4 (Normal Float 4)**：QLoRA 中使用的 4bit 量化。它不是均匀量化，而是利用正态分布的分位数来做量化点，更适合预训练模型权重通常服从近似正态分布的实际情况。

量化到 INT4 的影响：
- **显存**：以 LLaMA-2 7B 为例，FP16 部署需要约 14GB（权重 14GB + KV cache 等额外开销）。INT4 权重约 3.5GB，加上 KV cache 和中间激活，整体显存下降到 6-8GB。这意味着单张 24GB 的 4090 可以轻松部署 7B 甚至 13B 模型；单张 A100-80G 可以部署 70B 模型（在 INT4 下约 40GB 权重 + 额外开销）。
- **延迟**：量化主要优化显存带宽瓶颈。因为每个 token 产生时，模型都要从显存读取所有参数做矩阵乘法。参数变小（4bit vs 16bit），读取时间减少，所以 token 生成速度提升明显。具体来说：在 7B 模型上，INT4 量化后生成速度约为 FP16 的 1.5-2.5 倍。在 70B 模型上，因为 IO 瓶颈更严重，提升更大。
- **精度**：INT4 量化的精度损失通常是可接受的。以 MMLU 为例，LLaMA-2 7B FP16 精度约 46%，INT4 (GPTQ) 约 45.2%，损失在 1% 以内。但对于复杂推理或生成任务（代码、数学），损失可能更大。

如果精度下降明显，排查步骤应该是：
1. **确认基线**：先上量化前 FP16 的结果，确认是量化导致的退化还是模型本身有问题。
2. **校准集检查**：GPTQ 等依赖一个校准集，校准集不符合模型训练分布会导致量化误差大。换一个校准集或组合多个数据集试试。
3. **逐层检查**：用工具（比如 AutoGPTQ 的对比模式）或手动测试，看哪个 Transformer 层的输出在经过量化后误差最大。误差大的层可以保留 FP16（混合精度量化）。
4. **换成更温和的量化方案**：比如从 4bit 退回到 8bit（GPTQ 的 8bit），或者用 AWQ 代替 GPTQ。
5. **终极手段：QAT**：如果以上都不行，就做量化感知训练，让模型权重在学习过程中适应量化。不过这需要训练数据和时间，代价较大。

**常见追问与回答：**

追问 1：AWQ 和 GPTQ 哪个更好？
回答：两者定位不同。AWQ 的优势是简单且速度快，不需要大量校准数据和复杂的 Hessian 计算，适合快速部署。GPTQ 对精度更可控，因为它用二阶优化，通常精度优于 AWQ，但耗时更长。实际项目中，如果部署时间紧迫，我会优先跑 AWQ 看结果，如果精度不满足再换 GPTQ。

追问 2：量化到 4bit 后，模型还有必要用 FP16 做 attention 计算吗？
回答：有用。现在的常见做法是 W4A16：权重是 INT4，但计算时动态解量化为 FP16 做矩阵乘法，attention 部分仍然是 FP16。如果进一步把激活值也量化为 INT8，就是 W4A8，显存和计算更快，但对量化的误差更敏感，精度损失更大。

追问 3：如果显存特别小（比如 8GB 显存的旧卡），量化到 4bit 也放不下 7B，怎么办？
回答：可以用“offload”技术，比如把不常用的层卸载到 CPU 内存，需要计算时才加载回来。或者用模型并行/流水线并行，把模型切分到多张卡。也可以用更激进的量化，如 2bit 或 1.5bit，但精度损失会很大。实际工程中，8GB 显存建议直接上 1.5B-3B 的小模型加 RAG，而不是死磕大模型。

**项目联系：**

我在准备一个部署项目时，需要把 LLaMA-2 7B 部署到一张 16GB T4 卡上做推理。直接加载 FP16 模型会 OOM。我做了一套量化流程：先用 GPTQ 量化到 4bit，校准集从我的 RAG 知识库中随机采样了 128 条 document，确保校准集和实际业务分布一致。量化后权重只有 4GB，加上 KV cache 和 overhead，总显存约 7GB，完全放下。然后我用 MMLU 测试了精度，从 46% 降到 44.8%，下降了 1.2%，在可接受范围。为了进一步优化，我排查了误差最大的层，发现是最后几层 transformer layer 的 FFN 矩阵量化误差较大，于是我把最后 2 层的 FP16 保留，这样显存增加到 8GB，但精度恢复到 45.5%，只比基线下降 0.5%。面试中我会说“这让我学会了量化不是非黑即白，混合精度量化是工程中很实用的技巧”。

**记忆关键词：**
GPTQ, AWQ, NF4, W4A16, 混合精度量化, 校准集

### 题目 3：为什么大模型推理时 KV cache 能大幅提升速度？它的内存占用怎么估算？如果序列特别长（比如 32K 上下文），KV cache 会变得多大，你怎么优化？

**来源依据：**
- AgentGuide/docs/04-interview/12-company-interview-cases.md at main · adongwanai/AgentGuide · GitHub - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 技术博客 - 需进一步验证
- 大模型面试题 | 小林面试笔记 - https://xiaolinnote.com/ai - 技术博客 - 需进一步验证

**可信度：** 中

**考察点：** KV cache 原理、显存计算、长序列优化

**面试口头版回答：**

KV cache 是推理过程中每生成一个 token 时，之前所有 token 的 Key 和 Value 向量。在自回归生成中，预测下一个 token 时，模型需要计算当前 token 与之前所有 token 的注意力。如果不缓存，每次都要重新计算整个序列的 K 和 V，计算量随序列长度平方增长。KV cache 缓存后，每次计算只需要新 token 的 K 和 V，然后和新 token 的 Q 做注意力计算即可，避免了重复计算。它的内存占用公式是：batch_size * seq_len * num_layers * hidden_size * 2 * memory_per_elem（FP16 是 2 bytes）。以 LLaMA-2 7B (32 layers, 4096 hidden) 为例，batch=1，seq_len=4096 时，KV cache 占用：1 * 4096 * 32 * 4096 * 2 * 2 = 2.1GB。32K 序列时，这个值会膨胀到约 17GB —— 比模型权重本身还大。优化方案包括 MQA/GQA（分组查询注意力，减少 K/V head 数量）、PagedAttention（按需分配，减少碎片）、KV cache 量化（INT8 或 4bit）、以及窗口注意力（只存最近 N 个 token）。

**深入理解版回答：**

**KV Cache 为什么能提升速度？**

Transformer 自回归生成是逐 token 生成的。在第 t 步生成第 t 个 token 时，注意力计算需要计算当前 token 的 query 与前面所有 token 的 key 之间的相似度。如果每步都重新计算第 1 到第 t-1 个 token 的 K 和 V，复杂度是 O(t^2)（因为有 t 个 token，每步要计算 t 次注意力）。KV cache 解决了这个问题：在第 1 步时，计算 token1 的 K1、V1，并缓存起来。第 2 步时，只需要计算 token2 的 K2、V2，然后用 K2、V2 加上缓存的 K1、V1 一起做注意力。这样第 t 步的复杂度降为 O(t)，总共 O(t^2) 的复杂度被优化为总 O(t^2) 拆成每一步 O(t) 的累积，但实际上因为 K和V的重复计算被消除了，提速非常显著。

**KV cache 的内存估算：**

KV cache 占用的显存 = batch_size * num_layers * 2 * seq_len * hidden_size * bytes_per_elem。

以 LLaMA 7B 为例：
- num_layers = 32
- hidden_size = 4096
- 每层有 2 组矩阵：K 和 V，所以乘以 2
- seq_len = 4096
- bytes_per_elem（FP16）= 2

所以 1 个请求：1 * 32 * 2 * 4096 * 4096 * 2 = 2.1 GB。

当 seq_len = 32K 时，KV cache 为 1 * 32 * 2 * 32768 * 4096 * 2 = 17.2 GB。这意味着一个 32K 上下文请求的单 KV cache 就比 7B 模型权重（约 14GB）还大，两个这样的请求就能撑爆一张 80GB A100。

**优化方案：**

1. **GQA / MQA**：Multi-Query Attention 和 Grouped Query Attention。MQA 中所有注意力头共享相同的 K 和 V，只保留不同的 Q 头。KV cache 显存可降低到原来的 1/num_heads。GQA 是折中方案，把 Q 头分成若干组，每组内共享 K 和 V。如今的主流模型（LLaMA 2/3）都用了 GQA，说明这是业界公认的有效优化。
2. **PagedAttention**：vLLM 的核心。把 KV cache 切成固定大小的块，按需分配。这可以减少“提前分配但没用到的显存”浪费，但不会减少理论最小值。
3. **KV cache 量化**：把 KV cache 从 FP16 量化到 INT8 或 INT4。INT8 可以让显存减半，INT4 减到 1/4。但量化会引入精度损失，需要精细处理（比如用离群值感知量化）。
4. **窗口注意力（Sliding Window Attention）**：只存最近的 N 个 token 的 KV cache。Mistral 和最近的 LongLoRA 都用了这个思路。但需要注意：对于长文本中的远距离依赖，窗口注意力会丢失上下文。
5. **Streaming LLM**：结合窗口注意力和“注意力汇聚”token，保留早期值得记忆的 token。

**常见追问与回答：**

追问 1：GQA 和 MQA 对模型精度有影响吗？
回答：有，但通常影响不大。MQA 相比原始 MHA（Multi-Head Attention）精度下降了约 0.5-1%，主要体现在复杂推理和长上下文任务上。GQA 是折中，精度损失更小（控制在 0.3% 以内）。LLaMA-2 7B 用 8 个 Q 头共享 1 组 K/V（GQA-8），LLaMA-3 8B 用 8 个 Q 头共享 8 组 K/V，相当于全分组，精度几乎没有下降。

追问 2：如果用 INT8 量化 KV cache，相比 FP16 精度会下降吗？
回答：对于常见的 Chat 和 QA 场景，KV cache 量化到 INT8 的精度损失几乎可以忽略，因为 KV cache 本身“更耐量化”。但对数学、代码等高精度任务，建议做 AUC 回归测试。量化到 INT4 时，损失会更明显，需要结合 SmoothQuant 等激活值量化技术。

追问 3：32K 上下文不用 GQA 也不量化，KV cache 占 17GB，还有办法吗？
回答：可以用“稀疏注意力”，只计算部分 token 对之间的注意力。或者用“分段处理”，把 32K 拆成 4 个 8K 的片段，每个片段独立计算，最后融合。但这些方法都不是无损的，通常有 1-2% 的精度损失。

**项目联系：**

我项目里 Agent 的记忆模块需要处理长对话历史，用户对话了 50 轮后，上下文长度接近 10K token。最开始我简单地把全部历史拼进 prompt，结果每次推理都变慢。我分析了 KV cache 的显存占用（当时用 LLaMA-2 7B，10K 长度 KV cache 约 5GB），发现接近总显存 24GB 的 1/5。我的优化方案是：1）批量部署时限制了最大 batch 数；2）用窗口注意力策略，保留最近 8 轮对话的完整 KV cache，更早的历史只保留摘要文本（摘要由 LLM 在后台异步生成），这样上下文长度控制在 4K 左右。虽然不是严格的量化，但减轻了 KV cache 压力的思路是一致的，而且效果很好。

**面试口头版回答：**

我以 Mixtral 8x22B 为例来说明。这是一个典型的稀疏 MOE 模型。总参数量 235B，但一次推理时只会激活部分参数（对于 Mixtral 8x22B，是 8 个 expert 中的 2 个）。所以模型占用的显存可以分为两部分：1) 所有参数的权重（即使不激活也要加载到显存）：235B * 2 bytes（FP16）= 470 GB。这太大了，单卡放不下。2) 实际计算仅需要激活的专家参数。部署时，如果使用模型并行（比如 Tensor Parallel），把权重分布到多张 GPU 上。如果用 8 张 A100-80GB，总显存 640GB，足够存放 470GB 的权重+KV cache+overhead。所以可行的部署方案：8 张 A100 (80G) 或 4 张 H100 (96G)。另外，如果使用 INT8 量化，权重减半到 235GB，4张A100-80G够了。如果用 INT4，权重约 117GB，2 张 A100-80G 也可以跑。关键是要合理使用模型并行和量化技术。

题目 2：QLoRA 为什么选择 NF4 和 FP16 的组合？NF4 的分布拟合逻辑是什么？

**面试口头版回答：**

NF4（Normal Float 4）是一种专为预训练模型权重分布设计的 4bit 量化方法。它的核心理念是：预训练模型的权重值大致服从正态分布（均值0，方差小）。一般的均匀量化（INT4）在这种分布下效果不好，因为量化步长只覆盖中心部分。NF4 用正态分布的分位数作为量化点，确保量化值的分布和真实权重的分布更匹配。举个例子，如果权重的 99% 落在 [-2,2] 之间，NF4 的 16 个量化点就按正态曲线的形状优先排布在中心，两头稀疏。这样中心区域的量化误差大大降低。QLoRA 为什么用 NF4 + FP16 组合：把高精度的预训练权重（FP16）用 NF4 压缩存储，节省显存；但在 forward 计算时，把 NF4 权重反量化回 FP16 做运算。同时，LoRA 适配器（低秩矩阵）本身保持 FP16（因为是新的可学习参数，要准确更新）。这样既保留了接近全精度的计算精度，又把存储占用减少了 4 倍（FP16 -> NF4）。NF4 是专门为此优化的，效果优于直接 INT4。

## 3. 今日必背问答卡片

### 卡片 1：模型的 FLOPS 利用率（MFU）是什么？为什么 GPU 实际发挥出的算力远低于理论值？

- **Q：** 什么是模型的 FLOPS 利用率（MFU）？为什么 GPU 实际发挥出的算力远低于理论值？
- **A：** MFU = 实际达到的 FLOPS / GPU 理论峰值 FLOPS。推理时 MFU 通常只有 10%-30%，因为大模型是“显存带宽瓶颈”而不是“计算瓶颈”：每个 token 生成时都要把模型权重从显存搬到计算单元（显存带宽远低于计算速度），导致计算单元空等。优化 MFU 的方法是：增大 batch size（让权重读一次后做更多计算），使用量化（减小权重大小，减少搬运），或者用 Tensor Parallel 提高带宽利用率。预训练时可以通过大 batch、微 batch 和序列并行提高 MFU。
- **一句话记忆法：** MFU 低是因为模型在做“搬砖”而不是“砌墙”，带宽不够，计算在等砖头运来。
- **复习标签：** #模型部署 #推理优化

### 卡片 2：Continuous Batching 的“连续”体现在哪里？为什么能提高吞吐量，但不一定能降低单个请求的延迟？

- **Q：** Continuous Batching 的“连续”体现在哪里？为什么能提高吞吐量，但不一定能降低单个请求的延迟？
- **A：** 传统 Static Batching 需要等待一个 batch 的所有请求的 prefill（预填充，即计算输入阶段的 attention）完成之后，才一起进入 decode 阶段。Continuous Batching 则是在一个请求完成生成（遇到 EOS token）后，立即从队列中拉一个新请求进来，填补空位。同时，不同请求的 prefill 和 decode 阶段可以交错进行。这样 GPU 始终满载，没有等待空闲，所以吞吐量（每秒处理请求数）提高。但单个请求的延迟可能没变甚至微增（因为有时要等 GPU 做完其他请求的 token），不过整体系统 QPS 提升了 2-4 倍。
- **一句话记忆法：** 服务员不等一桌吃完再让下一桌坐下，而是谁吃完谁走，空位立刻有人。
- **复习标签：** #模型部署 #推理优化 #batching

### 卡片 3：推理时“prefill”阶段和“decode”阶段有什么不同？为什么 prefill 阶段是计算瓶颈，而 decode 阶段是显存带宽瓶颈？

- **Q：** 推理时“prefill”阶段和“decode”阶段有什么不同？
- **A：** prefill 阶段：模型处理用户输入的 prompt，并行计算所有 prompt token 的 KV cache，计算量大但可以高度并行（类似训练时的 forward）。是计算瓶颈（Compute-bound）。decode 阶段：逐 token 生成，每步只计算一个 token 的注意力，需要频繁读取 KV cache 和权重。因为每一步计算量小、数据搬运量大，所以是显存带宽瓶颈（Memory-bound）。了解这个区别很重要：优化 prefill 主要靠增大 batch 和精简 prompt 长度；优化 decode 主要靠减少读取数据量（量化、KV cache 优化、投机解码）。
- **一句话记忆法：** Prefill 是一次性煮一大锅饭（计算密集），Decode 是一个一个盛饭（搬运密集）。
- **复习标签：** #模型部署 #推理优化 #prefill #decode

### 卡片 4：什么是 Speculative Decoding（投机解码）？它为什么能加速？

- **Q：** 什么是 Speculative Decoding（投机解码）？为什么能加速？
- **A：** Speculative Decoding 的核心思想是：用一个轻量级的“草稿模型”（draft model）先快速生成 k 个候选 token，然后用目标大模型（target model）并行验证这 k 个 token 的合法性（拒绝不合理的）。如果误率低，则一次 forward 就能确认多个 token，从而降低目标模型的调用次数。比如草稿模型一次生成 5 个 token，目标模型一次 forward 验证后，可能接受 4 个，这样原来需要 4 次目标模型 forward 的计算，现在只需要 1 次草稿模型的 1 次 + 目标模型的 1 次，总时间减少。显存带宽瓶颈被缓解，因为目标模型的 forward 次数减少了。前提是草稿模型准确率高、误率低，否则 reject 多了就退化为普通生成。
- **一句话记忆法：** 派个“跑得快但容易错”的先试跑几步，大模型“严格验收官”一次检查多个，省得大模型反复慢跑。
- **复习标签：** #模型部署 #推理优化 #投机解码

### 卡片 5：模型部署时，计算“最小卡数”的公式是什么？考虑什么因素？

- **Q：** 模型部署时，如何计算最小需要多少张 GPU？考虑什么因素？
- **A：** 核心公式：总显存需求 = 模型权重显存 + KV cache 显存 + 中间激活显存 + 碎片和 overhead。模型权重显存 = 参数量 * bytes_per_param。KV cache 显存 = batch_size * seq_len * num_layers * hidden_size * 2 * bytes_per_elem。中间激活显存与显存带宽和 batch 并行策略有关。以 7B 模型 FP16 为例：权重 14GB，假设 batch=1, seq_len=4k，KV cache 约2GB，overhead 约2-3GB，总需求 18-19GB，一张 24GB 显卡刚好放下。如果需求 > 单卡显存，就需要用 Tensor Parallel （TP） 或 Pipeline Parallel （PP） 分发到多张卡。TP 对每张卡存部分权重，计算时通信；PP 分不同阶段。通常一张卡能处理 2-3 个并发请求，超出就需要横向扩。
- **一句话记忆法：** 先算“吃多少”（权重 + KV cache + overhead），再算“一个碗能装多少”，超了就要分碗（TP/PP）。
- **复习标签：** #模型部署 #显存计算 #TP #PP

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 推理服务中，continuous batching 和 speculative decoding 如何协同工作？两者一起使用是否会相互影响（比如减速）？

**面试口头版参考答案：**

两者可以协同工作，而且实际上是互补的。Continuous batching 解决的是系统层面的空闲等待问题，确保 GPU 始终在处理一些请求。Speculative decoding 解决的是单次 token 生成效率问题，让一次 forward 生成多个 token。两者结合的场景是这样的：系统使用 continuous batching 管理多个请求的动态调度，每个请求在 decode 阶段，系统会调用一个草稿模型来生成候选 token（草稿模型很小，计算快）；然后目标模型并行验证这些 token。因为 continuous batching 会交错处理多个请求，草稿模型的 token 生成和目标模型的验证可以混在一起调度，不会冲突。需要注意一点：Speculative decoding 在草稿模型误率高时会退化，增加无效计算，此时与 continuous batching 配合时，无效的验证步骤会浪费 slot，导致其他请求等待。所以需要根据场景（如模型 size、草稿模型质量）来决定开启与否。

**深入理解版参考答案：**

要回答这个问题，需要理解 continuous batching 和 speculative decoding 各自的 duty 和它们之间的依赖关系。

**Continuous Batching 的调度机制**：
在推理服务中，一个 iteration 会处理多个请求。每个请求处于不同的生成阶段（prefill 或 decode）。Continuous Batching 在这些请求之间做“拼单”：把处于 prefill 阶段的 prompt token 拼成一个大的 batch 矩阵运算；把处于 decode 阶段的不同 token 的 KV cache 和注意力计算也拼成一个 batch。这样 GPU 的计算单元始终忙于不同的任务。

**Speculative Decoding 的工作方式**：
草稿模型（draft model）是一个小模型（例如 LLaMA-68M），先对当前请求生成 k 个连续的 token。目标模型（例如 LLaMA-7B）接收到这 k 个 token 和原始 prompt 后，可以并行执行 k+1 次 forward 验证，计算所有 token 在当前位置的对数概率。然后通过拒绝采样算法确定接受多少个 token。例如，如果目标模型接受 4 个，skip 了第 5 个，则一次迭代生成了 4 个 token，而不是 1 个。

**两者协同的关键——如何拼进一个 iteration**：
步骤 1：Continuous Batching 的一个 iteration 到来，调度器收集所有处于活跃状态的请求。
步骤 2：对于每个请求，调度器先检查它是否处在 speculative decoding 模式。如果是，请求携带来自草稿模型的 k 个候选 token。
步骤 3：对于候选 token 被接受的请求，它们可以直接向前移动；被拒绝的，回退到上一个正确的 token 继续生成。
步骤 4：连续 batching 把这些不同状态的请求（有的需要 prefill prompt、有的需要验证 token、有的正常 decode）一起提交给 GPU 执行。

**是否相互影响？**
- 正面影响：两者共同提高系统吞吐量。Continuous batching 消除了 GPU 空闲，Speculative decoding 减少了目标模型的调用次数，使同一个带宽下的 token 产出更多。
- 负面影响：如果草稿模型质量差（比如 accept rate < 50%），speculative decoding 会增加前向传播数（验证不通过需要重新计算），这会挤占 Continuous Batching 中其他请求的 slot，反而降低系统吞吐。所以需要一个 threshold，设置一个 e3 指标（有效接受率），低于它的请求退化为普通 decode。

总的来说，现代推理框架（vLLM、SGLang）已经原生支持 speculative decoding 和 continuous batching 的联动。面试中可以说“我了解它们不是互斥的，而是在调度层面叠加优化，关键在于草稿模型质量”。

**可能追问与回答：**

追问 1：如果草稿模型和目标模型 tokenizer 不一致，投机解码还能用吗？
回答：可以用，但需要 tokenizer 对齐映射或者用 embedding 层对齐，比较复杂。一些工作（如 Medusa）不使用草稿模型，而是用目标模型的自预测头来生成候选 token，避免了 tokenizer 不一致问题。

追问 2：如果使用 SGLang 的 RadixAttention + speculative decoding，怎么处理前缀的 KV cache 复用？
回答：这是一个很好的结合点。SGLang 在 speculative decoding 中也复用前缀：草稿模型生成的候选 token 序列的 KV cache 也一并计算并存入 Trie 树。如果候选 token 被接受，这些 cache 就会被保留；如果被拒绝，它们会被淘汰。这进一步减少了验证阶段所需的显存和计算。面试中能提到这个点会加分。
