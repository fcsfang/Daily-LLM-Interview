### 题目 1：如何设计一套 RAG 系统的离线评测体系？你会选用哪些指标，怎么采集？

**来源依据：**
- 资料 [1] 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 资料 [17] RAG常见面试题和答案 - https://adg.csdn.net/695238785b9f5f31781b36a9.html - 技术博客 - 需进一步验证

**可信度：** 中（面经案例提供了实际操作经验，技术博客补充了理论框架）

**考察点：** RAG 评测体系、离线评测、指标选择、bad case 分析

**面试口头版回答：**
我会从检索质量和生成质量两个维度设计评测体系。检索侧主要看召回率（Recall@K）和命中率（Hit Rate），生成侧看答案准确率、忠实度（Faithfulness）和引用准确率。具体操作上，我会先构建一个小规模的标注评测集，包含典型 query 和标准答案。然后固定其他变量（比如同一个生成模型），对比不同检索配置下的指标变化。比如在我项目中，我们攒了一个几百条真实多轮对话的小评测集，对比了带记忆+Query改写和纯单轮两种配置的答案正确率，这样每改一处都能看到数字涨没涨，而不是凭感觉。

**深入理解版回答：**
一套完整的 RAG 离线评测体系应该覆盖三个层面。第一层是检索质量评估：常用的指标包括 Recall@K（前 K 个结果中相关文档的占比）、MRR（平均倒数排名，关注第一个相关结果的位置）、NDCG（归一化折损累积增益，考虑排序位置重要性）。这些指标需要离线标注 query-document 的相关性作为 ground truth。实际工程中，我们会用 Hit Rate 替代 Recall——只要前 K 里有一个相关文档就算命中，因为 RAG 只要有一条相关片段就能生成答案。第二层是生成质量评估：包括答案准确性（模型是否用了正确的检索内容）、忠实度（生成内容是否完全基于检索片段，没有编造）、引用准确率（模型引用的片段是否真的在其上下文中）。这里需要用 LLM-as-judge 或人工评估。第三层是 bad case 分析：将错误案例分类，比如"检索不相关"、"检索相关但模型忽视"、"模型误解了检索内容"等。我们项目中用了一个脚本，把每轮 query 的检索结果、模型答案和标注答案对齐，用 LLM 判定错误原因类型，按月产出 bad case 分布报告。值得注意的坑是：指标不能只看平均分，要关注长尾分布。很多系统的平均分数好看，但在一些高频但刁钻的 query 上表现很差，需要分层抽样评测。

**常见追问与回答：**
- **追问：** 你们是怎么构建这个评测集的？人工标注成本很高吧？
  - **回答：** 对，所以我们会先聚类采样。把用户日志里的 query 聚类，从每类里抽有代表性的 query，配合少量难例（历史上回答差的 query）形成评测集。标注用众包+专家复审，大概几百条就够了，关键是要覆盖不同类型和难度。
- **追问：** LLM-as-judge 做生成质量评估时，怎么避免 judge 模型本身的偏好和偏见？
  - **回答：** 我们会用多个 judge 模型做投票，同时设计对比式 prompt（让 judge 比较模型输出和参考答案的相似度），而不是直接打分。另外定期用人工抽检来校准 judge 的分数，如果发现偏差就调整 prompt 或换 judge 模型。

**项目联系：**
在搭建 RAG 项目时，可以先花 1-2 天构建一个 200-500 条 query 的评测集，包含常见问题和边界情况。然后为每个配置组合跑一次完整的评测流程（检索→生成），产出指标报告和 bad case 汇总。这个流程可以脚本化，每次优化后跑一遍，用数字验证效果。比如我在项目中发现，将 Top-K 从 5 改为 3 后，答案准确率反而提升了 8%，因为减少了无效上下文对模型的干扰。

**记忆关键词：** #RAG #离线评测 #Recall@K #忠实度 #bad case分析 #LLM-as-judge

### 题目 2：在 Agent 系统中，你怎么评估 Agent 整体的执行效果？会用到哪些方法和指标？

**来源依据：**
- 资料 [1] 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 资料 [5] 2026大厂大模型/Agent面试题 - https://gitcode.csdn.net/6a236d7b662f9a54cb7a2bf1.html - 面经或社媒 - 需进一步验证

**可信度：** 中（来自面经汇总，反映了面试高频考点）

**考察点：** Agent 评测、任务完成率、工具调用评估、多轮交互

**面试口头版回答：**
我会把 Agent 评测拆成三个维度：任务完成度、效率和稳定性。任务完成度看最终是否能达成用户目标，比如企业内部客服 Agent 中，用户问题是否被正确解决。效率看执行步数和单步耗时，如果一个简单的查政策问题要调用 5 个工具、花 30 秒才能完成，那体验就很差。稳定性尤其重要，包括工具调用失败率、是否进入死循环、是否产生幻觉。实践中，我会构建一个覆盖常见场景和异常情况的评测集，跑完之后人工或自动打标。比如在项目中，我们给每个测试 case 标注了"预期工具调用链"，然后将 Agent 实际走出的调用链与之对比，算出一个"路径准确率"，发现改进后从 60% 提升到了 85%。

**深入理解版回答：**
Agent 评测比纯 RAG 更复杂，因为它涉及多步决策和工具执行。我一般从四个层面设计。第一层是任务级：用 Success Rate（成功率）和 Goal Completion Rate（目标完成率）来衡量 Agent 最终是否达成用户预期。需要在评测集中为每个 query 预设"达成条件"，比如"用户要求查看华东区销售数据，回答应包含具体数字"。第二层是步骤级：评估每一步决策的合理性——工具选择是否正确、参数是否准确。这里可以用 Tool Call Accuracy（工具调用准确率）和 Plan Adherence（计划遵循度）。第三层是效率指标：包括平均步数（Steps per Task）、平均响应时间、最大步数（是否超限）。我曾经遇到过 Agent 因为工具参数理解错误，连续重试 8 次才成功，这就需要设置最大重试次数的兜底策略。第四层是鲁棒性：包括 Recovery Rate（从错误中恢复的能力）——当工具调用失败后，Agent 能否通过重试或换策略自救。在实现时，我们会为每个 Agent 流程添加完整的日志链路（Logging Pipeline），记录每一步的 prompt、思维链、工具调用结果。然后写一个自动化脚本，从日志中提取指标并生成报告。典型坑：很多 Agent 在正常 case 下表现很好，但在模糊查询（比如"给我看看那个"这种指代模糊的问题）或边界条件（比如工具返回空结果）下极易出错，这些应该在评测集中重点覆盖。

**常见追问与回答：**
- **追问：** 如果 Agent 陷入死循环，你们怎么检测和兜底？
  - **回答：** 我在项目里设了两个防线：一是最大步数限制（比如 10 步），超过就返回当前结果并打上 "incomplete" 标签；二是循环检测，记录最近 5 步的 tool name+参数摘要，如果出现重复模式就打断，切换到降级策略——比如直接调用一个"通用问答"工具来回答。
- **追问：** 工具调用的评测具体怎么做？需要标注每个参数吗？
  - **回答：** 我会先按工具级别评估。比如对"查询员工信息"工具，标注一个正确的参数 JSON，然后用 Agent 实际生成的 JSON 与标注对比，计算字段完整性和值准确度。更细致的做法是评估"参数理解正确但工具本身返回了空结果"vs"参数理解错误导致返回无关结果"两种情况，它们的 bad case 修复策略不同。

**项目联系：**
在做 Agent 项目时，可以写一个评测框架，输入测试 query 列表，自动跑 Agent，输出成功/失败、步数、工具调用序列。关键是要预设"成功的标准"，比如"最终回答中包含事实 x"。刚开始可能只有 20-30 个测试 case，但随着开发深入可以逐步扩充到 100+ 条，覆盖工具调用的各种边界条件（参数缺失、工具返回空、多步依赖等）。

**记忆关键词：** #Agent评测 #任务完成率 #工具调用准确率 #死循环检测 #自动化评估

### 题目 3：什么是 LLM-as-judge？它有哪些局限性？你怎么在实际项目中弥补这些局限？

**来源依据：**
- 根据上述资料 [1] 和 [17] 延伸出的经典面试题，需进一步验证

**可信度：** 中（作为模型评测的核心话题，资料中隐含提及，需要结合实践补充）

**考察点：** LLM-as-judge、自动化评测、评估偏差、校准策略

**面试口头版回答：**
LLM-as-judge 是指用一个大语言模型充当裁判，来评估另一个模型的输出质量。常见做法是给 judge 模型一个评估 prompt，让它给候选回答打分或做比较。它的局限主要是三个方面：一是位置偏差，比如 judge 偏向选择第一个或最后一个结果；二是自我偏好偏差，judge 更青睐和自己风格相似的输出；三是 judge 本身的幻觉和偏见也会影响评分。我在项目中做了三个弥补：第一，用多个 judge 模型投票，比如同时用 GPT-4 和开源模型；第二，采用"对比式评估"而不是直接打分——让 judge 比较模型输出和参考答案的相似度，而不是独立打分；第三，建立人工抽检机制，每周抽检 10% 的评测结果，用校对后的数据来校准 judge 的偏差。

**深入理解版回答：**
LLM-as-judge 本质上是把评估看作一个生成任务，而不是一个度量任务。它的优点是可以灵活评估各类文本质量，但缺点也同样明显。主要问题包括：
1. 位置偏差：当要求 judge 从多个输出中选择最佳时，它倾向于选择第一个（或最后一个）出现的选项。解决方案是随机打乱选项顺序，跑多次取共识。
2. 自我偏好偏差：judge 模型对自己同系列模型（比如 GPT-4 judge 评估 GPT-4 的输出）的评分偏高。需要交叉评估——用不同家族的模型做 judge。
3. 细粒度不准确：judge 对"轻微差异"不敏感，比如两个回答只在语气或结构上不同但语义等价，judge 可能错误判分。
4. 触发对抗攻击：用户可以通过 prompt 注入让 judge 给出高分或低分。
工程上我做了三件事：一是标准化评估 prompt，让 judge 先输出推理过程再给出分数（CoT），减少随机性；二是为 judge 提供少量标注好的"金标示例"作为参考（few-shot）；三是建立 Score Calibration 流程，用人工标注的 100 条数据来拟合一个校准函数，把 judge 的打分映射到真实分数空间。此外，还需要注意不要用同一个模型既做生成又做评估——这会产生严重的 overestimation bias。

**常见追问与回答：**
- **追问：** 你们项目中 LLM-as-judge 的评分一致性（Inter-rater reliability）怎么样？
  - **回答：** 我们用 Cohen's Kappa 来衡量，刚开始只有 0.5-0.6 左右，说明不一致性很高。后来通过统一评估 prompt 格式、加入 few-shot 示例、以及采用"比较式评估"（pairwise comparison），一致性提升到了 0.8 以上。
- **追问：** 如果需要用开源模型做 judge，你选什么样的模型？多大参数？
  - **回答：** 一般选 7B-13B 的参数范围，太小的模型评估能力不够。选型时要注意：模型要有良好的指令遵循能力，不能有太强的 safety alignment（防止自主拒绝评估）。我们常用 Qwen-2.5-14B 或者 Llama-3-8B，它们在文本评估任务上表现不错。

**项目联系：**
在项目中可以用 LLM-as-judge 来自动化评估 RAG 回答的忠实度。具体做法是：给 judge 模型输入（检索片段 + 模型回答），让 judge 判断回答中每一个事实点是否能在检索片段中找到支持。这样可以精确量化"幻觉率"。关键是要定义清楚评估的粒度——是全局打分还是逐句打分，后者更准确但成本更高。

**记忆关键词：** #LLM-as-judge #位置偏差 #自我偏好 #校准 #自动化评测 #CoT评估

**可转化面试题及参考答案：**

**面试题 1：** 你认为 RAG 系统最难优化的地方是什么？为什么？
**参考答案：**
我认为最难的点是检索质量和生成质量之间的匹配。有时候检索到的文档很相关，但模型生成时没有使用或者错误使用了这些信息。我在项目中遇到的典型问题是：当检索内容较多时（比如 Top-K=6），模型容易被中间位置的相关片段淹没——这就是 "lost in the middle" 问题。为了解决它，我们做了两件事：一是压缩 Top-K（从 6 降到 3），只保留最相关的；二是把最相关的片段放在 prompt 的首尾位置，因为模型对两端的信息记忆更强。另一个难点是 chunk 粒度的选择——chunk 太小信息不完整，chunk 太大又混入了噪音，需要根据文档类型逐领域调参。

**面试题 2：** 你通过什么方式来评估 RAG 系统的检索质量？
**参考答案：**
我主要用 Recall@K 和 Hit Rate 两个指标。Recall@K 衡量前 K 个结果中包含多少相关文档；Hit Rate 更宽松，只要前 K 里有一个相关文档就算命中。因为 RAG 只需要一条相关片段就能支持生成，所以 Hit Rate 更适合评估生成质量。具体做法是：先构建一个 query-相关文档对的小数据集（人工标注），然后跑检索，计算指标。我们发现一些 Embedding 模型在不同领域的表现差异很大，所以需要针对自己的文档领域做评估和选型。

**可转化面试题及参考答案：**

**面试题 1：** 你在项目中是如何发现"多轮失忆"问题的？怎么量化它的严重程度？
**参考答案：**
我抽样了脱敏后的多轮会话做人工标注，发现追问类问题（依赖上文）里有相当一部分答得不对或答非所问。根因基本都是丢了上文的约束或指代没还原。为了量化，我攒了一个几百条真实多轮对话的小评测集，标好理想答案，然后固定其他变量，对比带记忆+Query改写和纯单轮两种配置的答案正确率。有了这个评测集，后面每改一处——记忆策略、改写开关、检索参数——都能看到数字涨没涨，而不是凭感觉判断。

**面试题 2：** 记忆检索时，你怎么平衡"相关性"和"时近性"？注入多少记忆信息比较合适？
**参考答案：**
我给长期记忆的条目也做了向量化，检索时先按相关性召回相关条目，再用时近性做加权重排——相同相关度下更新优先。注入量我卡得很死，只拼 3-5 条最相关的进上下文，外加短期记忆里最近的两三轮原文。原因和 Top-K 一样：记忆塞太多同样会稀释当前问题、引入噪声、浪费 token。硬约束类型的记忆（比如"只看华东区"）是例外，只要还生效就每轮都带，不参与"取一小批"的竞争。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** 什么是模型离线评测？和在线评测有什么区别？
- **A：** 离线评测是在模型上线前，使用标注好的测试集评估模型的检索或生成质量，不涉及真实用户流量。在线评测是通过 A/B 测试或用户反馈收集来衡量线上效果。离线评测成本低、可重复，适合快速迭代；在线评测能反映真实用户场景，但周期长、不确定因素多。实际工作中两者结合：先离线筛选配置，再线上验证最终效果。
- **一句话记忆法：** 离线是实验室，线上是战场，先去实验室做好武器再上战场。
- **复习标签：** #评测 #离线评测 #在线评测

### 卡片 2
- **Q：** RAG 评估中常用的检索质量指标有哪些？简述它们的含义。
- **A：** 常用三个：Recall@K（前 K 个结果中相关文档的占比）、MRR（平均倒数排名，关注第一个相关结果的位置）、NDCG（归一化折损累积增益，考虑排序的权重）。在 RAG 场景中，我更常用 Hit Rate（只要前 K 里有一个相关就算命中），因为它与生成质量更强相关——只要有一条相关片段，模型就能生成正确答案。
- **一句话记忆法：** Recall 看量，MRR 看首个，NDCG 看排序权重，Hit Rate 看有没有。
- **复习标签：** #RAG #评测 #检索指标 #Recall@K

### 卡片 3
- **Q：** LLM-as-judge 的局限性有哪些？如何缓解位置偏差？
- **A：** 主要局限：位置偏差（偏向首尾选择）、自我偏好偏差（偏好同类模型）、细粒度不准确（对轻微差异不敏感）。缓解位置偏差的方法：随机打乱选项顺序，跑多次取投票结果；采用 pairwise 比较而非直接打分；使用 CoT 让 judge 先推理再打分，增加可控性。
- **一句话记忆法：** 打乱再跑，比较再打，推理再判。
- **复习标签：** #评测 #LLM-as-judge #位置偏差

### 卡片 4
- **Q：** 在 Agent 评测中，你怎么定义和计算"任务成功率"？
- **A：** 任务成功率 = 成功完成任务的 case 数 / 总测试 case 数 × 100%。成功的标准需要在评测集中预定义，例如"回答正确包含了查询的数据"或"工具调用链与预期一致"。关键是定义要明确、可自动判断，避免主观。我一般会结合自动检查（如判断关键数据是否出现在回答中）和人工抽检两种方式。
- **一句话记忆法：** 先定终点，再算到达率——没有明确定义，成功率就是伪命题。
- **复习标签：** #Agent评测 #任务成功率 #自动化评估

### 卡片 5
- **Q：** 做 bad case 分析时，常见的 RAG 错误类型有哪些？分别怎么定位原因？
- **A：** 三种常见类型：1）检索不相关：检索到的文档与 query 无关，看召回结果就能发现；2）检索相关但模型忽视：模型没有使用检索到的正确信息，需要检查 prompt 指令是否明确；3）模型误解检索内容：模型读取了正确的文档但错误理解了含义，可以在 prompt 中加入"请逐句核对"的约束。定位方法：对齐每一步——query、检索结果、最终回答——逐层排查。
- **一句话记忆法：** 检索不到是召回的问题，检索了不用是 prompt 的问题，用了但错了是理解的问题。
- **复习标签：** #RAG #bad case分析 #错误分类 #排查

## 4. 今日面试专项训练

### 4.1 大模型理论追问题：LLM-as-judge 的评分偏差会影响评测结果的可信度，你怎么设计一套校准机制？

**问题：**
假设你在项目中使用 GPT-4 作为 judge，来评估另一个 LLM（如 Llama-3-8B）的 RAG 答案质量。但你发现 judge 给 Llama 的分数普遍偏低，且和人工评估结果不一致。请设计一套校准机制，提高 LLM-as-judge 的可信度。

**面试口头版参考答案：**
我会做三件事。第一，建立校准集：用 100 条样本同时获取 judge 分数和人工分数，拟合一个映射函数，比如把 judge 的 1-5 分映射到人工的评分分布上。第二，采用 pairwise 比较代替绝对打分：每次给 judge 两个回答（Llama 和一个参考模型），让它比较哪个更好，而不是独立打分——这样能减少 judge 本身的偏见。第三，定期重检校准效果，比如每周用 20 条新样本做一次人工校准验证，如果偏差超过阈值就重新拟合校准函数。

**深入理解版参考答案：**
LLM-as-judge 的偏差本质上是 judge 模型的"先验偏好"导致的系统误差，无法完全消除，但可以通过校准机制来显著降低。核心思路是：**不允许 judge 直接输出绝对分数，而是输出相对偏好，再通过统计模型映射到绝对分数空间。**
具体步骤：
1. 构建校准集：从评测池中随机抽取 100-200 条样本，每条样本包含（query, 真实回答 A, Judge 对 A 的打分, 人工对 A 的打分）。人工打分需要 3 人标注取中位数，确保一致性。
2. 设计校准模型：使用线性回归或有序逻辑回归（Ordinal Logistic Regression），以 Judge 的打分（以及可能的影响因素，如 query 长度、回答长度、主题类别）为特征，预测人工打分的期望值。这样得到的校准函数可以调整 Judge 的系统偏差——比如如果 Judge 总是给长回答偏高的分，校准模型会降低长回答的权重。
3. 进阶方案：引入"困难度调节"。如果 judge 在某些主题（比如法律条款类）上偏差严重，则给该主题的样本加一个"难度权重"，将这些样本在校准模型中的影响变大，提升校准效果。
4. 交叉验证：将校准集分成 5 折，做交叉验证评估校准函数的效果，确保不过拟合。
5. 监控机制：在生产中持续收集新的人工标注，加入校准集，每周或每两周重新拟合一次校准模型。
这个方案已经被多个开源评测框架（如 LM-based Evaluation via Probabilistic Calibration）验证有效。局限在于需要持续的人工投入来维护校准集，但对于追求评测可信度的项目来说是值得的。

**可能追问与回答：**
- **追问：** 如果人工标注成本太高，无法维持校准集怎么办？
  - **回答：** 可以采用"主动学习"策略：只抽取 judge 和人工评分差异最大的样本进行人工标注，因为校准模型需要的是"错误区域"的信息，而不是均匀分布的样本。这样可以显著降低人工标注成本。
- **追问：** 校准后 judge 的一致性和准确率能提升多少？
  - **回答：** 在我的实践和文献中，校准后 Spearman 相关系数通常能提升 0.1-0.2（从 0.6 提升到 0.8 以上）。具体效果取决于 judge 的初始偏差程度和校准集的质量。关键是要做交叉验证，避免校准函数的过拟合。
