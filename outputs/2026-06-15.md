### 题目 1：长文档切片（Chunking）时 chunk_size 怎么定？如何保证语义完整性？

**来源依据：**
- 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证，但内容详细，可引用

**可信度：**
中（面经搬运，无官方背书，但内容结构与真实面试场景一致）

**考察点：**
RAG 数据处理、Embedding 模型 token 限制、语义完整性与检索性能权衡

**面试口头版回答：**
我会根据 Embedding 模型的 token 上限来定 chunk_size。比如 BGE-small 最大 512 tokens，我会设 chunk_size=400 tokens，留出冗余避免截断。同时用“重叠切分”，前后重叠 10-15%，防止关键信息落在边界被切散。如果文档结构明显，还要用“语义切分”，比如按 Markdown 标题、段落来切，比固定长度保留更多上下文。最后通过小评测集调参：对比不同 chunk_size 下的召回率 + 最终答案正确率，选择 trade-off 最优的值。

**深入理解版回答：**
chunk_size 的选择本质是“粒度”与“语义完整性”的平衡。太小的 chunk（如 128 tokens）容易丢失上下文，导致检索时语义不连贯；太大的 chunk（如 1024+）虽然信息完整，但会引入噪声，且超过 Embedding 模型窗口后不得不截断，反而丢失关键内容。工程上通用做法是：先确定 Embedding 模型最大输入长度（如 512/1024），取 70-80% 作为 chunk_size；采用“重叠切分”（overlap=10-20%）缓解边界问题；对结构化文档（PDF、FAQ）使用“语义切分器”按标题、段落切分。还需要考虑下游生成模型的上下文窗口——chunk 太长可能挤占“other chunks”的容量。常见坑：只依赖固定长度，忽略文档层级；overlap 过大会导致重复检索、浪费 token。优化方法：对同类业务场景离线评测不同 chunk 配置（召回率、MRR、Faithfulness 分数），选出最优值后固化到配置，上线后持续监控。

**常见追问与回答：**
- **Q：如果文档包含表格或图片，Chunking 怎么处理？** A：纯文本管线会丢失图、表信息。我会先用 OCR 或解析器提取表格和图片中的文字描述，作为 chunk 的一部分或单独建索引。也可用多模态 Embedding 模型，更直接。
- **Q：chunk_size 会影响 Rerank 效果吗？** A：会。Rerank 模型（Cross-Encoder）输入有长度限制（通常 512 tokens），如果 chunk 太长会被截断导致评估不准。所以一般在 Rerank 阶段也会控制 chunk 长度，或先分小段再 Rerank。

**项目联系：**
在 RAG 项目中，我对企业产品手册做 Chunking 优化：先用 LayoutParser 解析 PDF 结构，按标题语义切分，再对每个 chunk 做固定长度分块（256 tokens，overlap 32）。对比了 128/256/512 的召回率，发现 256 在召回和生成质量上最优，并写入系统配置。

**记忆关键词：**
chunk_size、overlap、语义切分、token 限制、离线评测

### 题目 2：RAG 中如何做稠密向量检索与稀疏向量检索的混合？RRF（Reciprocal Rank Fusion）的原理和优缺点？

**来源依据：**
- 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- RAG常见面试题和答案 - https://adg.csdn.net/695238785b9f5f31781b36a9.html - 技术博客 - 需进一步验证

**可信度：**
中

**考察点：**
混合检索设计、稀疏/稠密检索对比、Rerank 前融合策略

**面试口头版回答：**
混合检索是我用来平衡“精确匹配”和“语义理解”的标准方案。具体做法是：用户 query 并行发给 BM25（稀疏）和向量检索（稠密），各取 Top-K 结果；然后用 RRF（Reciprocal Rank Fusion）融合两个结果集：每个结果的分数 = 1/(rank + k)，k 是平滑参数，我一般设 60。最后按 RRF 分数重排，取 Top-N 进入生成阶段。RRF 的优点是简单、不需训练、效果好；缺点是没有考虑分数分布，高分的 BM25 结果可能被低分向量结果挤下去。所以我在项目中还会在 RRF 后加一个 Rerank 层来精排。

**深入理解版回答：**
混合检索解决的是单一检索方式覆盖不足的问题。稀疏检索（BM25）依赖关键词匹配，对专有名词（如“消音螺旋”）效果好；稠密检索（向量检索）理解语义，能召回“那个零件有没有替代品”这类同义问题。工程上通常用“分治融合”策略：两种检索并行执行，各自返回 Top-K（K 一般取 50-100），然后用 RRF 或加权求和融合。RRF 的不依赖于分数绝对值，只依赖排序位置，因此对不同检索结果的分数尺度不敏感，工程上比较健壮。参数 k 控制信号强度：k 越小，排名靠前的结果权重越大。混合检索的缺点是需要同时维护两个索引系统，增加了运维成本和延迟。常见优化：对高频 query 预计算混合结果缓存；或先用 BM25 粗筛再用向量精排（级联式），但级联可能丢失语义结果。此外，RRF 后一般会再接 Rerank（Cross-Encoder）进行精排，解决“交互不足”的问题。

**常见追问与回答：**
- **Q：RRF 中的 k 值怎么调？** A：k 越大，排名靠后的结果权重相对提升。我通常通过离线评测（MRR、Recall@N）在 20-100 之间搜索，比如 k=60 时平衡性好。也可用网格搜索。
- **Q：混合检索的延迟怎么优化？** A：把 BM25 和向量检索做成并行调用，用 asyncio 异步 IO 同时发起，然后同时等结果。如果其中一种检索特别慢，可降级为只用另一种。

**项目联系：**
在 RAG 项目中，我用 Elasticsearch 做 BM25 索引，用 Milvus 做向量检索，RRF 融合后 Rerank。并发请求使用 Python asyncio，延迟从 800ms 降到 450ms。离线测试中混合检索 recall@10 比单一向量检索提升 12%。

**记忆关键词：**
混合检索、BM25、稠密向量、RRF、融合排序、延迟优化

### 题目 3：Rerank（Cross-Encoder）解决了向量召回的什么问题？有哪些工程落地注意事项？

**来源依据：**
- 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 一文读懂：大模型RAG - https://zhuanlan.zhihu.com/p/675509396 - 技术博客 - 需进一步验证

**可信度：**
中

**考察点：**
Rerank 原理、Cross-Encoder vs Bi-Encoder、工程成本

**面试口头版回答：**
向量召回（Bi-Encoder）把 query 和文档分别编码为向量，只比较向量的余弦相似度，缺乏 query 和文档之间的交叉交互，所以可能把语义相似但实际不直接相关的文档召回。Rerank 用 Cross-Encoder 同时对 query 和文档做 Attention，能精确判断相关性。工程上我通常在召回 TOP 50-100 后做一次 Rerank，只保留 Rerank 分数最高的 10 个进上下文。但 Cross-Encoder 计算成本高，一般用轻量模型如 BGE-Reranker-v2 微调，或对产品长文本做滑动窗口剪枝。关键坑是：Rerank 的输入长度受限，如果文档太长会被截断，影响判断；另外 Rerank 的时延是 O(N)，要控制 N 在 50 以内。

**深入理解版回答：**
向量召回阶段使用 Bi-Encoder：query 和文档被独立编码为固定向量，通过余弦相似度匹配。这种方式假设语义相似意味着相关性，但实际中“相似”不等于“相关”——比如用户问“退款流程”，文档“退货政策”语义相似但可能不包含操作步骤。Cross-Encoder 将 query 和文档拼接后做 2 分类（相关/不相关）或打相关性分数，能捕捉单词级别的交互，精度更高。工程落地时：① 召回 Top-K 不宜过大（K 建议 30-50），否则 Rerank 延迟线性增长；② 使用异步批量推理，提高吞吐；③ 对长文档，先做 Chunk 筛选或长文本截断，避免超过 Rerank 模型窗口（常见 512 tokens）；④ 可考虑使用“两阶段 Rerank”：先轻量模型粗排再重模型精排。Rerank 的缺点：需要额外部署模型、推理耗时、训练需要标注数据。生产环境中通常结合 AB 测试评估引入 Rerank 后对最终回答准确率（如 Faithfulness）的提升。

**常见追问与回答：**
- **Q：Rerank 模型怎么训练？** A：用历史 query+chunk 对，人工标注相关分数（如 0/1/2）。也可以用 LLM-as-judge 自动生成弱标签，再用小模型蒸馏。训练目标是二分类或排序损失（如 ListMLE）。
- **Q：如果用户 query 非常长，Rerank 怎么处理？** A：截断 query 到模型最大长度，保留关键意图。也可以先用 query 的 embedding 做粗筛，再对保留的 chunk 做 Rerank。

**项目联系：**
在我的 RAG 项目中，刚开始没有 Rerank，发现用户问“报销流程”时模型引用了“请假流程”，因为两者语义相似度都高。加了 BGE-Reranker-v2 后，相关准确率从 78% 提升到 92%。线上部署使用 ONNX 量化，20 个候选 Rerank 延迟控制在 30ms。

**记忆关键词：**
Rerank、Cross-Encoder、Bi-Encoder、交互缺失、延迟优化

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** 为什么向量检索（Bi-Encoder）召回后还需要 Rerank（Cross-Encoder）？
- **A：** Bi-Encoder 将 query 和文档独立编码，缺乏交互，可能把语义相似但实际不相关的文档召回。Cross-Encoder 让 query 和文档做全 Attention，能更精确判断相关性。所以在召回出 Top-K 后，用 Cross-Encoder 精排，保留 Top-N 进上下文。
- **一句话记忆法：** ✅ 向量检索是相亲看照片，Rerank 是见面聊两句。
- **复习标签：** #RAG #Rerank #Cross-Encoder

### 卡片 2
- **Q：** chunk_size 太大或太小分别有什么后果？
- **A：** 太小（如 128）容易截断关键信息，导致检索不完整；太大（如 1024+）会引入噪声，且可能超过 Embedding 模型窗口被迫截断，同样丢失信息。一般取 Embedding 最大长度 70-80%，配合 10-15% 重叠切分。
- **一句话记忆法：** ✅ 切太碎是盲人摸象，切太大是囫囵吞枣。
- **复习标签：** #RAG #Chunking #工程

### 卡片 3
- **Q：** 什么是 RRF（Reciprocal Rank Fusion）？它的优点是什么？
- **A：** RRF 是一种融合多个搜索结果排序的算法。对每个结果，如果它在某个检索中排名 rank，则贡献分数 1/(rank + k)（k 一般为 60）。把多路检索的结果按分数求和再排序。优点是简单、无参数训练、不依赖分数尺度、效果好。
- **一句话记忆法：** ✅ 排名靠前的给高分，Rerank 混合不用调分，RRF 一锅端。
- **复习标签：** #RAG #混合检索 #RRF

### 卡片 4
- **Q：** RAG 上线后如何监控幻觉？（新角度：在线实时检测）
- **A：** 我采用“引用率 + 忠实度”两个指标。引用率：检查生成回答是否有 cite 来源原文中的 statement，如果引用少可能幻觉。忠实度：用 LLM-as-judge 或轻量模型评估回答是否被检索支持。实时检测可用 Guardrails 层，如果忠实度低于阈值，触发降级（如删幻觉句或返回“无法回答”）。
- **一句话记忆法：** ✅ 不引用就是耍流氓，没证据就让模型闭嘴。
- **复习标签：** #RAG #幻觉控制 #在线监控

### 卡片 5
- **Q：** RAG 评测中，如何做到“活”的评测，避免只跑一次就丢掉？（新追问）
- **A：** 我建立了“评测回归池”：包含历史 bad case 和周期性新采用户会话。每次系统迭代（模型升级、检索参数调整）都跑一遍回归池，对比引入前后各指标变化。并用 CI/CD 流水线自动执行，如果关键指标（如 Faithfulness）下降超过 5%，阻断上线。
- **一句话记忆法：** ✅ 评测不是一次考试，是持续的体检 + 守门员。

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 从 Attention 机制原理角度解释 RAG 中为什么会出现 “Lost in the Middle” 现象？你有什么工程手段缓解它？

**面试口头版参考答案：**
Lost in the Middle 指的是大模型在做长上下文推理时，会倾向于关注开头和结尾的信息，而忽略中间的内容。本质原因是 Transformer 的注意力机制中，每个 token 对所有其他 token 做加权，但位置编码（尤其是 RoPE）对相对距离敏感，中间 token 接收两端的信息不如两端接收中间的信息充分。另外 LLM 训练时 query 和关键信息的相对位置随机，导致模型学不到中间位置的强信号。工程上缓解方法：1. 限制输入上下文长度，检索只留头部最相关的 5-7 个 chunk；2. 对 Chunk 按相关性排序，把最关键的内容放在开头；3. 在 prompt 中明确要求模型按顺序仔细阅读中间段落。

**深入理解版参考答案：**
“Lost in the Middle” 由 Liu et al. (2024) 命名，实验发现当相关信息出现在输入中间 1/3 时，模型性能下降最多。原因有：1. 注意力分数受位置偏置影响，中间 token 的相对位置对称，信号更弱；2. 训练数据中以信息靠前或靠后为主，中间位置的监督信号少；3. 模型倾向于用首尾的强位置编码记忆位置。工程上缓解手段包括：① 主动信息排序：将来自检索的 Top-K chunk 按预估相关性降序排列，把最相关放开头；② 上下文长度裁剪：只保留必要数量（3-5）的 chunk，越长越容易 Lost；③ 使用结构化 prompt 将中间段落标记为重要，如 “Instructions: Carefully read ALL sections from middle to end”；④ 利用 Rerank 二次排序，确保进入上下文的都是高相关内容；⑤ 对下游生成模型做长上下文微调（如位置插值扩展训练数据中中间位置相关性）。在 RAG 系统中，我通过控制 Top-K 数量（从 10 降到 5）缓解了 20% 的 Lost 相关 bad case。

**可能追问：**
- **Q：如果在 128K 上下文中，是否所有模型都有 Lost in the Middle？** A：研究表明大多数模型都存在，但经过长上下文优化的模型（如 GPT-4-128K）通过位置插值和更多长上下文训练数据时，有一定改善。但彻底解决很难。
- **Q：你如何在线监测 Lost in the Middle 的影响？** A：我会跟踪回答中引用的来源位置，如果模型引用来自开头和结尾的比例显著高于中间，可能发生了 Lost。通过日志分析引用 chunk 的索引位置分布，若中间引用很少，则触发告警并提示需要优化排序或裁剪策略。
