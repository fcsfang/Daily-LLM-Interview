### 题目 1：LoRA 微调的原理是什么？训练时 Transformer 更新哪些参数？低秩矩阵怎么初始化的？

**来源依据：**
- 2026大厂大模型/Agent面试题-CSDN博客 - https://blog.csdn.net/weixin_44151034/article/details/161729146 - 面经或社媒 - 中，来源为牛客网面经摘录，需进一步验证
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 中，来源为 GitHub 面经集合，需进一步验证

**可信度：** 中

**考察点：** LoRA 原理、参数更新范围、初始化策略

**面试口头版回答：**
LoRA 的核心思想是冻结预训练权重，在 Transformer 的某些层上插入低秩矩阵作为可训练参数，用这个低秩矩阵的输出去近似全量微调时的权重更新量。训练时，原始权重不更新，只有插入的 A 和 B 两个矩阵参与梯度计算。初始化时，A 用高斯分布随机初始化，B 用全零初始化，这样初始状态下低秩矩阵的输出是零，模型行为与原始权重一致，不会造成训练初期的剧烈扰动。在实践中，我通常对 Attention 的 Q 和 V 投影层施加 LoRA，秩 r 一般取 8 或 16，α 设为 16 或 32。

**深入理解版回答：**
LoRA 基于一个关键观察：预训练模型在微调时，权重变化 ΔW 具有低秩性质，即可以用两个小矩阵的乘积来近似。具体来说，对于原始权重矩阵 W₀（维度 d×k），LoRA 引入 A（维度 r×k，r << min(d,k)）和 B（维度 d×r），那么前向计算变为 h = W₀x + BAx，训练时只更新 A 和 B。秩 r 是核心超参数：r 越大，表达能力越强，但参数量和过拟合风险也上升；r 太小则可能无法捕捉领域特化知识。

初始化方式有工程考量：A 用随机初始化保证各向异性，B 用全零确保初始输出为 0，这样第一轮前向结果与原始模型完全一致，loss 不会突变，训练更稳定。实际项目中，我遇到过因为 B 没用全零初始化导致 loss 一开始就炸掉的情况，排查了很久。

LoRA 的优点包括显存节省（只需存 A、B 的梯度和优化器状态，对 7B 模型可以降低 60-70% 显存占用）、训练速度快（约是全量微调的 2-3 倍）、便于切换（不同任务的 adapter 可以独立保存，推理时动态切换）。局限性在于表达能力受限于秩，对于需要大幅度修改模型行为（如改变输出格式或风格）的任务，效果可能不如全量微调。常见的坑包括：适配器合并后的精度损失（尤其是混合低比特量化时），以及在长文本生成中 LoRA 层可能成为性能瓶颈。

**常见追问与回答：**
- Q：为什么通常选择在 Attention 的 Q 和 V 上加 LoRA，而不是 FFN？A：实验表明 Q 和 V 的权重变化对下游任务影响更大且更具领域特异性，FFN 的权重偏向通用知识，改动后容易破坏预训练学的语言能力。不过我在项目中发现，对于代码生成任务，在 FFN 上加 LoRA 反而效果更好，需要根据任务实验确定。
- Q：LoRA 的秩 r 和 α 怎么取值？A：r 一般取 8、16、32，α 通常设为 r 的 2 倍。α 是用来缩放低秩矩阵输出的系数，实际效果是控制 ΔW 的贡献大小。α 过大导致训练不稳定，过大则 adapter 几乎不生效。

**项目联系：**
在我做的一个面向企业内部文档问答的对话 Agent 中，我使用 LoRA 对基座模型进行领域知识微调。具体做法是：收集 5000 条产品手册问答对，对 Llama-2-7B 的 Q 和 V 层加 LoRA（r=16, α=32），用 4 块 A100 训练了 2 小时。训练后模型在领域问答的 BLEU 和 ROUGE-L 上分别提升 12% 和 8%。我还做了消融实验，发现只微调 Q 层的效果比 Q+V 同时微调差 5% 左右，验证了 V 层在领域特化中的重要性。

**记忆关键词：**
低秩分解、ΔW = BA、零初始化、Q 和 V 层、秩 r、adapter 切换

### 题目 2：全量 SFT 和 LoRA SFT 的 learning rate 设置有什么技巧？为什么会有这种差异？

**来源依据：**
- 2026大厂大模型/Agent面试题-CSDN博客 - https://blog.csdn.net/weixin_44151034/article/details/161729146 - 面经或社媒 - 中，来源为牛客网面经摘录，需进一步验证

**可信度：** 中

**考察点：** LoRA 训练细节、优化器策略、工程经验

**面试口头版回答：**
全量 SFT 的 learning rate 通常设置在 1e-5 到 5e-5 之间，而 LoRA SFT 的 lr 通常要高出 1-2 个数量级，一般在 1e-4 到 1e-3 之间。原因有两个：第一，LoRA 只更新极少量参数（通常占总参数的 0.1%-1%），每个参数的“学习压力”更大，需要更大的步长才能有效改变模型行为；第二，LoRA 的低秩矩阵是随机初始化的，初始梯度很小，如果用全量微调的 lr，几乎学不动。我在实践中常用的策略是：先用 2e-4 作为起步 lr，配合 cosine scheduler 逐步衰减到 2e-5，并监控前 100 步的 loss 下降情况，如果下降太慢就翻倍 lr。

**深入理解版回答：**
LoRA 和全量 SFT 的 lr 差异本质上源于参数空间的“密度”不同。全量 SFT 更新所有参数，每个参数承担的信息密度低，大 lr 容易导致灾难性遗忘——模型会迅速忘记预训练学到的通用能力。LoRA 只更新低秩矩阵，参数总量少但每个参数需要承载的信息密度高，因此需要更大的 lr 来驱动学习。

从优化器角度看，LoRA 的 A 和 B 矩阵在初始化时输出接近 0，这意味着初始梯度接近 0，如果 lr 太小，梯度被优化器的 momentum 和 adaptive 机制淹没，实际上学不动。我做过实验：LoRA lr 设为 1e-5 时，前 500 步 loss 几乎没有下降，而设为 2e-4 时，loss 在 100 步内就降了 30%。

工程上的建议是：不直接复用全量 SFT 的 lr 配置。我通常先用一个小数据子集（200 条）做快速实验，扫描 lr 在 [1e-4, 1e-3] 范围的前 100 步 loss 曲线，选最陡的。另外，不同秩的 LoRA 也需要不同 lr：r 越小，需要 lr 越大。

常见的坑：有些人直接把全量微调的 lr 套到 LoRA 上，发现 loss 根本不降，然后盲目加训很多步，既浪费资源又可能导致过拟合。另一个坑是学习率过大导致 loss 炸裂，这时需要检查是否初始化有问题（B 是否全零）。

**常见追问与回答：**
- Q：你监控 loss 下降，但看到过拟合的依据是什么？A：除了 loss，我还会看验证集上的困惑度（Perplexity）和下游任务指标。如果训练 loss 持续降低但验证 PPL 开始回升，就说明过拟合了，这时要么减少训练步数，要么增大秩 r 或增加 dropout（LoRA 本身不带 dropout，但可以在 adapter 输出上加）。
- Q：LoRA 微调时，如何设置 optimizer 参数？A：我通常用 AdamW，weight decay 设为 0.01，betas=(0.9,0.999)。LoRA 本身参数少，weight decay 不需要太大，否则容易把低秩矩阵的范数压得太小，制约表达能力。

**项目联系：**
在我用 LoRA 微调对话 Agent 的意图分类模块时，我记录了 lr 扫描实验数据：lr=1e-4 时，100 步后分类准确率 72%；lr=2e-4 时，准确率 85%；lr=5e-4 时，准确率 82% 但训练出现波动。最终选择 lr=2e-4，配合 warmup ratio 0.03。这个过程也帮我理解了为什么面试官喜欢问“你踩过什么坑”——因为我真的踩过 lr 太小学不动的坑。

**记忆关键词：**
lr 差异、参数密度、LoRA 学习率、初始梯度、快速实验

### 题目 3：QLoRA 为什么选择 NF4 和 FP16 这个组合？NF4 的分布拟合逻辑是什么？

**来源依据：**
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 中，来源为 GitHub 面经集合，需进一步验证

**可信度：** 中

**考察点：** QLoRA 原理、NF4 量化、混合精度训练

**面试口头版回答：**
QLoRA 选择 NF4 + FP16 组合是因为 NF4 是正态分布适配的 4-bit 量化格式，它能用 4-bit 表示近似正态分布的权重值，而大模型的权重分布恰好处处是正态或类正态分布。NF4 采用了一个双级量化策略：先将参数分组归一化到标准正态区间，然后用非均匀的 4-bit 码表去拟合 N(0,1) 分布，使得量化精度损失最小。而 FP16 用于保持模型输出的精度，因为 Activation 和 LoRA 适配器的精度要求更高，4-bit 不足以支持反向传播。简单说，NF4 存权重、降显存，FP16 算梯度、保精度，各司其职。

**深入理解版回答：**
QLoRA 的核心创新在于它能够在 4-bit 量化后的基座模型上施加 LoRA 微调，同时保持接近 16-bit 的微调效果。这个“近无损”的关键就在于 NF4（NormalFloat4）数据格式。

NF4 的设计思路源于对权重分布的先验知识：预训练模型各层的权重近似正态分布，均值附近密度高、尾部密度低。传统的 INT4 是均匀量化，各区间等宽，对于正态分布会浪费尾部区间的精度。NF4 采用“分位数量化”思路：先把权重归一化到均值为 0、方差为 1 的区间，然后按照标准正态分布的分位数切分出 16 个区间（4-bit 对应 16 个值），每个区间用中心值表示。这样在概率密度高的区域（均值附近）量化更精细，尾部分辨率低，整体量化误差最小。

具体工程实现上，QLoRA 做了双重量化：第一层用 NF4 量化权重，第二层将 NF4 的量化常数（scale）又用 FP8 量化一次，进一步压缩显存。这样对于 7B 模型，基座权重从 FP16 的 14GB 降到 NF4 的约 3.5GB，节省了约 75% 显存。同时，前向计算时用 NF4 权重以 4-bit 计算，反向传播时梯度用 FP16 的 adapter 计算，确保梯度精度不损失。

为什么选 NF4 不选 INT4？实验表明，对于大模型正态分布的权重，NF4 的量化精度损失（perplexity 增加）仅为 0.5% 左右，而 INT4 会导致 2-3% 的损失。对于 FP16 梯度保持：NF4 的 4-bit 数值范围覆盖不了梯度大小，用 FP16 是必要的取舍。

**常见追问与回答：**
- Q：NF4 量化后，模型在推理时怎么快速解量化？A：QLoRA 使用基于 lookup table 的解量化方式：预计算好 NF4 码表（16 个 FP16 值），前向时直接按 4-bit 索引查表得到中精度的 FP16 值然后计算。这一步可以通过 CUDA kernel 优化，解量化开销很小。
- Q：QLoRA 微调后，能不能把 NF4 权重和 LoRA adapter 合并导出？A：可以，但需要将 NF4 量化参数和解量化后的权重与 LoRA 矩阵合并，得到一个 FP16 的完整模型权重。导出时需要做反量化（dequantize）和合并操作，在推理框架里以 FP16 加载。如果保持 NF4 分开加载，推理时需要跑解量化逻辑，多了步开销但显存更省。

**项目联系：**
在我的 RAG 项目中，我需要在一台单卡 A100-40G 上部署和微调 13B 模型。使用 QLoRA（NF4 + FP16 + LoRA r=16）后，基座模型仅占 6.5GB，加上 adapter 约 0.5GB，总共 7GB，远低于全量微调所需的 ~26GB。这使得我能够用更小的 GPU 完成领域微调实验，快速迭代领域语料效果。

**记忆关键词：**
NF4、正态分布分位数、双重量化、FP16 保梯度、QLoRA 显存节省

### 题目 4：LoRA 微调时，你是如何选择冻结层的？依据是什么？

**来源依据：**
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 中，来源为 GitHub 面经集合，需进一步验证

**可信度：** 中

**考察点：** LoRA 层选择策略、微调深度对效果的影响

**面试口头版回答：**
我选择冻结层的依据是层级有效性与资源效率的平衡。通常做法是只在 Attention 层的 Q 和 V 投影上加 LoRA，因为已有研究（LoRA 原论文）和对齐实验表明这两个矩阵的权重变化对下游任务影响最大。如果资源允许，我会进一步微调所有 Attention 层的 Q、K、V、O，以及 FFN 层的中间投影。实际项目中，我习惯于先做一个小规模的 probing 实验：用固定 r=8，分别测试不同层组合（只 Q、Q+V、所有 Attention、全部层）在验证集上的指标，选择性价比最高的方案。对于 7B 模型，通常 Q+V 就能达到全量 SFT 的 90% 以上效果而参数量仅增加 0.1%。

**深入理解版回答：**
冻结层的选择本质上是在问：模型的不同层在下游任务中的重要性排序如何？LoRA 原论文做了消融实验，结论是 Q 和 V 层最敏感，而 FFN 层的权重变化对任务影响不大，可能是因为 FFN 存储的是通用语言知识，改动后容易引发灾难性遗忘。

从训练稳定性来看，如果选择的层太少（比如只 Q 层），适配器能力不足，模型学不到领域特有的模式，出现欠拟合；如果选择的层太全（所有层都加 LoRA），适配器参数总量变大，如果训练数据不够多，容易过拟合。我遇到过一个案例：只微调 Q 层的 LoRA 在代码生成任务上 F1 只有 0.65，同时微调 Q+V+O 后提升到 0.78，再微调 FFN 反而掉到 0.73（原因是非 Attention 层的扰动破坏了泛化性）。

工程上，还有一种做法是渐进式解开冻结：先用 LoRA 微调 Q、V，固定其他层，观察 loss 下降如果慢了，就逐步解开 K、O，最后再考虑 FFN。这样训练更可控，可以随时监控是否过拟合。

常见的坑：有人不加验证直接在 embed 层上加 LoRA，这通常没意义，因为 embed 层只是查表，没有“知识”可以微调。另一个坑是对不同层用相同 r 值，实际上浅层（靠近 embed）的权重变化通常更大，应该设置更大的 r。

**常见追问与回答：**
- Q：你提到渐进解开冻结，具体怎么实现？A：在训练代码里设计一个冻结层控制列表，如 `[“q_proj”, “v_proj”]` 前期只挂这两层 LoRA。每训练 N 步后在验证集上评测，如果 PPL 下降变缓，就加 `[“k_proj”, “o_proj”]`，再训练 N 步，最后加 FFN。这种策略在数据量小的场景非常有效，能在防止过拟合的同时逐步释放模型能力。
- Q：你怎么知道是这个模块需要微调而不是数据量不够？A：我会监控每层适配器的梯度范数。如果某层的梯度范数长期为 0（接近 0），说明该层对当前任务不敏感，加不加都一样；如果梯度范数很大但 loss 不降，说明该层的学习信号是噪音，可能是数据质量的问题。这个诊断方法在我排查微调失败时帮了大忙。

**项目联系：**
我在微调一个客服对话模型时，用了渐进式冻结策略：首先只挂 Q+V（r=8），训练 200 步后 PPL 从 8.0 降到 6.2；再接 K+O 层，再 200 步到 5.5；最后接 FFN 但 PPL 反而回升到 5.8，所以回退到最后一步，最终模型取 Q+V+K+O 的配置。最终模型在客服意图识别上 F1 0.92（全量 SFT 为 0.94），很接近全量微调的效果但节省了 70% 显存。

**记忆关键词：**
层选择、梯度范数、渐进式冻结、Q+V 高敏感、过拟合

### 题目 5：SFT 训练数据是怎么构造的？如何保证样本多样性和质量？

**来源依据：**
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 中，来源为 GitHub 面经集合，需进一步验证

**可信度：** 中

**考察点：** SFT 数据工程、数据清洗、多样性平衡

**面试口头版回答：**
SFT 数据构造的核心是让模型学会“在上下文中按预期方式输出答案”。我的流程分四步：第一，从领域文档中通过规则提取问答对（如从企业 IT 工单记录中提取“问题-解决方案”对）；第二，用 LLM 做数据增强，将已有样本改写为不同问法（如同义改写、句式变换、多轮对话拆分），增加多样性；第三，质量过滤，包括去重（基于文本相似度）、去低质量（语义不完整或答案过于短小的）、去敏感内容；第四，平衡各意图类别的样本比例，防止模型对占比过大的类别过拟合。对于质量，我引入人工抽样标注和 LLaMA-Guard 式自动过滤，确保符合安全规范。

**深入理解版回答：**
SFT 数据的质量和多样性直接决定微调后模型的表现。我总结为“三要三不要”：要多样、要干净、要领域聚焦；不要依赖单一来源、不要容忍错别字和逻辑矛盾、不要忽略安全边界。

数据构造的工程流程是：源数据获取（公开领域语料、用户对话日志、人工标注） -> 规则清洗（去 HTML 标签、统一编码、去短于 10 个词的样本） -> 自动提取（用 LLM 或正则提取“指令+输入+输出”三元组） -> 质量打分（基于流畅度、信息密度、安全性） -> 多样性采样（将每条样本编码为向量后做聚类，每类取一定比例，防止单一模式数据主导）。我还在数据集中混入 5% 的拒绝回答样本（如“我不知道”、“请咨询专业人士”），帮助模型学会在不确定时 say no。

多样性保证方面，我采用“数据池”策略：维护一个大规模候选池（几十万条），训练时按意图分布采样。同时将不同来源的数据颜色编码，训练时检查每个 batch 中各类别占比，如果某类占比超过 30%，就从该类别中随机丢弃样本，迫使模型接触更多样例。

常见的坑是：很多人只关注数据量而忽略质量。100 万条模板化的低质数据，不如 5 万条精心构造的高质数据效果好。另一个坑是：微调数据与模型基座的数据分布偏差过大（比如用纯产品文档去微调一个基座模型但预训练权重是百科数据），导致模型学不到有效对齐。

**常见追问与回答：**
- Q：你提到了自动去重，具体用什么方法？A：我用了 SimCSE 把每条样本编码为向量，基于余弦相似度的 DBSCAN 聚簇，每簇内只留 1 条。另外，句子级别的 n-gram 去重（5-gram 以上）清除模板化重复。
- Q：处理倾向性标签（bias）时你做了什么？A：在人工标注环节，我引入多标注员一致性校验（简单说就是要求 3 个人对同一条 QA 打“质量标签”，多数投票才通过）。另外，用 Fairness metrics 评估微调后模型在敏感属性上的表现差异。

**项目联系：**
在我构建客服 Agent 的 SFT 数据时，从 3 万条历史工单中提取了 8000 条 QA 对，经过 LLM 改写扩充到 25000 条（每种问法 3-5 个变体），再用 BERT 向量去重，剩余 15000 条。按“用户查询 = 产品升级 / 故障排查 / 账单问题”三类平衡采样。微调后模型首轮正确率提升 18%。

数据清洗、多样性采样、聚类去重、拒绝回答样本、质量打分

## 3. 今日必背问答卡片

### 卡片 1

- **Q：** LoRA 的低秩矩阵初始化方式是什么？为什么这样设计？
- **A：** A 矩阵（d×r）用随机高斯初始化，B 矩阵（r×k）用零初始化。这样初始输出 BAx = 0，模型行为与原始权重一致，loss 和梯度不会突变，训练更稳定。如果 B 也用随机初始化，第一轮前向输出会带随机噪声，可能导致 loss 炸裂。
- **一句话记忆法：** A 随机取色彩，B 全零保初始；初态不改原始力，训练才稳不炸梯。
- **复习标签：** #LoRA #初始化 #训练稳定性

### 卡片 2

- **Q：** 全量 SFT 和 LoRA SFT 的 learning rate 为什么有数量级差异？
- **A：** 全量 SFT 更新所有参数，lr 通常 1e-5~5e-5；LoRA SFT 只更新少量低秩参数（约总量 0.1%），每个参数承担更大学习压力，需要更大步长（1e-4~1e-3）驱动。如果 LoRA 用全量微调的 lr，梯度几乎为 0，模型学不动。
- **一句话记忆法：** 人多步小（全量参数多 lr 小），人少步大（LoRA 参数少 lr 大）。
- **复习标签：** #LoRA #SFT #学习率 #优化策略

### 卡片 3

- **Q：** QLoRA 为什么选 NF4 + FP16 的组合？NF4 的分布拟合是什么逻辑？
- **A：** NF4 是正态分布适配的 4-bit 格式，用分位数法拟合 N(0,1) 分布，量化误差小；FP16 用于保持梯度和 LoRA 适配器的精度。双重量化（NF4 存权重、FP8 量化 scale）大幅节省显存（7B 模型从 14GB 降到 3.5GB）。
- **一句话记忆法：** NF4 存权降显存，FP16 算梯保精度，正太分位靠拟合，双重量化显存省。
- **复习标签：** #QLoRA #NF4 #量化 #显存优化

### 卡片 4

- **Q：** LoRA 微调时选择冻结层的依据是什么？
- **A：** 通常只选 Attention 的 Q 和 V 层，因为实验表明这两层的变化对下游任务最敏感。更精细的策略是渐进式：先 Q+V，指标不升再加 K+O，最后加 FFN，避免过拟合。不同层用不同 r 值（浅层 r 大）。
- **一句话记忆法：** QV 主力军，渐进派工兵；过拟合时往回撤，加层要看梯度信号。
- **复习标签：** #LoRA #冻结层 #渐进式微调 #过拟合

### 卡片 5

- **Q：** SFT 训练数据是怎么构造的？如何保证多样性和质量？
- **A：** 四步法：1. 源数据清洗（去重、去短、去敏感）；2. LLM 增强（改写不同问法）；3. 质量打分（流畅度、信息密度、安全性）；4. 多样性采样（聚类后按类比例取样）。混入 5% 拒绝回答样本教模型说“不知道”。

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 在 QLoRA 微调中，NF4 量化的权重在前向计算中是如何做到“4-bit 权重的计算精度接近 FP16”的？请从量化、反量化和数值稳定性角度解释。

**面试口头版参考答案：**
QLoRA 的关键在于设计了 NF4 格式，它能用 4-bit 表示近似正态分布的值，同时配合一个高效的反量化机制。前向计算时，QLoRA 不会真的用 4-bit 做矩阵乘法，而是先把 NF4 权重“解量化”回 FP16，然后用 FP16 做运算。这个解量化步骤通过预先计算好的查表实现，速度很快。同时，NF4 的分位数量化保证了绝大多数权重值（均值附近）的量化精度很高，只有很靠尾巴的值才有大误差，但这些权重数量极少，所以整体误差极小。反向传播的梯度全部基于 FP16 的 adapter 计算，所以梯度精度不受 NF4 影响。

**深入理解版参考答案：**
背后有三个机制协同工作：

1. **NF4 的分位数量化**：传统 Uniform 量化为每个值分配等宽的区间，但训练好的权重是“中间密、两头稀”的正态分布，均匀量化中间这些密集值会丢精度。NF4 把标准正态分布的累积分布函数（CDF）的取值区间 [0,1] 等分为 16 个值，逆向计算出对应分位点的真实数值（如 -0.5, -0.3, -0.1, 0.1, 0.3, 0.5...），然后每个 NF4 的 4-bit 码对应其中一个分位点。比如权重 0.15 落在 [0.1, 0.3] 之间，它就就近量化为“0.1”对应的码。因为正态分布在均值附近概率密度高，这些值在分位数上反而在较密集的小区间，所以量化精度高；尾部的值概率密度低，量化精度差一点，但权重值极少，对整体影响很小。数学上这相当于在概率空间中做均匀量化，而在数值空间做非均匀量化。

2. **双重量化**：QLoRA 第一步对每 64 个权重做 NF4 量化并存储其 scale（该组的最大值），这个 scale 本身用 FP16 存的话，会额外消耗显存。所以第二步，再把这个 FP16 scale 自己做一次 8-bit 量化（FP8），就形成“双重量化”。这步大约能再省 0.5 比特/参数。

3. **解量化和计算**：前向计算时，先通过查表（预计算好的所有码对应的 FP16 值）快速解量化出 FP16 权重，然后进行标准的 FP16 矩阵乘法。解量化只在内存引入一次小开销，计算仍在 GPU 的 FP16 Tensor Core 上跑，所以吞吐基本不受影响。

**稳定性保障**：由于 base 权重被 NF4 量化后会有微小误差，QLoRA 通过以下方式补偿：① LoRA adapter 的 FP16 参数在学习中修正因为量化引起的微小误差。② 训练过程中，QLoRA 会定期把 FP16 的 LoRA 权重“合并”回量化后的 base 权重（通过误差反传更新 base 量化值），但这步在实现中通常每几百步做一次，不是每步都做。

**常见追问与回答：**

- Q：NF4 量化会引入多少额外的推理延迟？A：解量化的查表操作在 GPU 内存带宽上是 O(n) 的，相比矩阵乘法的 O(n²)，开销可忽略。实际测量显示，NF4 量化的 LLM 推理延迟仅比 FP16 版本高 1-3%。但因为显存减半，可以塞更大的 batch，所以总体吞吐（throughput）反而是提高的。

- Q：QLoRA 微调后的模型，能不能以纯 NF4 形式部署做推理？A：可以。QLoRA 微调后，你可以选择把 LoRA adapter 合并进 NF4 权重（重新量化合并后的权重），部署时只加载 NF4 模型即可。但这样做需要将 NF4 的 scale 也重新更新。也有框架支持直接将 LoRA adapter 作为 FP16 侧挂接，推理时同时加载 NF4 base 和 FP16 adapter，计算前反量化 base 权重后再加 adapter。但这样会增加一点内存和计算开销。
