### 题目 1：如何设计 RAG 中的文档切片（Chunking）策略？chunk_size 怎么定？如何评估效果？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 这几道高频RAG面试题你都会了吗 - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 高可信度（腾讯云开发者社区）

**可信度：** 中

**考察点：** RAG 链路、文档预处理、chunk 策略、工程经验

**面试口头版回答：**
我通常采用“语义切块”而不是固定字数切分。chunk_size 没有绝对最优值，需要根据文档类型和检索任务实验。我一般先定一个基准，比如 256 或 512 tokens，然后做滑动窗口加 overlap（比如 10-20%）。评估效果不能只靠感觉，我用 Hit Rate 和 MRR 来量化检索质量：Hit Rate 衡量正确答案是否出现在 top-k 里，MRR 关注排位。例如把一个企业文档集切成不同 size，对比两种指标，选择能让召回和精排都高效的 size。如果文档有自然段落，我会优先按段落切，避免在句子中间断开。

**深入理解版回答：**
Chunking 策略的核心是在“信息完整性”和“检索粒度”之间做权衡。切得太细（比如 64 tokens）会导致语义碎片化，关键信息被截断；切得太粗（比如 1024 tokens）会使单个 chunk 包含噪声，且更容易触发模型的“Lost in the Middle”问题，因为大模型对长上下文中部的信息利用度低。工程上常用滑动窗口 + overlap 保证上下文连贯。更高级的做法是使用语义分块（如基于句向量变化检测边界）或按文档结构（段落、章节）切分，但需要额外计算。评估方法：构建一个标注好的 QA 测试集，每个问题关联正相关文档段。然后固定检索模型和生成模型，只变 chunk 策略，计算 Hit Rate@K、MRR、以及最终答案的 Faithfulness。通过 A/B 实验选择最优配置。常见坑：忽略 overlap 导致边界信息丢失；仅凭直觉选择 size 而不做量化验证。

**常见追问与回答：**
- **追问：你见过哪些因为 chunk 没切好导致的 bad case？**
  答：用户问“合同第三条第2款是什么”，结果 chunk 刚好从第2款中间断开，模型找不到完整条款，容易漏答或编造。
- **追问：多文档混合检索时，chunk 策略需要不同吗？**
  答：是的。比如纯文本、PDF、表格数据应该分别用不同策略：表格按行或按表格块切分，代码按函数切分。

**项目联系：**
在我的 RAG 项目中（企业内部资料问答），初期使用固定 512 tokens 切分，结果发现长合同的条款经常被切断。后来改为按段落 + 滑动窗口（256 tokens，overlap 64），Hit Rate@5 提升了 12%。我还写了一个自动调优脚本，遍历多个 size 组合，结合 MRR 自动选出最佳配置。

**记忆关键词：**
语义切块、chunk_size、overlap、Hit Rate、MRR、量化评估

### 题目 2：什么是 Rerank（重排序）？在工程中如何实现？它解决了什么问题？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 这几道高频RAG面试题你都会了吗 - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 高可信度

**可信度：** 中

**考察点：** RAG 链路、向量检索 vs 交互式排序、rerank 工程实践

**面试口头版回答：**
Rerank 是在向量召回之后增加的一层精排环节。向量检索很快但只做语义近似匹配，缺乏查询与文档间的深层交互。Rerank 通常用 Cross-Encoder 模型，同时输入 query 和候选文档，输出一个相关性分数。它弥补了向量检索的不足，能更精准地判断哪些文档真正相关。工程实现上，我一般先用向量检索召回 top-50 或 top-100 的候选，再用一个轻量级的 ReRanker（比如 BGE-Reranker）计算分数，最后取 top-3 到 top-5 送给 LLM 生成。这样做既保证了召回覆盖率，又通过精细排序提升了生成质量。

**深入理解版回答：**
Rerank 是 RAG 中“粗排 + 精排”双阶段检索的关键。第一步用双编码器（Bi-Encoder）做向量检索，速度快但缺乏 query 和文档间的词级交互。第二步用 Cross-Encoder 做精排，它把 query 和每个文档拼接后一起经过 Transformer，输出一个相关性分数，计算更精确但开销大（O(n) 推理）。所以必须控制召回阶段送回的数量（如 top-100）。模型选型：常用的有 BGE-Reranker、Cohere Rerank、bge-reranker-v2-m3 等，参数量一般几亿到十亿，单卡可跑。优化点：可以批量推理、将 rerank 请求放在 GPU 上做 batch；还可以对结果做阈值过滤，丢到低于 0.3 分的文档。实际效果：在工业级项目中，添加 Rerank 后 top-3 正确率通常能提升 5-10 个百分点。

**常见追问与回答：**
- **追问：Rerank 的延迟怎么优化？**
  答：控制召回数量（如 top-50），使用小模型（如 BGE-Reranker 300M），并用批量推理（一次送入多个 query-doc pair 到 GPU）提升吞吐。
- **追问：能不能用 LLM 做 rerank？**
  答：可以，但成本高。只在极关键场景下用 GPT-4 或类似模型做“LLM-as-judge”对候选文档打分，日常用轻量级 Cross-Encoder。

**项目联系：**
在我的项目中，初期只用向量检索，发现有些语义相近的文档实际内容不相关（比如“苹果手机”和“红富士苹果”）。增加 BGE-Reranker 后，将 top-20 候选重排到 top-3，最终的答案忠实度从 78% 提升到 89%。我们还做了基于 Triton Inference Server 的 Rerank 服务，批量推理延迟控制在 50ms 以内。

**记忆关键词：**
Cross-Encoder、双阶段检索、粗排+精排、BGE-Reranker、延迟优化

### 题目 3：如何评估 RAG 系统的检索和生成质量？具体用哪些指标？如何自动化计算？尤其是忠实度（Faithfulness）？

**来源依据：**
- 这几道高频RAG面试题你都会了吗 - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 高可信度
- LLM Interview Questions and Answers (2026) - https://www.interviewbit.com/llm-interview-questions-answers - 技术博客 - 中高可信度

**可信度：** 高

**考察点：** RAG 评测体系、指标选择、自动化评估

**面试口头版回答：**
我把 RAG 评估分为三层。第一层是检索质量：用 Hit Rate 和 MRR 衡量是否召回正确文档以及它们的排序。第二层是生成忠实度（Faithfulness）：指模型回答是否基于检索到的文档，有没有无中生有。自动化计算忠实度一般用 NLI 模型（比如 TrueTeacher）或 LLM-as-judge：将回答拆成原子声明，检查每个声明是否能从给定文档中推理出来。第三层是答案正确性：对比标准答案，用 ROUGE-L、BLEU 等，但更推荐用语义相似度或 LLM-as-judge。实际工程中，我直接用开源的 RAGAS 框架，它能自动计算 context_precision、context_recall、faithfulness 和 answer_relevance。

**深入理解版回答：**
构建一个可靠的 RAG 评测体系需要结合自动化和人工。自动化指标：检索侧用 Hit Rate@K（召回命中率）、MRR（平均倒数排名）；生成侧用 Faithfulness（忠实度）和 Answer Relevancy（答案相关性）。Faithfulness 是核心指标，因为 RAG 的价值在于可溯源。自动计算 Faithfulness 的常见方法是：把 LLM 回答拆成若干 claim，然后让一个 NLI 模型判断每个 claim 是否被文档片段支持（entailment）。如果不支持，即认为有幻觉。RAGAS 框架就是采用这个思路。注意：NLI 模型本身也有偏差，所以可以结合 LLM-As-Judge 做双重校验，或用投票机制。另外，需要构建一个标注集定期评估指标与人工判断的一致性。避免单一指标陷阱，比如回答“我不知道”可能 Faithfulness 高但 Answer Relevancy 低，要综合看待。

**常见追问与回答：**
- **追问：用什么模型来做 Faithfulness 的自动判断？**
  答：可以用开源的 DeBERTa 微调 NLI 模型（如 TrueTeacher），或直接用 GPT-4 在少量样本上做 prompt 评估。最稳妥的方式是混合：小模型做批量初筛，大模型做难例二次判决。
- **追问：你如何保证评测指标与用户满意度一致？**
  答：定期做 A/B 线上实验，将离线指标（Faithfulness、Hit Rate）与线上指标（用户点赞率、会话长度、反馈率）关联，建立回归模型。如果某个离线指标和线上正相关度高，就以它为核心。

**项目联系：**
在我的项目中，我搭建了基于 RAGAS + 人工抽样的评测流水线。离线跑完指标后，每隔一周抽 100 条 bad case 人工标注，发现 Faithfulness 低于 0.7 时用户投诉率明显偏高。因此我把 Faithfulness 设为红线指标，版本发布前必须达标。还做了一个 dashboard 实时监控。

**记忆关键词：**
Hit Rate、MRR、Faithfulness、NLI 评估、RAGAS、LLM-as-judge

### 题目 4：RAG 中的引用（Citation）机制如何设计？如何验证引用的准确性？

**来源依据：**
- 这几道高频RAG面试题你都会了吗 - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 高可信度
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证

**可信度：** 中

**考察点：** 引用设计、溯源、工程实现

**面试口头版回答：**
引用机制是 RAG 相比纯生成模型的核心优势。我通常让 LLM 在生成时显式标注引用来源，比如“根据文档1第3段，答案是...”。有两种实现方式：一种是强制模型在输出格式中包含 `[source_id]`，通过 prompt 约束；另一种是在后处理阶段将生成文本与检索到的文档片段做按句对齐，利用语义相似度或文本匹配自动打上引用标签。验证引用准确性，需要构建评估集，人工标注每个 claim 对应的正确来源，然后用“引用命中率”指标：看模型输出的引用是否指向正确的文档片段。特别要注意的是，防止模型编造引用——必须保证引用的文档确实是检索出来且传递下去的。

**深入理解版回答：**
设计引用机制的关键是“可证实”和“可追溯”。工程上，我推荐在 prompt 中给文档编号（如 `[Doc1]`、`[Doc2]`），并明确要求输出时引用编号。但 LLM 可能不遵守格式，所以还要做后处理校验：用规则正则提取引用标记，检查标记是否在传入的文档编号范围内，超出则丢弃。另一种更可靠的办法是“检索-生成-验证”链路：生成后，将每个句子通过小模型与文档片段计算相似度，相似度最高的片段作为该句的引用；如果相似度低于阈值（如 0.6），就标记为“无引用来源”，触发兜底策略。评测引用准确性：可以计算“引用精准度”（模型引用的文档是否真包含该句信息）和“引用召回率”（正确信息有多少被引用了）。常见坑：模型引用与内容矛盾（如说A引用的是B），需要做一致性校验。

**常见追问与回答：**
- **追问：如何防止模型伪造引用？**
  答：在后处理阶段强制检查：如果模型引用了`[Doc3]`，但实际只给了2个文档，则丢弃该引用，并用提示词修正输出。
- **追问：引用格式如何影响用户体验？**
  答：对 C 端用户，可设计成超链接悬浮提示；对 B 端合规场景，要求每句都精确标到段落，并提供原文预览。

**项目联系：**
我的项目是企业知识问答，合规要求每句话必须有出处。我采用混合方式：prompt 强制引用 + 后处理对齐。用 BGE 模型计算每个句子与文档片段的相似度，低于 0.7 的句子视为“无引用”，然后将该句重新送入 LLM 要求重写或标注为“模型推测”。上线后引用准确率达到 96%，用户投诉减少 50%。

**记忆关键词：**
引用标记、后处理对齐、引用精准度、引用召回率、反幻觉

### 题目 5：什么是 HyDE（假设文档嵌入）？它在零样本检索中起什么作用？

**来源依据：**
- 40 Generative AI Interview Questions That Actually Get Asked in 2026 - https://pub.towardsai.net/40-generative-ai-interview-questions-that-actually-get-asked-in-2026-with-answers-b4c647f1e2e8 - 技术博客 - 中高可信度
- - YouTube - https://www.youtube.com/watch?v=BpG21SftCrE - 其他 - 需进一步验证

**可信度：** 中

**考察点：** HyDE 原理、零样本检索、Query Rewrite

**面试口头版回答：**
HyDE 全称是“假设文档嵌入”，用于解决用户查询简短模糊导致检索不准确的问题。核心思路：先用一个 LLM 根据用户查询生成一段“假设的理想答案”或“假设文档”，然后把这段假设文档编码成向量去检索。因为假设文档在语义空间上更接近真实文档，所以检索效果更好。它是一个零样本方法，不需要训练，只需要一个通用 LLM 和嵌入模型。但缺点是需要额外一次 LLM 调用，增加了延迟，且如果 LLM 生成的内容跑偏，反而会引入噪声。

**深入理解版回答：**
HyDE 出自论文《Precise Zero-Shot Dense Retrieval without Relevance Labels》。用户在搜索框中输入 short query（如“苹果手机对比华为 ”），向量检索直接基于短 query 编码，往往与文档空间存在语义 gap。HyDE 的做法是：调用 LLM 基于该 query 生成一段详细的“假设文档”（hypothetical document），比如一段包含技术参数、优缺点的对比文本。然后将这段假设文档送入同一个 embedding 模型得到向量，再去向量库检索。因为假设文档的向量空间与真实文档更对齐，检索结果更精准。实际使用中，需要控制生成的质量和长度：太短不够区分，太长可能引入幻觉。也可以生成多个假设，取平均向量或加权融合。在 RAG 项目中，HyDE 可用于第一轮检索提升 recall，但成本是额外一次 LLM 调用（通常用小模型如 llama-8B 即可）。与 Query Rewrite 的区别：Rewrite 是将 query 改写成更明确的版本，然后用改写后的 query 编码；HyDE 输出的是完整的假设文档而不仅是 query。两者可互补。

**常见追问与回答：**
- **追问：HyDE 的延迟怎么办？**
  答：可以用更小的、专为句子生成优化的模型（如 FLAN-T5-small），或在低峰期批量生成假设文档并缓存。另外，只在 query 长度 < 10 tokens 的场景启用 HyDE。
- **追问：什么情况不适合用 HyDE？**
  答：当 query 本身已经足够详细（如包含具体实体、属性）时，HyDE 的增益有限，甚至可能引入噪声。只在短 query 或指代模糊时推荐。

**项目联系：**
在我做的内部知识库问答中，用户常输入“最新促销政策”这类模糊 query，导致召回混乱。我在第一版接入 HyDE，用 Qwen2.5-7B 生成一段假设政策文档，再用 bge-large-zh 编码。离线评测 Hit Rate@10 从 67% 提升到 79%。但线上延迟增加了约 200ms，所以只针对特定低延迟场景（如客服助手）保留 HyDE，其他场景用传统 Query Rewrite。

**记忆关键词：**
HyDE、假设文档嵌入、零样本、语义 gap、短 query 优化

- **可转化面试题：**
  - **问题1：** 在 RAG 中，余弦相似度和欧氏距离在向量检索中有什么区别？为什么常用余弦相似度？
    **参考答案：** 余弦相似度关注方向一致性，对向量长度不敏感；欧氏距离关注绝对距离。在 embedding 空间中，长短不一的向量往往只有方向有意义（长度多代表句子长度或语调），因此余弦相似度更稳定。实际工程中，如果将向量归一化后，余弦相似度和内积等价，所以很多向量数据库默认用内积。
  - **问题2：** 增量索引时如何处理残留文档和版本一致性？
    **参考答案：** 我采用 chunk 粒度 upsert：deletion 时标记 chunk 级失效而非删除物理数据，利用 vector+metadata 的版本号做乐观锁。检索时过滤掉失效版本，确保只有最新版本被召回。同时定期做 compaction 清理。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** 在 RAG 中，chunk_size 一般选多大？该如何决定？
- **A：** 没有固定值，通常从 256 tokens 开始实验。优先按文档自然段落切分，配合滑动窗口（overlap 10-20%）。要用 Hit Rate 和 MRR 做量化对比，选择能让召回率最高的配置。太细导致碎片化，太粗引发 Lost in the Middle。
- **一句话记忆法：** chunk 不是切西瓜，大小要实验定，overlap 保边界，Hit Rate 说了算。
- **复习标签：** #RAG #Chunking #HitRate

### 卡片 2
- **Q：** 为什么 RAG 中需要 Rerank（重排序）？它和向量检索的区别是什么？
- **A：** 向量检索（双编码器）快速但缺乏 query-doc 交互，Rerank 用 Cross-Encoder 对候选文档逐个打分，精度高但慢。所以先粗排（向量检索取 top-50~100），再精排（Rerank 取 top-3~5）。Rerank 能过滤掉语义相似但实际不相关的文档，提升生成准确度。
- **一句话记忆法：** 粗排海选靠向量，精排决赛靠 Rerank。
- **复习标签：** #RAG #Rerank #CrossEncoder

### 卡片 3
- **Q：** 如何自动化计算 RAG 的 Faithfulness（忠实度）？
- **A：** 先将模型回答拆成原子声明（利用 LLM 或规则分割），然后对每个声明，用 NLI 模型（如 TrueTeacher）判断是否被检索到的文档片段支持（entailment）。支持率即为 Faithfulness 分数。开源框架 RAGAS 已实现此流程。
- **一句话记忆法：** 拆句对文档，NLI 当裁判，支持率越高越忠实。
- **复习标签：** #RAG #评测 #Faithfulness #NLI

### 卡片 4
- **Q：** RAG 中引用（Citation）设计的关键点是什么？如何防止模型胡编引用？
- **A：** 关键点：prompt 中给文档编号并强制引用，后处理做引用合法性校验，确保编号在范围内。更可靠的做法：生成后用句子与文档片段计算相似度，低于阈值则丢弃引用并触发兜底。引用准确性用“引用精准度”和“引用召回率”评估。
- **一句话记忆法：** 编号强制引，后处理校验，相似度兜底，假引用无处逃。
- **复习标签：** #RAG #引用 #Citation #反幻觉

### 卡片 5
- **Q：** HyDE（假设文档嵌入）是怎么工作的？适用场景是什么？
- **A：** 先用 LLM 根据用户 query 生成一段假设文档，再用该文档的向量去检索。目的是弥补短 query 与文档之间的语义 gap。零样本，无需训练。适用于模糊短查询场景，缺点是多一次 LLM 调用增加延迟，且生成质量差时引入噪声。
- **一句话记忆法：** query 短、语义远，HyDE 先生假设文，再编码检索更精准。
- **复习标签：** #RAG #HyDE #零样本检索 #QueryRewrite

## 4. 今日面试专项训练

### 4.1 大模型理论追问题：HyDE 的局限性及工程适配

**问题：**
请深入分析 HyDE（假设文档嵌入）在实际 RAG 系统中的局限性，并说明你会如何从工程上缓解这些局限？

**面试口头版参考答案：**
HyDE 的局限主要有三点：一是依赖 LLM 生成假设文档的质量，如果 LLM 生成的内容偏离主题或编织事实，反而会污染检索结果。二是多一次 LLM 调用，增加延迟和成本。三是假设文档的长度和风格难以把控，太短不足以消除语义 gap，太长可能引入幻觉。工程上，我采取以下缓解：1）用小而稳定的生成模型（如 Qwen2.5-7B）降低延迟，同时控制生成格式固定为关键词+摘要；2）只在 query 较短（<10 tokens）或 recall 低于阈值时启用 HyDE；3）生成多个假设文档取平均向量，或者与原始 query 向量做加权融合，减少单次误差。另外，我会在离线评测集上持续监控 HyDE 的增益，决定开关。

**深入理解版参考答案：**
HyDE 的局限性可以从理论层面和工程层面分析。理论上，HyDE 假设 LLM 能生成与真实文档分布接近的文本，但 LLM 生成的样本存在“采样偏差”，特别是在 domain-specific 领域（如医疗、法律），通用 LLM 可能生成不符合领域规范的假设文档，导致检索偏差。此外，embedding 模型对生成文本的编码质量也与真实文档存在差距。工程层面，瓶颈在于延迟和成本。一次额外的 LLM 推理调用（即使是小模型）会显著增加端到端耗时（约数百ms），对于高并发低延迟场景不可接受。同时，生成假设文档的长度越长，LLM 输出越耗时，且嵌入向量维度固定但内容冗余度高，可能降低检索区分度。缓解方案如下：
1. **选择性触发**：在 RAG 流水线中加入一个预判断模块，用 query 的 token 数、语义复杂度（如是否有实体）来判断是否启用 HyDE。仅在低 recall 风险时启用。
2. **生成质量优化**：使用结构化 prompt 限制输出为“技术规范描述”等短且特点分明的格式；或利用 few-shot 示例锚定领域风格。
3. **多假设融合**：生成 2-3 个假设文档，分别嵌入后取平均或加权（权重根据 LLM 的置信度打分）。然后用融合向量检索，提高鲁棒性。
4. **缓存与预计算**：对于高频出现的热门 query（如“促销政策”），可预计算假设文档并缓存结果向量。或在离线阶段对所有已知 query pattern 计算 HyDE 向量。
5. **轻量替代**：如果延迟敏感，可以用简单的 Query Expansion（基于同义词或模板）替代 HyDE，虽效果略差但成本极低。

**可能追问与简短回答：**
- **追问：** 你提到生成多个假设文档取平均，如何确定权重？如果其中一个假设完全错误，如何避免污染？
  **答：** 可以用 LLM 输出时的 logit 概率或自带的 confidence 作为权重。更鲁棒的方法是先用一个 NLI 模型判断假设与 query 的一致性，丢弃不一致的假设后再融合。
- **追问：** 在你的项目中，实际测出来 HyDE 提升了多少 recall？有对比过传统 Query Rewrite 吗？
  **答：** 提升 Hit Rate@10 约 12%。与 Query Rewrite（基于模板+小模型）对比，HyDE 在短 query 上更好，但延迟高 200ms。所以最终选择混合方案：短 query 用 HyDE，长 query 用 Query Rewrite。
