### 题目 1：RAG、微调、Agent 三种方案如何选型？

**来源依据：**  
- [8 LLM & RAG Interview Questions That OpenAI, Anthropic & Google Actually Ask | CallSphere Blog](https://callsphere.ai/blog/llm-rag-interview-questions-2026-openai-anthropic-google) — 技术博客，可信度高  
- [大厂RAG面试题：24个RAG八股文](https://www.cnblogs.com/crazymakercircle/p/18980652) — 个人博客，需进一步验证  
- 根据上述资料延伸，需进一步验证（经典高频题补充）

**可信度：** 中（资料高可信，但本题为综合延伸，缺少一手面经）

**考察点：** 架构决策能力、RAG vs 微调 vs Agent 适用场景、工程选型思维

**面试口头版回答：**  
我会用“RAG first，微调 last”的决策框架。第一步判断：是否需要访问外部实时知识或私有文档？需要的话上 RAG。第二步判断：输出格式是否固定、风格是否特殊（比如 JSON、法律文书）？需要的话加微调。第三步判断：是否需要多步推理、工具调用或长期记忆？需要的话上 Agent。举例：做一个内部知识问答系统，我先用 RAG，因为文档每周更新；只有当 RAG 召回了相关片段但模型依然不能按固定格式输出时，我才会考虑对生成模型做轻量 LoRA 微调。这个顺序保证最低成本和最快迭代。

**深入理解版回答：**  
三者不是互斥关系，而是层次叠加的关系。RAG 解决“知识从哪里来”，微调解决“输出风格和格式”，Agent 解决“复杂任务流程”。  
- **RAG**：核心是检索 + 生成。优势是知识可更新、可追溯、不依赖训练数据。局限性：对长尾知识检索精度敏感，对格式约束弱。  
- **微调（LoRA）**：改变模型行为偏好，比如输出格式、语气、结构化约束。成本比全参微调低，但需要构建高质量数据集。  
- **Agent**：在 RAG 基础上加入工具调用、多步规划、记忆管理。本质是任务编排，不改变模型本身。  
工程上推荐“RAG 先行，80% 场景够用；不够时对 Embedding 模型做领域微调（20% 成本获得 80% 效果）；只有终极格式约束才微调生成模型。Agent 则是在交互复杂时叠加”。常见坑：一上来就同时搞微调和 Agent，导致无法定位问题。我的项目里先用 RAG 做到 recall@5 80%，然后发现模型回复不总按模板输出，才用 QLoRA 微调了 300 条样本的回复部分，格式准确率从 72% 提升到 94%。

**常见追问与回答：**  
- Q：RAG 和 Agent 在什么场景下必须同时用？  
  A：当需要多步推理，且每步都需要查外部资料时，比如“分析最近三个月竞品融资新闻并生成报告”。RAG 负责每步检索，Agent 负责规划步骤和调用工具。  
- Q：你说 RAG 优先，但若知识库巨量（1 亿文档），检索延时高，怎么办？  
  A：可以做层级索引（先粗搜再精排），或把高热度知识预加载到模型上下文（类似 Cache），这本质是 RAG+ 缓存策略，不是放弃 RAG。

**项目联系：**  
假如我做一个科研论文问答 Agent，我会先用 RAG（检索论文段落），当用户要求“总结成三段式格式”时，我会判断是否需要微调生成模型（LoRA 微调 200 条总结样本）。如果用户说“帮我对比三篇论文的结论”，我就叠加 Agent 的多步规划能力。

**记忆关键词：** RAG first → 微调 last → Agent 叠加 → 成本优先 → 可定位问题 → 分层决策

### 题目 2：LoRA 微调时如何选择要更新哪些层？r 值怎么定？

**来源依据：**  
- [国内大厂大模型岗面试八股系列二：LoRA篇](https://www.1point3acres.com/bbs/thread-1085490-1-1.html) — 面经搬运，需进一步验证  
- [AI Agent常考面试题汇总- LoRA篇](https://www.nowcoder.com/discuss/876512639952683008) — 面经搬运，需进一步验证  
- [AgentGuide 公司面试案例](https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md) — GitHub 面经汇总，需进一步验证  
- [LoRA (Low-Rank Adaptation) · Hugging Face](https://huggingface.co/learn/llm-course/chapter11/4) — 官方文档，高可信

**可信度：** 高（Hugging Face 官方文档 + 多份面经交叉验证）

**考察点：** LoRA 原理、冻结层选择策略、rank 调参经验、工程判断

**面试口头版回答：**  
LoRA 不是固定选择所有层，而是优先更新 Attention 的 Q、K、V、O 投影层，因为这些层对下游任务语义影响最大。选择冻结层的依据主要有两点：一是看任务类型——分类任务更依赖 Attention 输出层，生成任务更依赖 FFN 层；二是看资源——如果显存紧张，只更新 Query 和 Value 两层。r 值一般从 8 或 16 开始尝试，r 太小则适配能力不足，r 太大则失去低秩优势。我一般在 7B 模型上对 QKV 层用 r=16，对 O 层用 r=8，然后在验证集上观察 loss 收敛速度，若欠拟合则增大 r。

**深入理解版回答：**  
LoRA 本质假设是微调时权重变化集中在低秩子空间。选择冻结层时，不推荐“全部层都加 LoRA”，因为不同层重要性不同。  
- **常见策略**：  
  1. 只加在 Attention 的 Q 和 V 上（经典做法，参数量最少）。  
  2. 加在 Q、K、V、O 上（效果更好，参数量增加有限）。  
  3. 有论文（AdaLoRA）提出根据层的重要性自动分配 rank，避免手工选择。  
- **r 值选择**：r 控制低秩矩阵的维度，相当于“学多少信息”。  
  - r=1：极度约束，只适合简单分类。  
  - r=8~16：通用推荐，平衡容量和效率。  
  - r=64：接近全参数微调，但显存和速度都恶化。  
- **经验**：先用 r=8 做一轮小数据训练，观察 attention 输出层的梯度范数。若高梯度集中在前几层，可以只在那几层加 LoRA。  
常见坑：不同数据集对 rank 敏感度不同，不能用同一个 r 值套所有任务。我的项目里用 QLoRA 微调 7B 模型做代码生成，对 Q、K、V、O 各加 r=8 Adapter，比只加 QV 的 BLEU 提升 3 个点，且参数量只增加 8%。

**常见追问与回答：**  
- Q：你如何评估不同层加 LoRA 的效果差异？  
  A：做 ablation study：固定其他层，只加某一层组，比较验证集性能。也可以用 Fisher Information 矩阵估计每层重要性。  
- Q：LoRA 合并后还能分开吗？  
  A：可以保留单独 Adapter 权重（几 MB），合并后变成单层权重（无法分离）。想保留多任务切换能力就不要 merge。

**项目联系：**  
在我做的一个指令跟随微调项目中，我对 7B 模型用 LoRA，只微调 Q、V 层（r=8），训练 1000 条数据，可训练参数量约 4.2M，保存的 Adapter 权重仅 16MB。我会在项目中记录每个实验的 rank、层选择、loss 曲线，形成经验表格。

**记忆关键词：** QKV 优先 → r=8~16 起步 → AdaLoRA 自动分配 → ablation 实验 → 合并 vs 分离 → 梯度范数辅助

### 题目 3：如何设计一个能够自我纠错的 RAG-Agent 系统？

**来源依据：**  
- [Comprehensive RAG Interview Guide: 60 Questions & Answers | LinkedIn](https://www.linkedin.com/posts/mohammadkshah_50-rag-interview-questions-answers-activity-7404980802406666240-h3e_) — 个人 LinkedIn，需进一步验证  
- [大模型 & 智能体（Agent）求职与面试手册](https://www.cnblogs.com/limingqi/p/19848278) — 个人博客，需进一步验证  
- [LLM & VLM & Agent 面试问题总结](https://github.com/datawhalechina/hello-agents/blob/main/Extra-Chapter/Extra01-%E9%9D%A2%E8%AF%95%E9%97%AE%E9%A2%98%E6%80%BB%E7%BB%93.md) — GitHub 面经汇总，需进一步验证  
- 根据上述资料延伸，需进一步验证

**可信度：** 低（多份资料交叉但无一手面经，本题为综合设计题）

**考察点：** Agent 架构设计、RAG 纠错、自我反思机制、工程容错

**面试口头版回答：**  
我会采用“检索 → 生成 → 校验 → 纠正”四步闭环。第一步：根据用户问题检索 top-5 片段；第二步：LLM 基于片段生成回答；第三步：用一个轻量校验器（可以是 NLI 模型或 LLM 自评）判断回答是否忠实于检索结果；第四步：如果校验不通过，则触发纠正流程——可能是重新检索（用改写后的 query）、或降低温度重生成、或直接返回“无法确定”。例如在客服场景，如果答案引用了过时文档，校验器发现时间戳不对，就会触发纠正。整个流程通过 Agent 框架（如 LangGraph）编排，每个步骤可观测、可回滚。

**深入理解版回答：**  
自我纠错 RAG-Agent 本质是“验证后生成”（Verify-then-Generate），比传统 RAG 多了一层校验。  
- **校验方式**：  
  1. **LLM-as-Judge**：用强模型（如 GPT-4）评价弱模型的回答，但成本高。  
  2. **NLI model**：训练一个蕴含模型判断回答是否被检索内容支持。  
  3. **事实性检查**：用命名实体识别 + 知识图谱评估答案中的实体是否存在。  
- **纠正策略**：  
  1. **重检索**：用最新 query 重新检索，或换用不同检索模型（如从向量检索切换到 BM25）。  
  2. **重生成**：修改 prompt 增加约束（如“请只引用下面片段”），或降低温度。  
  3. **拒绝回答**：若校验多次失败，返回“请提供更多信息”。  
- **工程实现**：  
  使用 Agent 框架定义状态机：  
  - 状态1：检索 → 若空，进入 query 改写子 agent  
  - 状态2：生成 → 存入缓存  
  - 状态3：校验 → 若 pass，输出；否则进入纠正子 agent  
  每个状态记录日志，便于回溯。  
常见坑：校验器本身可能误判（假阳性/假阴性），需要人工采样评估校准阈值。

**常见追问与回答：**  
- Q：如果一个 query 反复触发纠正但依然错误，怎么防止循环？  
  A：设置最大重试次数（如 3 次），达到后自动降级为“抱歉无法回答”，避免死循环。  
- Q：校验器的准确率如何保证？  
  A：用人工标注的忠实性数据集来评估，定期通过人机协作反馈修正。

**项目联系：**  
如果我做一个企业知识库助手，我会在 RAG 链路中插入一个“忠实性评分”模块，使用预训练的 Nutrasum 模型或微调一个小 Bert 做 NLI。如果评分低于阈值，就触发 query 改写重新检索。我会在项目中记录纠正触发率和最终准确率。

**记忆关键词：** 检索-生成-校验-纠正 → NLI 校验 → 重检索/重生成/拒绝 → 重试上限 → 可观测性 → 人工采样校准

### 博客 1：LoRA (Low-Rank Adaptation) · Hugging Face

- **链接：** https://huggingface.co/learn/llm-course/chapter11/4  
- **来源依据：** Hugging Face 官方文档，高可信  
- **文章类型：** 理论/论文 + 工程实践  
- **核心内容总结：**  
  - LoRA 冻结预训练权重，注入可训练的低秩分解矩阵  
  - 可训练参数减少约 90%，在 GPT-3 175B 上可降低 10000x 可训练参数量  
  - 训练时 Adapter 单独保存，推理时可合并到原模型，无额外延迟  
  - Hugging Face PEFT 库提供了 `load_adapter()`、`set_adapter()`、`unload()` 等方法管理多 Adapter  
  - QLoRA 将基模型量化到 4bit 再挂 LoRA，进一步节省显存  
- **对面试有什么用：**  
  - 提供 LoRA 在 Hugging Face 生态的具体操作，可用于回答“如何用代码实现 LoRA 微调”  
  - 包含模型合并（merge）与分离的实践细节，面试常问  
- **可转化面试题：**  
  1. LoRA 推理时如何合并权重？合并后有什么利弊？  
  2. QLoRA 为什么比 LoRA 更省显存但可能更慢？  
- **对应参考答案：**  
  **Q1：** LoRA 合并权重就是把训练好的低秩矩阵 A 和 B 的乘积加到原线性层权重上。合并后推理时不需要额外 LoRA 分支，计算效率与原始模型相同，但失去了切换不同 Adapter 的能力。工程上可以用 `peft` 的 `merge_and_unload()` 方法。  
  **Q2：** QLoRA 将基模型量化到 4bit NF4，减少了基模型的显存占用（如 7B 模型从 14GB 降到 3.5GB），但每次前向传播需要先反量化权重，增加了计算开销。所以它主要优化显存瓶颈而非推理速度，适合显存有限的场景。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** RAG 系统中最关键的评测指标有哪些？分别对应什么目的？  
- **A：** 关键指标分两类：检索端——Recall@k、MRR（检索相关度）；生成端——忠实性（Faithfulness，用 NLI 模型判断）、答案完整度（Coverage）。工程上建议先保证检索 recall 达标（>80%），再优化生成忠实性。  
- **一句话记忆法：** 检索看“找没找到”，生成看“说没说对”。  
- **复习标签：** #RAG #评测

### 卡片 2
- **Q：** LoRA 微调完成后，推理时能否合并权重？如何实现？  
- **A：** 可以合并。将训练好的低秩矩阵 A×B 加到原模型权重上，得到新的权重矩阵。之后推理时不再需要额外 LoRA 分支，无额外延迟。Hugging Face PEFT 提供 `merge_and_unload()` 接口。但合并后无法灵活切换多个 Adapter。  
- **一句话记忆法：** 合并不分离，快但不能换；分离可换慢，取舍看场景。  
- **复习标签：** #LoRA #模型部署

### 卡片 3
- **Q：** Agent 工具调用失败后，你会如何设计回退策略？  
- **A：** 分三级：第一级重试——间隔 1s 重试原调用（最多 3 次）；第二级降级——换用备用工具（如从 WebSearch 降级到本地知识库）；第三级告知用户“该功能暂时不可用”。同时记录失败日志，便于事后分析是工具故障还是 Agent 意图识别错误。  
- **一句话记忆法：** 先重试、再降级、最坏直说。  
- **复习标签：** #Agent #容错

### 卡片 4
- **Q：** RAG 中 chunking 的策略如何选择？  
- **A：** 核心原则是“语义完整性 + 上下文窗口约束”。常用：固定大小（256-512 tokens）加重叠（overlap=10-20%）；结构化分割（按 Markdown 标题、段落）；语义分割（用 Embedding 相似度判断边界）。我的经验是：先试 512 固定 chunk + 20% overlap，再根据 recall 调优。  
- **一句话记忆法：** chunk 太大记不准，太小缺逻辑，重叠保衔接。  
- **复习标签：** #RAG #预处理

### 卡片 5
- **Q：** 模型量化为 INT4 后回答质量下降，你会怎么做精度补偿？  
- **A：** 首先定位下降原因：是激活值范围被截断？还是权重精度损失在关键层？常用补偿：保留部分层为 FP16（混合精度量化）；对敏感层（如 embedding、lm_head）不做量化；或者用 QAT（量化感知训练）微调少量数据恢复精度。  
- **一句话记忆法：** 量化像压缩图片，关键区域保留高清（FP16），其余用低清（INT4）。  
- **复习标签：** #模型部署 #量化

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** DPO 相比 SFT 有哪些优劣？它在 Agent 任务上效果提升明显吗？如何构造偏好对？

**面试口头版参考答案：**  
DPO（直接偏好优化）直接使用偏好对优化策略，不需要训练 Reward Model，简化了 RLHF 流程。相比 SFT，DPO 能更好地对齐人类偏好（如让模型更简洁、更安全），但需要高质量的偏好数据。在 Agent 任务上，DPO 提升尤其体现在工具调用顺序和结果选择上——比如让模型选择更合适的 API 而非胡乱调用。构造偏好对时，我通常从用户行为日志中抽取：同一个 query 下，成功完成任务的行为 vs 失败的行为作为一对，正样本是最终成功的那条轨迹。

**深入理解版参考答案：**  
DPO 核心思想是用 Bradley-Terry 模型直接优化策略，损失函数包含两个部分：正样本与负样本的 log 概率差，以及一个 KL 散度正则项防止策略偏移过远。  
- **与 SFT 对比**：SFT 只学习“正确答案”，不学习“为什么这个答案比另一个好”；DPO 学习相对偏好，因此在开放生成任务（如对话、Agent 工具调用）中更鲁棒。  
- **Agent 任务上的效果**：Agent 任务中，模型需要从多个工具中做选择、决定调用顺序。DPO 通过偏好对学习“什么调用顺序是合理的”，效果提升明显（实验中任务成功率提升 5-10%）。但需要大量高质量的轨迹数据，且正负样本的边界可能模糊（比如一次成功但绕了弯路）。  
- **偏好对构造方法**：  
  1. 自动构造：用 reward function 或成功率对多条轨迹打分，取最高和最低各一条。  
  2. 人工标注：标注员判断两条回复哪个更好（成本高）。  
  3. 对抗构造：用当前模型生成两条回复，让人类选择更好的。  
- **常见坑**：DPO 对偏好数据质量敏感，如果负样本太差（明显胡扯），模型容易学会“糊弄”；如果正负样本差异太小，训练不收敛。

**可能追问 1：** DPO 和 PPO 的核心区别是什么？  
**回答：** PPO 需要训练 Reward Model + 策略模型 + 价值模型（四个模型），训练流程复杂；DPO 直接使用偏好数据优化策略，省去 Reward Model 和采样步骤，更轻量。PPO 适合在线策略更新（可探索新行为），DPO 适合离线偏好对齐。

**可能追问 2：** 如果偏好对中正样本和负样本来自不同 prompt（不同意图），还能用 DPO 吗？  
**回答：** 不能。DPO 要求正负样本必须对应同一个 prompt（同一输入），否则模型无法学习“对于这个 query 哪个回答更好”。因此偏好对必须同 query 下。
