## 1. 今日面试真题精讲

### 题目 1：SGLang 和 vLLM 在推理加速上的核心区别是什么？各自有什么适用场景？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证，但知识点具备实操价值
- LLM RAG面试问题大全! - https://www.53ai.com/news/RAG/2024100162935.html - 技术博客 - 低可信度，仅为背景参考

**可信度：** 中

**考察点：** 模型部署、推理优化、前缀复用、内存管理、框架选型

**面试口头版回答：**

SGLang 和 vLLM 都是大模型推理框架，但核心设计理念不同。vLLM 的核心创新是 PagedAttention，它借鉴操作系统分页思想，把 KV Cache 切成固定大小的块来管理，解决了显存碎片化和 OOM 问题。SGLang 则提出了 RadixAttention，专为多轮对话和前缀复用场景优化，它把历史对话的 KV Cache 用 Radix Tree 组织，遇到相同前缀时直接复用缓存，避免了重复计算。简单说，vLLM 的强项是通用推理，适合高并发场景；SGLang 在需要大量前缀复用（比如多轮对话、少样本问答、Agent 的多次工具调用）时优势更明显。我在项目里就对比过，SGLang 在多轮对话中首 token 延迟能降低 30% 左右。

**深入理解版回答：**

要理解两者的本质区别，需要从大模型推理的瓶颈入手。自回归生成时，每个 token 都要计算注意力，但前面所有 token 的 K 和 V 已经算过了，于是有了 KV Cache 复用。问题在于 KV Cache 的显存管理：每个请求生成的序列长度不同，如果提前分配最大长度的空间，会浪费大量显存；如果按需分配，又会产生碎片。

vLLM 用 PagedAttention 解决了这个问题：将 KV Cache 切成固定大小的 block（一般 16 或 32 个 token），每个请求只需要按需申请 block，减少了显存浪费，也支持了动态的 batch。但 PagedAttention 对前缀复用的支持不够原生——如果你有一个 1000 token 的系统提示词，每个新请求都得重新计算这 1000 token 的 KV Cache。

SGLang 的 RadixAttention 直接瞄准了这个空白。它把 KV Cache 当成一个 Radix Tree 来存储，树上的每个节点对应一个 token 或一段 token 序列。当新请求到达时，它从根节点开始匹配前缀，如果找到共享的前缀，直接从那里继续计算，不再重复。这对多个对话共享相同系统提示词、Agent 多次调用同一工具描述的场景收益极大。工程上，SGLang 还提供了 `@function` 装饰器和结构化调度，让框架能提前规划好生成路径。

对比而言：
- **vLLM 优势**：通用性好、API 兼容 OpenAI、生态成熟、支持各种量化方案、社区活跃。
- **SGLang 优势**：前缀复用场景下首 token 延迟更低、Agent 链式调用更高效、结构化输出支持更强。
- **代价**：SGLang 的 API 接口更新较慢，对非主流的模型架构支持不如 vLLM 广泛。

实际选型时，可以先评估自己应用的前缀复用率：如果 80% 的请求共享 60% 以上的前缀（如多轮对话、批量 QA），SGLang 收益明显；如果是随机查询密集场景，vLLM 更适合。

**常见追问与回答：**

1. **追问：RadixAttention 怎么处理序列长度不一致的问题？**
   - 回答：Radix Tree 会自动分裂和合并节点。如果两个请求共享前 20 个 token 然后在第 21 个分叉，树就在第 21 个 token 处分裂。框架支持节点级别的引用计数，共享节点被引用就保留，没有引用时释放。

2. **追问：如果 SGLang 用 RadixAttention 也做不了前缀复用，退化后的性能和 vLLM 比谁更好？**
   - 回答：退化场景下两者性能相近。但 SGLang 额外要维护 Radix Tree，内存开销略大。严格来说，如果你确定几乎没有前缀复用，vLLM 是更稳妥的选择。

**项目联系：**
在 Agent 项目中，我设计了一个多轮对话系统，每轮都需要重传系统提示词（包含了角色设定和可用工具列表）。上线后发现首 token 延迟偏高，排查后发现每个请求都在重复计算相同的系统提示词 KV Cache。把后端推理引擎从 vLLM 切换为 SGLang 后（保持 API 接口一致），首 token 延迟从 800ms 降至 500ms，吞吐量提升了 30%。同时，因为 Agent 工具调用往往在同一段工具描述上重复触发，SGLang 的前缀缓存作用非常明显。

**记忆关键词：** RadixAttention、PagedAttention、前缀复用、KV Cache 管理、框架选型、首 token 延迟

### 题目 2：你在部署大模型推理服务时，如何系统性地优化吞吐和延迟？请从量化、KV Cache、Batching 三个维度展开。

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证，但问题为经典高频
- LLM Interview Questions and Answers (2026) - https://www.interviewbit.com/llm-interview-questions-answers - 技术博客 - 低，综合性题库
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 技术博客 - 低，独立整理内容

**可信度：** 中（经典问题，有资料支撑）

**考察点：** 模型部署、推理优化、吞吐 vs 延迟权衡、量化原理、Batching 策略、KV Cache 优化

**面试口头版回答：**

我的优化思路是“先做粗粒度策略，再做细粒度调参”。首先是 Batching，这是最有效的吞吐提升手段，核心是把多个请求拼成一个 batch 一起前向推理，利用 GPU 的并行计算能力。但 batch 不是越大越好，请求到达率不高时只能攒 Batch，这会增加延迟。所以我用动态 Batching + 超时策略（比如 50ms 内凑满 8 个请求就发，否则超时强制发）。然后是量化，我常用 W8A16 或 W4A16，把模型权重复制到 8bit 或 4bit，显存占用降一半到四分之一，同一个 GPU 能装下更大模型或更大 batch。量化还降低了访存带宽需求，间接加速。最后是 KV Cache 优化，除了常见的 PagedAttention 减少碎片，我还用过 KV Cache 量化（如 FP8），以及 attention 的 GQA/MQA 架构来减少 KV Cache 大小。三管齐下后，我服务的 QPS 从 50 提升到了 200，P50 延迟 < 200ms。

**深入理解版回答：**

要系统性地理解这三个维度，需要回归到推理的物理约束：**计算瓶颈 vs 访存瓶颈**。在 Prefill 阶段（计算用户输入生成第一个 token），主要是计算瓶颈，优化重点是减少计算量（量化后计算更快、更小的 batch 用 FlashAttention）；在 Decode 阶段（生成后续 token），模型是 memory-bound，每步都从 HBM 加载模型权重和 KV Cache，优化重点是减少访存量（量化减少权重尺寸、KV Cache 优化减少内存访问、更大的 batch 分摊权重加载成本）。

具体到每个维度：

1. **Batching（连续 batching）**
   - 原理：GPU 的矩阵计算吞吐远高于多个小请求串行。把 K 个请求拼在一起做一次前向，计算量几乎不变但总吞吐翻倍。
   - 工程实现：vLLM 的 continuous batching 允许 batch 动态增长，已完成的请求（已生成到长度）可以被新请求替换，不用等整个 batch 跑完。
   - 代价：单个请求的延迟会因排队和 batch 同步而增加。收益公式：吞吐增长 ≈ batch_size，延迟增加 ≈ batch 等待时间 + 同步开销。
   - 常见坑：把 batch_size 推到极限导致 OOM；请求输出长度差异大时，短请求被长请求拖长（处理完的请求必须在 batch 里等，直到最长请求结束）。

2. **量化**
   - 原理：将模型权重和激活值映射到更低精度，用更少的比特表示参数。
   - 常用方案：W8A16（权重 8bit，激活 16bit）降低显存和访存，精度损失约 0.5%；W4A16 显存再减半，但对小模型精度损失较大；KV Cache 量化（FP8 或 INT8）直接减少 Cache 占用的显存。
   - 工程诀窍：对量化敏感的关键层（如 attention 的 QKV 投影）可以保持 FP16，其他层低精度；用 GPTQ 权重量化比简单 round 效果好。
   - 代价：量化误差累积可能导致输出质量下降，需要在评测集上验证。

3. **KV Cache 优化**
   - 结构优化：GQA（分组查询注意力）和 MQA（多查询注意力）通过共享 KV 头减少 KV Cache 大小，比如 LLaMA-70B 用 GQA 后 KV Cache 减少约 8 倍。
   - 缓存管理：PagedAttention 避免碎片，RadixAttention 复用前缀。
   - 显存量化：KV Cache 用 FP8 存储，精度损失可控但显存减半。

最终的权衡原则是**先保证延迟满足 SLA，再压吞吐**。我的经验是：先用量化确保模型能装在可用显存内，然后用 Batching 推高吞吐，最后用 KV Cache 量化进一步提升并发。每一步都要有压测数据，不要凭感觉调参。

**常见追问与回答：**

1. **追问：你说量化精度损失约 0.5%，这个数字怎么来的？**
   - 回答：这是一个经验值，一般用 WikiText-2 困惑度对比或者下游任务评测。以 LLaMA-2 7B 为例，用 GPTQ 做 W8A16 量化后，在多个 benchmark 上平均损失通常 < 0.5%，但具体到特定任务（比如代码生成、数学推理）损失可能更大，需要独立验证。

2. **追问：如果延迟要求很苛刻（比如必须 P99 < 100ms），你怎么取舍？**
   - 回答：这种情况下我会牺牲吞吐来保延迟。首先把 batch_size 限制在 1 或 2；其次开启 vLLM 的 early stopping，确保请求不会因为等待 batch 而增加延迟；最后考虑用 SGLang 进行更细粒度的调度。如果还不行，就需要升级硬件了。

**项目联系：**
在部署 LoRA 微调后的 7B 模型时，我用 vLLM + W8A16 量化 + continuous batching 做服务化。一开始 batch_size 设为 16，QPS 到了 80，但 P99 延迟飙到 800ms 以上。我用逐步压测的方式，发现当 batch_size = 4 时 QPS = 50 且 P99 = 200ms。然后用 KV Cache 量化（FP8），把 batch_size 可以提升到 8 同时保持同等延迟。最后的配置是 batch_size = 6 + 动态超时 100ms + KV Cache 量化，现场压测 QPS 75 的同时 P99 = 180ms。这个迭代过程让我深刻理解了“先量化吃内存，再 batching 吃计算，最后优化缓存”的调参顺序。

**记忆关键词：** 连续 batching、量化（W8A16/W4A16）、KV Cache 量化、吞吐 vs 延迟、Prefill vs Decode、vLLM continuous batching

### 题目 3：在推理部署中，LoRA 模型如何实现低成本、低延迟的切换？你了解哪些工程方案？

**来源依据：**
- Tune and Deploy LoRA LLMs with NVIDIA TensorRT-LLM - https://developer.nvidia.com/blog/tune-and-deploy-lora-llms-with-nvidia-tensorrt-llm - 技术博客 - 高，NVIDIA 官方技术博客
- Efficient LLM Fine-Tuning with LoRA - https://medium.com/@raquelhvaz/efficient-llm-fine-tuning-with-lora-e5edb88b64a1 - 技术博客 - 中，Medium 个人博客

**可信度：** 高

**考察点：** LoRA 部署、推理优化、多任务切换、模型打包、内存复用

**面试口头版回答：**

LoRA 的部署优势在于：推理时可以把 LoRA 权重合并（merge）回基座模型权重，这样推理服务看到的仍然是一个完整模型，不增加延迟。但如果要支持多个不同任务的 LoRA 动态切换，合并方案就不太灵活了。更好的方案是使用 TensorRT-LLM 或 vLLM 的多 LoRA 支持：基座模型加载一次到显存，多个 LoRA adapter 也预加载到显存里，推理时根据请求头部指定的 adapter 名称动态注入对应 LoRA 的结果。这个方案的显存开销是“基座模型 + 所有 adapter 大小之和”，相比于加载 N 个完整模型，省了大量显存。我在项目里用 vLLM 的 LoRA 支持实现了三个不同领域 adapter 的动态切换，每个 adapter 只有几十 MB，切换延迟几乎为零。

**深入理解版回答：**

LoRA 的核心思想是冻结预训练权重 W₀，训练两个低秩矩阵 A 和 B，最终推理时的有效参数为 W' = W₀ + BA。基于这个性质，部署时有三种方案：

1. **合并部署（Merge + Deploy）**：提前计算 W₀ + BA，保存成一个完整模型文件，然后像部署普通模型一样部署。这是最简单的方案，推理延迟和原始模型完全一致。缺点是每个任务都需要一个完整模型，如果任务多则显存开销线性增长。

2. **动态注入（Runtime Injection）**：基座模型只加载一份到显存，多个 LoRA adapter（只有几 MB 到几十 MB）也加载上来。推理时，每个 batch 里面的请求可能来自不同 adapter，TensorRT-LLM 或 vLLM 能够为每个请求分别注入对应 adapter 的 output。这种方案有两个技术要求：
   - 支持 per-request LoRA selector：请求 API 中需要携带 adapter 标识。
   - 支持 batch 内异构 LoRA：不同请求走不同 adapter 时，计算要能并行处理。
   - 显存开销：基座 + 所有 adapter 大小之和。
   - 切换延迟：适配器查找和取指时间，微秒级。

3. **基于前缀的 prompt 注入**：更轻量，把 LoRA 微调的内容变成 system prompt 或 few-shot 示例，不需要真正的 LoRA 适配器，适合小规模任务。

工程上，NVIDIA TensorRT-LLM 对 LoRA 做了深度优化：它支持 LoRA 插件的提前编译，可以在 runtime 动态挂载多个 LoRA checkpoint。官方博客测试显示，在 8xA100 上，动态注入 8 个不同 LoRA 时相对于基座的延迟增长 < 5%。vLLM 的 LoRA 支持还在持续完善中，目前支持 batch 内异构，但对大规模适配器的管理（例如上千个 adapter）仍需优化。

常见坑：
- adapter 和基座模型版本不对齐导致精度下降。
- 多个 adapter 同时使用时，全局 batch size 不能等效于原始 batch size（因为 adapter 切换带来额外同步开销）。
- 极小的 rank（如 r = 4）可能因为过拟合，部署后效果反而差于大 rank。

**常见追问与回答：**

1. **追问：如果有上千个不同任务的 LoRA adapter，怎么管理？**
   - 回答：需要做 adapter 的预加载和缓存策略。TensorRT-LLM 支持 LRU 换出策略，请求频率高的 adapter 常驻显存，低频的从磁盘加载。对于极端情况，可以用统一的 LoRA hub + 用户级路由。

2. **追问：动态注入的反向计算（训练）怎么做？LoRA 部署和训练是同一套代码吗？**
   - 回答：训练时通常用 PEFT + Transformers，推理时用 TensorRT-LLM/vLLM。前者做了梯度计算和 optimizer state，后者只做推理。所以部署框架一般不直接支持 LoRA 训练，需要把训练好的 adapter checkpoint 格式转换。

**项目联系：**
项目需要为一个企业内部系统部署三个 LoRA 适配器：合同审核、合规检查、客户对话。如果用合并方式部署，每一份都包含完整的 7B 模型，三个副本占用显存极大。改用 vLLM 的多 LoRA 后，仅加载一次基座模型（7B 约 14GB FP16）和各 adapter（每份约 50MB），总显存不到 15GB。推理时，请求中指定 `--lora-module=contract` 即可切换，客户反馈切换响应时间无明显差异。这块我在项目中专门写了多 LoRA 的压力测试脚本。

**记忆关键词：** LoRA 合并、动态注入、TensorRT-LLM、vLLM LoRA、per-request selector、切换延迟

### 题目 4：在 Agent 系统中，如何设计一个高吞吐、低延迟的工具调用（Tool Calling）链路？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证

**可信度：** 低（面经延伸题，需进一步验证）

**考察点：** Agent 系统设计、工具调用优化、推理加速、缓存策略、异步并行

**面试口头版回答：**

Agent 工具调用卡在两点：一是大模型规划慢，二是工具执行耗时。优化思路分三层。第一层是缓存和前缀复用：用 SGLang 的 RadixAttention，让不同工具调用共享相同的系统提示词和工具描述前缀，减少首 token 延迟。第二层是工具并行度：如果 Agent 计划调用多个独立工具，用 async/await 并行执行，而不是串行。第三层是推理和工具的 pipeline：模型生成调用参数时，工具输入已经准备好了，工具执行可以跟下一轮模型推测重叠。我在项目中用这三点，把多工具调用场景的端到端延迟从 3 秒降到了 1.5 秒。

**深入理解版回答：**

Agent 工具调用可分为“决策 → 调用 → 推理”的循环。每个循环中，大模型不仅要去理解工具描述，还要决定调用哪个工具、构造参数、接收结果、继续生成。这是典型的计算密集型 + IO 密集型混合场景。

优化的具体方案：

1. **缓存与复用**：
   - 工具描述（function schema）通常是固定的，系统提示词也固定。如果 Agent 每轮都把这些内容作为模型输入，每次都需要重新计算前缀的 KV Cache，非常浪费。使用 SGLang 的 RadixAttention 可以缓存这部分，后续请求直接复用，首 token 延迟降低 30%-40%。
   - 工具调用结果也可以缓存：同一种参数的同种工具，如果结果可以复用，直接返回，跳过模型推理。

2. **异步并行与流水线**：
   - 当 Agent 决定调用多个互不依赖的工具时（比如“查询天气 + 查询新闻”），可以使用 asyncio.gather 并行执行工具，而不是依次执行。
   - 更进一步的流水线：假设 Agent 第一轮、第二轮都要调用工具。第一轮工具执行时，第二轮的系统提示词前缀仍可以预先计算（类似 GPU 和 CPU 的 overlap）。虽然目前框架支持有限，但这是高频优化的方向。

3. **模型层面提速**：
   - 使用更小的工具调用专用模型（比如 3B 级别）来替代 70B 模型做工具选择，仅当小模型不确定时才 fallback 到大模型。
   - 使用投机解码（speculative decoding）加速生成：用一个小草稿模型生成候选 token 序列，大模型并行验证，多个工具调用的 token 都可以加速。

4. **服务化与连接池**：
   - 对 HTTP 工具调用，维护 persistent connection 和连接池，避免每次请求都建连。
   - 对 gRPC 工具调用，使用流式调用，减少网络往返。

常见的坑：Agent 推理的 latency guzzler 往往是“工具调用结果太大”。如果工具返回长文本（比如文档查询结果），模型要读完所有内容才能生成下一步，这会让 decode 变慢。解决方案是工具结果摘要化，让工具返回摘要而不是全文。

**常见追问与回答：**

1. **追问：你说工具结果可以缓存，但缓存怎么解决过期问题？**
   - 回答：缓存的 TTL（Time-to-Live）策略很关键。对实时性要求高的工具（如天气），设置短 TTL（30秒）；对更新频率低的工具（如企业规章制度），设置较长 TTL 或手动失效。我在项目里给每个工具定义了缓存策略并写入工具描述，让 Agent 在调用时就知道是否要用缓存。

2. **追问：如果工具执行和模型推理是两个不同 GPU 或 CPU，怎么协调？**
   - 回答：通过消息队列或 Redis pub/sub 做 inter-process 通信。模型推理推理出的 action + input 发给工具 Worker，工具 Worker 执行后把结果写回队列。整个流程可以通过后端 Python 框架（如 FastAPI + asyncio）做异步编排，不阻塞模型服务的主循环。

**项目联系：**
在 Agent 项目中，我处理了一个多工具客服场景：用户询问“帮我查一下订单状态，顺便看看仓库有没有货”。Agent 决策后需要调用订单查询、库存查询两个独立 API。我用了 asyncio.gather + 持久化连接池 + 工具结果摘要，两个工具总共耗时 400ms，而模型推理耗时 600ms，端到端 1 秒内完成。优化之前是串行调用工具，总耗时 2.2 秒。

**记忆关键词：** Agent 调用优化、前缀复用、异步并行、工具缓存、流水线、小模型 fallback

### 题目 5：在评估一个模型部署方案时，你会关注哪些指标？如何基于这些指标进行决策？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证，但问题经典且通用
- LLM 大模型存在哪些问题？RAG 优化有哪些方法？ - https://www.cnblogs.com/crazymakercircle/p/18841145 - 技术博客 - 低，偏架构知识科普

**可信度：** 中（经典问题，有资料佐证）

**考察点：** 模型部署、评测体系、吞吐 vs 延迟、成本、稳定性

**面试口头版回答：**

我重点关注的指标是 P50/P99 延迟、吞吐（QPS）、显存占用和成本（每 token 成本）。先给一个决策流程：第一步，确定延迟 SLA（比如 P99 < 500ms），然后选择模型量化方案，确保模型能装进预算显存。第二步，压测确定最大 QPS 拐点——即在 SLA 约束下的最大吞吐，这个点之前吞吐随并发线性增长，之后延迟急剧恶化。第三步，算实际成本：一台 GPU 一天的批处理数就是吞吐*24h*3600，除以模型推理一次的成本（电费 + 折旧），得到每千次请求的成本。基于这个数据，决定是用更大显存的 GPU 做更高 batch 的量化还是用多台小 GPU 做水平扩展。我在项目中就是按这个逻辑做的 A100 vs H100 选型决策。

**深入理解版回答：**

模型部署评估的本质是对抗“算力-显存-延迟”的三角约束。一个全面的评估体系应该包括：

1. **延迟指标**
   - **首 token 延迟（TTFT）**：从请求发出到生成第一个 token 的时间。Prefill 阶段决定。RAG 场景下需要大 chunk 输入时，首 token 延迟会变长。
   - **解码延迟（TPOT）**：每个 token 的平均生成时间。Decode 阶段决定。通常几十到几百毫秒不等。
   - **端到端延迟**：生成完整输出的总时间。重点要考虑输出长度差异带来的影响，长输出会拖长 P99。

2. **吞吐指标**
   - **QPS**（每秒请求数）：需要指定输出 token 数分布（比如平均 200 tokens 的 QPS），否则口径不一致。
   - **request throughput** vs **token throughput**：后者更能反映算力利用率。但面试时多按前者算。
   - **吞吐拐点**：找到 SLA 约束下的最大并发数，超过了延迟就炸。

3. **显存指标**
   - **模型加载显存**：量化后模型的显存 = 参数数量 × 精度字节数。
   - **KV Cache 预留显存**：= 最大 batch size × 输出最大长度 × 每层 KV Cache 大小（取决于架构）。
   - 经验公式：7B 模型 + 4K 上下文 + batch=8 ≈ 总共占用 18-20GB。
   - **OOM 风险点**：KV Cache 在长上下文时是显存消耗大头。

4. **成本指标**
   - **每千 token 成本** = (GPU 小时成本 × 同时运行实例数) / 每千 token 吞吐量。
   - **推理成本 vs 训练成本**：一个 7B 模型每天可以服务数万到数十万请求，量化后成本可以降到每百万 token 几毛钱人民币。

5. **稳定性指标**
   - **P99 延迟抖动幅度**：小于 20% 为稳定。
   - **P99 空转率**：排队但无请求的比值。
   - **error rate**：包含 OOM、超时、模型生成异常的比率。

决策流程：
- 先用小压测工具（如 `locust` + FastAPI 封装）在目标 GPU 上跑一遍，得到基础延迟数据。
- 然后用量化方案对比：W8A16 vs W4A16 vs FP16，记录每项指标。
- 再决定 batch_size 上限和并发数。
- 最后根据 QPS 拐点核算实际成本。

常见坑：生产环境请求的输出长度分布严重影响延迟和显存，但很多压测都假设所有请求输出长度一致，这样测出来的数据不可靠。最好是对真实流量采样，标出输出长度的分布特征再做压测。

**常见追问与回答：**

1. **追问：你说“吞吐拐点”，怎么高效找到这个拐点？** 
   - 回答：用二分法。比如从并发 1 开始，以 2x 倍数增加（1, 2, 4, 8...），直到延迟超过 SLA；然后在过界区间内继续二分。再加一段实际流量 replay，确保拐点设置合理。

2. **追问：如果成本预算是一台 80GB 的 A100，你要部署 70B 模型，怎么取舍？** 
   - 回答：70B 模型 FP16 约 140GB，80GB 显存不可能直接部署。方案一：用 4bit 量化（如 GPTQ q4），模型降到约 35GB，加上 KV Cache 预留，可以支持 batch=1 部署。方案二：用 AWQ 6bit 量化，但显存可能会超一点。方案三：用 QLoRA 的 NF4 量化 + CPU offloading，但延迟会增加。

**项目联系：**
我在项目初期评估过使用 8xA100 还是 4xH100 部署 70B 模型。我写了压测脚本，分别用 batch=1/2/4/8 测试，收集 TTFT、TPOT、显存。输出是：在同等 batch_size 下，H100 的 token 吞吐是 A100 的 1.7x，但 H100 功耗高导致电费高 30%；不过 H100 可以在显存内多跑一个 batch。最终结论是，对于我们的日均 10 万次请求的场景，H100 的实际使用成本反而比 A100 更低 15%。这个结论让我在选型中更有底。

**记忆关键词：** P50/P99 延迟、TTFT、TPOT、QPS、显存评估、成本核算、吞吐拐点、A100 vs H100

## 3. 今日必背问答卡片

### Q1：为什么说 KV Cache 是推理优化的核心？它到底怎么省计算？

**A：** KV Cache 的核心思想是空间换时间。自回归生成时，模型每步都要算注意力，但前面所有 token 的 K 和 V 已经算过了，如果重新算，随着序列变长计算量呈平方增长。KV Cache 把每一步的 K、V 矩阵都存下来，下一步生成时直接复用，只算当前 token 的和历史所有 K、V 的注意力。这让自回归生成的计算量从 O(n²) 降为 O(n)，每多一个 token 只做一个额外的注意力查询。当然，代价是显存占用：每个 token 存储 K、V 两个向量，模型层数、头数越多，缓存越大。

**一句话记忆法：** KV Cache 就是“算一次，存一辈子”的注意力套餐，靠牺牲显存换来自回归推理的极速。

**复习标签：** #模型部署 #推理优化 #KV Cache

---

### Q2：模型量化的本质原理是什么？为什么它可以在几乎不掉点的情况下减少显存？

**A：** 模型量化的本质是将高精度浮点数值（如 FP32 的 32 位、FP16 的 16 位）映射到低精度整数（如 INT8 的 8 位、INT4 的 4 位）。映射公式是 `q = round(r / s) + z`，其中 r 是原值、s 是缩放因子、z 是零点偏移。模型参数在训练后分布的方差往往不大，很多权重集中在 0 附近的小范围内，因此低位宽量化保留了足够的精度。GPTQ、AWQ 等方案通过混合精度（敏感层高精度、其他层低精度）、校准集优化等技巧，能将精度损失控制在 0.5% 以内，而显存直接减半（FP16->INT8）或减四分之三（FP16->INT4）。但激进量化（如 INT4 小模型）在复杂推理任务上精度损失可能较大。

**一句话记忆法：** 量化就是给模型权重“瘦身”，FP32 换 INT8——精度勉强保，显存大减半。

**复习标签：** #模型部署 #量化 #INT8 #W8A16

---

### Q3：PagedAttention 和 RadixAttention 有什么本质区别？哪个更好？

**A：** PagedAttention 和 RadixAttention 解决的是不同维度的优化问题。PagedAttention 解决显存分配碎片化和利用率低的问题——把 KV Cache 按 block 切分、按需分配，类似于操作系统的分页存储。RadixAttention 解决前缀重复计算的问题——将 KV Cache 用前缀树组织，共享相同前缀的请求可以复用缓存，避免首 token 重复计算。所以没有绝对的好，要看场景。高并发随机查询场景（前缀小）用 PagedAttention；多轮对话或批量 QA 场景（高频复用长前缀）用 RadixAttention 收益更大。

**一句话记忆法：** PagedAttention 治“碎片化”，RadixAttention 治“重复造轮子”——两把钥匙开不同的锁，别问哪个万能。

**复习标签：** #模型部署 #推理优化 #PagedAttention #RadixAttention

### Q4：连续 Batching（Continuous Batching）相比静态 Batching 提升了什么？

**A：** 静态 Batching 是等固定大小的 batch 填满后再做一次前向，空闲时 GPU 空转；而且 batch 内所有请求的输出长度必须一致，短请求要等长请求结束。连续 Batching（vLLM 首创）允许 batch 动态变化：一个请求生成完就移出 batch，立即插入一个新请求，GPU 一直满负荷运转。同时，每个请求可以有不同的输出长度，互不影响。工程效果：在相同延迟约束下，连续 Batching 能把吞吐提升 2-4 倍。代价是实现复杂，需要仔细控制 cache 的分配和回收。

**一句话记忆法：** 静态 Batching 是“等人齐再开饭”，连续 Batching 是“先到先吃、来新人就加座”。

**复习标签：** #模型部署 #推理优化 #Batching #吞吐

### Q5：生产环境推理上来了但延迟超标，怎么排查？优先查哪里？

**A：** 按“最可能导致问题”的顺序：一查 KV Cache 是否 OOM 导致请求被踢出/回退；二查 batch 是否过大导致长请求拖长 P99（输出长度分布不均匀）；三查量化方案是否导致模型输出变长或反复重试（量化误差）；四查磁盘 IO 或 CPU 瓶颈（模型加载、tokenizer 处理）。优先排查思路：先看监控面板，显存利用率 > 90% 时先降 batch 或启用 KV Cache 量化；输出长度 P90 远大于平均值时加 max_tokens 限制或做 response trimming；如果是模型反复生成空/重复内容，回退量化方案测试。

**一句话记忆法：** 延迟超标不用慌，显存排队输出长——四步排查法，先杀显存再杀 batch。

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 量化（INT8 权重量化）是如何影响模型推理的精度和速度的？请从数学原理和工程实践两个维度解释。

**面试口头版参考答案：**

量化核心就是高精度转低精度，涉及两个数学概念：**映射函数**和**精度损失**。以 INT8 对称量化为代表，就是把 FP16 权重 `w` 映射到 INT8 的 `q = round(w / scale)`，其中 `scale = max(|w|) / 127`。推理时，计算从矩阵乘法 `W·x` 变成 `round(W / scale_W)·x`，再乘回 `scale`。量化后的矩阵可以用 INT8 精度做矩阵乘法（Tensor Cores 支持），比 FP16 快 2-4 倍，显存也减半。精度方面，大部分权重都分布在 ±2 倍 std 之内，因此量化后损失的主要是尾部异常值的信息。用 GPTQ 或 AWQ 这种自适应量化方案，可以保留下重要权重，把损失降到最低。

**深入理解版参考答案：**

量化对推理的影响要分两步讲：**Prefill** 和 **Decode**。

**Prefill 阶段（计算瓶颈）**：量化后权重从 FP16 变成 INT8（或 INT4），显存缩小一半。GPU 的 Tensor Core 对 INT8 矩阵乘法的算力是 FP16 的 2 倍（A100 上为例，FP16 TFLOPS 约 312，INT8 约 624）。这意味着同样吞吐的模型，Prefill 延迟会降低约一半。但注意：输入激活（x）通常保持 FP16，所以 W8A16 量化只对权重量化，激活仍 FP16。

**Decode 阶段（访存瓶颈）**：Decode 时模型是 memory-bound，主要耗时在从 HBM 搬运权重到 SRAM。权重缩小一半意味着需要的访存带宽减半，所以 token 生成延迟也可以降低 30%-50%（取决于带宽利用率）。

**数学原理**：量化误差分为两个来源：
- **rounding error（舍入误差）**：真实值 `r` 和量化值 `q` 的差。量化前的均匀采样下，rounding error 在 [-0.5, 0.5] × scale 范围内。
- **outlier（异常值）损失**：超出 quantization range（-127 ~ 127 之外的权重）被截断，信息完全丢失。这是 W8A16 量化精度损失的主要来源。

**工程实践**：
- **对称量化 vs 非对称量化**：对称量化（zero-point=0）计算简单，Tensor Core 原生支持；非对称量化（有 zero-point）更灵活，但计算复杂。常用对称量化 + per-channel。
- **校准（calibration）**：用一小批数据（几百条）统计权重分布，确定最佳 scale。GPTQ 用“最优化满足扰动最小”的策略，AWQ 通过观察权重和激活的交互关系，保留对激活敏感的 weight channel。
- **混合量化**：对 attention 的 Q、K、V 层保持高精度，因为它们的权重统计分布更宽；对 MLP 层可以大胆量化。
- **常见坑**：量化后模型输出长度可能变长（因为模型对一些不确定性增大了随机性），需要增大 max_new_tokens 限制或者在评测时确认。

**可能追问 1：** 量化的 scale 怎么选？为什么不用简单平均而是用 max 方法？

**回答：** 用 max 方法（`scale = max(|w|) / 127`）确保所有值都在 [-127, 127] 范围内，没有超出范围的截断误差。缺点是如果有个别异常值非常大，会导致 scale 变大，大多数权重的分辨率变低。所以更好的方法是去除 top 0.1% 异常值后计算 scale（clip），或者用 GPTQ/AWQ 自动找最优 scale。简单平均方法无法保证无截断。

**可能追问 2：** 如果量化后模型出现“理解力下降”（比如逻辑推理变差），怎么修复？

**回答：** 首先回退到 FP16 确认是量化的问题。然后尝试：1）使用更高精度的量化方案，如 W8A16 变回 W8A16+F16 混合；2）对关键层（如 QKV 投影、前几层 attention）保持 FP16，其余层量化；3）用校准数据集（来自目标任务的 few-shot 数据）而不是通用文本做 GPTQ 校准；4）如果量化的目的是服务化加速，也可以尝试用投机解码或 SGLang 的前缀优化来提升速度，而不是继续降精度。
