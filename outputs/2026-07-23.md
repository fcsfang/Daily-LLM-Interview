### 题目 1：如何评估一个 RAG 系统的性能？请从检索端和生成端分别说明指标和评测方法。

**来源依据：**
- RAG Interview: 40 Questions to Go from Beginner to Advanced - https://www.analyticsvidhya.com/blog/2026/02/rag-interview-questions-and-answers - 技术博客 - 较高可信度
- LLM Interview Questions and Answers (2026) - https://www.interviewbit.com/llm-interview-questions-answers - 技术博客 - 较高可信度
- AI 智能体与大模型应用开发面试题库 - https://github.com/didilili/ai-agents-from-zero/blob/main/AI智能体与大模型应用开发面试题库.md - 技术博客 - 需进一步验证

**可信度：** 高

**考察点：** RAG 链路理解、评测维度的系统性、指标选择与权衡

**面试口头版回答：**

RAG 评测必须拆成检索端和生成端。检索端看召回率和精度，常用指标是 Recall@K、MRR、NDCG。比如我会对每个问题标注黄金 chunk，然后算 top-5 召回里命中多少。生成端主要看忠实度、答案相关性和答案正确性。忠实度用 NLI 模型或 LLM-as-judge 打分，看回答是否被检索内容支撑；答案正确性需要人工或自动对比标准答案。我习惯先离线建一个几百条的真实评测集，固定几个变量做对比实验，比如对比不同 chunk_size 或 reranker 的效果，这样每个改进都能看到数字涨跌，而不是凭感觉。

**深入理解版回答：**

一个完整的 RAG 评测方案应该覆盖四个维度：

1. **检索质量**：通常用 Recall@K 衡量正确答案是否在 top-K 里，用 MRR 评估第一个正确结果的位置，用 NDCG 考虑排序。需要标注黄金 chunk，标注成本高。工程上常用「基于答案的反向标注」：对每个 query，把生成正确答案所需的 chunk 标为正例。
2. **生成质量**：
   - **忠实度（Faithfulness）**：回答是否严格基于检索内容。常用 True-NLI 模型（如 AlignScore）或 LLM-as-judge，让模型判断回答中的每个声明是否被上下文支撑。
   - **答案相关性（Answer Relevancy）**：回答是否针对问题，不跑偏。
   - **答案正确性（Answer Correctness）**：需要参考答案，计算语义相似度或人工打分。
3. **鲁棒性**：测试检索噪声（混入无关 chunk）时回答是否仍能拒绝或正确回答。
4. **效率**：延迟、吞吐、成本。

工程落地建议：不要一次性追求全量指标。建议先建 100-200 条高质量评测集（人工标注黄金 chunk 和参考答案），每次改 chunk 策略或 reranker 时，固定其他变量跑一次离线评测。线上再配合反馈数据（点赞/点踩、用户重问率）做闭环。

**常见追问与回答：**

- Q：如何标注黄金 chunk？  
  A：对每个 query，检索出 top-20 后人工标注哪些是真正有用的，或者用「答案反推法」：把正确答案丢给检索器，找最相似的 chunk 设为正例。
- Q：评测集样本量不够怎么办？  
  A：可以先从线上日志采样，用 LLM 自动生成 query-chunk 对，再人工抽检 30% 保证质量。初始 100 条足够发现问题。

**项目联系：**  
我在上一版 RAG 项目里，先离线做了 200 条涵盖“多轮指代”“领域术语”“图表引用”的评测集。每次修改检索参数后，用这个评测集跑一遍 Recall@5 和忠实度分数，能把“感觉有提升”变成“实际提升了 3 个点”。这个思路也用在后续 Agent 评测中。

**记忆关键词：**  
RAG 评测、Recall@K、忠实度、LLM-as-judge、离线评测集、指标闭环

### 题目 2：你如何设计一个 Agent 的评测方案？特别是针对工具调用和多轮对话场景，如何进行自动化评估？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- GenAI & Agentic AI Interview Questions (2026) - https://agentswarms.fyi/interview-questions - 技术博客 - 需进一步验证
- AI 智能体与大模型应用开发面试题库 - https://github.com/didilili/ai-agents-from-zero/blob/main/AI智能体与大模型应用开发面试题库.md - 技术博客 - 需进一步验证

**可信度：** 中（多来源交叉验证）

**考察点：** Agent 交互行为评估、工具调用正确性、多轮一致性、自动化评测

**面试口头版回答：**

Agent 评测比纯 RAG 复杂，它涉及行为序列。我分成三个维度：一是工具调用正确性，比如参数格式、调用的时机、对失败的重试逻辑是否合理；二是任务完成率，用多轮模拟环境跑 50-100 条测试用例，看最终是否达成目标；三是交互质量，比如回复是否冗余、是否主动澄清歧义。自动化评测我会分两层：底层用 rule-based 检查工具调用参数（比如 JSON schema 校验），上层用 LLM-as-judge 给多轮对话的流畅度、代理忠诚度打分。然后在线上埋点追踪任务完成率和回退次数。

**深入理解版回答：**

Agent 评测的核心难点是“行为链”的评估。具体方案：

1. **构建模拟环境**：为每个 Agent 任务（如订机票、查库存）设计一个 mock 工具接口，返回可控数据。这样我们可以预定义“期望的工具调用序列”作为标准答案。
2. **离线评测集**：每条测试用例包括：用户意图 → 期望的工具调用链（含参数） → 期望的最终回复；同时标注失败情况（如工具超时、返回空值）下的期望应对。
3. **自动化评估指标**：
   - **工具调用准确率**：对比 Agent 实际调用与期望调用，精确到函数名和参数键值对。参数可以用 Jaccard 相似度或模糊匹配。
   - **路径正确率**：整个工具链顺序是否符合预期，是否有冗余调用。
   - **恢复能力分数**：当工具返回错误时，Agent 是否主动重试、修正参数或告知用户。
   - **最终任务成功率**：二元指标（成功/失败）。
4. **多轮评估**：引入会话级别的指标，比如指代消解的正确性、记忆一致性（上一轮说过的偏好本轮是否记住）。
5. **LLM-as-judge**：用强大模型（如 GPT-4）对 Agent 的回复做多维度打分，包括离题、重复、攻击性等。注意要提供结构化评分 Prompt 和 few-shot 样例。

工程上，可以基于 LangSmith 或者自建日志系统，把每一步的输入输出、工具调用记录都保存下来，然后跑离线脚本计算上述指标。

**常见追问与回答：**

- Q：条件组合爆炸怎么解决？  
  A：用组合测试（pairwise testing）选择典型工具调用路径，先覆盖最常用路径，再用错误注入测试异常路径。
- Q：LLM-as-judge 打分不准确怎么办？  
  A：先对 LLM 裁判做校准，用 100 条人工标注数据对比，若一致率低于 70% 就要改 Prompt 或换模型；可结合 rule-based 校验提升可靠性。

**项目联系：**  
我做的 Agent 项目里，构建了 30 条典型“订会议室”用例，覆盖正常流程、时间冲突、权限不足等异常。离线跑工具调用准确率，从 72% 优化到 91%。线上日志里每月抽样分析 bad case，发现 80% 的失败源于参数类型错误，于是加了 JSON schema 前置校验。

**记忆关键词：**  
Agent 评测、工具调用准确率、模拟环境、路径正确率、LLM-as-judge、错误注入

### 题目 3：在实际项目中，你如何处理 bad case？请结合一个具体案例说明你如何发现、分析并修复 bad case。

**来源依据：**
- 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证（文中提到“先有评测集再优化”的实践）
- AI 智能体与大模型应用开发面试题库 - https://github.com/didilili/ai-agents-from-zero/blob/main/AI智能体与大模型应用开发面试题库.md - 技术博客 - 需进一步验证（提到 badcase 建设和离线评测闭环）

**可信度：** 中

**考察点：** 问题发现机制、根因分析能力、闭环优化习惯

**面试口头版回答：**

我的 bad case 处理流程分三步：发现、定位、修复。发现阶段靠线上日志和用户反馈，比如收集用户重问、负面反馈，或者监控回答的各项指标（忠实度分数、延迟）的异常波动。定位阶段我会把 bad case 拆到系统各模块：是检索错了（召回 top-1 不相关）、还是上下文组装错了（塞了噪声）、还是模型生错了（忽视上下文）。比如有一次用户问“华东区销售数据”，系统答了全国数据。我分析发现检索阶段混入了全国综述文章，因为 chunk 切分时把全国和华东的段落切在了一起。修复方法是改 chunk 策略：按小标题分割，并加标题级 metadata 过滤，在检索时优先匹配包含“华东”关键词的 chunk。

**深入理解版回答：**

一个可落地的 bad case 闭环持续优化流程：

1. **发现机制**：
   - 线上埋点：用户点击“不满意”按钮、短时间内重复提问、调出/重新生成操作。
   - 自动监控：通过 LLM-as-judge 对回答的忠实度做实时抽样打分，低于阈值自动告警。
   - 定期抽样：每周从日志里抽 200 条对话，人工看 20-30 条典型 bad case。
2. **定位方法**：对于每个 bad case，建立一个“根因排查表”，逐层检查：
   - 用户 query（是否歧义）
   - Query 改写（是否正确）
   - 检索（top-5 召回质量）
   - Reranker（排序是否合理）
   - 上下文拼接（是否超长、是否丢失关键 chunk）
   - 生成（是否忠实）
   - 若涉及 Agent：工具调用是否正确
3. **修复动作**：
   - 快速修复（hotfix）：改规则、改 Prompt、测一批样例上线。
   - 系统修复（长期）：更新 chunk 策略、调整 reranker 模型、加入异常检测模块。
4. **验证闭环**：将修复后的 case 加入回归评测集，跑一轮确保不引入新问题。

案例：在一个 RAG 系统中，发现大量 case 回答中包含与检索内容矛盾的陈述。根因是检索到的 chunk 本身含有过时信息。修复方式：在知识库更新流程中增加“版本时间戳”，检索时优先取最新版本；同时在生成 Prompt 中加指令“如果检索内容存在矛盾，请指出并优先采纳更新时间最近的内容”。

**常见追问与回答：**

- Q：如何量化修复效果？  
  A：对这批 bad case 逐个打分（0/1 是否修复），统计修复率；并回测 100 条正常用例看无退化。
- Q：遇到无法复现的 case 怎么办？  
  A：可能是模型采样随机性或上下文长度截断导致。我会尝试固定 seed、温度参数重现，如果仍不复现，则记录为“不稳定 case”，短期内加更多约束（如用确定性解码）并监控。

**项目联系：**  
我做 RAG 项目时，每周从日志抽样，发现一个典型 case：用户问“保修政策”，系统回答包含“本产品保修三年”，但知识库最新版本是“保修两年”。根因是知识库更新时没有清除旧 chunk，检索排名上旧 chunk 得分更高。修复后加了版本过滤，并在回答中增加来源时间戳，用户可见。

**记忆关键词：**  
Bad case 闭环、根因分析、回归测试、忠实度监控、知识库版本

### 题目 4：什么是 LLM-as-judge？在实际评测中如何应用？有哪些风险和缓解措施？

**来源依据：**
- 根据经典面试题延伸，今日资料中未直接详细解释，需标注。资料中有“LLM-as-judge”概念应用（如 Agent 评测打分），但无系统性介绍。
- 需进一步验证

**可信度：** 低（但该知识点为大模型评测领域公认重要，且今日资料中多次暗示使用 LLM 作为裁判）

**考察点：** 自动化评测方法理解、LLM 偏好偏差认知、工程落地经验

**面试口头版回答：**

LLM-as-judge 就是用一个语言模型（通常是更强的模型，如 GPT-4）来评估另一个模型的输出质量。常见做法是写一个结构化评分 prompt，让模型给回答的忠实度、相关性、连贯性等维度打分，并给出理由。优点是自动化、可扩展；缺点是有偏好偏差：裁判模型可能偏爱更长的回答、更自信的语气，或者与自己训练数据风格相似的输出。缓解措施包括：使用多个裁判模型投票、提供 few-shot 评分样例、要求裁判模型先分析再打分、以及定期校准（用人工标注集对比一致率）。

**深入理解版回答：**

LLM-as-judge 已成为大模型评测的重要工具，尤其在离线评测中替代大量人工。

1. **应用方式**：
   - **打分式**：输入（问题、模型输出、参考标准），输出 1-5 分。
   - **比较式**：输入（问题、模型A输出、模型B输出），输出谁更好或平局。
   - **分类式**：判断输出是否存在特定问题（如幻觉、离题）。
2. **Prompt 设计关键**：
   - 明确的评价维度定义（忠实度：回答中的每个声明都能被上下文支持）。
   - 要求先引用证据（“请从回答中引用具体句子来说明你的判断”）。
   - 使用 few-shot 样例（特别是边界 case）。
   - 控制输出格式（JSON 结构化）方便后续处理。
3. **风险与缓解**：
   - **位置偏差**：比较式评测中，优先展示的输出可能被偏爱。缓解：重复两次（A/B 和 B/A）取一致结果。
   - **风格偏差**：偏爱更详细、更正式的回答。缓解：在评测时给裁判模型提供“什么是好的回答”的具体定义，而不是笼统描述。
   - **自褒偏差**：裁判模型倾向于给自己的模型打高分。缓解：使用弱模型作为裁判；或者只用“比较式”而非“打分式”。
   - **一致性**：同一 case 多次评测结果不稳定。缓解：固定温度和 seed，多次评测取平均。
4. **工程实现**：封装一个 `Judge` 类，接受评测集和评分 Prompt，支持并发调 API，输出打分结果和置信度。每天跑一次批量评测，监控分数趋势。

**常见追问与回答：**

- Q：如何验证 LLM-as-judge 的可靠性？  
  A：与人工标注对比，计算 Cohen's Kappa 或一致率。如果低于 0.6，则需要优化 Prompt 或更换裁判模型。
- Q：能否用开源的弱模型做 judge？  
  A：可以，但需要更严格的校准。比如用 100 条人工标注 finetune 一个打分模型，或者用多数投票。

**项目联系：**  
我没完全实现过 LLM-as-judge 生产化，但在实验室项目里用它来评估多个微调版本的应答质量，避免了招多人标注。我设计了对比式 judge，每次比较两个版本，并比较人工抽检的结果，一致率约 78%，基本可用。

**记忆关键词：**  
LLM-as-judge、偏好偏差、校准、结构化评分、比较式评测、自动评测

- **链接：** https://www.analyticsvidhya.com/blog/2026/02/rag-interview-questions-and-answers
- **来源依据：** 同上（技术博客，高可信度）
- **文章类型：** 技术博客（工程实践 + 面试准备）
- **核心内容总结：**
  - RAG 的基础概念和流程（离线索引 → 在线检索+生成）
  - 详细解释了检索器和生成器的耦合关系
  - 文档块（chunking）的粒度对性能的影响：太大则噪音多，太小则丢失语义
  - 评测指标：Retrieval metrics（Recall@K, MRR, NDCG）和 Generation metrics（Faithfulness, Relevance, Answer Correctness）
  - 混合检索（稀疏+稠密）的优势和实现方法
- **对面试有什么用：**
  - 可以系统性地回答“RAG 怎么评测”的问题，直接给出 6 个左右指标并解释为什么选它们
  - 了解 co-relation 问题：检索质量直接影响生成质量，面试官可能追问“如果你只能选一个指标，你会选哪个？”
- **可转化面试题：**
  - **Q1：你认为在 RAG 评测中，最关键的一个指标是什么？为什么？**  
    **参考答案：** 我认为最关键的是忠实度（Faithfulness）。因为 RAG 的核心价值就是让回答基于检索内容，即使检索错了，如果模型能拒绝或指出矛盾，也比自信地编造好。忠实度直接衡量回答是否被检索内容支撑。如果忠实度低，说明系统没有发挥 RAG 的作用；如果忠实度高但答案不准，至少说明是检索问题，可以定向优化。
  - **Q2：离线评测和在线评测在 RAG 场景下各自的侧重点是什么？**  
    **参考答案：** 离线评测侧重指标对比，如 Recall@K、忠实度、答案正确性，适合快速迭代验证；在线评测侧重用户体验，如用户点击率、重问率、平均任务完成时间。离线评测不能替代在线评测，因为离线评测集可能覆盖不全，且用户的真实 query 分布和离线模拟不同。所以我习惯先离线验证再到线上小流量实验。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** 评估 RAG 忠实度（Faithfulness）时，有哪些自动评测方法？
- **A：** 常用三类：① NLI 模型：将回答声明拆成原子句子，与检索上下文做蕴含判断，计算未蕴含比例；② QA 回测：根据回答生成问题，再检索上下文看能否回答；③ LLM-as-judge：让大模型判断回答是否被上下文完全支撑，并给出理由。
- **一句话记忆法：** 拆句做 NLI，反向生成 QA，直接问大模型——三种保真度验尸法。
- **复习标签：** #评测 #RAG #忠实度

### 卡片 2
- **Q：** Agent 评测中，为什么不能只用最终任务成功率？
- **A：** 最终成功率掩盖了过程问题：可能碰巧猜对参数，也可能走了冗余路径但成功，或者工具调用失败但用户让步。需要拆解中间行为指标，如工具调用格式正确率、参数正确率、重试次数、回退率，才能定位具体瓶颈。
- **一句话记忆法：** 只看进球不算好教练，还要看传球、跑位和失误。
- **复习标签：** #评测 #Agent #过程指标

### 卡片 3
- **Q：** LLM-as-judge 的常见偏差有哪些？如何缓解？
- **A：** ①位置偏差（先看到的答案得分高）：双向比较消除；②风格偏差（偏爱长答案）：在评分 Prompt 中明确定义“好答案”标准，不靠感觉；③自褒偏差（偏爱自己模型）：用弱模型做裁判或比较式；④一致性差：多次运行取平均，固定温度。
- **一句话记忆法：** 裁判偏爱、位置、风格、一致性——四害，用双向、标尺、弱化、固定来治。
- **复习标签：** #评测 #LLM-as-judge #偏差

### 卡片 4
- **Q：** 构建离线评测集时，样本数量和分布有什么要求？
- **A：** 至少 100-200 条，覆盖主要场景（正常、边界、异常）。分布尽量逼近线上真实 query 分布（可从日志采样），但也需要专门补充 20% 的极端 case（如长尾问题、多轮指代）。每条需标注：query、理想答案、支撑 chunk 索引、分类标签。
- **一句话记忆法：** 100 条打底，采样子看日志，补极端 20%，标注四件套。
- **复习标签：** #评测 #数据集构建 #离线评估

### 卡片 5
- **Q：** RAG 系统中，如何衡量检索质量对最终答案的影响？
- **A：** 通过对比实验：固定生成模型和 Prompt，只改变检索策略（如不同 chunk_size、不同 embedding、不同 reranker），观测生成端指标（忠实度、答案正确性）的变化。同时做消融研究：逐步加入检索噪声（混入无关 chunk），看模型容忍度。
- **一句话记忆法：** 控制变量测检索，加噪声看抗干扰——检索好不好，生成说了算。
- **复习标签：** #评测 #RAG #检索质量 #消融实验

## 4. 今日面试专项训练

### 4.1 大模型理论追问题：对比传统基于规则的评测与基于 LLM 的评测在 Agent 场景下的优劣

**问题：** 在设计 Agent 评测方案时，你会如何权衡使用规则评判（如正则匹配工具调用参数）和使用 LLM-as-judge 评判各自的适用场景？并举例说明。

**面试口头版参考答案：**

规则评判适用于确定性强的检查，比如工具调用 JSON 格式是否正确、参数类型是否合规，速度快且无偏差。LLM-as-judge 适用于语义层面的判断，比如回复是否礼貌、是否偏离用户意图、工具调用链是否逻辑合理。我通常组合使用：底层用规则卡硬性条件（格式、超时），上层用 LLM 做软性评分。比如一个 Agent 回答中包含了工具调用，我先用规则校验 JSON schema，若格式错直接判负；格式正确后，再用 LLM 判读调用顺序是否合理。

**深入理解版参考答案：**

传统规则评测的优点是确定性强、成本低、可解释。缺点是无法处理复杂语义（如“修改会议室”和“将会议改为线上”可能是同一种意图但表述不同），且需要手动编写大量规则，维护困难。
LLM-as-judge 的优点是灵活，能覆盖语义相似性和复杂行为序列。缺点是成本高（调用 API 的 token 和延迟）、有偏好偏差、稳定性差（多次评测结果不一致）。
在 Agent 场景下，我的建议是：
- 工具调用准确率（参数是否匹配 schema）：**规则**。用 JSON Schema 校验器和 keyset 匹配即可。
- 工具调用顺序合理性：**LLM-as-judge**。因为“先确认用户身份再查询”和“查询后提示用户身份需要确认”可能都是合理路径，规则难以枚举。
- 多轮记忆一致性：**LLM-as-judge**。因为要判断“前面说过的约束在后续是否被遵守”，需要语义理解。
- 安全性检查（是否输出了敏感信息）：**规则（白名单/黑名单）+ LLM**。先用正则匹配敏感词，再让 LLM 判断上下文是否构成泄露。
工程上，我建议先跑规则过滤掉绝对错误的 bad case，再随机采样 30% 的剩余 case 用 LLM 评估，既节省成本又能捕获语义问题。

**可能追问与简短回答：**

- Q1：规则评测和 LLM 评测的结果不一致时，你以哪个为准？  
  A：分别记录。规则评测是硬约束，LLM 评测是软建议。如果规则判通过但 LLM 判差，会人工抽检，看是否为规则遗漏或 LLM 偏差。若规则判差但 LLM 判好，几乎不存在（因为规则通常更严格）。
- Q2：能不能训练一个专门的评分模型来代替 LLM-as-judge？  
  A：可以，用人工标注的 500-1000 条分数数据微调一个轻量模型（如 BERT-large），可以作为 Agent 评测的快速打分器。但初期数据不足时，用 LLM-as-judge 更灵活。
