# 今日大模型实习面试日报

## 1. 今日面试真题精讲

### 问题 1：请详细解释 LoRA 微调的原理，并说明它与全参数微调相比的优缺点。

- **考察点：**
  - 对高效微调技术（PEFT）的深入理解。
  - 对 LoRA 数学原理（低秩分解）的掌握。
  - 能够从工程和效果两个角度客观评价技术方案。
- **面试口头版回答：**
  “LoRA，全称 Low-Rank Adaptation，是一种高效的模型微调技术。它的核心思想是，大模型在下游任务上做全参数微调时，权重更新量 ΔW 虽然是高维的，但它的‘有效秩’通常很低。因此，我们不用直接更新原始的权重矩阵 W，而是冻结它，额外引入两个低秩矩阵 A 和 B 来模拟这个变化。最终的前向传播就变成了 h = Wx + BAx。这样，训练时只需要优化 A 和 B 这两个小矩阵，显存占用和训练时间都大大降低了。它的优点是训练快、显存省、易于切换任务（只需换个 adapter），缺点是对于一些非常复杂的任务，可能无法达到全参数微调的极限性能，而且 rank、alpha 这些超参数对效果影响比较大。”
- **深入版回答：**
  “更深入地讲，LoRA 源于一个假设：大模型在下游任务中的权重更新具有‘低秩性’（Low Intrinsic Rank）。如果我们想微调一个预训练权重矩阵 W₀ ∈ R^(d×k)，其更新量 ΔW 也是 d×k 维的。LoRA 将 ΔW 分解为两个可训练的低秩矩阵 B ∈ R^(d×r) 和 A ∈ R^(r×k) 的乘积，其中 r << min(d, k)。这样，需要训练的参数量从 d×k 降到了 d×r + r×k，通常是原来的万分之一甚至更少。从梯度更新的角度看，LoRA 实际上是在一个低维子空间中搜索最优解，这大大降低了显存需求，因为优化器状态（如 Adam 的 momentum 和 variance）都是针对低秩矩阵存储的。此外，LoRA 可以与量化技术结合，比如 QLoRA，通过将基础模型量化到 4-bit，进一步压缩显存，使得在单张消费级显卡上微调 7B、13B 甚至更大的模型成为可能。
  
  关于缺点，除了回答中提到的，还有一点是‘合并’操作。虽然推理时可以把 BA 合并进 W₀，不增加延迟，但这会失去灵活切换不同任务 adapter 的能力。这在需要同时服务多个下游任务的场景下是个需要考虑的 trade-off。”
- **常见追问：**
  - “LoRA 的 rank r 通常取多少？对效果有什么影响？”
    - **回答：** “通常取 8、16、64 等。r 越大，可训练参数越多，模型拟合能力越强，但也越容易过拟合，显存占用也越大。r 太小则可能欠拟合，无法充分学习任务特征。实际中需要通过实验来调参。
  - “QLoRA 和 LoRA 有什么区别？”
    - **回答：** “QLoRA 是 LoRA 的一种极致优化。它把预训练模型权重量化到 4-bit（通常用 NF4 格式），然后在量化后的模型上挂载 LoRA adapter。这样做大幅降低了显存占用，使得小显存卡也能微调大模型。但 QLoRA 并不保证训练速度更快，因为量化/反量化操作本身有额外开销，它主要解决的是显存瓶颈。”
- **项目联系：**
  “在我的项目中，因为只有一块 24G 显存的 GPU，无法直接对 7B 模型进行全参数微调。因此我选择使用 LoRA 来微调模型以适应特定领域的问答任务。通过这种方法，我成功在有限资源下完成了模型适配，最终模型在领域测试集上的 F1 得分提升了 8%。”
- **记忆关键词：**
  `冻结原权重` + `低秩矩阵AB` + `模拟增量ΔW` + `省显存快训练` + `低秩假设`

### 问题 2：请详细阐述一个完整的 RAG 系统流程，并说明每个环节的关键设计点。

- **考察点：**
  - 对 RAG 系统工程实现的理解广度。
  - 对数据准备、索引、检索、生成、评估各环节的把握。
  - 对系统瓶颈和优化方向的认知。
- **面试口头版回答：**
  “一个典型的 RAG 系统分为离线流程和在线流程。离线是为了构建知识库：首先对文档进行清洗和切块（Chunking），然后使用嵌入模型将每个块转化为向量，最后把这些向量存入向量数据库并建立索引。在线流程就是处理用户请求：用户输入问题后，系统首先将问题也用相同的嵌入模型转为向量，然后去向量数据库中进行最近邻搜索，找到最相关的 top-k 个文档块。接着，通常会用一个重排序模型（Reranker）对这些文档块进行精排，去除噪声。最后，将用户问题和检索到的相关文档组合成一个 Prompt，交给大语言模型（LLM）生成最终答案。关键设计点包括：1. 分块策略（大小、重叠度、分割方式）；2. 嵌入模型的选择（质量、维度、语言支持）；3. 检索算法（近似最近邻搜索）；4. 重排序的必要性；5. Prompt 的设计和优化。”
- **深入版回答：**
  “深入来看，RAG 系统的核心挑战在于‘检索质量和上下文管理’。在离线阶段，我倾向于使用‘语义分块’策略，即利用 LLM 或 NLP 技术识别文档的语义边界，确保每个块是一个完整的语义单元，而不是简单地按字数截断。同时，为了应对‘Lost in the Middle’问题，在生成时，我会尝试将检索到的文档按相关性排序，最重要的放在 Prompt 的开头和结尾，而不是一股脑地丢进去。我还关注‘检索后处理’（Post-Retrieval），比如使用‘压缩’（Compression）技术，让 LLM 仅提取文档中与问题相关的部分，以减少上下文长度、降低噪音。在评估层面，我会使用 RAGAS（Retrieval Augmented Generation Assessment）框架来量化评估系统的忠实度（Faithfulness）、答案相关性（Answer Relevancy）和上下文精度（Context Precision），而不仅仅是看最终的答案。”
- **常见追问：**
  - “什么是‘Lost in the Middle’问题？如何缓解？”
    - **回答：** “这是大型语言模型对长上下文中位置信息的偏好问题。模型倾向于记住上下文开头和结尾的信息，而忽略中间部分的内容。缓解方法有：调整检索到的文档排序，将最相关文档放在首位；使用‘上下文压缩’技术提取关键信息；或者限制传递给 LLM 的上下文长度，只保留最精华的部分。”
- **项目联系：**
  “在构建一个企业内部知识库问答系统时，我建立了一个完整的 RAG 管道。使用 LangChain 框架进行流程编排，选择了 bge-large-zh-v1.5 作为嵌入模型，FAISS 作为向量数据库。实践中发现，简单的 embedding 相似度检索有时会召回语义相似但与原问题无关的文档。因此我在检索后增加了一个 cross-encoder 的 re-ranker 模块，显著提升了最终答案的准确率。”
- **记忆关键词：**
  `离线建库（清洗切块-嵌入-入库）` + `在线检索（问题嵌入-召回-重排序-生成）` + `分块策略` + `Lost-in-the-Middle` + `RAGAS评估`

### 问题 3：RAG 与微调（Fine-tuning）的区别和适用场景是什么？

- **考察点：**
  - 对两种主流技术路线的优缺点有清晰的认知。
  - 能够根据实际业务场景做出合理的技术选型。
- **面试口头版回答：**
  “RAG 和微调都是为了提升大模型在特定任务上的表现，但它们的思路完全不同。RAG 是一种‘开卷考试’，它不修改模型参数，而是通过动态检索外部知识来给模型提供‘参考资料’。而微调是一种‘闭卷考试’，是通过更新模型参数，让模型自己‘记住’新的知识。RAG 的优势在于：知识更新成本低（只需更新外部数据库）、可以溯源（答案有出处）、对抗幻觉能力强。微调的优势在于：可以让模型学习到新的行为模式和输出风格、减少推理延迟（不需要检索步骤）、对私有数据更安全（知识内化于模型内部）。因此，如果任务是事实性问答、需要引用最新资料，选 RAG。如果任务是学习特定格式的输出（如写诗歌、代码风格转换）、需要让模型适应新的交互范式，选微调。在企业中，通常是 RAG-First 策略，先用 RAG 快速验证效果，如果发现模型‘语感’或输出格式不符预期，再考虑用微调来打补丁。”
- **深入版回答：**
  “更深入地说，RAG 和微调可以互补。RAG 处理‘知识’问题，微调处理‘能力’问题。例如，一个智能客服系统，可以使用 RAG 来从最新的产品手册中寻找答案，确保信息的准确和时效；同时，可以使用微调来让模型学会客服的‘话术’和‘礼貌用语’。另一个关键区别在于‘数据效率’。微调需要一定数量的标注数据才能有效；而 RAG 只需要文档本身，零样本即可工作。在‘数据安全’维度，微调将知识写入了模型参数，理论上存在被攻击者通过特定 Prompt 抽取出来的风险；而 RAG 可以做到‘数据不出域’，只返回检索结果。当前很多先进的系统，如 Agentic RAG，就是将两者结合，Agent 负责规划，RAG 负责提供知识，实现了更强大的能力。”
- **常见追问：**
  - “在什么场景下，微调是 RAG 无法替代的？”
    - **回答：** “当你希望模型学会一种全新的、隐含在大量数据中的‘风格’或‘行为模式’时，微调是必要的。比如，让模型从机器翻译的正式风格微调成社交媒体上的口语化风格，或者让模型学会像某个特定作家一样写文章。这很难通过几段检索到的文档来解决。”
- **项目联系：**
  “在我的项目中，我先是用 RAG 解决了知识问答的准确性问题。但发现模型回答风格过于死板。于是我用几百条高质量的客服对话数据对模型进行了 LoRA 微调，让它的回复变得更亲切、更人性化。最终，这个‘RAG + 微调’的组合方案在用户满意度上比纯 RAG 方案提升了 15%。”
- **记忆关键词：**
  `开卷（RAG：查资料）vs 闭卷（微调：背知识）` + `RAG强在知识更新和溯源` + `微调强在风格和行为学习` + `RAG-first，微调打补丁`

### 问题 4：你是如何评估一个 RAG 系统效果的？请列举具体指标。

- **考察点：**
  - 对 RAG 系统评估体系的认知深度。
  - 能够区分评估检索效果和生成效果。
- **面试口头版回答：**
  “评估 RAG 系统必须从‘检索’和‘生成’两个维度分开进行。对于检索模块，核心指标有：召回率（Recall），看正确答案是否在返回的结果中；还有平均倒数排名（MRR），看正确答案排在第几位。对于生成模块，最重要的是忠实度（Faithfulness），检查生成的答案是否与检索到的文档内容一致，有没有‘胡说八道’；其次是答案相关性（Answer Relevancy），看答案是否与用户问题相关；最后也可以使用 ROUGE-L、BLEU 等指标来评估答案与标准答案的文本相似度。在实践中，我强烈推荐使用 RAGAS 评估框架，它可以自动化地计算这些指标，帮助我们快速迭代系统。”
- **深入版回答：**
  “更专业的评估要关注‘上下文’的质量和‘系统’的鲁棒性。除了 RAGAS 的五维指标（忠实度、答案相关性、上下文相关性、上下文召回率、上下文精确度），我还会关注‘噪声鲁棒性’：当检索结果中混入大量无关文档时，模型是否还能抓住重点去生成。此外，‘否定鲁棒性’也很重要：当检索结果中明确没有答案时，模型是否能够诚实地回答‘不知道’，而不是强行编造。在生产环境中，我们还会使用 A/B 测试，通过收集用户的点击、点赞、修正等隐式和显式反馈，来评估系统的最终用户体验。好的评估体系应该是‘离线指标 + 在线指标’的闭环。”
- **常见追问：**
  - “如果模型答案忠实度很高，但答案相关性不好，说明了什么问题？如何解决？”
    - **回答：** “这说明检索到的文档虽然与问题语义上相关，但并非用户真正需要的那个‘答案’。比如，用户问‘苹果公司的股价’，系统检索到了一篇讨论苹果公司产品的文章。这说明嵌入模型对语义的理解不够精确。解决方法包括：优化分块策略，让文档块更聚焦；使用混合检索（向量检索 + 关键词检索）；或者在检索后使用更精细的 Reranker 模型。”
- **项目联系：**
  “在我优化的 RAG 项目中，最初仅凭感觉观察回答效果。后来引入了 RAGAS 框架，发现忠实度指标只有 75%。经过分析，发现是 Prompt 设计的问题，模型经常‘自由发挥’。于是我在 Prompt 中强制要求模型‘只根据提供的信息回答，如果信息中没有，就说不知道’。修改后，忠实度指标提升到了 95%。”
- **记忆关键词：**
  `检索评估（Recall, MRR）` + `生成评估（忠实度、相关性）` + `RAGAS框架` + `鲁棒性评估` + `离线+在线闭环`

- **可以转化成的面试题：**
  1. 你的 RAG 系统上线后，用户反馈经常找不到正确的信息，你会如何排查和优化？
     - **参考答案：** “首先，我会按照‘切块策略→召回方式→精排兜底’的三步排查法来定位问题。第一步，检查数据切块是否合理，是否因为切块太大导致信息不聚焦，或切块太小导致语义不完整。我会尝试语义切块，确保每个块是完整的语义单元。第二步，检查召回方式。如果仅用向量检索，可能会出现语义漂移（比如用户问‘苹果’系统召回‘水果’）。我会考虑增加 BM25 关键词检索，实现混合检索。第三步，引入重排序。在初步检索返回的几十条结果后，使用一个 Cross-Encoder 的 Reranker 模型进行精排，将最相关的文档排在最前面，再交给 LLM 生成。如果这三步依然无法解决，我会排查嵌入模型是否适合当前领域，以及文档本身的质量问题（如信息缺失或矛盾）。”

## 3. 今日必背问答卡片

- **卡片 1：**
  - **Q：** 什么是 Transformer 模型中的 LayerNorm，为什么它比 BatchNorm 更适合 NLP？
  - **A：** Layer Normalization 是对每个样本的所有特征维度进行归一化，而 Batch Normalization 是对每个特征维度在一个 batch 的所有样本上进行归一化。在 NLP 中，序列长度可能不同，且 Batch size 通常较小（受显存限制），BN 的统计量不稳定。LayerNorm 的计算独立于 batch 和序列长度，训练和推理行为一致，更稳定，因此更适合**需进一步验证**Transformer。
  - **一句话记忆法：** `LN 跨特征，BN 跨样本；NLP 序列长，BN 不稳，LN 香。`

- **卡片 2：**
  - **Q：** 在 LoRA 微调时，如何选择 rank r 的值？有哪些经验法则？
  - **A：** R 值通常设为 8、16、32 或 64。根据经验，对于简单的任务（如分类），较小的 r（如 8）即可；对于复杂的任务（如生成、对话），可能需要更大的 r（如 64）。r 越大，可训练参数越多，拟合能力越强，但过拟合风险和 GPU 显存占用也越高。通常建议从较小的 r（如 8）开始尝试，通过实验观察模型在验证集上的表现来调整。
  - **一句话记忆法：** `任务简单用小 r（8），任务复杂用大 r（64），实验是检验真理的唯一标准。`

- **卡片 3：**
  - **Q：** 什么是 Agent 中的 ReAct 框架？请简述其核心思想。
  - **A：** ReAct 全称 Reasoning + Acting，是 Agent 的一种经典设计模式。核心思想是让 LLM 在每一步交替进行两个动作：**推理（Reasoning）**：思考当前状态，分解任务，并制定下一步需要做什么（例如：“我需要查询用户所在地的天气信息”）；**行动（Acting）**：根据推理结果，调用一个工具（Tool）来执行具体操作（例如：调用天气 API）。LLM 观察到工具返回的结果后，再进行下一轮的推理和行动，直到完成任务。
  - **一句话记忆法：** `想（Reason）了再做（Act），做了再想，循环往复直到任务搞定。`

- **卡片 4：**
  - **Q：** LLM 训练时，SFT（指令微调）的 Loss 是如何计算的？为什么要忽略 padding token 的 Loss？
  - **A：** SFT 的 Loss 是交叉熵损失。在训练时，模型输入通常需要被 padding 到统一长度。计算 Loss 时，我们会使用`ignore_index`参数（如 `-100`）来标记 Loss Tensor 中的 padding token 位置。这样，在计算 Loss 均值时，只有“真实回答”部分的 token 贡献 Loss，padding token 的部分被完全忽略。这样做是因为 padding token 不是模型需要学习的内容，如果计算它们的 Loss，会引入噪声，并鼓励模型去预测无意义的 padding 标记，从而干扰模型对真实语言模式的学习。
  - **一句话记忆法：** `SFT 计算回答部分 Loss，padding 占位用 -100 标记忽略，不让无关噪声影响学习。`

- **卡片 5：**
  - **Q：** 为什么 MoE（混合专家）架构可以在不显著增加推理成本的情况下，有效扩大模型参数规模？
  - **A：** MoE 的核心是稀疏激活。模型由多个“专家”子网络（FFN）和一个“门控网络”组成。在推理时，门控网络会根据输入 token 的特点，只激活其中少数几个（比如 top-2）专家。这意味着，虽然模型的总参数量很大（可能包含几十个专家），但对于每一个输入的 token，实际参与计算的参数量只相当于一个小模型。因此，推理时的计算成本（FLOPs）远小于总参数量对应的计算成本，实现了“大容量、低成本”的推理。
  - **一句话记忆法：** `MoE 用门控网络挑专家，每次只让少数专家干活，总参数量大，但实际计算量小，省力又强大。`

- **1 道算法题：**
  - **题目：** LeetCode 25: K 个一组翻转链表。给你链表的头节点 `head` ，每 `k` 个节点一组进行翻转，请你返回修改后的链表。`k` 是一个正整数，它的值小于或等于链表的长度。如果节点总数不是 `k` 的整数倍，那么请将最后剩余的节点保持原有顺序。
  - **思路：**
    1.  **计算长度**：首先遍历一遍链表，算出链表的总长度 `len`。
    2.  **确定翻转组数**：计算需要翻转的组数 `times = len // k`。
    3.  **迭代翻转**：进行 `times` 次翻转。在每组翻转中，可以使用**头插法**或者**递归法**。最常用的是迭代式的头插法。
        - 设置一个虚拟头节点 `dummy`，方便处理边界。
        - 在每组翻转前，记录好当前组的前驱节点 `pre` 和当前组的第一个节点 `cur`。
        - 在当前组内部，进行 `k-1` 次翻转操作，将 `cur` 后面的节点不断移动到 `pre` 后面。
    4.  **移动指针**：一组翻转完成后，更新 `pre` 和 `cur` 指针，指向下一组的前驱和第一个节点。
    5.  **返回结果**：返回 `dummy.next`。
  - **参考代码（Python）：**
    ```python
    def reverseKGroup(self, head: ListNode, k: int) -> ListNode:
        dummy = ListNode(0)
        dummy.next = head
        pre = dummy
        cur = head
        
        # 计算链表长度
        length = 0
        node = head
        while node:
            length += 1
            node = node.next
        times = length // k
        
        for _ in range(times):
            # 对当前 k 个节点进行翻转
            for _ in range(k - 1):
                temp = cur.next
                cur.next = temp.next
                temp.next = pre.next
                pre.next = temp
            # 翻转完成，移动指针到下一组
            pre = cur
            cur = cur.next
        return dummy.next
    ```

- **1 道大模型理论题：**
  - **题目：** 请简述 Self-Attention 的计算过程，并写出其时间复杂度。
  - **参考答案：**
    1.  **计算 Q, K, V：** 对于输入序列 X (长度为 n，维度为 d)，通过三个不同的权重矩阵 Wq, Wk, Wv 分别计算出 Query 矩阵 Q，Key 矩阵 K，Value 矩阵 V。形状均为 (n, d)。
    2.  **计算注意力分数矩阵：** 计算 Q 和 K 的点积，得到注意力分数矩阵 S = Q * K^T。形状为 (n, n)。
    3.  **缩放：** 为了防止内积过大导致 softmax 后梯度消失，将 S 除以 √d_k（其中 d_k 是 Query/Key 的维度，通常与 d 相同），即 S' = S / √d_k。
    4.  **Softmax：** 对矩阵的每一行应用 softmax 函数，得到概率化的注意力权重矩阵 A = softmax(S')。形状为 (n, n)。
    5.  **加权求和：** 将注意力权重矩阵 A 与 Value 矩阵 V 相乘，得到最终的输出 O = A * V。形状为 (n, d)。
    **时间复杂度：** 主要开销在三个矩阵乘法上。Q*K^T 和 A*V 的时间复杂度均为 O(n²·d)。因此，Self-Attention 的总时间复杂度为 **O(n²·d)**。其中 n 是序列长度，d 是特征维度。
