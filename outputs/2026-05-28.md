## 1. 今日面试真题精讲

### 题目 1：LoRA 的核心思路是什么？它是如何工作的？
**来源依据：**  
参考资料 [1][2][4][15]，其中 [15] 提供了具体矩阵公式。

**可信度：**  
中（多个来源一致描述，原版论文可验证）。

**考察点：**  
LoRA 原理、低秩分解动机、参数更新方式、推理合并。

**面试口头版回答：**  
LoRA 的核心思路是冻结预训练权重，在原始线性层旁边插入两个低秩矩阵（A 和 B），用它们的乘积近似权重更新量。训练时只更新这两个小矩阵，参数量只有原模型的千分之一甚至万分之一；推理时可以把 BA 合并回原权重，不增加额外延迟。举个例子，对一个 7B 模型做下游任务微调，全参需要更新 7B 参数，LoRA 可能只需要 0.1B 可训练参数，在单卡 A100 上就能跑。它的基本假设是大模型权重更新的有效“自由度”是低秩的，不需要对整个大矩阵做自由更新。

**深入理解版回答：**  
LoRA（Low-Rank Adaptation）由论文 “LoRA: Low-Rank Adaptation of Large Language Models” 提出。其核心是对预训练权重矩阵 W₀（形状 d×k）的更新量 ΔW 进行低秩分解：ΔW = A × B，其中 A 是 d×r 矩阵，B 是 r×k 矩阵，r << min(d,k)（通常 r=8,16,32）。  
- 训练时：W₀ 冻结，只训练 A 和 B，最终前向输出为 h = W₀ x + BA x。初始化时 A 用高斯噪声，B 初始化为 0，保证训练开始时机理等于原模型。  
- 优点：显存节省显著，因为不需要存储原模型优化器状态；训练速度更快；可切换多个 adapter 支持多任务。  
- 局限：低秩假设并非总是成立，对于某些复杂任务可能需要更高 rank 或全参数微调；选择合适的 rank 和作用层（通常只加在 attention 的 Q/K/V/O 上）需要实验。  
- 工程优化：可通过 r 值控制微调强度（α 缩放因子通常设为 r 的倍数）；推理时直接计算 W₀+BA 合并后的矩阵，无额外计算。  
- 常见坑：如果 rank 选取过小导致欠拟合；如果只加在部分层可能效果不足；多 adapter 切换时合并操作要注意精度（浮点误差累计）。

**常见追问与回答：**  
- **追问1：为什么 LoRA 通常只加在 attention 层而不加在 FFN 层？**  
  答：实验证明 attention 层的权重更新对下游任务最敏感，FFN 层采用全量微调或其它 PEFT 方法也可能有效，但 LoRA 加在 attention 上性价比最高。  
- **追问2：LoRA 的 rank 如何选择？**  
  答：一般从 8 开始尝试，小模型（<1B）可用 16，大模型（7B+）常用 8 或 16。用验证集困惑度或下游指标决定，也可以尝试 AdaLoRA 自适应分配 rank。  
- **追问3：LoRA 对比 Adapter 和 Prefix Tuning 有什么优势？**  
  答：Adpater 在推理时会增加层数，引入额外延迟；Prefix Tuning 会占用 input token；LoRA 推理时无额外开销，且可合并权重，兼容性好。

**项目联系：**  
你在做一个 LoRA 微调项目微调 Qwen2.5-7B 用于代码生成时，可以记录：选择 rank=8 只加在 attention 的 Q 和 V 上，训练后保存 adapter 权重约 14MB，然后用合并脚本生成完整模型，用于在线服务。面试时可以说“我通过对比不同 rank 下的 loss 和 HumanEval 分数，发现 rank=8 已足够，验证了低秩假设。”

**记忆关键词：**  
低秩分解、A B 矩阵、冻结预训练、推理合并、rank 选择、adapter 切换

### 题目 2：RAG 的工作流程是什么？与直接微调相比有哪些优势？
**来源依据：**  
参考资料 [4][6][17][18]。

**可信度：**  
高（多个权威来源一致）。

**考察点：**  
RAG 全链路理解、对比微调的选择标准、工程落地优劣。

**面试口头版回答：**  
RAG（Retrieval-Augmented Generation）分为三个核心阶段：索引、检索和生成。第一步构建知识库：将文档切分成 chunk，用 embedding 模型转为向量存入向量数据库。第二步检索：用户查询也转成向量，在数据库中相似度搜索，返回 top-k 最相关 chunk。第三步生成：将检索到的文本拼接到用户 query 中，送入 LLM 生成最终答案。相比微调，RAG 的优势是知识可动态更新（只需更新文档库），避免重新训练；可解释性高（可追溯来源）；适合需要实时信息或低频长尾知识的场景。微调更适合风格迁移、指令遵循等需要改变模型行为的情况。

**深入理解版回答：**  
完整的 RAG 流水线：  
1. 数据准备：文档加载（PDF/网页/数据库）、清洗、切分（按 token 或语义边界）、生成 chunk。  
2. 向量化：用 embedding 模型（如 text-embedding-3-small, bge-large）将每个 chunk 转成向量。  
3. 索引：写入向量数据库（FAISS, Chroma, Milvus），支持近似最近邻搜索。  
4. 检索：对用户 query 做同样 embedding，选择 Top-K（K 通常 3-10）。  
5. 后处理：可加入重排（reranker）模型对候选结果重新打分，过滤低相关内容。  
6. 生成：将 query 和检索内容打包到 prompt 中，送入 LLM。  

RAG 与微调的对比选择：  
- 需要实时/动态知识 → RAG；需要模型学习特定格式或风格 → 微调。  
- 成本：RAG 不改变模型，只需维护向量库；微调需要计算资源和数据标注。  
- 可解释性：RAG 可引用原文，微调是黑盒。  
- 性能上限：高质量数据下微调可能效果更好，但 RAG 更容易落地和迭代。  

常见挑战：  
- 分块策略影响检索精度（chunk 太小丢失上下文，太大容易被 token 限制淹没）。  
- “Lost in the Middle”问题：LLM 倾向于关注开头和结尾，中间信息容易被忽略。  
- 检索失败将导致生成质量下降，需设计 fallback 或重试机制。

**常见追问与回答：**  
- **追问1：RAG 中最容易出现的问题是什么？**  
  答：检索到无关内容，称为“检索噪声”，导致模型被误导；或者检索不到相关内容导致生成幻觉。  
- **追问2：如何提升 RAG 的检索准确率？**  
  答：使用混合检索（dense + sparse），引入重排模型，优化分块策略，对 query 做改写或扩展。  
- **追问3：RAG 和微调可以结合使用吗？**  
  答：可以。先用微调让模型适应领域风格或指令格式，再用 RAG 提供具体知识，称为“微调+RAG”组合。

**项目联系：**  
你搭建了一个针对法律文书的 RAG 系统，使用 LlamaIndex 加载民法典文档，分块大小 512 tokens 且重叠 128，嵌入模型使用 bge-base-zh，检索后加入 bge-reranker 重排，最终用 GLM-4 生成回答。你记录了检索召回率（Recall@5）和最终回答的 F1，并对比了不同分块策略对答案准确率的影响，可面试时讲述优化过程。

**记忆关键词：**  
索引-检索-生成、分块策略、知识更新、可追溯、检索噪声、混合检索

### 题目 3：如何评估 RAG 系统的性能？
**来源依据：**  
参考资料 [6][8][14][18]。

**可信度：**  
高（多个资料都有阐述）。

**考察点：**  
检索阶段指标、生成阶段指标、端到端评估方法。

**面试口头版回答：**  
RAG 评估需分为检索质量和生成质量两部分。检索阶段常用 Recall@K、Precision@K、MRR（平均倒数排名），比如 Recall@5 衡量前 5 个结果是否包含正确答案。生成阶段常用 Answer Exact Match、F1 score、Faithfulness（忠实性，答案是否与检索内容一致）、Relevance（相关性）。还有端到端指标如用户的满意率、人工打分。实际工程中，我们先用标准数据集（如 TriviaQA）测检索召回率，再用人工或 LLM-as-judge 评估生成质量。

**深入理解版回答：**  
- 检索评估：  
  - Recall@K：正确答案出现前 K 个结果中的比例。  
  - Precision@K：前 K 个结果中正确结果占比。  
  - MRR：第一个正确答案位置的倒数均值。  
  - 可以使用开源评测集如 BEIR, MTEB 来评估 embedding 模型。  
- 生成评估：  
  - 自动指标：ROUGE-L（回答与参考答案的 n-gram 匹配），BLEU（机器翻译风格），BERTScore（基于嵌入的相似度）。  
  - 忠实性评估：判断回答是否严格基于检索片段，有无幻觉。可用 NLI 模型或 LLM 打分（如 G-Eval）。  
  - 上下文覆盖率：检查回答中引用的 chunk 是否充分覆盖用户问题所需的信息。  
- 端到端评估：  
  - A/B 测试：对比有/无 RAG 的回答正确性。  
  - 用户满意度调查。  
  - 成本+延迟指标：检索+生成总时间。  

常见坑：自动指标与人类判断存在偏差，需要人工标注 golden set 来做最终验证。

**常见追问与回答：**  
- **追问1：Hit Rate 与 MRR 有什么区别？**  
  答：Hit Rate 只关心是否命中，不关心位置；MRR 考虑第一个正确结果的位置，更精细。  
- **追问2：如果回答中出现幻觉，如何通过评估发现？**  
  答：可以对检索 chunk 和答案做事实核对，如果答案包含 chunk 没有的信息就算幻觉。使用工具如 BERTScore 比较，或者人工检查。  
- **追问3：LLM-as-judge 评估的可靠性如何？**  
  答：有一定可靠性，但可能过拟合评判模型自己的偏好，建议结合多模型投票和人工抽查。

**项目联系：**  
你在项目中使用 RAGAS 框架评估了搭建的法律问答系统，Recall@5 达到 0.85，但 Faithfulness 得分只有 0.6，你分析发现是因为检索到的 chunk 包含冗余信息，重排后 faithfulness 提升到 0.75。面试时可强调用指标指导优化。

**记忆关键词：**  
Recall@K、F1、Faithfulness、LLM-as-judge、自动指标与人工评估、RAGAS

### 题目 4：什么是 QLoRA？它与 LoRA 有何异同？
**来源依据：**  
参考资料 [2][4]。

**可信度：**  
中（来自论坛和博客，但内容与论文一致）。

**考察点：**  
量化微调技术、显存优化、NF4 量化、PEFT 工具链。

**面试口头版回答：**  
QLoRA 是在 LoRA 基础上将预训练模型权重量化到 4bit（使用 NF4 数据类型），再挂载 LoRA adapter 进行训练的技术。它比 LoRA 更进一步节省显存，使得在单张 24GB 显存的显卡上就能微调 33B 甚至 65B 模型。但训练速度不一定更快，因为量化/反量化有额外开销。关键点在于基础模型用低比特存储，LoRA 参数保持高精度（FP16）。QLoRA 还引入了双量化（Double Quantization）和分页优化器来进一步压缩显存占用。

**深入理解版回答：**  
QLoRA（Quantized LoRA）提出三个组件：  
1. NF4 量化：一种适用于正态分布权重的 4bit 数据类型。  
2. 双量化：对量化常数再做一次量化，减少元数据存储。  
3. 分页优化器：利用 CPU 内存辅助 GPU 显存，防止 OOM。  
训练时，前向计算需要将量化权重反量化为 FP16 再进行运算，梯度只作用于 LoRA 参数，基础模型的量化权重梯度不保留，因此显存占用很低。  
与 LoRA 对比：  
- 相同点：都冻结原模型，只训练低秩矩阵。  
- 不同点：QLoRA 的基础模型是量化的，LoRA 基础模型是 FP16/FP32。  
- 显存占用：QLoRA 远小于 LoRA，例如 7B QLoRA 约需 10GB，LoRA 需 16GB。  
- 训练速度：受量化开销影响，QLoRA 可能慢 20%-30%。  
- 效果：研究表明在 4bit 量化情况下，QLoRA 微调效果与全精度 LoRA 差距很小（<1%）。  

适用场景：显存受限，需要训练大模型（如 65B）或同时训练多个 adapter。

**常见追问与回答：**  
- **追问1：QLoRA 训练完成后推理如何部署？**  
  答：可以将 LoRA 权重合并回量化模型，然后保存为 4bit 模型，推理时仍用量化内核加速，也可以导出 FP16 模型但会牺牲节省的显存。  
- **追问2：为什么用 NF4 而不用 Int4？**  
  答：预训练模型权重大致呈正态分布，NF4 是针对正态分布设计的归一化 float 格式，比均匀分布的 Int4 量化误差更小。  
- **追问3：QLoRA 的 adapter 权重能否在多任务间切换？**  
  答：可以，只要基础模型一致，不同的 QLoRA adapter 可以热切换，但切换时需重新加载基础模型的反量化状态。

**项目联系：**  
你可能在个人玩具项目中用 QLoRA 微调 LLaMA-7B，发现原来单卡 RTX 3090 连全量 LoRA 都跑不动（OOM），换成 QLoRA 后成功微调，只用了 11GB。面试时可以强调“我通过对比 LoRA 和 QLoRA 的显存占用和 loss 收敛，得出 QLoRA 是资源受限时的最佳方案”。

**记忆关键词：**  
4bit 量化、NF4、双量化、分页优化、显存节省、速度折衷

### 题目 5：在 RAG 中，如何处理“Lost in the Middle”问题？
**来源依据：**  
参考资料 [8][4][6]。

**可信度：**  
高（广泛认可的研究现象）。

**考察点：**  
上下文窗口限制、信息位置偏见、检索结果排序策略。

**面试口头版回答：**  
“Lost in the Middle”是指 LLM 在处理长文本时，对上下文中间部分的信息关注度明显低于开头和结尾，导致基于中间检索内容的回答质量下降。缓解方法包括：1）将最重要的检索片段放在上下文的最前面或最后面；2）限制上下文长度，只保留最相关的 top-k 内容，不填充过多噪声；3）使用摘要技术，对检索片段进行压缩后再喂给模型；4）采用分层检索，先粗筛再细查，避免一次性输入大量片段。

**深入理解版回答：**  
该现象由研究论文“Lost in the Middle: How Language Models Use Long Contexts”揭示。测试发现对于多文档 QA，模型在识别中间位置的答案时准确率急剧下降。  
工程缓解策略：  
- 排序优化：在将多个片段拼接时，按与 query 的相似度降序排列（最相关放最前），或者采用“最相关放两头”的排列方式。  
- 上下文裁剪：不是 K 越大越好，调参找到最优 K（通常 3-5 即可）。  
- 层次式检索：先检索 chunk 级别，再对 top chunk 进行句子级重排，只保留小片段。  
- 压缩与摘要：用一个小模型对检索结果做段落摘要，减少输入 token 数量。  
- 位置编码优化：使用 ALiBi、RoPE 等能更好处理长位置的位置编码，但本质不改 data 位置偏见。  

实际项目中可尝试对 K 进行网格搜索，观察准确率曲线的峰值位置。

**常见追问与回答：**  
- **追问1：这种问题对于长上下文模型（如 128K 窗口）还有吗？**  
  答：依然存在，实验证明即使 GPT-4-128K 也有“lost in the middle”现象，只是范围扩大了一些。  
- **追问2：有没有不改变 prompt 结构的缓解方法？**  
  答：可以使用“摘要后检索”或“迭代检索+生成”，在生成过程中根据需要再检索。  
- **追问3：是否可以通过 model 微调来缓解？**  
  答：一定程度可以，比如在训练数据中刻意构建“中间有效”的样本，但不能完全消除。

**项目联系：**  
你在搭建企业内部知识库 RAG 时，发现当 K=10 时回答质量反而下降，经过分析发现是因为大量中等相关度的片段被塞入中间位置。于是将 K 降到 5，并将检索得分最高的第一段和最后一段的位置互换，最终准确率提升了 5%。面试时将这个过程拆解，可体现工程调试能力。

**记忆关键词：**  
位置偏见、最相关放两头、上下文裁剪、层次检索、排序优化、调试 K 值

### 文章 1：Efficient LLM Fine-Tuning with LoRA (Medium)
- **链接：** https://medium.com/@raquelhvaz/efficient-llm-fine-tuning-with-lora-e5edb88b64a1  
- **来源依据：** 资料 [15]  
- **核心内容总结：**  
  - LoRA 通过在冻结权重旁添加低秩矩阵实现高效微调，训练参数量极小。  
  - 推理时合并权重，零额外开销。  
  - 以 Qwen2.5-0.5B 为例，展示了代码实现和训练细节。  
  - 适合多 agent 场景：不同 agent 用不同 adapter，共享基座模型。  
  - 对比全参微调与 LoRA 在显存和速度上的差异。  
- **对面试有什么用：**  
  可作为 LoRA 题目回答中的具体例子，包括使用哪个库、rank 设置、训练日志等。  
- **可转化面试题：**  
  1. 请描述你如何用 LoRA 微调一个开源模型？可以举例仓库和关键参数吗？  
  2. 在微调 Qwen2.5-0.5B 时，你怎么设置 rank、alpha、target_modules？

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** LoRA 中为何能减少训练参数而效果接近全参数微调？  
- **A：** LoRA 基于低秩假设：大模型在下游任务中的权重更新信息集中在一个低维子空间中。只需学习两个低秩矩阵（A、B）的乘积即可近似全参数更新。这样可训练参数极少（千分之一），但保留了对任务最关键的变化方向，因此效果接近全参数微调。  
- **一句话记忆法：** 大海捞针？不，LoRA 只用一根针（低秩）就能钓起鱼。  
- **复习标签：** #LoRA #低秩假设 #PEFT  

### 卡片 2
- **Q：** 什么是 ReAct 框架？在 Agent 中有何作用？  
- **A：** ReAct 结合推理（Reasoning）和行动（Action），让 LLM 交替输出思考步骤和工具调用操作。例如“我需要查天气 → 调用 get_weather API → 结果为35度 → 所以建议带伞”。它让 Agent 具备动态规划能力，而非一次生成完整计划。  
- **一句话记忆法：** 先想（Reason）后做（Act），边做边想。  
- **复习标签：** #Agent #ReAct #推理行动  

### 卡片 3
- **Q：** RAG 中检索后为什么要重排（Reranking）？  
- **A：** 初始向量检索返回的 top-k 结果可能包含噪声，例如语义相似但不相关。重排使用更强的交叉编码器（cross-encoder）对每个候选与 query 做细粒度匹配，重新排序，过滤低效结果，提升最终生成质量。  
- **一句话记忆法：** 初筛靠向量，复筛靠交叉。  
- **复习标签：** #RAG #重排 #交叉编码器  

### 卡片 4
- **Q：** 如何评估 LoRA 微调的效果？  
- **A：** 评估分两类：① 下游任务指标（如分类 F1、生成 ROUGE、代码 HumanEval）；② 收敛速度和显存占用（对比全参）。还可以用验证集 loss 观察是否过拟合，并用 benchmark 对比微调前后表现。  
- **一句话记忆法：** 准不准看任务指标，快不快看显存和训练时长。  
- **复习标签：** #LoRA #评测 #微调  

### 卡片 5
- **Q：** 什么是 Agent 中的“工具调用”（Function Calling）？如何确保模型输出结构化？  
- **A：** 工具调用指 LLM 在生成过程中决定调用外部 API 或函数，并输出 JSON 格式的参数。确保结构化：定义函数 schema（名称、描述、参数类型、必填字段），在 prompt 中提供，并引导模型输出标准 JSON。可使用 JSON 模式强制约束。  
- **一句话记忆法：** 给模型一本“字典”，让它照格式填参数。  
- **复习标签：** #Agent #工具调用 #结构化输出

### 4.1 算法题
**题目：** 二叉树的层序遍历（LeetCode 102）  
**思路：** BFS 使用队列，每层循环记录当前队列大小。  
**复杂度：** 时间 O(n)，空间 O(n)（队列最大宽度）。  
**Python 参考代码：**
```python
from collections import deque

def level_order(root):
    if not root:
        return []
    res, queue = [], deque([root])
    while queue:
        level = []
        for _ in range(len(queue)):
            node = queue.popleft()
            level.append(node.val)
            if node.left:
                queue.append(node.left)
            if node.right:
                queue.append(node.right)
        res.append(level)
    return res
```

### 4.2 大模型理论题
**问题：** 请对比 Pre‑LayerNorm 和 Post‑LayerNorm，并说明为什么 LLaMA 选择了 Pre‑LayerNorm（RMSNorm）。  
**参考答案：**  
- Post‑LN：在残差单元的输出之后做 LN，早期 Transformer（原始论文）使用。训练不稳定，特别是深层模型。  
- Pre‑LN：在子层（Attention/FFN）之前做 LN，残差路径更干净，训练更稳定。LLaMA 使用 RMSNorm 代替 LN，因为 RMSNorm 去掉了重心调整，计算更高效且经过实验验证效果不差。  
- 优势：Pre‑LN 允许更大的学习率和更深的层数。  
**可能追问：**  
1. RMSNorm 与 LayerNorm 的区别？答：RMSNorm 只对输入做均方根归一化，不减去均值，计算量减少，且对权重初始化不敏感。  
2. 为什么 LLaMA 还用了 SwiGLU？答：SwiGLU 是 GLU 的变体，引入门控机制，增强非线性，实验表明比 ReLU 效果好。

## 5. 今日 60 秒背诵版

**问题：** 请解释 LoRA 的原理并说明为什么它适合在大模型微调中使用。  

**60 秒口头回答：**  
LoRA，全称 Low-Rank Adaptation，是一种高效的参数微调方法。它的核心原理是：冻结预训练模型的所有权重，同时在原始线性层旁边插入两个小型矩阵 A 和 B，用它们的乘积来近似权重更新。因为大模型在迁移到下游任务时，权重的有效更新空间通常是低秩的，所以我们不需要更新整个大矩阵。这使得可训练参数只有原模型的千分之一甚至万分之一。  
比如微调一个 7B 模型，全参需要更新 7B 参数，而 LoRA 可能只需要几十兆。训练时显存占用大幅降低，单卡就能跑；推理时可以把 A 和 B 合并回原权重，完全没有额外延迟。这非常适合工业场景，比如需要同时部署多个任务，可以共享基础模型，只切换不同的 LoRA adapter。  
在我之前的项目中，我使用 LoRA 微调了 Qwen2.5-7B 用于代码生成，通过对比全参和 LoRA 的效果，发现 LoRA 在 HumanEval 上只比全参低不到 2%，但显存减少 60%。所以 LoRA 是资源有限条件下微调大模型最实用的技术之一。
