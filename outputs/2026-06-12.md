### 题目 1：在部署大模型时，你如何理解并权衡“吞吐率（Throughput）”和“延迟（Latency）”？常用的优化手段有哪些（比如量化、KV Cache、Batching）？

**来源依据：**
- [9] LLM Interview Questions and Answers (2026) | Large Language Models - https://www.interviewbit.com/llm-interview-questions-answers - 技术博客 - 高
- 根据上述资料延伸，需进一步验证 - 这与2026年大模型部署的基础常识一致。

**可信度：**
高

**考察点：**
模型部署、推理优化、量化、Batch、成本控制

**面试口头版回答：**
理解吞吐率和延迟的权衡是部署的核心。吞吐率是单位时间处理请求数，延迟是单个请求的响应时间。优化时，我会从三方面下手：第一，**量化**，比如将模型从 FP16 转成 INT8，显存占用减半，计算速度翻倍，这能直接提升吞吐率。第二，**KV Cache 管理**，使用 PagedAttention 技术（如 vLLM）避免 KV Cache 碎片化，能提升 Batch 中活跃请求的数量，从而提升吞吐。第三，**Continuous Batching**，在 GPU 上不再等待一个 Batch 全部处理完再接收新请求，而是动态地完成一个请求就立刻插入新请求，最大化了 GPU 利用率。通常，我们要在 SLA（延迟容忍度）内最大化吞吐，比如设定 P99 延迟 < 1s，然后通过调整 Batch Size 压测找到最优值。

**深入理解版回答：**
这是一个典型的系统设计权衡问题。在工程上，我们追求的是“在满足延迟约束下的最大吞吐率”。
1.  **核心概念辨析**：
    *   **延迟**: 直接决定用户感知，常以 P50/P99 衡量。瓶颈可能在模型计算（前向传播）、内存搬运（显存带宽瓶颈）、或 I/O（网络）。
    *   **吞吐率**: 决定服务容量和成本，常用 QPS (Queries Per Second) 或 tokens/sec 衡量。瓶颈通常在 GPU 算力和显存带宽。
    *   **Batch Size**: 增大 Batch Size 是提升吞吐最直接的方法，但也线性增加显存占用和单次请求的等待时间（延迟抖动）。
2.  **关键优化手段**：
    *   **量化 (Quantization)**：本质是通过降低权重/激活值的精度来压缩模型。FP16 -> INT8 是常用手段。INT8 不仅可以减少显存占用（模型和计算时的数据搬移更少），还能利用 GPU 上的 Tensor Core 的 INT8 计算单元，获得 2-4 倍的吞吐提升。但要注意量化精度损失，需要对量化后的模型进行评测（比如在特定业务数据上跑一遍看 BLEU、ROUGE 或人工打分）。
    *   **Continuous Batching**: 相比 Static Batching（等待满 Batch 再推理），Continuous Batching 在一个推理阶段内动态调度请求，完成后立即退出。这可以极大提升低并发场景下的吞吐。vLLM、TensorRT-LLM 都实现了此技术。
    *   **KV Cache 优化**: 显存中，KV Cache 占据了绝大部分。通过 PagedAttention（vLLM）将 KV Cache 分页，解决了显存碎片和共享问题。通过 Prefix Caching 可以让共享前序上下文的请求复用 KV Cache，提升首 Token 延迟。
3.  **工程取舍**：没有绝对最优。例如，高精度场景（如医疗）必须使用 FP16；而高并发场景（如智能客服）则优先 INT8。开启连续 Batching 会增加显存碎片（即使有 PagedAttention），需要监控显存利用率。

**常见追问与回答：**
*   **追问1：你如何选择 Batch Size？**
    *   **回答**：通过实验。我会从 `batch_size=1` 开始，逐步增大，观察显存和延迟阈值。对于 LLM，受限于 KV Cache 大小，最大 Batch Size 受模型参数量、输入/输出序列长度和显存共同制约。理想情况是在不发生 OOM 的前提下，压到 P99 延迟满足 SLA 的最大 Batch Size。
*   **追问2：用 INT8 量化后模型效果下降太多怎么办？**
    *   **回答**：可以采用混合精度量化（如 SmoothQuant），对敏感层（如 Attention）保持 FP16，对不敏感层（FFN）用 INT8。或者在量化后做一小步蒸馏微调，恢复部分精度。

**项目联系：**
在我之前做的智能客服 RAG 项目中，我们部署了一个 7B 模型。最初用 FP16 推理，QPS 很低，导致响应延迟超过 3s，用户体验很差。我引入了 vLLM 框架，先通过 INT8 量化将模型显存占用从 14GB 降到 7GB，然后利用 PagedAttention 和 Continuous Batching 将吞吐率提升了 3 倍，使 P99 延迟控制在 1.5s 以内，很好地满足了业务需求。这个优化过程完全可以作为项目中“模型部署”部分的亮点。

### 题目 2：大模型部署时，显存为什么会很快被占满？除了模型权重，还有什么？什么是 KV Cache？它是如何影响部署成本的？

**来源依据：**
- [4] 2024 | 大模型算法工程师相关面试题汇总_牛客网 - https://www.nowcoder.com/feed/main/detail/bd5cb38466e04ec787505e1d866203bd - 面经或社媒 - 中（牛客网数据可信度较高，但需筛选）

**可信度：**
中

**考察点：**
模型部署、KV Cache、显存管理、成本

**面试口头版回答：**
这是一个很常见的部署问题。显存被占满的主要原因有：1. **模型权重**：这是最基础的部分，比如 7B 模型 FP16 需要约 14GB。2. **优化器状态和梯度**：训练时才需要，推理不需要。3. **中间激活值**：推理过程中计算的变量，只在一个 Transformer 层内存活，层数深时很可观，但可以释放。4. **最关键的部分是 KV Cache**：推理自回归生成时，需要缓存注意力层的 Key 和 Value 矩阵，避免重复计算。它的大小 = `Batch_Size * Sequence_Length * (Layers * 2 Dim_KV)`。部署时，KV Cache 占据服务器显存的很大一部分，它直接限制了服务的最大并发数和序列长度。所以，优化 KV Cache 是控制部署成本的关键。

**深入理解版回答：**
这道题考察对推理引擎的理解。
1.  **显存占用的四个核心部分**：
    *   **模型参数**：静态占用，取决于模型大小和精度。
    *   **KV Cache**: 动态且需求巨大。它随着 `batch_size` 和 `seq_len` 增长而线性增长。在大多数生产环境中，KV Cache 占比可高达 50%-80%。
    *   **中间激活 (Intermediate Activations)**: 计算过程中的输出，在反向传播时需要，推理时不需要。好的推理框架（如 vLLM, TensorRT-LLM）会优化中间激活，使其在单层计算完后立即释放，从而节省显存。
    *   **输入/输出**: 模型输入和输出的张量，占比很小。
2.  **KV Cache 与成本的关系**：
    *   **单机并发上限**：假设单卡 A100 显存为 80GB，模型权重占 15GB，剩余 65GB 用于 KV Cache。假设每请求序列长度平均 4K tokens，每个请求占 15MB KV Cache，那么理论上最大并发只有 65GB / 15MB ≈ 4300 个请求。但实际上，你还需要留显存给 Continuous Batching 的中间激活，所以实际可用并发更少。这个“并发上限”直接决定了你的服务器数量和总成本。
3.  **工程上的优化手段**（不仅限于概念）：
    *   **模型架构层面**：采用 Multi-Query Attention (MQA) 或 Grouped-Query Attention (GQA)。它们通过共享 Key/Value heads，将 KV Cache 大小减少数倍。这是 Llama 2/3 和 Qwen2 等主流模型采用的关键部署优化。
    *   **推理框架层面**：使用 vLLM 的 PagedAttention，避免 KV Cache 的内存碎片。使用 Prefix Caching，让共享上下文的请求复用 KV Cache，相当于“白嫖”了计算和显存。
    *   **服务策略层面**：限制 Max Output Length，为 KV Cache 设置上限。

**常见追问与回答：**
*   **追问：能直接说出 MHA, MQA, GQA 的区别吗？**
    *   **回答**：MHA（原始模型）是每个 Head 都有自己的 Key 和 Value。MQA 是所有 Head 共享同一对 Key/Value，显存节省巨大但效果可能受损。GQA（如 Llama 2）是折中方案，将一个 Query Group 共享一对 Key/Value，平衡了速度和效果。
*   **追问：如果我只想调一个参数来降低部署成本，你推荐调什么？**
    *   **回答**：我会优先调“**最大输出长度 (Max Output Length)**”。因为 KV Cache 是序列长度的线性函数。很多场景下，用户不需要很长的回复，将最大输出长度从 2048 降到 1024，KV Cache 用量直接减半，能显著提升并发和降低成本。

**项目联系：**
在我们的 RAG 项目中，我发现模型回复经常很长，且回答引用大段原文。我通过分析用户数据，发现 95% 的有效回复都在 256 tokens 以内。于是我果断将最大输出长度从 512 调整为 256。这个简单的修改直接让我们单卡支持的并发数提升了约 30%，成本显著下降。这证明了对业务细节的理解能直接转化为工程优化。

### 题目 3：请描述一个多 LoRA Adapter 部署方案。当有上千个不同业务线的 LoRA 权重需要同时服务时，你会如何设计？

**来源依据：**
- [4] 2024 | 大模型算法工程师相关面试题汇总_牛客网 - https://www.nowcoder.com/feed/main/detail/bd5cb38466e04ec787505e1d866203bd – 面经或社媒 – 中 (题目中提到LoRA部署相关问题)
- [13] Multi-LoRA deployment — Ray 2.55.1 - https://docs.ray.io/en/latest/serve/llm/user-guides/multi-lora.html - 官方文档 - 高

**可信度：**
高

**考察点：**
LoRA、模型部署、微调、服务编排

**面试口头版回答：**
部署多 LoRA 的核心是“**基座模型共享 + Adapter 按需加载**”。我不会为每个 Adapter 都部署一个完整的模型，那样成本太高。我会采用类似 Ray Serve 或 HuggingFace TGI 的 Multi-LoRA 方案。具体思路是：
1.  **部署**：只部署一份基座模型（比如 Qwen-14B）在 GPU 上。
2.  **存储**：所有 LoRA Adapter（每个通常只有几十 MB）全部存储在共享磁盘或对象存储（如 S3）上。
3.  **调度**：当请求进来时，请求会携带 `lora_adapter_name`。调度器根据这个标签，将请求路由到当前加载了该 Adapter 的 GPU 节点上。
4.  **缓存**：如果该请求需要的 Adapter 不在当前节点上，节点会通过网络从存储中动态加载到显存中。Ray Serve 等框架会用 LRU (Least Recently Used) 策略来管理 Adapter 的缓存，避免显存爆炸。
5.  **推理**：在 GPU 上，推理时会将基座模型的权重冻结，只对 Adapter 部分的参数进行计算。这一步通常非常快，性能损失可以忽略。**内存和调度是关键**，平台会通过负载均衡保证高可用和低延迟。

**深入理解版回答：**
这是 LoRA 微调落地到生产常见的工程问题，尤其适合 SaaS 平台提供个性化模型服务。其核心思想是“计算与存储分离 + 按需加载”。
1.  **架构设计**：
    *   **管理层**: 一个元数据服务，记录每个 LoRA Adapter 的位置（存储路径）、关联的用户/任务元信息、以及当前所在的 GPU 节点。
    *   **推理节点层**: 一个节点池，每个节点上运行一个多 LoRA 推理引擎（如 vLLM、TensorRT-LLM 的多 LoRA 功能）。每个节点维护一个 LRU Adapter Cache。
    *   **请求路由层**: 根据请求头中的 Adapter ID，将请求转发到正确的节点或能快速加载该 Adapter 的节点。
2.  **关键权衡**：
    *   **基座模型*N vs 单基座模型*LoRA Cache**：前者是线性成本扩张，后者是控制成本的选择。后者关键在于 Adapter 的加载延迟，如果在缓存中，加载隐形；如果 miss，需要从存储（可能是 HDD 或网络）加载，延迟可能达到几百毫秒，这也是主要挑战。
    *   **缓存策略与规模**：当有上千个 LoRA 时，必须依赖缓存。可以采用“最近最少使用（LRU）”+“频次（LFU）”结合的策略，或者更高级的“预热”：预测监控系统，对高频次访问的 LoRA 预先加载到节点的缓存中。
3.  **成本与性能**：
    *   **成本**：极大节省，因为你可以服务 1000 个任务而只需要 1 个基座模型，而不是 1000 个模型。
    *   **性能**：主要受限于 Adapter 热切换的速度和 Cache Miss 导致的延迟抖动。通常来说，Adapter 的推理计算量极小，对首次 Token 延迟和总延迟的影响可以忽略不计。但要密切监控 `p99_latency` 的抖动。

**常见追问与回答：**
*   **追问：如果所有业务线都集中在一个 LoRA 上怎么办？**
    *   **回答**：如果集中，可以考虑是否需要合并 Adapter，或者对该 Adapter 进行专门的独立部署（直接合入基座模型输出或者在多 LoRA 引擎上对该 Adapter 的 Cache 进行 pinned（常驻），避免被淘汰）。
*   **追问：如何保证安全性？不让 A 用户拿到 B 用户的 LoRA 权重？**
    *   **回答**：从请求调度层做鉴权。请求路由必须校验 `用户ID` 和 `LoRA_ID` 的绑定关系。管理层面的元数据服务也要做访问控制，只有授权节点才能从存储系统下载对应的权重。采用加密存储或 VPC 内网也是可选的。

**项目联系：**
如果我负责一个 SaaS 客服平台，为不同商家提供定制化回答风格的 LLM 服务，这就是典型的多 LoRA 场景。我会用 Ray Serve 搭配 vLLM 实现一个 Multi-LoRA 推理集群，每个商家只需提供少量的微调数据（客服对话），我就能生成一个 LoRA Adapter，然后通过请求负载（包含 `merchant_id`）动态切换。这个设计能让我以一份模型权重服务数百家商家，极大降低成本和运维复杂度。

### 博客 1: Multi-LoRA deployment — Ray 2.55.1

*   **链接：** https://docs.ray.io/en/latest/serve/llm/user-guides/multi-lora.html

    *   **请求路由机制**：详细描述了当请求到来时的路由逻辑，包括检查本地缓存、负载均衡、动态加载（Cache Miss时）。

    *   **配置要点**：展示具体如何在 Ray Serve 的配置文件中启用 `dynamic_lora_loading` 功能。
*   **对面试有什么用：**
    *   能直接回答“百万级用户下的个性化微调如何部署”这类系统设计题。
    *   提供了官方的、工业级的架构参考，回答会显得非常专业。
    *   理解热切换、负载均衡、缓存淘汰策略在部署中的应用，非常贴合面试中“工程化”的要求。
*   **可转化面试题：**
    *   **问题1：** 字节跳动-Agent开发实习生面试题中提到“LoRA权重是否可以合入原模型？”你觉得在部署时如何处理这种合并？跟 Multi-LoRA 部署相比，优劣是什么？
    *   **问题2：** 当多 LoRA Cache Miss 时，如何尽可能降低服务抖动？假设需要在 200ms 内完成模型加载和首次推理。
*   **对应参考答案：**
    *   **问题1答案：** 可以合入原模型。一种方式是离线合并：将 LoRA 权重与原模型权重进行合并，生成一个全新的模型权重文件，然后按照标准模型来部署。
        *   **优点**：推理性能最优，因为不涉及 Adapter 切换的计算开销。适合单一高流量场景。
        *   **缺点**：模型文件变大（等于原模型大小，Adapters 权重占小头，但整体还是很大），无法动态切换，必须为每个不同任务部署一整个模型，成本高。
        *   **Multi-LoRA 方式**：Adapters 不合并，以插件形式存在。优点是灵活、成本低。劣势是多了动态加载和切换的微小计算开销。
        *   **选择**：如果下游任务固定且流量极大（如核心 App 的问答功能），离线合并+独立部署是更高性能的方案。对于多租户、低频访问的个性化服务（如各种垂直小助手），Multi-LoRA 必须是首选。
    *   **问题2答案：** 有多种策略搭配使用：
        1.  **预热（Pre-warming）**：基于请求的历史统计（频次），在服务启动或闲时将高频 LoRA 预加载到显存 `pinned`（锁定）起来，避免竞争淘汰。
        2.  **异步加载**：Cache Miss 时，主请求不阻塞。服务立即返回一个“暂不可用”或使用兜底回答（如基座模型），后台异步加载 Adapter，等待下个请求使用。
        3.  **分级存储**：请求路由时，尽量将同一 Adapter 的请求路由到同一个节点，增加 Cache 命中。
        4.  **性能调优**：将 LoRA Adapter 存储在 NVMe SSD 或 Ceph RBD 上，并利用 RDMA 或高速网络加载。几百 MB 的加载在毫秒级可以完成。

## 3. 今日必背问答卡片

*   **Q：** 什么是“Continuous Batching”？它如何提升推理吞吐率？
    *   **A：** Continuous Batching（连续批处理）是一种请求调度策略。与Static Batching等待整个批次完成后才处理新请求不同，Continuous Batching允许实时地将新到达的请求插入到正在Batch中等待的GPU计算流中，一旦有请求输出完毕，就立即将其移出并插入新请求。这最大限度地压榨了GPU算力，减少了空闲时间，显著提升了系统在高并发和低并发场景下的吞吐率。
    *   **一句话记忆法：** 就像餐厅厨师不等满一整桌客人的菜再炒，而是谁点好了先炒谁的，一直炒到关门。
    *   **复习标签：** #模型部署 #推理优化 #Batching #吞吐率

*   **Q：** PagedAttention（分页注意力）的核心思想是什么？
    *   **A：** PagedAttention 受虚拟内存的分页管理机制启发，将KV Cache从连续物理内存映射到不连续的Page（虚拟内存块）。这使得KV Cache可以像文件系统一样按需分配和回收，彻底消除了传统实现中的**显存碎片**（无法被充分利用的内存空洞）问题。同时，它也支持跨请求的**KV Cache共享**（如前缀缓存），进一步提升显存利用率和吞吐率。
    *   **一句话记忆法：** 就像把图书馆的书从不连续的架子上搬回连续的格子间，书就能多放几倍，找书也更快。
    *   **复习标签：** #模型部署 #推理优化 #KV_Cache #vLLM

*   **Q：** 在LoRA微调中，为什么通常选择微调 Attention 层的 Q/K/V/O 矩阵，而不是 FFN 层？
    *   **A：** 这是因为Attention层是模型**信息流动和交互**的关键，负责捕捉长距离依赖。微调 Attention 层可以更直接地改变模型如何**关注和整合输入信息**，这对于适配下游任务（如风格、知识、格式）非常有效。而 FFN 层更多负责**知识和模式的存储与变换**。实验表明，微调 Attention 层能以更少的参数获得更好的迁移效果，在效果和泛化性上更优。实践中通常会 **全层** 微调，但 Attention 被普遍认为更重要。
    *   **一句话记忆法：** 微调模型就像调整方向盘（注意力）比调整发动机（知识）更容易改变整车走向。
    *   **复习标签：** #LoRA #模型微调 #PEFT #Attention #FFN

*   **Q：** RAG系统的“忠实度（Faithfulness）”评估的是什么？工程上如何评估？
    *   **A：** 忠实度衡量的是大模型生成的回答是否**严格基于**检索到的上下文，没有添加或编造信息（即没出现幻觉）。工程上常用 LLM-as-judge 方法：准备一个评估 prompt，将【问题】、【检索到的上下文】和【生成的回答】一起输入一个裁判大模型，让它回答“回答是否完全基于上下文？”并给出分数。开源框架 RAGAS 现成支持该指标，它会自动抽取生成回答中的声明，并与上下文做匹配验证。
    *   **一句话记忆法：** 忠实度就是看大模型有没有“照本宣科”，而不是“天马行空”。
    *   **复习标签：** #RAG #模型评测 #忠实度 #Faithfulness #LLMasJudge

*   **Q：** 什么是 Agent 设计中的 ReAct 框架？请简述其工作流。
    *   **A：** ReAct（Reasoning + Acting）是一种将**推理**（Reasoning）和**行动**（Acting）结合起来的Agent设计模式。其核心思想是让LLM交替输出 **“思考（Thought）”**和 **“动作（Action）”** ，Action 会调用外部工具（如搜索、查数据库），然后将工具的返回结果（Observation）作为下一步的输入。如此循环，直到模型认为已收集足够信息并给出最终答案（Finish）。这解决了LLM仅靠自身知识无法与外部世界交互的问题。
    *   **一句话记忆法：** 想一步，干一步，看一步效果，再想下一步，直到任务完成。
    *   **复习标签：** #Agent #ReAct #推理 #工具调用 #LangGraph

## 4. 今日面试专项训练

### 4.1 大模型理论追问题：量化（INT8 vs FP16）对推理的数学依据和性能差异

*   **问题：** 请从理论和计算层面解释，为什么 INT8 推理比 FP16 更快？既然更快，为什么不是所有地方都用 INT8 推理？这种精度降低对 Token 生成的概率分布会产生什么影响？

*   **面试口头版参考答案：**
    INT8 比 FP16 快，原因有两点：
    1.  **计算速度**：GPU 上的 Tensor Core 专门为低精度计算进行了硬件加速。计算一次 FP32 乘加操作的时间，可以计算**两次甚至更多**次 INT8 乘加操作。虽然 GPU 计算一个 FP16 和 INT8 的绝对时间都很短，但在高吞吐率的场景下，INT8 的并发计算优势会被放大。
    2.  **内存带宽**：INT8 的显存占用和计算数据量是 FP16 的一半。在 LLM 推理中，计算瓶颈往往是显存带宽（Memory Bound），因为需要不断读取模型权重和 KV Cache。INT8 减少了从显存到计算核心的数据搬运量，这比单纯提升计算单元速度带来的提升更关键。
    *   **至于为何不是所有地方，是因为量化会产生精度损失**。INT8 的表示范围远小于 FP16，会导致更大概率的数值溢出（Overflow）或精度截断。例如，输入的激活值分布不同，直接使用 INT8 可能会把一些小数值直接舍掉，导致模型输出分布变陡峭，表现为输出质量下降，甚至出现幻视/语义错误。所以需要权衡性能和精度。

*   **深入理解版参考答案：**
    量化（INT8）更快，但其背后不只是简单的单位运算次数多。其性能提升的核心是**缓解了“内存墙”问题**。在 LLM 的 Attention 计算中，计算核心需要等待从 GM（显存）或 SRAM（芯片缓存）加载权重和 KV Cache 数据，这个加载时间往往是瓶颈。INT8 直接将需要加载的数据量减半（权重从 2 bytes 降到 1 bytes），这就等于内存带宽等效地翻倍了。因此，单次计算的延迟可能没变太多（因为实际计算时间很短），但在**Batch 场景**下，多个计算可以同时发出数据加载请求，之前由于带宽不够导致的排队等待时间大幅减少，从而提升了整体吞吐率。

    *   **精度损失的影响**：INT8 的均匀量化假设权重和激活值在整个区间都均匀分布。但实际上，模型中的某些层（如输出层、Attention 的 Softmax 后）的数值分布范围可能非常小或非常不均匀。直接将整个张量映射到 [-128, 127] 会让很多值被归零或噪声，导致模型最终的 logits 分布变得“尖锐”或“平坦”。
        *   **分布尖锐**：模型“不确定性”降低，对非最优 Token 的概率估计失准，导致重复性变高，多样性变差。
        *   **分布平坦**：模型“混乱度”增加，回答可能变得不连贯，甚至出现语法或逻辑错误。
        *   **极端情况**：对于某些特定任务（如代码生成、数学问题），量化后准确率可能会显著下降。
    *   **缓解方法**：先进量化技术（如 SmoothQuant, LLM.int8()， AWQ）并非简单的直接量化。它们会引入 per-channel 或 per-tensor 的 **scale** 和 **zero-point**，并针对 Attention 输出等关键敏感区域使用更高精度（FP16）混合量化。所以，合格的 INT8 部署会通过一定量的校准数据（calibration data）来寻找最佳的量化参数，最大限度保留精度。

*   **可能追问与简短回答：**
    *   **追问1：** 除了 INT8，你了解更低比特量化吗（如 INT4）？部署中会考虑吗？
        *   **回答：** 了解，比如 GPTQ 和 AWQ 都支持 4-bit 量化。主要用在**超低显存和带宽场景**。但精度损失会更大，推理时可能需要借助 **Group Quantization**（分块量化）来提升精度，这会增加一点点计算量。一般用于 13B 等大模型在显存有限的消费级显卡上部署。
    *   **追问2：** 量化后模型跑出乱码（gibberish）了，你作为工程师，第一步会做什么调试？
        *   **回答：** 第一步是排查是否是**量化溢出**。我首先会检查模型每个层的权重和激活值的分布。如果某个层的值范围特别大/小，我会对该层单独做 per-channel 量化，或者增加 `scale` 的敏感度。如果不奏效，我可能将该层的 Attention 层或 Output 层单独回退到 FP16（混合精度量化）。通过二分法定位是哪一层的信息流失了。
