### 题目 1：在 Agent 系统中，你如何设计工具调用失败后的反馈与重试策略？

**来源依据：**
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 需进一步验证（GitHub 个人项目，内容来自网络收集）
- AI智能体与大模型应用开发面试题库 - https://github.com/didilili/ai-agents-from-zero/blob/main/AI%E6%99%BA%E8%83%BD%E4%BD%93%E4%B8%8E%E5%A4%A7%E6%A8%A1%E5%9E%8B%E5%BA%94%E7%94%A8%E5%BC%80%E5%8F%91%E9%9D%A2%E8%AF%95%E9%A2%98%E5%BA%93.md - 面经或社媒 - 需进一步验证

**可信度：** 中（面经汇编，没有官方来源，但题目方向普遍）

**考察点：** 工具调用、错误恢复、降级策略、幂等设计

**面试口头版回答：**
我会从三个层面设计工具调用失败后的策略。第一是**重试机制**：对于网络抖动等临时错误，自动重试 1-2 次，但需要设置指数退避和最大超时时间。第二是**结果验证**：收到工具返回值后，用 LLM 判断结果是否合法（比如 API 返回 500），不合法则触发重试或降级。第三是**降级与补偿**：如果多次重试依然失败，Agent 需要用自然语言告知用户“当前工具不可用，建议手动操作”。对于幂等性要求高的场景（如下单），我会在调用前生成唯一请求 ID，防止重复提交。我还在项目中用 LangGraph 的**条件边**实现了一个失败节点路由，成功则继续，失败则进入“重试/告知用户”的循环。

**深入理解版回答：**
工具调用失败是 Agent 生产环境中最高频的故障之一，设计策略需要考虑错误分类、状态一致性和用户体验。首先，**错误分类**：分可自动恢复（如超时、限流）和不可恢复（如权限不足、参数错误）。对于可恢复错误，实现**指数退避重试**，每次重试间隔逐渐增加，并设置最大重试次数（通常 2-3 次）。重试时需要使用**幂等性 Token**（如 UUID），避免重复执行副作用操作（如发送邮件、扣款）。其次，**结果验证**：不是所有工具返回的 HTTP 200 都是正确结果，所以需要在 Agent 的“观察”阶段增加一个校验步骤——让 LLM 检查返回内容是否符合预期格式，如果不符合，则标记为失败。第三，**降级策略**：当重试耗尽，Agent 应进入“告知 + 补偿”模式。例如，在工单系统中，如果自动创建工单失败，可以降级为“记录日志 + 通知管理员 + 给用户显示抱歉信息”。对于有状态的 Agent（如下单流程），失败时需**回滚或标记状态**：使用工作流框架（如 LangGraph）的 State 管理，在状态中写入失败标记，后续步骤根据该标记选择补偿分支（如取消订单、释放库存）。最后，**监控与告警**：上报失败率、失败类型、重试次数，用于优化工具稳定性和 prompt 设计。一个常见陷阱是忘记**超时控制**：如果工具是第三方 API，必须设置客户端超时，防止 Agent 卡死。

**常见追问与回答：**
- **追问：** 如果一个工具调用被成功执行但 LLM 认为失败（误判），你怎么处理？
  - **回答：** 我会在 prompt 中明确工具返回的格式规范和成功标准，同时引入“可信度”打分机制。如果 LLM 经常误判，考虑取消 LLM 验证，改为规则校验（如检查 JSON 字段是否存在）。
- **追问：** 幂等性在非 RESTful 工具中如何实现？
  - **回答：** 对于任意工具，可以在调用前生成一个全局唯一的请求 ID，并在工具侧（如数据库）做去重。对不支持 ID 的工具，可以记录调用参数哈希，在重试前查询历史记录。

**项目联系：**
我在做一个智能客服 Agent 项目时，需要调用查询库存、创建工单等多个 API。我设计了统一的 ToolExecutor 类，封装了重试（retry 插件）、幂等性（request_id）、超时降级, 并在 LangGraph 中增加了一个 `error_handler` 节点，将失败信息写入 State，让后续的 Planner 感知并调整后续计划。

**记忆关键词：** 重试退避、幂等性、结果验证、降级补偿、状态标记、LangGraph 条件边

### 题目 2：Agent 的长期记忆如何存储？如果历史记录量非常大，怎么优化查询效率？

**来源依据：**
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 需进一步验证
- AI智能体与大模型应用开发面试题库 - https://github.com/didilili/ai-agents-from-zero/blob/main/AI%E6%99%BA%E8%83%BD%E4%BD%93%E4%B8%8E%E5%A4%A7%E6%A8%A1%E5%9E%8B%E5%BA%94%E7%94%A8%E5%BC%80%E5%8F%91%E9%9D%A2%E8%AF%95%E9%A2%98%E5%BA%93.md - 面经或社媒 - 需进一步验证

**可信度：** 中

**考察点：** Memory 设计、向量检索、混合检索、摘要压缩、上下文管理

**面试口头版回答：**
我倾向于将 Agent 记忆分为三层：**工作记忆**（当前会话的短期上下文，直接放入 LLM 上下文窗口），**长期记忆**（跨 session 的用户偏好、历史结论），**结构化记忆**（用户画像、标签）。对于长期记忆，我一般使用**向量数据库 + 摘要文本**的组合存储。每次对话结束后，将关键信息（用户意图、工具调用结果、最终回答）编码成向量存入向量库，同时生成一段自然语言摘要存入单独的文档。当新对话需要检索时，先用最近几轮的摘要做快速语义匹配，再精排检索历史。如果历史量巨大（百万级），我会做**分层存储**：热层（最近 30 天对话）用向量库 + 快速索引，冷层（超过 30 天）用更粗粒度的摘要向量或直接存文本压缩包。查询时先搜热层，未命中再搜冷层，并限制 top-k 数量。

**深入理解版回答：**
Agent 长期记忆的本质是**持久化的、可查询的上下文**。常见方案包括：纯文本存 Redis（适合小规模）、向量数据库（如 Chroma、FAISS、Pinecone）、以及关系型数据库（存结构化属性）。工程上要解决三个问题：**存储什么**、**如何索引**、**如何检索**。

存储内容上，最优实践是存储**事件三元组**：(时间、类型、摘要)，并额外存储向量。例如一个用户询问“帮我查上周订单”，存为 `{time: "2026-07-01 10:00", type: "query_orders", summary: "用户查询2026年6月最后一周的订单状态", embedding: [...], metadata: {user_id: "u123"}}`。

索引优化方面，不能只靠向量搜索。当历史量很大时，需要加入**时间窗口过滤**（只搜最近 N 天）、**实体提取过滤**（根据用户 ID 先过滤）、**分类标签过滤**（如只查询“偏好类”记忆）。实际工程中常用的混合检索策略是：先做 metadata 过滤（时间、用户 ID）缩小候选集，再做向量相似度排序，最后用 cross-encoder 重排 top-5。

另一个优化点是**记忆压缩**：当某段历史被多次检索但信息冗余时，LLM 可以定期将多条相似记忆归纳成一条，减少存储量。例如每周生成一次用户行为总结，并删除该周内重复的细粒度记录。

性能瓶颈上，如果向量库规模超过 100 万条，需要考虑**聚类或分片**。比如按用户 ID 哈希分片，让每个用户历史只在单个分片中查询，避免全库扫描。此外，读 QPS 高时，可以为热数据开启缓存（如 Redis 缓存最近 100 条记忆的向量和摘要）。

**常见追问与回答：**
- **追问：** 如果用户突然改变意图，但历史记忆中有大量相反的信息，Agent 会混淆吗？
  - **回答：** 需要在 prompt 中添加指令，让 LLM 以当前对话中的最新意图为准，历史记忆仅作参考。同时，可以为记忆打上“置信度”标签，旧记忆置信度低。
- **追问：** 在 LangChain 或 LangGraph 中，如何实现多用户记忆隔离？
  - **回答：** 使用 session_id 作为命名空间。在存入向量库时，将 session_id 作为 metadata；查询时强制过滤该 session_id。对于分布式场景，使用 Redis 的 key 前缀隔离。

**项目联系：**
在之前的 Agent 项目中，我使用 LangGraph 的持久化功能（checkpointer）配合自定义的 MemoryStore 接口，后端使用 ChromaDB 存储向量 + MongoDB 存储原始记忆文本。每个用户有一个独立的 collection，查询时先按用户 ID 过滤，再按时间降序取最近 50 条，然后做语义检索。这样可以保证检索效率，同时支持跨 session 的场景恢复。

**记忆关键词：** 三层记忆、向量库、摘要压缩、metadata 过滤、时间窗口、冷热分层、session 隔离

### 题目 3：LangGraph 中如何管理 Agent 的状态？特别是多轮对话中的状态隔离与持久化？

**来源依据：**
- LinkedIn AI Engineer 面经 - https://www.linkedin.com/posts/bilal-momin_generativeai-llm-langgraph-activity-7353048174648840192-C0CR - 面经或社媒 - 需进一步验证（个人 LinkedIn 帖子）
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 需进一步验证

**可信度：** 中（LinkedIn 内容需验证，面经汇总可信度一般）

**考察点：** LangGraph 状态设计、State 数据结构、节点通信、多用户隔离、持久化 checkpoint

**面试口头版回答：**
在 LangGraph 中，Agent 的状态由用户定义的 State 对象管理。State 是一个 TypedDict，包含 messages、current_step、memory 等字段。每个节点可以读写 State，节点之间的数据传递通过 State 完成。对于多轮对话中的状态隔离，我使用**会话级 checkpoint**。LangGraph 的 `MemorySaver` 或自定义 checkpointer 可以根据 `thread_id` 区分不同会话。每次节点执行后，框架自动保存 State 快照。当用户发出新对话时，用之前的 thread_id 恢复整个 State，保证上下文连续性。持久化方面，我倾向于将 checkpoint 存入数据库（如 SQLite 或 PostgreSQL），支持历史回溯和故障恢复。对于高并发场景，我会为每个 thread_id 加锁，避免并发读写冲突。

**深入理解版回答：**
LangGraph 本质是一个有向图（Graph），State 是图执行时的全局数据总线。在工程上，你需要定义 State 结构，例如：
```python
class AgentState(TypedDict):
    messages: list  # 消息历史
    agent_scratchpad: str  # 中间思考过程
    current_node: str  # 当前所在节点
    user_session_id: str
```
节点函数接受 State，返回 State 的部分更新。每次节点执行后，框架调用 checkpointer 保存 State。

**多用户隔离的核心是 thread_id**。LangGraph 的 `CompiledGraph` 在运行时接受 `config` 参数，其中包含 `configurable` 字典，`thread_id` 作为标识。不同 thread_id 的 State 完全隔离。持久化时，checkpointer 需要支持按 thread_id 存储和恢复。推荐使用 `SqliteSaver`（开发环境）或 `PostgresSaver`（生产环境）。

**优化点：**
- **增量保存**：只保存 State 的变化部分，节省存储和 I/O。
- **过期清除**：定期清理超过 7 天未活跃的 thread 的 checkpoint。
- **状态回滚**：用户可以“回到上一步”，利用 checkpoint 的版本号实现。
- **状态压缩**：对于超长对话，可以在保留关键信息的前提下压缩 messages，例如用 LLM 生成历史摘要，替换旧的消息列表。

**常见追问与回答：**
- **追问：** 如果 Agent 被多个集群实例处理同一个 thread，如何避免状态冲突？
  - **回答：** 使用分布式锁（如 Redis 锁）确保同一时刻只有一个实例能修改该 thread 的 State。或者将同一 thread 的所有请求路由到固定实例（Sticky Session）。
- **追问：** State 中如果包含大段文本（如检索结果），会影响序列化和反序列化性能，怎么优化？
  - **回答：** 可以将检索结果存储在外部存储（比如 Redis 临时 key），State 中只存引用 key。需要使用时再从外部加载。

**项目联系：**
我曾在 LangGraph 中设计过 IT 工单处理 Agent。State 中除了消息列表，还包含 `order_status`、`pending_actions`、`error_count`。使用 `SqliteSaver` 持久化到本地文件，并用 thread_id 区分每个工单。当 Agent 重启后，可以恢复所有未完成的工单状态，继续处理。多用户隔离通过不同 thread_id 实现，生产环境中则会迁移到 PostgreSQL 以支持并发读取。

**记忆关键词：** State TypedDict、thread_id、checkpointer、持久化、增量保存、锁、外部引用

**可转化面试题：**
1. 比较 LangChain 和 LangGraph 的适用场景，为什么越来越多的 Agent 项目转向 LangGraph？
2. 在设计 Agent 系统时，你会如何权衡使用 context window 还是向量数据库来存储长期记忆？

**对应参考答案：**

**问题 1：LangChain 和 LangGraph 的适用场景对比**
**回答：** LangChain 提供了链式调用和简单工具组合，适合固定流程或者顺序执行的场景，比如一个先检索再生成的 RAG 流程。LangGraph 则是图式编排，支持条件路由、循环、分支，更适合 Agent 这种需要动态决策、来回推理的任务。如果我的 Agent 需要根据当前状态决定下一步调用什么工具，甚至需要自我反思和重新规划，我会选 LangGraph。因为它可以显式建模循环和状态转移，而 LangChain 的链式结构在处理复杂逻辑时很容易变成混乱的条件判断。实际项目中，我们先用 LangChain 快速验证 RAG 链，然后在需要 Agent 能力时迁移到 LangGraph。

**问题 2：context window vs 向量数据库存储长期记忆**
**回答：** 我会根据记忆的规模和时效性做选择。如果记忆量很小（< 几轮对话），直接放入 context window 即可，简单高效。但一旦跨 session 或者记忆量很大，context window 会溢出或导致“lost in the middle”问题。我倾向于用向量数据库存储高价值记忆（关键结论、用户偏好），检索时只取 top-5 放入上下文。这样既控制了 token 消耗，又保证了信息密度。不过有一个常见的坑：如果检索到的记忆之间互相矛盾或者与当前意图不符，需要在 prompt 中教导 LLM 以最新信息为准。

### 精读 2：Efficient LLM Fine-Tuning with LoRA（Medium 技术博客）

**文章标题：** Efficient LLM Fine-Tuning with LoRA
**链接：** https://medium.com/@raquelhvaz/efficient-llm-fine-tuning-with-lora-e5edb88b64a1
**来源依据：** Medium 博客文章 - 技术博客 - 可信度中（个人博客，但内容基于论文）
**文章类型：** 理论或论文

**核心内容总结：**
- LoRA 通过注入低秩矩阵 A 和 B 冻结原权重，只训练极小部分参数。
- 训练速度更快、显存占用更少，推理时可将 adapter 合并回原模型，零额外延迟。
- 详细展示了用 LoraConfig、TrainingArguments、SFTTrainer 进行实践。
- 与多 Agent 场景结合：可以为每个 Agent 微调一个小模型（如 Qwen、DeepSeek）做专用任务。
- 关键超参数：rank（通常 8-16）、alpha、target_modules。

**对面试有什么用：**
- 虽然 LoRA 不是 Agent 专属，但面试中经常结合 Agent 提问：“你会如何让每个 Agent 保持低成本？”——答案就是用 LoRA 微调小模型替代大模型。
- 可以回答关于模型部署和 adapter 切换的问题，比如多任务部署时如何通过切换 LoRA adapter 实现不同能力。
- 提供了 lora rank 和 alpha 的具体经验值，可以在回答中提及增加可信度。

**可转化面试题：**
1. 在 Agent 系统中，你会如何使用 LoRA 来降低推理成本？微调时 rank 如何选择？
2. 如果已经有一个微调好的 LoRA adapter，推理时如何与基础模型合并？合并后还能切换其他 adapter 吗？

**对应参考答案：**

**问题 1：Agent 系统中使用 LoRA 降低推理成本**
**回答：** 我会将复杂任务拆分为多个专用子任务，例如一个 Agent 专门做意图识别，另一个做实体提取，再一个做工具调用。这些子任务可以用小模型（如 Qwen-1.5B）通过 LoRA 微调得到专用模型，推理成本远低于每次都调用 GPT-4。rank 的选择通常从 8 开始试，如果任务简单可以降到 4，如果任务复杂提到 16。alpha 一般设为 rank 的两倍。要注意 target_modules 的选择，一般选择 q_proj 和 v_proj，也可以加上 o_proj 和 up_proj 来提升效果。

**问题 2：LoRA adapter 合并与切换**
**回答：** 推理时有两种模式：第一种是**不合并**，每次推理时额外计算低秩矩阵的输出与原始权重相加。这样可以在同一基础模型上动态切换多个 adapter（例如根据用户意图加载不同的 adapter）。第二种是**合并**，将 adapter 权重合并到原始线性层中，此时推理速度与原始模型完全一致。但合并后就不能在运行时切换 adapter 了，需要重新加载。我通常在生产中采用不合并部署，用 vLLM 或 TGI 的 LoRA 加载能力，实现多个 adapter 的低成本切换。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** Agent 中 tool calling（工具调用）的典型流程是怎样的？
- **A：** 首先 LLM 根据用户输入和目标决定是否调用工具，输出结构化 JSON（如 function_call），包含工具名和参数。然后 Agent 调用对应工具 API，获取结果。最后将结果作为观察反馈给 LLM，LLM 据此生成最终回答或继续下一轮调用。流程核心是“决定-调用-观察”循环。
- **一句话记忆法：** LLM 决定用什么工具，Agent 去调用，观察结果后再决定下一步。
- **复习标签：** #Agent #工具调用 #tool calling

### 卡片 2
- **Q：** ReAct 范式中的“思考-行动-观察”循环具体在代码中如何实现？
- **A：** 在代码中，将 LLM 的生成嵌入一个 while 循环。每次 LLM 输出解析为 thought（思考）和 action（行动，包含工具调用），Agent 执行行动并得到 observation（观察），然后将 thought + action + observation 追加到消息历史，再次输入 LLM。循环直到 LLM 输出 Finish 或达到最大步数。
- **一句话记忆法：** 先让模型想，再让它做，看了结果再想，直到它说做完了。
- **复习标签：** #Agent #ReAct #循环

### 卡片 3
- **Q：** Agent 记忆系统通常如何分类？
- **A：** 按作用时长分三类：**短期记忆**（当前会话的 messages，存于上下文窗口）；**长期记忆**（跨 session 的知识，存于向量库或数据库）；**工作记忆**（当前正在处理的中间结果，如工具调用返回值，随节点流转）。另一种分类是：episodic memory（事件记录）、semantic memory（语义知识）、procedural memory（操作流程）。
- **一句话记忆法：** 记说话的叫短期，记历史的叫长期，手里正在用的叫工作。
- **复习标签：** #Agent #记忆系统 #分类

### 卡片 4
- **Q：** 在 LangGraph 中如何实现条件分支（conditional edges）？
- **A：** 在定义 graph 时，使用 `add_conditional_edges` 方法，传入源节点名称和一个路由函数。路由函数接收当前 State，返回下一个节点名称的字符串。例如，根据 `State["next_step"]` 的值决定是去 call_tool 还是 finish。条件分支是 LangGraph 支持复杂决策流的核心。
- **一句话记忆法：** 加一条特殊边，写一个函数看 State 决定走哪条路。
- **复习标签：** #LangGraph #状态管理 #条件分支

### 卡片 5
- **Q：** 如何设计 Agent 的降级策略？
- **A：** 降级策略分三层：**服务层**——如果 LLM API 超时，自动切换到备用模型（如从 GPT-4 降级到本地小模型）；**工具层**——如果某个工具不可用，Agent 应告知用户并跳过该步骤，或用替代方案；**体验层**——当所有依赖失败时，返回预设的兜底回复，并记录错误日志供人工跟进。降级逻辑应写在工作流图中，用条件分支判断失败标志。
- **一句话记忆法：** 顶不住就降一层：模型换小的，工具跳过，最后给用户道歉记录日志。
- **复习标签：** #Agent #错误恢复 #降级

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 在 Agent 的 Planner/Executor 架构中，如果执行器在执行某个步骤时成功，但返回的结果与 Planner 的预期不一致（例如查询的库存数量为负数，但 API 返回了成功状态码），Planner 应该如何感知并处理这种“隐式失败”？与显式失败（网络错误）相比，处理方式有什么不同？

**面试口头版参考答案：**
首先，我会在被执行步骤的观察结果节点中增加一个**结果验证器**。验证器可以用 LLM 或规则来判断输出是否合理，比如检查数值范围、是否包含异常关键词。如果发现隐式失败，我会在 State 中标记该步骤为 `"failed_implicit"`，并附上错误原因。然后，通过 LangGraph 的条件边，让后续节点判断：如果 `step_status == "failed_implicit"`，就走重试或重新规划分支。与显式失败不同的是，隐式失败不允许自动重试，因为它可能是逻辑错误，重试可能会重复错误。正确的做法是让 Planner 重新生成后续步骤或直接进入人工兜底流程。

**深入理解版参考答案：**
隐式失败是 Agent 生产环境中最隐蔽的陷阱。显式失败有明确的异常抛出（如 HTTP 500 或超时），可以编写标准的重试和降级逻辑。而隐式失败发生在工具成功返回结果，但结果在语义上不合理，比如返回了负数、空字符串、或者违反业务规则的数据。处理隐式失败需要引入**结果验证层级**：

1. **规则验证**：直接在代码中写校验逻辑，比如检查是否为空、数值范围、枚举值是否合法。
2. **LLM 验证**：将工具返回结果和预期描述发送给 LLM，让 LLM 判断是否符合预期。但这会增加延迟和成本，所以只用于关键步骤。
3. **对比验证**：如果有历史正常数据，可以用 embedding 相似度检测当前结果是否异常。

检测到隐式失败后，不能简单重试（因为参数可能正确但业务逻辑不对），需要让**Planner 重新决策**。具体做法：在 Agent State 中维护一个 `step_status` 字典，记录每一步的执行状态。Planner 在执行计划时，会检查前一步的状态。如果前一步是 `success` 但结果在验证后被标记为 `potentially_invalid`，Planner 可以选择：
- 尝试用不同参数重新调用该工具（如更换查询维度）。
- 跳过该步骤，用默认值替代（需明确告知用户）。
- 终止当前计划，进入“请求人工介入”分支。

工程实现时，我会在 LangGraph 的 State 中增加 `trust_map` 字段，记录每个工具调用的信任级别。如果某个工具的隐式失败率高于阈值，后续调用时会自动降级为“仅使用规则验证”，并提升重试触发条件。此外，所有隐式失败实例都应该记录到日志和评测数据集，用于改进 prompt 或工具描述。

**2 个可能追问与简短回答：**

- **追问：** 如果隐式失败频繁发生，LLM 验证的额外成本太高，你怎么优化？
  - **回答：** 可以实施**采样验证**策略：对每个工具调用，随机抽取 10% 验证，其余全部放行。或者仅在工具返回的数据量较小且字段简单时做 LLM 验证，对于大批量数据只做规则验证。还可以用**小模型**做验证，比如用 DistilBERT 分类器判断结果是否异常。

- **追问：** Planner 重新规划时，如何保证不陷入无限重试循环？
  - **回答：** 在 State 中记录 `replan_count` 和 `max_retries`（比如 3）。如果重新规划次数超限，直接进入“人工接管”分支。同时在每次重新规划时，让 LLM 输出“尝试的不同方案”，不能重复相同参数。此外，可以用**手动阈值**——如果连续两次都隐式失败，就放弃自动处理。
