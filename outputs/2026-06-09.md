### 题目 1：请详细说说你设计过的 Agent 系统中，Tool Calling 的流程和错误处理策略。

**来源依据：**  
- AgentGuide/docs/04-interview/12-company-interview-cases.md - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 来源类型：技术博客/开源指南 - 可信度：中（需进一步验证，但内容系统化）  
- 京东物流实习面经（Agent项目深度拷问） - https://mj.mianlingai.com/tag/%E5%A4%A7%E6%A8%A1%E5%9E%8B - 来源类型：面经或社媒 - 可信度：低（需进一步验证）

**可信度：** 中（综合两份来源，逻辑自洽）

**考察点：**  
Agent 工程、Tool Calling 设计、错误恢复、超时与重试

**面试口头版回答：**  
我设计的 Agent 系统中，Tool Calling 分为三步：第一步，LLM 根据用户意图生成工具调用请求（function call），通常是一个 JSON 格式的 {“function”: “search_order”, “parameters”: {“order_id”: “123”}}。第二步，执行引擎校验参数、调用真实 API，并捕获结果或错误。第三步，将结果作为新消息回传给 LLM 继续推理。  
错误处理我用了三层兜底：第一层，给每个工具设置超时阈值（例如 5 秒），超时后返回“工具超时”信号；第二层，LLM 收到错误后可以自主重试，但限制最大重试次数（比如 3 次），防止死循环；第三层，如果多次失败，Agent 会切换到人工介入（human-in-the-loop）或返回兜底回复“该功能暂时不可用”。在项目中，我曾遇到一个天气查询工具偶尔返回空数据，我增加了“重试 1 次 + 换源”的 fallback 逻辑，显著提升了成功率。

**深入理解版回答：**  
Tool Calling 是 Agent 与外部世界交互的桥梁，工程实现的核心挑战在于：准确性（LLM 生成正确的参数）、可靠性（外部服务可能出错）、安全性（防止注入或权限越界）。  
典型流程：  
1. **LLM 工具选择与参数生成**：通过 system prompt 描述工具 schema（名称、描述、参数类型），LLM 输出 tool_call 请求。实践中需要用严格的 JSON Schema 约束，并用输出解析器校验。  
2. **工具执行**：执行引擎维护一个注册表，根据工具名称路由到具体实现。需要做参数类型转换（如日期字符串→datetime）、权限校验（该用户是否有权调用）、速率限制。  
3. **错误处理**：  
   - **超时**：使用 asyncio.wait_for 或线程池超时，避免一个慢工具阻塞整个 Agent。  
   - **重试策略**：区分临时错误（网络）和永久错误（参数错误）。临时错误可重试 3 次，指数退避；永久错误直接返回错误信息给 LLM。  
   - **循环检测**：记录工具调用历史，如果连续 3 次相同的工具/参数组合，强制打断并输出“检测到重复错误，请修改请求”。  
   - **回退**：降级到简单规则（如基于关键词的回复）或人工介入。  
4. **结果返回**：工具输出可能需要后处理（如截断长文本、格式化），然后作为 assistant 消息的 tool_result 字段回送入 LLM 上下文。  
**局限**：LLM 有时会虚构不存在的工具（幻觉），需要增加白名单校验；参数生成可能缺失必填字段，需要 prompt 约束 + 校验层补全。  
**评估**：可通过追踪 tool_call 的成功率、重试次数、平均耗时、用户满意度来持续优化。

**常见追问与回答：**  
- **问：** 如果 LLM 连续调用同一个工具并不断失败，你怎么避免死循环？  
  **答：** 我在执行引擎维护一个“失败计数器”，每个工具/参数组合失败 3 次后，我会在 prompt 中加入“该工具暂时不可用”的提示，并屏蔽该工具 1 分钟。  
- **问：** 如何确保工具调用的安全性，比如防止注入？  
  **答：** 对用户输入的参数进行净化，比如 SQL 查询工具中只允许预编译参数；文件操作用沙箱目录限制路径。另外所有工具执行都在隔离进程/容器中运行。  
- **问：** 你提到 human-in-the-loop，具体怎么实现？  
  **答：** 当 Agent 不确定或多次失败时，它会生成一个“需要人工确认”的暂停信号，由后端推送到客户端，用户通过 UI 输入或确认后才继续执行。

**项目联系：**  
我在实习项目中实现了一个电商客服 Agent，包含查订单、退款、物流查询等 10 个工具。我负责设计了工具注册表、超时重试和错误兜底逻辑。例如，当物流查询工具返回“未找到物流信息”时，Agent 会自动转调“订单状态查询”工具获取最新状态，避免直接向用户说不知道。

**记忆关键词：**  
工具 schema、超时+重试、循环检测、human-in-the-loop、降级策略、安全性

### 题目 2：你在 Agent 中如何实现长期记忆（Long-term Memory）？如果历史记录量非常大，怎么优化查询效率？

**来源依据：**  
- 京东科技AI后端研发一面（日程实习） - https://mj.mianlingai.com/tag/%E5%A4%A7%E6%A8%A1%E5%9E%8B - 来源类型：面经或社媒 - 可信度：低（需进一步验证）  
- AgentGuide 面试题库 - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 来源类型：技术博客/开源指南 - 可信度：中（需进一步验证）

**可信度：** 中

**考察点：**  
Memory 设计、向量数据库、上下文压缩、性能优化

**面试口头版回答：**  
我的 Agent 长期记忆采用分层存储：工作记忆（短期）放在 Redis 中保存最近几轮对话，长期记忆则通过向量数据库（如 Chroma）存储关键信息。每次用户提问时，先检索历史中最相关的片段加入上下文。当历史记录量非常大时，我会做三点优化：第一，只保留高价值记忆，比如用户偏好、已完成的订单编号，而非全量日志；第二，对记忆做自动摘要，用 LLM 将多轮对话压缩成一条摘要，减少 token 占用；第三，设置检索的 top-k 限制（比如 5 条），并加上时间衰减权重，让近期记忆有更高优先级。这样既能保持个性化的交互，又不会撑爆上下文窗口。

**深入理解版回答：**  
Agent 的长期记忆核心解决的是“跨会话记忆”问题。工程实现通常有两种方式：  
1. **显式存储**：将重要信息（如用户名称、偏好、关键事实）以键值对形式存储到数据库（如 Redis、PostgreSQL），每次对话开始时加载。优点：精确、低成本；缺点：需要预定义 schema，灵活性差。  
2. **语义检索**：将历史对话片段或摘要向量化后存入向量数据库（如 Milvus、Weaviate、Chroma），运行时用当前问题检索最相关的记忆。优点：灵活，可以召回意外相关的历史；缺点：延迟较高，存在召回噪声。  
实践中常混合使用：结构化的元数据（如用户 ID、会话 ID）用键值存储，自由文本记忆用向量检索。  
**大规模优化策略**：  
- **摘要压缩**：每 N 轮对话后，让 LLM 生成一段代表本次会话的摘要，替换原始对话内容。摘要可以分层（小时摘要→天摘要→周摘要），类似 RAG 中的层次索引。  
- **记忆压缩**：通过 MCP（Model Context Protocol）或类似技术，将记忆打包成压缩 token 或特殊占位符，减少上下文占用。  
- **检索优化**：使用混合检索（向量 + BM25）+ 重排序，提高召回准确性；对记忆分桶（按主题或时间），减少搜索空间。  
- **过期清理**：设置 TTL（time-to-live），例如 30 天未访问的记忆自动归档或删除。  
**常见坑**：检索到的记忆可能包含过时或错误信息，导致 Agent 产生幻觉。解决方案：对每条记忆记录置信度分数，低置信度的不返回；或者要求 Agent 在引用记忆时要标注来源时间。  
**评估**：可用记忆力准确率（Agent 能否回忆起之前用户说过的关键信息）和上下文压缩比（记忆后 token 数 / 原 token 数）来评估。

**常见追问与回答：**  
- **问：** 记忆的摘要会丢失细节，你怎么处理？  
  **答：** 我使用分层摘要：保留最近的 5 轮对话原始内容，只对更早的做摘要。用户如果需要回溯细节，可以从原始日志中检索。  
- **问：** 多用户并发时，如何保证记忆隔离？  
  **答：** 通过用户 ID 或会话 ID 作为 partition key，存储在 Redis 或向量数据库时带上该 key，检索时先过滤。另外线程安全的连接池需要正确配置。  
- **问：** 能不能用知识图谱来做长期记忆？  
  **答：** 可以，比如将用户的重要实体（订单、商品、地址）构建成知识图谱，用图查询获取关系。不过开销较大，适合高稳定场景。

**项目联系：**  
我在项目中使用 Redis 存储短期会话，用 Chroma 存储长期兴趣点。例如用户曾经问过“我上次为什么退货”，Agent 能检索到之前对话中的退货原因，避免重复解释。我还在后台做了一个定期压缩的定时任务，每天凌晨对前一天对话做摘要，减少存储成本。

**记忆关键词：**  
分层记忆、语义检索、摘要压缩、时间衰减、并行隔离

### 题目 3：在 Planner/Executor 架构中，Planner 的任务计划如果被中途打断（比如模型输出格式错误或工具超时），你是如何恢复的？

**来源依据：**  
- AgentGuide 面试题库（多Agent通信与编排） - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 来源类型：技术博客/开源指南 - 可信度：中（需进一步验证）  
- 40 Generative AI Interview Questions ... - https://pub.towardsai.net/40-generative-ai-interview-questions-that-actually-get-asked-in-2026-with-answers-b4c647f1e2e8 - 来源类型：技术博客 - 可信度：中（个人博客，需进一步验证）

**可信度：** 中

**考察点：**  
Planner/Executor 模式、错误恢复、状态持久化、任务编排

**面试口头版回答：**  
我设计的 Agent 中，Planner 先生成一个步骤列表，Executor 逐个执行。如果中途出错，比如工具调用返回错误或 Planner 输出格式不对，我会分情况处理：  
第一，对于格式错误，我让 Executor 重新向 Planner 发送“上一步输出格式无法解析，请修正”，Planner 会修正后再提交。第二，对于工具执行失败，如果 Planner 定义了依赖关系（比如步骤 2 依赖步骤 1 的结果），我会让 Executor 标记步骤 1 为失败，并通知 Planner 生成替代方案。第三，我会持久化每一步的执行状态到数据库，这样即使整个 Agent 进程重启，也能从断点继续执行，而不是从头再来。  
举个例子，在我做的旅行规划 Agent 中，如果“查航班”工具超时，Planner 会重新生成一个包含“查火车”的计划，保证用户能继续。

**深入理解版回答：**  
Planner/Executor 架构将复杂任务分解为可执行的子步骤，但分布式系统下的失败是常态。工程上的恢复策略包括：  
1. **局部重试**：Executor 捕获到临时错误（如网络抖动、服务返回 503），可以自行重试 2-3 次，不影响 Planner。  
2. **反馈修正**：当 Executor 无法处理（如工具不存在、参数验证失败），它会将错误信息作为观察结果反馈给 Planner，Planner 重新规划。这类似 ReAct 的思考-行动-观察循环。  
3. **全局 checkpoint**：在每一步执行后，将当前任务快照（包括已完成步骤、待执行列表、状态变量）序列化到数据库（如 PostgreSQL）。如果进程崩溃，重启后读取最新 checkpoint，跳过已完成步骤继续。  
4. **超时熔断**：为整个任务设置总超时（如 60 秒），超时后回滚部分已执行的副作用（如已创建的订单需要取消），并通知用户失败。  
5. **上下文压缩**：如果失败导致 Planner 的上下文太长（因为积累了错误历史），可以用 LLM 对失败历史和当前状态做一次摘要，再喂给 Planner。  
**局限**：Planner 模型如果智力不足，可能反复给出错误计划。此时需要人工介入或切换到更简单的规则计划（如固定流程）。  
**评估**：可以通过成功率（任务完整执行的比例）、平均重试次数、平均完成时间、回滚次数来监控。

**常见追问与回答：**  
- **问：** 持久化 checkpoint 会不会引入性能瓶颈？  
  **答：** 可以异步写入，或者只在关键步骤（如涉及资金操作）时写 checkpoint。另外使用 Redis 等内存型数据库可以减少延迟。  
- **问：** 如果 Planner 一直输出无效计划，怎么跳出循环？  
  **答：** 设置最大规划次数，比如 5 次。同时检测计划是否与之前高度相似（如 Jaccard 相似度 > 0.8），若是则强制跳出。  
- **问：** 回滚副作用如何实现？  
  **答：** 我的做法是遵循 Saga 模式：每步工具调用都定义对应的补偿动作（如创建订单→取消订单）。执行器记录已执行步骤，回滚时逆序执行补偿。

**项目联系：**  
在智能客服 Agent 中，我实现了 Checkpoint 机制。当用户请求“取消订单并退款”时，Planner 生成三步：查验订单→取消订单→发起退款。如果取消订单成功后，退款接口超时，系统会标记阶段为“部分完成”，并在用户下次进来时从第三步继续。同时补偿动作：如果退款失败，提醒人工客服手动处理。

**记忆关键词：**  
Planner 反馈、checkpoint 持久化、Saga 补偿、超时熔断、重试限制

### 题目 4：LangGraph 中的状态管理（State）是如何工作的？你用它解决过什么工程问题？

**来源依据：**  
- LinkedIn 面经（Bilal Momin） - https://www.linkedin.com/posts/bilal-momin_generativeai-llm-langgraph-activity-7353048174648840192-C0CR - 来源类型：面经或社媒 - 可信度：低（需进一步验证）  
- AgentGuide - https://github.com/adongwanai/AgentGuide - 来源类型：技术博客/开源指南 - 可信度：中（需进一步验证）

**可信度：** 中

**考察点：**  
LangGraph 框架、状态图、条件路由、工程应用

**面试口头版回答：**  
LangGraph 使用图结构定义 Agent 流程，每个节点是逻辑处理单元（如工具调用、决策），节点之间通过边连接，边可以带条件。状态是全局共享的 TypedDict，所有节点可以读写。状态变化会触发图的执行流程。  
我在一个文档处理 Agent 中用 LangGraph 解决了“多步流水线”的问题：我们需要先总结文档，然后基于总结生成报告，最后翻译。传统 LangChain 链式调用很难处理“如果总结缺失就重新生成”的条件逻辑，而 LangGraph 让我可以添加一个条件边：检查总结是否为空，若是，则回到生成节点重试。状态管理避免了显式传递上下文，所有节点共享同一个 state 对象，简化了代码。

**深入理解版回答：**  
LangGraph 是对 LangChain 的扩展，核心概念是 StateGraph：  
- **State**：一个继承自 TypedDict 的对象，包含应用状态（如 messages、当前步骤、临时数据）。每个节点都会接收 state 并返回 state 的部分更新。  
- **Nodes**：函数或可调用对象，接收当前 state，执行逻辑，返回 state 更新。例如一个工具调用节点接收 messages 列表，调用工具，将结果追加到 messages。  
- **Edges**：有向连接，可以附带条件函数（conditional edge），根据 state 的某个字段决定下一步走到哪个节点。  
- **Checkpointing**：LangGraph 内置支持 checkpoint，可以将每一步的 state 保存到数据库，用于故障恢复和重放。  
**工程优势**：  
1. **可视化**：图的拓扑结构清晰，易于设计复杂工作流（循环、分支、并行）。  
2. **错误恢复**：借助 checkpoint，如果执行中途崩溃，可以从最后一个 checkpoint 重启。  
3. **可测试性**：每个节点是独立函数，可以单独单元测试。  
**常见坑**：  
- 状态体积过大导致内存和序列化开销：建议只保留关键字段，其他如原始文档可存为文件路径。  
- 条件边逻辑复杂：最好用枚举定义可能的分支，避免字符串硬编码。  
- 并发执行时状态冲突：LangGraph 默认支持单线程流程，如果需要并行节点，需用 ParallelNode 并手动处理合并。  
**评估指标**：可通过流程执行成功率和平均完成时间来衡量。

**常见追问与回答：**  
- **问：** LangGraph 与 ReAct 框架的区别？  
  **答：** ReAct 是一种 pattern（思考→行动→观察），LangGraph 是实现这个 pattern 的图框架。LangGraph 更灵活，可以实现循环、分支，而 ReAct 通常是固定循环。  
- **问：** 你遇到过状态数据过大导致性能问题吗？怎么解决的？  
  **答：** 是的，早期我把所有历史消息都放在 state 中，导致序列化很慢。后来我改为只保留最近 10 轮消息，更早的消息压缩为摘要并单独存储 key。  
- **问：** 如何调试 LangGraph 流程？  
  **答：** 使用 LangGraph 自带的 LangSmith trace，可以看到每个节点的输入输出、耗时和错误。也可以手动打印 state 快照。

**项目联系：**  
我做了一个多步推理 Agent：用户提问后，Agent 先检索知识库，如果结果不足，则调用 Web 搜索，然后综合回答。我用 LangGraph 的 conditional edge 实现“如果检索结果置信度高则直接回答，否则走 Web 搜索”。状态中保留 search_results、confidence 等字段，清晰易维护。

**记忆关键词：**  
StateGraph、条件边、checkpoint、可视化、节点测试

### 题目 5：你的 Agent 系统如何保证多轮对话中上下文的正确性？如果中间有一轮用户突然切换话题，如何避免混淆？

**来源依据：**  
- 京东AI后端日常实习二面（Agent项目深度拷问） - https://mj.mianlingai.com/tag/%E5%A4%A7%E6%A8%A1%E5%9E%8B - 来源类型：面经或社媒 - 可信度：低（需进一步验证）  
- AgentGuide 面试题库 - https://github.com/adongwanai/AgentGuide/blob/main/docs/04-interview/12-company-interview-cases.md - 来源类型：技术博客/开源指南 - 可信度：中（需进一步验证）

**可信度：** 中

**考察点：**  
上下文管理、话题切换、意图识别、状态隔离

**面试口头版回答：**  
我主要靠三种机制：第一，在每次 LLM 调用时，我将当前上下文明确限定在“当前任务”范围内，把历史消息按时间分段，并给每段打上标签（比如“订单查询”、“投诉处理”）。第二，当检测到用户新消息与前一话题无关时，我会启动一个“话题切换检测”，如果相似度低或包含明显的新意图词，我就会重置上下文，只保留系统 prompt 和最新的 query。第三，对于可能混淆的历史信息，我在 prompt 中明确写“请基于最新的用户意图和当前对话历史回答问题”。这样 Agent 也能根据语义推理是否引用旧信息。  
例如，用户先问“我的订单咋样了”，然后突然问“今天天气好吗”，Agent 会通过意图分类识别出这是新话题，不会再把天气问题和订单绑定。

**深入理解版回答：**  
多轮对话中上下文管理的核心挑战是：如何让 Agent 既能记住关键的历史信息，又不被无关历史误导。  
工程实践方法：  
1. **意图与话题分割**：在每一轮用户输入后，先运行一个轻量级的话题边界检测模型（可以用小模型或规则）。如果检测到新话题，将历史消息中的“旧话题消息”移出 main context，只保留摘要或完全丢弃。  
2. **上下文窗口剪裁**：使用滑动窗口，只保留最近 N 轮对话（如 5 轮）。如果需要引用更早的信息，通过检索系统召回相关记忆。  
3. **结构化上下文**：在 prompt 中显式构建“当前任务”、“历史摘要”、“当前用户的直接问题”三个区域。Agent 被训练成优先关注当前任务。  
4. **状态重置机制**：当用户明确表示开启新会话（如“我们换个话题”）或连续两次低相似度时，清除工作记忆，仅保留长期记忆中的关键信息（如用户身份）。  
**常见坑**：  
- 旧话题中的实体可能被二次讨论，比如用户先说“我要买 A 手机”，新话题“推荐耳机”中不能轻易忘记 A 手机，因为可能关联。解决方案：通过长期记忆保留重要实体，但在短期上下文中不突出。  
- Agent 可能过于僵化，用户聊着聊着回到旧话题。可以使用“话题切换”不是硬切断，而是按比例保留上下文，用 attention 机制让模型自己决定权重（但实际工程中需 prompt 约束）。  
**评估**：可用跨话题的准确率（新话题下旧信息干扰度）和用户满意度来评估。离线评测可构造多轮切换对话集。

**常见追问与回答：**  
- **问：** 你提到话题边界检测模型，具体用什么？  
  **答：** 我用的是一个小型的基于 BERT 的意图分类器，预训练后微调。也可以用规则如关键词匹配。  
- **问：** 如果用户新话题与旧话题有隐含关联怎么办？  
  **答：** 我会让 LLM 自行判断是否需要引用旧信息。在 prompt 中加上“如果历史信息与当前问题相关，可以引用；否则忽略”。  
- **问：** 如何保证用户隐私，不跨用户混用上下文？  
  **答：** 每个用户/会话有自己的 context 对象，存在 Redis 中，key 为 user_id:session_id。用户切换时按 session 隔离。

**项目联系：**  
我在实习中负责的客服 Agent 支持多话题。用户可能先问“退款政策”，然后问“有哪些商品促销”。我用了一个基于 sentence-BERT 的话题检测器，当检测到余弦相似度低于 0.3 时，自动清理短期上下文，只留系统 prompt 和 query。同时将退款政策的关键信息（如 30 天无理由）存入长期记忆，下次用户问“能退吗”时仍能召回。

**记忆关键词：**  
话题分割、滑动窗口、意图分类、状态隔离、显式 prompt

我主要用过 LangGraph 和 AutoGen。LangGraph 是基于图的框架，适合定义复杂的流程控制，比如循环、条件分支、并行。它的优势在于状态可视化和 checkpoint 支持，并且兼容 LangChain 的生态。AutoGen 则更侧重于多 Agent 对话，内置了演讲者选择、对话历史管理等，适合需要多个 Agent 协作的场景（比如一个写代码、一个审查）。但 AutoGen 的图表达能力较弱，不适合细粒度控制。项目选型时，如果流程简单单 Agent，推荐 LangChain；如果需要精确的步骤图，选 LangGraph；如果需要多 Agent 角色扮演，用 AutoGen。我自己的项目用的是 LangGraph，因为需要条件分支和记忆管理。

**面试口头版答案：**  
MCP 是一个开放标准，定义了 Agent 如何发现工具、调用工具和获得工具结果的协议。它解决了两个核心问题：第一，工具发现的标准化，Agent 不需要硬编码每个工具的 API 地址和 schema，而是通过 MCP 服务器动态获取；第二，可扩展性，新工具只要实现 MCP 接口就可以被任何兼容的 Agent 使用。在实际工程中，使用 MCP 后，我们团队切换某个天气 API 时，只需要更新 MCP 服务器配置，不需要修改 Agent 代码。MCP 与 Function Calling 区别在于 Function Calling 是 LLM 侧的一个机制，而 MCP 是跨 Agent 和工具的通信协议。

#### 对应参考答案：

**问题 1：RAG 中的 Hybrid Search（混合检索）是如何工作的？它解决了什么单一检索的不足？**  
**面试口头版答案：**  
Hybrid Search 结合了稠密检索（基于嵌入向量语义相似度）和稀疏检索（基于 BM25 关键词匹配）。稠密检索擅长语义相近但字面不同的查询，比如“买手机用什么好”能召回到“智能手机推荐”，但它可能漏掉精确匹配的关键词。稀疏检索正好相反，它能精确匹配“苹果 iPhone 14”，但不理解同义词。两者先用各自的检索器得到 Top-K 结果，然后用 RRF（倒序排名融合）或加权融合重新排序。在 Agent 场景中，Hybrid Search 能显著提升召回率和鲁棒性。我曾在项目中把 Hit Rate 从 85% 提升到 93%。

**问题 2：当你构建 RAG 作为 Agent 的工具时，如何确保检索结果对 Agent 决策是可靠、及时的？**  
**面试口头版答案：**  
我在 Agent 中封装了一个“RAG 查询工具”，该工具返回的结构不仅包括文本，还包括元数据（时间戳、来源文档、置信度分数）。Agent 在决策时会检查：如果查询的是股票价格等时效性要求高的内容，RAG 工具会优先返回最新数据源的结果，并标注采集时间。此外，RAG 结果在返回前会经过一个“验证器”判断相关性：如果检索结果与查询主题相关性低于阈值，则工具返回“未找到可靠信息”，Agent 可以选择改用其他工具（比如 Web 搜索）。这样能避免 Agent 被过时或无关的文档误导。

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 请解释 ReAct 框架中“思考（Thought）- 行动（Action）- 观察（Observation）”循环的工程实现原理。为什么有时候需要限制循环次数？如果限制后仍未完成任务，你有什么备选方案？

**面试口头版参考答案：**  
ReAct 让 LLM 每次输出一段思考、一个工具调用（Action）和需要观察的字段。系统解析 Action 并执行工具，将结果作为 Observation 填入下一轮 prompt。循环可以继续直到 LLM 输出最终答案。限制循环次数是为了防止无限循环或消耗过多 token，通常设 max_iterations=10。如果达到上限仍未完成，我会让 Agent 输出当前已收集的信息和失败原因，并请求用户进一步澄清或转人工。工程中还会加上“思考内容是否在重复”的检测，如果连续 3 次思考相似，也会提前终止。

**深入理解版参考答案：**  
ReAct 是 Agent 的基础模式。工程实现步骤：  
1. **Prompt 构造**：包含工具描述、格式要求（如 “Thought: …\nAction: tool_name\nAction Input: {…}”）、系统指令（如 “你可以调用以下工具…”）。  
2. **循环执行**：  
   - 调用 LLM 获取输出。  
   - 解析输出：用正则或结构化提取 Thought、Action、Action Input。若格式错误，返回错误信息给 LLM 让其修正。  
   - 执行工具：根据 Action 找到并调用工具，捕获结果或错误。  
   - 将 Action Input、Observation 追加到对话历史，准备下一轮。  
   - 检测是否遇到 Final Answer 关键字，若出现则结束。  
3. **终止条件**：  
   - 达到最大轮次（如 10）。  
   - 检测到重复的 Thought/ Action 组合（避免死循环）。  
   - 用户主动取消。  
4. **备选方案**：  
   - 如果轮次耗尽，Agent 输出“无法完成，已收集以下信息…”并提供部分结果。  
   - 使用降级方案：切换到简单规则回复（如“抱歉，我暂时无法处理”）。  
   - 启用 human-in-the-loop，将当前状态推送给管理员决断。  
**局限**：ReAct 没有显式的计划能力，只是“边想边做”。对于需要多步推理的任务，Plan-and-Execute 可能更适合。  
**评估**：可用成功完成率、平均轮次、平均 token 消耗、用户满意度评分。

**可能追问与简短回答：**  
- **问：** 如果工具返回的结果长度很大，如何处理？  
  **答：** 我会对结果进行截断（比如只保留前 2000 tokens）或摘要，然后放入 Observation。同时保留完整结果在 state 中，必要时让 LLM 引用。  
- **问：** 如何避免 Agent 在 Observation 中误将部分内容当作最终答案？  
  **答：** 在 prompt 中明确“不要将工具返回的直接输出作为最终答案，除非它包含 Final Answer 标记”。也可以通过结构化的输出解析器，只接受 Final Answer 字段作为结束条件。
