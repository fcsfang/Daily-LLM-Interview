> **日期：** 2026-07-13
> **今日重点：**
> 1. Chunk 策略不是静态的，需要根据文档类型和下游任务调参，工程上建议用滑动窗口 + overlap + 元数据标注。

## 1. 今日面试真题精讲

### 题目 1：如何设计 RAG 中的 Chunk 策略？chunk_size 一般怎么定？不同文档类型（代码、表格、长文本）怎么处理？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- RAG Interview: 40 Questions to Go from Beginner to Advanced - https://www.analyticsvidhya.com/blog/2026/02/rag-interview-questions-and-answers - 技术博客 - 可信度中（知名博客但非官方）

**可信度：**
中

**考察点：**
RAG 文档处理、chunk 参数调优、工程经验

**面试口头版回答：**
Chunk 策略直接影响检索粒度。我一般遵循几个原则：第一，chunk_size 通常取 256-512 个 token，太小会丢失上下文，太大则可能混入噪声，导致 lost in the middle。第二，必须加 overlap，一般设为 chunk_size 的 10%-20%，保证跨片信息不丢失。第三，针对不同文档类型要差异化处理：代码可以用函数或类作为自然边界切分；表格先用行转文本描述再切；长文本先按段落或标题分块，再对每个块做二次切分。工程上我会用 sliding window 方式，同时保留每个 chunk 的元数据（比如来源文档、章节标题），便于引用溯源。上线后还要通过实验对比不同参数对召回率和回答准确度的影响。

**深入理解版回答：**
Chunk 是 RAG 知识库构建的起点，直接影响检索质量。首先需要明确：chunk_size 不是越大越好也不是越小越好。太小会导致 chunk 语义不足，检索到碎片；太大则多个主题混在一个 chunk 里，模型难以定位答案位置，且浪费上下文窗口。业界常见做法是围绕 256-512 token 区间做 grid search，结合人工评估的召回精度和生成质量综合打分。

具体工程流程：
1. 文档清洗（去重、格式转换、图/表格提取）。
2. 分层切片：先用标题/段落切大块，再用 langchain 的 RecursiveCharacterTextSplitter 按分隔符（\n\n、\n、句号）递归切到目标大小。
3. 添加 overlap，一般 10%~20%，并记录每个 chunk 的起始位置和所属章节。
4. 对代码文档：使用 Tree-sitter 解析 AST，按函数/类节点切分，确保语法完整性。
5. 对表格：先调用 OCR/PDF 解析，然后转换为自然语言描述（如 “产品A单价100元”），再切分，避免纯表格向量化丢失语义。
6. 对多模态（图文混排）：保留图片引用标记，在 chunk 元数据中记录关联图片路径，生成时注入。

常见坑：没有 overlap 导致答案被截断在两个 chunk 里；chunk_size 全文档统一导致段落短的文档被强切出不完整块；没有元数据导致模型无法回答来源。优化方向：引入语义切分（如使用 LLM 判断语义边界），或者用自适应 chunk（根据内容动态调整大小）。评测时可以用 RAGAS 的 context_precision 和 context_recall 来量化 chunk 策略好坏。

**常见追问与回答：**
- **追问1：你如何评估 chunk 策略的好坏？** 答：我会构建一个评测集，标出每个 query 对应的黄金 chunk ID，然后计算 hit rate（Top-K 命中率）和 MRR。同时离线用 RAGAS 指标（context_precision）看生成答案是否基于正确 chunk。
- **追问2：overlap 设多少合适？有没有理论依据？** 答：一般 10%-20%，保证句子不截断。如果文档是连续叙述式，overlap 可适当增大。没有标准理论，主要靠实验和观察 Bad Case。
- **追问3：chunk 里要保留句号边界吗？** 答：尽量按句子边界切，避免句子被拆成两半。可以用 splitter 的“先按段落切，再按句号切”逻辑。

**项目联系：**
在之前做的企业内部知识库 RAG 项目中，我负责知识库构建。文档包括产品手册（PDF）、技术规格（表格）、代码片段。我针对每种类型实现了不同的 splitter，并在元数据中加入 `doc_id`、`chapter`、`page_num`。上线后通过 A/B 测试对比了 chunk_size=256 和 512，发现 512 对多步骤查询效果更好，但加重了 token 消耗，最后折中为 384 + overlap 15%。

**记忆关键词：**
Chunk_size、overlap、RecursiveCharacterTextSplitter、语义切分、元数据、文档类型适配

### 题目 2：为什么需要 Rerank？Cross-Encoder 相比向量召回有什么优势？工程上如何部署 Rerank？

**来源依据：**
- 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 8 LLM & RAG Interview Questions That OpenAI, Anthropic & Google Actually Ask - https://callsphere.ai/blog/llm-rag-interview-questions-2026-openai-anthropic-google - 技术博客 - 可信度中

**可信度：**
中

**考察点：**
RAG 检索优化、rerank 原理、工程部署

**面试口头版回答：**
向量召回（如使用 embedding 模型）只能做浅层语义匹配，丢失了 query 与文档之间的交互信息。Rerank 用 Cross-Encoder 模型同时输入 query 和每个候选文档，输出一个相关性分数，能做到更细粒度的匹配。优势在于精度高，但计算成本也高，所以一般作为二阶段精排。工程部署时，我会把 Rerank 作为一个独立的微服务，通过 gRPC 暴露接口，资源分配上保证 GPU 推理，并设置 batch 大小控制延迟（例如每次 rerank 40 个 doc，batch=8）。同时加上超时和熔断，防止网络抖动拖慢整个链路。如果延迟敏感，还可以用知识蒸馏把 Cross-Encoder 压缩成更小的模型，或者用 ColBERT 的后期交互作为轻量替代。

**深入理解版回答：**
Rerank 解决的核心问题是：向量检索只比较 query 和文档各自 embedding 的余弦相似度（双塔架构），但 query 和文档之间没有发生细粒度的 token-level 交互。而 Cross-Encoder 将 query 和文档拼接起来喂给一个分类模型直接输出相关度分数，能够捕捉到“否定词、条件、反义词”等交互特征。例如用户问“不是苹果手机”，向量召回可能会召回苹果手机相关文档，而 rerank 能识别“不是”并降低其分数。

工程流程：
- 召回阶段：用双塔 embedding 模型（如 bge-base）召回 Top-K（比如 100 个）。
- 精排阶段：将 query 和每个候选 chunk 拼接成 [CLS] query [SEP] chunk [SEP]，送入 Cross-Encoder（如 bge-reranker-base），输出 0-1 之间的相关性分数。
- 取 rerank 后的 Top-N（比如 5 个）作为最终上下文。

部署考虑：
- GPU 推理：建议使用 ONNX 或 TensorRT 加速，显存占用大致为 2-4GB（base 模型）。
- 延迟优化：如果 Top-K=100，需要推理 100 次，可 batch 推理（如 batch_size=8），总延迟约 50-100ms。若仍不满足，可降级为只对 Top-30 做 rerank。
- 动态调用：设置 qps 阈值，当超时或负载过高时跳过 rerank 直接取向量召回 Top-N（降级策略）。

常见坑：Cross-Encoder 在大 batch 下显存溢出；未归一化导致分数不可比；直接用在长文本上（超过 512 token）需要 truncation。优化方向：可以用 DistilRerank 蒸馏出小模型；也可以使用 ListWise 损失训练 reranker 提升排序质量。

**常见追问与回答：**
- **追问1：你和 embedding 模型是共用一个 GPU 吗？** 答：通常分开部署，避免资源争抢。如果资源受限，可以共享 GPU 但用 MPS 或者 MIG 做隔离，或者时间片轮转。
- **追问2：Rerank 的分数如何与向量检索引结合？** 答：先向量召回拿到分数，但向量分数和 rerank 分数不在同一个量级，不能简单加权。工程上常用的是“先召回后 rerank，以 rerank 为准”，或者用 Dense + Sparse 混合召回后做 RRF（倒数排名融合），再用 rerank 做 Top-K 内重排。
- **追问3：有没有不做 rerank 也能提升精度的方案？** 答：用 Query 改写（如 HyDE 或假设性文档检索）可以提升召回质量，间接减小 rerank 必要性。但对于高精度场景（如医疗、法律），rerank 仍然是必选项。

**项目联系：**
在公司 AI 助手项目中，我负责检索链路优化。最初只用 bge-large 召回 Top-10 直接喂给生成，发现大量不相关内容被误召唤后导致幻觉。我加入了 bge-reranker-base 对 Top-30 做 rerank，最终取 Top-3。延迟从 80ms 升到 120ms（batch_size=8），但回答相关性指标（通过人工评测）提升了 15%。同时做了降级开关，当 rerank 服务不可用时自动回退到纯向量。

**记忆关键词：**
Cross-Encoder、双塔交互缺失、精排、Batch 推理、降级策略、ColBERT

### 题目 3：在 RAG 系统中如何让模型输出带引用的答案？如何保证引用的准确性？

**来源依据：**
- RAG Interview: 40 Questions to Go from Beginner to Advanced - https://www.analyticsvidhya.com/blog/2026/02/rag-interview-questions-and-answers - 技术博客 - 可信度中
- 8 LLM & RAG Interview Questions That OpenAI, Anthropic & Google Actually Ask - https://callsphere.ai/blog/llm-rag-interview-questions-2026-openai-anthropic-google - 技术博客 - 可信度中

**可信度：**
中

**考察点：**
RAG 引用设计、幻觉控制、Prompt 设计

**面试口头版回答：**
引用主要是通过 Prompt 设计让模型在生成答案时标注信息来源。具体做法：给模型的系统提示中加上“请用数字或方括号标记你所引用的段落编号，并在最后给出来源列表”。然后，将传入的检索 chunk 前面加上 `[1]`、`[2]` 等标签，并在 Prompt 里明确要求“引用格式如下：根据[1]...”。为了保障准确性，离线我会自动评测引用粒度：用精确匹配或正则看答案中是否出现 `[数字]`，以及引用的 chunk 是否真实包含答案内容。同时还可以做回答忠实度（faithfulness）评测，用 LLM 判断答案每个事实是否都能在指定 chunk 中找到对应句子。如果发现模型生成了 chunk 没有的内容，就记为不忠实，需要分析是 Prompt 约束不够还是检索带入了噪声。

**深入理解版回答：**
引用是 RAG 对抗幻觉的重要手段，能让用户和开发者追溯答案来源。工程实现分三步：

1. **注入检索上下文时加上标识**：例如 `参考文档1: [1] 段落内容` `参考文档2: [2] 段落内容`，并明确告知模型“请引用最相关的文档编号”。
2. **Prompt 约束：结构化输出**：可以要求模型同时输出 `{ "answer": "xxx", "citations": [1,3] }` 的 JSON 格式，方便下游解析。
3. **后处理验证**：检查引用的 chunk ID 是否在本次检索的上下文中，避免模型幻觉出虚假 ID。如果发现引用不存在的 ID，可以丢弃该回答，强制重生成。

评测引用准确性常用指标：
- **引用精确率**：模型引用了的 chunk 中，有多少确实包含答案所需信息（需要人工或 LLM 判断）。
- **引用召回率**：正确 chunk 中有多少被模型正确引用。
- **Faithfulness（忠实度）**：答案的每个陈述是否都能在引用 chunk 中找到对应文字。

常见坑：
- 模型对 `[1]` 当作普通 token 处理，不理解为引用，导致乱标。可以微调一个专门做引用的小模型。
- 多个 chunk 内容相似时引用错误。可以用交叉验证，要求模型对每个答案句子给出具体引用段落。
- Prompt 太长导致指令被稀释。可以拆成两步：先生成答案，再对答案做引用检索（self-RAG 方式）。

优化方向：引入 self-RAG 机制——模型在生成过程中可以主动检索并引用。或者使用 Citation-LLM 这类专门训练过的模型，如 Command-R+。

**常见追问与回答：**
- **追问1：如果要求 JSON 格式输出，但模型输出格式不符怎么处理？** 答：加 few-shot 示例，同时用函数调用（Function Calling）约束 schema。如果仍失败，则用正则回退或调用二次生成。
- **追问2：引用准确性怎么自动化评测？** 答：用 LLM-as-Judge 打分。给出一个答案和其引用的 chunk，让 judge 模型打分是否每个句子在 chunk 中有直接支持。也可以计算 NLI（自然语言推理）分数，即 chunk 蕴含答案的程度。
- **追问3：引用会不会增加 Token 消耗？** 答：是的，但可以控制：只让引用出现在 final answer 的末尾，或者在生成时用 context distillation 压缩 chunk 长度。

**项目联系：**
在 RAG 项目评测环节，我建立了引用准确率的离线评测集：100 条人工标注了“正确答案”和“必须引用的 chunk ID”。然后跑我的 pipeline 输出带引用的答案，计算引用精确率和召回率。最初只有 60%，通过改进 Prompt 约束（增加“按顺序引用最前面的文档”）提升到 85%。

**记忆关键词：**
引用标注、Faithfulness、结构化输出、Self-RAG、LLM-as-Judge

### 题目 4：如何降低 RAG 系统中的幻觉？请列举几种方法。

**来源依据：**
- 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- RAG Interview: 40 Questions - https://www.analyticsvidhya.com/blog/2026/02/rag-interview-questions-and-answers - 技术博客 - 可信度中

**可信度：**
中

**考察点：**
幻觉控制、RAG 优化

**面试口头版回答（增量回答，不与之前重复）：**
除了常规的检索质量优化，我主要从三个层面控制幻觉：第一是引用强制，让模型必须输出引用标记；第二是知识冲突检测，如果检索出的多个 chunk 互斥（比如两个文档定价不同），会调用一个外部的冲突仲裁服务或提示模型告知用户信息矛盾；第三是后处理验证，使用一个轻量 NLI 模型检查生成答案中的事实是否被检索 chunk 支持，不通过则拒绝该回答并触发重生成。另外，我还会用“不检索就不会”的 Prompt 策略，明确告诉模型“如果文档中没有相关信息，请回答不知道”，降低捏造。在离线评测中，即使召回正确，也要监控 faithfulness 指标，如果分数低于阈值，就做 Bad Case 分析。

**深入理解版回答：**
RAG 中的幻觉分为两类：一是模型不依赖检索结果，凭自身知识生成（retriever override）；二是模型过度依赖训练数据中的偏见，与检索结果冲突。针对性的解法包括：

1. **检索端优化**：提升召回精度和覆盖度。例如使用 Hybrid 检索（BM25 + 稠密）、Query 改写（HyDE）、Rerank。
2. **生成端 Prompt 约束**：加入“只基于提供的文档回答”、“如果文档中没有相关信息，请说不知道”。还可以使用 Chain-of-Thought 让模型先推理文档内容再回答。
3. **引用验证**：上文已谈。
4. **知识冲突处理**：当检索结果中存在矛盾时，可以执行一个 verify 步骤：让模型对矛盾点做出标注，或调用外部知识图谱核对。
5. **后处理拒绝**：使用一个评估器（可以是 LLM）判断回答是否忠实于检索文档。如果忠实度分数低于阈值，则返回“我无法确定答案”或触发重新检索。
6. **训练层面**：可以微调一个专门做 RAG 的模型，在训练中加入拒绝回答的样本（即 query 与上下文不匹配时回答不知道）。

工程上最实用的是组合“引用强制 + 忠实度检测”。我在项目中用 LangChain 的 `SelfRAG` 实现，即生成时对每个句子判断是否需要检索，检索后再验证。效果提升明显，但成本较高。

**常见追问与回答：**
- **追问1：忠实度检测用什么模型？** 答：可以用 TrueTeacher（Google 出品）或自蒸馏的小 BERT 模型，做 entailment 分类。也可以直接用 GPT-4-mini 做 judge，但成本高。
- **追问2：如果用户故意问文档中不存在的观点，如何防止模型随声附和？** 答：除了 Prompt 约束，可以加对抗样本做 red-teaming 测试，并针对这类 query 设置专门的 guardrail 拦截。
- **追问3：重生成会拖慢响应，怎么平衡？** 答：可以设置为异步任务：先返回第一版答案，后台做忠实度检测，如果不通过则 log 告警并缓存，下次相同 query 直接返回修正版。

**项目联系：**
我在项目中使用开源的 `deberta-v3-base` 做 NLI 模型，对生成答案与检索 chunk 做忠实度判断。阈值为 0.7，低于则触发重生成。上线后幻觉相关投诉减少 40%。

**记忆关键词：**
忠实度检测、知识冲突仲裁、拒绝机制、Self-RAG、NLI 验证

  - 帮助理解 RAG 全链路和各环节的 trade-off。
- **可转化面试题：**
  - 问题 1：如何评估 RAG 系统的检索质量？具体说几个指标。
  - 问题 2：RAG 中如何处理更新文档？有哪些策略？

**问题 1 参考答案（面试口头版）：**
我主要从三个维度评估：一是上下文精确率（context_precision），衡量检索出的 chunk 中到底有多少包含了答案所需的信息；二是上下文召回率（context_recall），即正确答案在检索结果中是否能被找到。这两个指标可以结合人工标注或 LLM-as-Judge 来计算。三是答案忠实度（faithfulness），看生成答案是否基于检索 chunk 的事实。工程上，我会离线构造标注好的 query-chunk 对，跑 pipeline 后计算指标，并监控 Bad Case。同时在线也记录用户反馈（点赞/踩）来辅助判断。

**问题 2 参考答案（面试口头版）：**
文档更新分全量刷新和增量更新两种。全量刷新适合小知识库，直接全部重新索引。增量更新则要监听文档变动（如文件修改时间、数据库 binlog），对变动的 chunk 做 upsert 操作：先删除旧 chunk（用 doc_id 匹配），再插入新 chunk 并重新 embedding。关键是要保证版本一致性——如果用户 query 时索引正在更新，可能会拿到新旧混合的结果，我一般通过版本号控制：每次更新后创建一个新的索引别名，完成索引后切换别名。还有就是在文档更新后，部分相关 query 的召回质量可能下降，因此我会在更新后自动跑一轮离线评测，对比更新前后的召回率。

**问题（lost in the middle）参考答案：**
Lost in the middle 是指当给模型的上下文中塞入过多 chunk 时，模型倾向于只关注开头和结尾，而忽略中间部分。在 RAG 中体现为：如果检索返回 Top-K=10，即使中间 chunks 相关，模型也可能忽略。我应对的方法：第一，优化 Top-K 值，比如只用 Top-3~5，减少噪声；第二，rerank 后只取保序靠前的；第三，在构建 prompt 时，将最相关的 chunk 放在开头或结尾（通过分数排序而非原始顺序）；第四，使用 Lost-in-the-Middle 感知的 prompt 模板，显式告诉模型“注意所有文档都可能包含关键信息”。离线评测中，我会统计模型对中间位置 chunks 的引用率，低于某阈值则调整排序策略。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** 什么是 HyDE（Hypothetical Document Embeddings）？它如何提升 RAG 召回？
- **A：** HyDE 先让 LLM 根据用户 query 生成一个假设性回答文档，然后用这个假设文档的 embedding 去检索，而非直接用 query 的 embedding。因为假设文档包含更完整的描述性词汇，与 doc 的嵌入空间更接近，能显著提升召回率，尤其适合 query 很短、语义稀疏的场景。工程上需要额外一次 LLM 调用，但检索精度提高明显。
- **一句话记忆法：** 先写答案再搜，搜得更准。
- **复习标签：** #RAG #召回优化

### 卡片 2
- **Q：** 什么是 CRAG（Corrective RAG）？它解决了什么问题？
- **A：** CRAG 是在标准 RAG 基础上增加了一个检索结果评估开关。先对检索出的 chunks 做相关性打分，如果得分低于阈值，说明检索质量差，则触发“纠正”操作——比如重写 query、重新检索，或者回退到仅用 LLM 知识。这解决了“检索质量差时还硬喂给模型”的问题。在实践中，我会用一个小模型（如 microsoft/deberta-v3-xsmall）做相关性分类器。
- **一句话记忆法：** 检索不行就重查，不给模型喂坏材料。
- **复习标签：** #RAG #检索优化

### 卡片 3
- **Q：** 什么是 RAG 中的 Self-Query？适用于什么场景？
- **A：** Self-Query 是指让 LLM 从用户自然语言中提取结构化过滤条件（如日期、类型），然后结合向量检索执行。例如用户问“去年华东区的销售额”，模型先提取 “region: 华东”、“time: 去年”，然后生成一个元数据过滤器传给向量数据库。这样可以用结构化过滤缩小范围，减少噪声。适合知识库中有丰富元数据的场景。
- **一句话记忆法：** 先让模型拆出条件，再用元数据精确过滤。
- **复习标签：** #RAG #检索优化

### 卡片 4
- **Q：** 什么是“In-Context Rerank”？它和传统 Rerank 有什么区别？
- **A：** In-Context Rerank 利用 LLM 本身的排序能力，在生成 prompt 之前让 LLM 对检索到的 chunks 进行排序，而不是用专门的 Cross-Encoder 模型。做法是把 chunks 混在一起放入 prompt，让 LLM 输出相关性高的 top-N。优势是不需要额外部署 reranker，节省资源；缺点：准确率通常低于专用模型，且消耗 LLM 的 token。一般作为低成本备选方案。
- **一句话记忆法：** 让大模型自己当裁判排序，省部署但费 token。
- **复习标签：** #RAG #Rerank

### 卡片 5
- **Q：** RAG 中“分块重叠（chunk overlap）”的作用是什么？如何确定大小？
- **A：** 重叠是为了避免因为切分边界导致关键信息被截断。例如一个句子被切成两半，分别放在不同的 chunk，不重叠就会丢失语义。常用做法：用 RecursiveCharacterTextSplitter 设置 `chunk_overlap` 为 chunk_size 的 10%~20%，且 overlap 区域尽量以句子为边界（所以分隔符优先级设为句号）。工程上通过实验对比不同 overlap 下的召回率来确定最优值。
- **一句话记忆法：** 切块时留点尾巴，句子不摔断。
- **复习标签：** #RAG #Chunk

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 在 RAG 的向量检索中，常用的相似度度量有余弦相似度和欧氏距离。请说出它们的区别以及你在工程中如何选择？如果 embedding 已归一化，两者有什么关系？

**面试口头版参考答案：**
余弦相似度计算两个向量方向的相似性，取值范围 [0,1]（归一化后），对量级不敏感。欧氏距离计算向量绝对距离，对量级敏感。如果 embedding 模型已经做了 L2 归一化（模长为 1），那么余弦相似度 = 1 - 欧氏距离² / 2，两者本质等价。工程上，我一般用余弦相似度，因为向量数据库（如 Milvus、FAISS）默认支持内积检索（归一化后内积等于余弦相似度），计算速度更快。如果 embedding 未归一化（比如有些模型同时保留方向和幅度信息），则需要根据任务特点选择：当向量幅度代表置信度时用欧氏距离更好；当只关心语义方向时用余弦。实践中我统一对 embedding 做 L2 归一化，然后使用内积，既快又等价于余弦。

**深入理解版参考答案：**
余弦相似度定义：cos(A,B) = A·B / (|A||B|)。如果 A 和 B 已归一化，则 cos(A,B) = A·B。欧氏距离定义：d(A,B) = √(∑(ai-bi)²)。对于归一化向量，|A|=|B|=1，那么 d² = 2(1-cos(A,B))，所以此时两者单调等价。选择原则：

1. **标准化策略**：大多数开源 embedding 模型（如 BGE、E5）输出向量已做归一化，因此直接使用内积即可。如果未归一化，可手动添加 normalization 步骤。
2. **场景偏好**：
   - 余弦相似度更适合语义搜索，因为它只关注方向，忽略长度。例如“苹果”和“苹果公司”的 embedding 长度可能不同但方向一致。
   - 欧氏距离可能对 embedding 质量更敏感：如果模型训练时隐式编码了频率或重要性到模长，欧氏距离能捕捉到这部分信息。
3. **稀疏向量场景**：对于 BM25 的稀疏向量（词频权重），通常用内积（点积），因为长度本身带有信息。
4. **混合检索（Dense + Sparse）**：融合不同检索分数时，需要先将它们标准化到同一尺度，例如使用 MinMax 归一化或倒数排名融合（RRF）。

**常见追问与回答：**
- **追问1：如果使用余弦相似度，但 embedding 没有归一化，结果会怎样？** 答：如果不归一化，余弦计算时自动做了归一化，所以结果与归一化后内积相同。但归一化后内积的计算效率更高，因为可以提前将 embedding 归一化存储，检索时只需计算点积，节省了分母计算。
- **追问2：有没有必要同时保留两种相似度？** 答：可以但少见。一个权衡方案是使用可学习的度量（如训练一个评分模型），但代价大。多数情况下，选取一种并固定即可。我建议用余弦 + 内积实现。
- **追问3：向量检索中，查询向量的归一化是否会影响搜索结果？** 答：会影响。如果库中向量已归一化，而查询向量未归一化，则直接内积不等价于余弦。所以必须保持查询向量和库向量归一化一致。工程上我会在 embedding 管道中统一添加 normalization 步骤。
