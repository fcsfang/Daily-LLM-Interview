## 1. 今日面试真题精讲

### 题目 1：SGLang 的 RadixAttention 和 vLLM 的 PagedAttention 有什么核心区别？你更推荐用哪个做推理部署？

**来源依据：**
- 标题：字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 标题：AI大模型八股面试题，覆盖Agent、RAG、LLM等面试题（600张图解+25万字答案解析） - https://www.cnblogs.com/xiaolincoding/p/20534325 - 技术博客 - 中

**可信度：** 中（面经还原内容，但技术细节与公开论文/文档一致，可信度较高）

**考察点：** 推理优化、KV cache 管理、显存调度、框架选型

**面试口头版回答：**
这两个框架最根本的区别在于管理 KV cache 的方式。vLLM 的 PagedAttention 把 KV cache 像操作系统的虚拟内存那样分页管理，解决了显存碎片问题，让 batch 内请求的 KV cache 大小可以动态分配，大幅提升显存利用率和吞吐。SGLang 的 RadixAttention 则更进一步，它把不同请求之间的 KV cache 共享看作“树状前缀”，比如两条请求共用相同的系统 prompt 或几个历史轮次，RadixAttention 可以直接复用已经算好的 KV cache，避免重复计算。所以在多轮对话、共享 system prompt 的场景，SGLang 的加速更明显。我倾向的选择是：如果是单轮高并发、需要跑满 batch，vLLM 成熟度更高；如果是多轮对话、前缀共享频繁，SGLang 的前缀缓存加速更显著。

**深入理解版回答：**
首先讲 vLLM 的标准方案。传统推理服务中，每条请求独立申请一块连续的显存来存放 KV cache，这会遇到两个问题：第一，你无法提前知道一条请求的生成长度，只能按最大长度预分配，造成显存浪费；第二，不同长度的请求拼成 batch 时，短的请求占用超过实际需要的连续显存，形成了外部碎片。PagedAttention 把 KV cache 切分成固定大小的“page”（通常是 16 或 32 token 一块），用页表来映射逻辑 key-value 到物理显存地址。这样每个请求只分配实际需要的 page 数，而且 page 可以离散存放，碎片最小。这套机制其实是把操作系统内存分页的思想移植到了 GPU 显存管理上。

SGLang 的 RadixAttention 则看到另一个层面的优化空间：很多请求之间有大量的公共前缀——比如所有请求都带着相同的 system prompt，或者多轮对话中前面几轮是相同的。传统方案不管前缀是否相同，每次都要重新算一遍 attention。RadixAttention 把这些 KV cache 组织成一棵“基数树”，相同的前缀共享同一段 KV cache 节点。当新请求进来，它沿着树匹配前缀，匹配到的部分直接拿已有的 KV cache，只计算分支新增的部分。这使得 SGLang 在对话类场景、Few-shot 评测、多轮 Agent 场景下有显著的首次 token 延迟优势。

从实际工程角度，vLLM 生态更成熟、支持更多模型、社区更活跃，适合通用场景落地。SGLang 最近版本也在快速成长，它内置的 RadixAttention 配合 Stream 模式，特别适合共享前缀密集型的场景（比如多轮 Agent、批量评测）。选择哪一个，要看你线上最主要的请求模式是什么。

**常见追问与回答：**
- **追问：RadixAttention 在处理长上下文时，树的分支越来越多，会不会导致匹配查找变慢？**
  回答：会的。匹配复杂度取决于前缀树的深度和当前批次前缀的长度。SGLang 内部用 hash 优化前缀匹配，并且在 batch 内通过“前缀缓存可被多个请求共享”来减少总体计算量。根据 SGLang 论文，在小 batch 下效果显著，极端大 batch 下优势会减弱。

- **追问：这两种方案能否互补？比如在 vLLM 上面加一层前缀缓存？**
  回答：这是一个很好的方向。实际上 vLLM 也引入了 prefix caching 的实验性功能（v1 版本后）。但它的实现是在页表层面做前缀匹配复用，思路和 RadixAttention 不同，效果取决于实现细节。更常见的做法是用 Nginx 层做路由，把相同前缀的请求路由到同一组推理实例上来最大化共享。

**项目联系：**
如果我在项目中做企业智能客服 Agent 部署，我每轮对话都需要把固定的 System Prompt + 工具描述 + 历史 N 轮对话拼接成 prompt。这个场景下，不同客户虽然背景信息不同，但 system prompt 和工具描述是相同的。如果使用 vLLM 的 PagedAttention，只能解决显存碎片，但每次都要从头计算前几轮的 KV cache。切换到 SGLang，RadixAttention 直接复用公共前缀，我实测发现多轮对话场景下的推理延迟可以降低 20-40%——这个数据在项目汇报中也能直观体现优化效果。

**记忆关键词：**
PagedAttention | RadixAttention | 前缀缓存 | 显存分页 | 框架选型 | 推理加速

### 题目 2：你如何理解模型量化？量化的主要类型（如 W8A16、W4A16、A8W8）各自的特点和适用场景是什么？量化后模型精度下降了你怎么办？

**来源依据：**
- 标题：字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 标题：AI大模型八股面试题，覆盖Agent、RAG、LLM等面试题（600张图解+25万字答案解析） - https://www.cnblogs.com/xiaolincoding/p/20534325 - 技术博客 - 中

**可信度：** 中

**考察点：** 模型部署、量化认知、精度恢复、推理成本权衡

**面试口头版回答：**
模型量化就是把浮点数表示的参数映射到更低位宽的整数，核心目标是减少显存占用和加速计算。常见的类型有 W8A16（权重 8bit、激活 16bit），W4A16（权重 4bit、激活 16bit），以及 W8A8（权重和激活都 8bit，可以走 INT8 矩阵乘法加速）。W8A16 对精度影响最小，基本持平 FP16，显存减半，适合追求高精度且显卡显存不极端紧张的场景。W4A16 能把模型压到 5-6GB（比如 7B 模型），但精度下降明显一些，需要结合卡顿感知或量化感知训练来恢复精度。W8A8 全量化后可以调用 GPU 上的 INT8 Tensor Core 加速，吞吐提升明显，但对校准数据集和量化方式（如 smoothquant）要求较高。如果量化后精度下降，我不会盲目调整量化策略，而是先做分模块的精度定位，找出精度掉得最多的层，对这些层回退到 FP16，其余层保持量化，这叫混合精度量化，实践中用来平衡精度和速度。

**深入理解版回答：**
从工程角度看，量化具体分“量化粒度”（per-tensor / per-channel / group-wise）和“数据类型”（INT8/INT4/NF4）。我们通常说的 W8A16 是权重量化成本最低的方式，只减小模型加载时的显存，但计算时权重会反量化为 FP16 再运算，所以计算速度几乎不变，主要收益在显存。W4A16 更进一步压缩权重到 4bit，典型实现是 GPTQ，用二阶信息校正量化误差；AWQ 的策略是保留少数重要通道的精度，其他通道做量化，效果比 GPTQ 稍好一些且更硬体不敏感。W8A8 全量化是最激进的加速方案，权重和激活都量化到 8bit，计算走 INT8 GEMM，延迟收益非常明显，但问题在于激活的量化对量化校准数据非常敏感，因为激活的分布方差大，容易出现 outlier。一个成熟的方案是 SmoothQuant，把部分量化难度从激活转移到权重，让激活分布变平滑。

当量化后精度下降时，我的工程排查路径是：第一步，用几组典型 benchmark（MMLU、GSM8K 或自己业务评测集）对比量化前后的准确率。第二步，做逐层精度回溯——单独把某一层量化、其他层保持 FP16，看哪一层掉精度最严重。经验上，开头几层（Embedding 层附近）和末尾几层（LM Head）对量化最敏感，因为这些层的权重分布不均匀、outlier 多。第三步，对敏感层采用更高精度的量化粒度（per-channel 替换 per-tensor）或者直接回退到 FP16。这种做法叫混合精度量化，在实际部署中非常有效，既保留了 70-80% 的量化加速收益，又把精度损失控制到 1% 以内。如果业务对精度极度敏感，还会用量化感知训练来 fine-tune，把量化误差模拟进训练过程。

**常见追问与回答：**
- **追问：GPTQ 和 AWQ 的区别是什么？你在项目中怎么选择？**
  回答：GPTQ 是一种后训练量化方法，通过在线化理论中引入 Hessian 矩阵来补偿量化误差，适合一次量化后冻结。AWQ 强调保留权重的少数重要通道，让它们保持 FP16，其余通道量化，这种做法更轻量且硬件兼容性更好。如果是要做一次性的模型压缩、精度要求高，我倾向 GPTQ；如果模型需要频繁换版本、不想反复跑校准，AWQ 更省事。

- **追问：Activation 量化为什么比 Weight 量化难做？**
  回答：激活值的分布是动态的，不同输入产生不同的分布，而且激活值的 outlier（极端值）比权重多得多。权重在训练后基本稳定，量化时可以离线校准到最优映射；激活必须在线完成量化，如果 outlier 没处理好，量化误差会逐层放大。所以 SmoothQuant 的工作就是找缩放因子，把 outlier 从激活转移到权重上来。

**项目联系：**
在我的 Agent 推理服务中，低延迟是核心指标，但 8 卡 A100 80G 的推理成本高。我先用 W8A16 把模型从 80G 压到 40G 左右，延时基本不变，显存省了一半，这样单卡上能跑更大的 batch。如果要求进一步降低单卡成本、上 W4A16，我必须先在自己的业务测试集上跑逐层精度回归，发现前 2 层和最后 1 层的输出偏差超过 5%，我就把这 3 层保持 FP16，其他层做 W4A16，结果精度损失从 3% 降到 0.5%，内存只多用 1GB。

**记忆关键词：**
模型量化 | W8A16 | W4A16 | W8A8 | 混合精度量化 | 量化感知训练

### 题目 3：什么是 Continuous Batching？它是怎么提升推理吞吐的？在实际服务中你如何配置 batch size 和调度策略？

**来源依据：**
- 标题：字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 标题：AI大模型八股面试题，覆盖Agent、RAG、LLM等面试题（600张图解+25万字答案解析） - https://www.cnblogs.com/xiaolincoding/p/20534325 - 技术博客 - 中

**可信度：** 中

**考察点：** 推理调度、吞吐优化、动态 batch、服务稳定性

**面试口头版回答：**
Continuous Batching（连续批处理）是解决传统静态 batch 调度缺陷的关键优化。在静态 batch 模式下，只有当一个 batch 中的所有请求都生成完，才能释放资源、编排下个 batch，这会导致 GPU 在生成长度不同的请求时会浪费大量时间等待最慢的请求。Continuous Batching 的核心思想是每生成一个 token 就进行一次 batch 调度：一旦某个请求生成结束（例如遇到 EOS 或达到最大长度），立刻把这个请求踢出 batch，同时从等待队列拉进一个新请求，填入空出来的显存 slot。这样 GPU 的计算单元每时每刻都在处理有效的 token，避免了“等待空闲”的显存浪费和计算空闲。本质上，这会带来“吞吐和延迟的近似线性提升”，尤其在请求长度标准差较大的场景下提升极为明显。

**深入理解版回答：**
我们先拆解传统 batch 的问题。假设同时对 8 个请求做 batch，它们有的只生成 30 个 token，有的要生成 2000 个 token。在静态 batch 里，这 8 个请求会被“绑定”在一起，GPU 要等最后那个 2000 token 的请求结束后，才能回收显存、开始下一批。这意味着前面 7 个请求空占显存，GPU 在大量做不必要的 padding 和无效计算。这在长尾分布的生产流量中非常普遍。Continuous Batching 也叫 iteration-level batching，最早在 ORCA 论文中被系统阐述，现在已经是所有主流推理框架的标配：vLLM、TGI、TensorRT-LLM 都实现了类似的机制。

具体实现上，框架会维护一个请求池，每个请求占若干个 page（PagedAttention 的显存页）。每轮 forward，调度器只处理当前仍处于“生成中”状态的请求。一旦某个请求生成了 EOS token，调度器立即标记这个请求完成，回收它的 page，并从等待队列拿一个新请求填入空白页。这个过程中，batch 的组成是动态调整的，每一轮 iteration 的 batch size 可能都不同，但 GPU 的利用率始终保持高位。

在实际部署里，我不会盲目调大 batch size。上限受两件事约束：一是显存容量（每请求的 KV cache 大小）；二是个别请求的延迟 TCO。如果 batch 太大，长尾请求的排队时间和计算时间都会增长，导致高延迟尾部恶化。我一般会给关键业务的请求设置一个“最大批处理等待时间”——比如超过 30ms 还没被调度进 batch，则立即用最小 batch 抢发一次，保证响应时间的 P99。同时在框架层面添加“分桶调度”：对短文本请求、长文本请求各自建立独立的 Continuous Batching 队列，减少长短混排造成的 GPU 计算碎片。

**常见追问与回答：**
- **追问：Continuous Batching 和 Dynamic Batching（比如在 TensorFlow Serving 中常见）有什么区别？**
  回答：Dynamic Batching 是等一个 batch 里的请求全部准备完毕后，一次性送入 GPU 前向。Continuous Batching 不是在 prefill 阶段就定死 batch，而是在 decode 阶段每个 token 步都动态调整 batch 成员。所以 Dynamic Batching 解决了“排队等待”问题，但没解决“batch 内不同生成长度拖累”的问题，而 Continuous Batching 两者都解决了。

- **追问：在你的项目中，Continuous Batching 带来的吞吐提升具体是多少？**
  回答：我服务中有大量用户多轮对话场景，请求生成长度差异很大（短到几个 token，长到 1k+ token）。切换 static batch 到 Continuous Batching 后，单卡吞吐从 200 请求/秒提升到 550 请求/秒，提升超过 2 倍，且 P99 延迟从 15s 降到 4s。主要收益来自于 GPU 计算不再被长请求“阻塞”住。

**项目联系：**
我之前的 RAG 问答 Agent 需要在 GPU 上同时服务多个部门，不同部门的请求来源、长度分布完全不同。为了让 GPU 利用率最大化，我使用了 vLLM 的 Continuous Batching，并且设置了分桶队列：对于检索质量高、回答短的知识库问答请求，用高频小 batch；对需要长推理的复杂 Agent 任务，用低频但大 batch 的方式。这样既保证了短请求的实时响应，又兼顾了长请求的吞吐。

**记忆关键词：**
Continuous Batching | 动态调度 | 显存利用率 | P99 延迟 | 分桶调度 | 推理框架

- **可转化面试题 1：** 在项目实践中，你如何决定是否使用投机解码（Speculative Decoding）？
  **面试口头版答案：**
  投机解码的核心思路是用一个更快的草稿模型先生成 K 个候选 token（比如用一个小型 LLM 甚至更轻量的 n-gram），然后让目标大模型一次验证这 K 个 token 的接受概率。如果大部分被接受，实际每步 decode 算力等价于草稿模型的加速比。我选择是否使用投机解码主要看三点：第一，业务场景是否对“首 token 延迟”敏感——因为投机解码的 prefill（草稿模型一遍生成 K 个候选）会增加首 token 的延迟；但随后 token 间延迟会大幅降低。第二，草稿模型和目标模型的“接受率”能否维持高位，如果草稿模型生成的候选经常被拒绝，那还不如直接走标准自回归。我的经验是：对生成长度长、且生成内容确定性较高的场景（如代码补全、结构化输出），投机解码收益大；对生成内容多样、难以预测的开放式对话，收益有限。第三，草稿模型的额外显存成本能否承受。如果在设备端做投机解码（如手机端），显存压力大，可能得不偿失。

- **可转化面试题 2：** 前缀复用（Prefix Caching）在实际服务中如何确保缓存命中？
  **面试口头版答案：**
  前缀复用的核心挑战是：怎么判断两个 prompt 有相同的前缀？常用的方法是“增量哈希”，即对 prompt 的 token ID 序列按固定窗口（如 256 token）计算哈希值，用哈希表存储 KV cache。新请求的 prompt 从最左侧起，看有多少连续 token window 的哈希已在缓存里。命中后，只计算后续多出来的 token 的 KV cache。在实现时，我通常会配合“前缀树”或“Trie 树”加速匹配，并且给缓存加 TTL（time-to-live）或 LRU 淘汰策略，避免缓存占用过多显存。在项目实践中，多轮对话 Agent 场景下，80% 的请求都带有相同的 System Prompt，前缀复用后 prefill 时间可以缩短 60% 以上。

  **面试口头版答案：**
  Flash Attention 的核心贡献不是减少浮点计算量，而是减少 GPU 高带宽内存（HBM）和片上 SRAM 之间的读写次数。传统注意力计算中，QK^T 的中间结果（S = QK^T，P = softmax(S)）需要完整写到 HBM 中，然后再读回做下一步 softmax 和 PV 乘积，这个过程产生大量的 IO。Flash Attention 通过 tiling（分块计算）和 recomputation（重计算前向的中间结果），把 softmax 拆成在片上分段近似，避免写回 HBM。它在计算时把 Q、K、V 切分成小块，每块在 SRAM 里做局部 softmax，最后再合并，全程只在结束时结果写到 HBM。这使推理和训练的显存消耗从 O(n^2) 降到近似 O(n)，实际速度在长序列上也大幅提升。在我部署 32K 甚至更长上下文的模型时，FlaxAttention 是必选项，否则显存和延迟都无法接受。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** SGLang 的 RadixAttention 和 vLLM 的 PagedAttention 本质区别是什么？
- **A：** PagedAttention 解决显存碎片问题，通过分页管理 KV cache 实现动态 batch；RadixAttention 解决前缀重复计算问题，通过树形缓存复用相同前缀的 KV cache。一个是显存调度，一个是计算复用。
- **一句话记忆法：** PagedAttention 管“管子”——显存不浪费；RadixAttention 管“树根”——前缀不重算。
- **复习标签：** #模型部署 #推理优化 #SGLang #vLLM

### 卡片 2
- **Q：** 你如何选择量化粒度——对什么层用 per-tensor，什么层用 per-channel？
- **A：** Per-channel 比 per-tensor 精度高但实现复杂。经验上：Embedding 层、LM Head、以及权重分布方差大的 linear 层（比如 attention output）适合 per-channel 或 group-wise 量化；中间层的权重分布大都均匀，per-tensor 就够了。先做逐层精度回溯，对精度损大的层增加量化粒度。
- **一句话记忆法：** 病重的层下猛药——per-channel，轻症层下温和药——per-tensor。
- **复习标签：** #模型部署 #量化 #精度恢复

### 卡片 3
- **Q：** 在推理服务中，Continuous Batching 怎么做到“每秒都在用 GPU 算有效 token”？
- **A：** 每生成一个 token，调度器就检查提前完成请求的 EOS，立即回收其显存并拉入新请求。batch 的组成每一 iteration 都在变，GPU 从不空闲等待长尾请求。这消灭了传统静态 batch 的“batch 内最慢请求拖累所有请求”的问题。
- **一句话记忆法：** 谁跑完谁下车，新客立刻上车，不让 GPU 停工。
- **复习标签：** #模型部署 #推理优化 #吞吐

### 卡片 4
- **Q：** 在连续批处理场景下，如何控制单个请求的尾部延迟（P99）？
- **A：** 设置“最大批处理等待时间”阈值，超过这个时间（如 30ms）该请求强制退队并用小 batch 先发。另外可以按请求长短分桶：短文本队列用小 batch 高频调度，长文本队列用大 batch 低频调度，避免长短混排拉高短请求的延迟。
- **一句话记忆法：** 不让你空等超过 30ms，再晚我就“单飞”发车。
- **复习标签：** #模型部署 #吞吐 #延迟

### 卡片 5
- **Q：** 投机解码（Speculative Decoding）的核心假设是什么？何时收益最大？
- **A：** 核心假设是草稿模型生成的候选 token 能被目标大模型高比例接受。收益最大的场景是：生成内容确定性高（如代码、结构化 JSON）、草稿模型和目标模型的重合度好、且生成长度较长。首 token 延迟会有增加，因为草稿需要先跑一遍生成候选。
- **一句话记忆法：** 小模型“打草稿”，大模型“速审通过”，高风险场景就是拒绝率太高。
- **复习标签：** #模型部署 #推理优化 #投机解码

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** Continuous Batching 要求每个请求的 KV cache 能够在推理过程中动态加入和退出 batch，这就带来一个问题：当多个请求共享相同的前缀（系统 prompt 或历史对话轮次）时，它们的 KV cache 之间如何隔离？你如何设计一个支持前缀共享且不泄漏跨用户数据的安全隔离机制？

**面试口头版参考答案：**
这道题考察我对“性能优化”和“安全隔离”的权衡能力。首先，前缀共享的核心是用一个 Trie 树或哈希表来匹配请求的公共前缀 token，共享对应的 KV cache 物理页。但共享会带来一个严重问题：如果请求 A 和请求 B 共享了同一个 system prompt 的 KV cache 物理页，那么 A 不能因为共享就访问到 B 后续生成的私有 KV cache。这就需要机制来保证两点：第一，前缀缓存只覆盖完全确定的、多用户一致的、不包含用户私有信息的 prompt 部分（例如固定 system prompt 和工具函数描述）；第二，每个请求从属于自己的 private prompt 开始，生成属于自己的私有 KV cache 路径。在实现上，我的做法是：一个 Trie 树节点代表一个固定 prefix token，节点有一个引用计数，当引用降为零时释放显存。每个请求维护一个“共享前缀长度 L”，长度为 L 的 token 走共享路径，后续 L+1 及之后的 token 写私有路径。L+1 对应的物理页是新分配的，不会和任何其他请求共享。这样前缀共享能让前 L 个 token 的推理加速，但后续的隐私数据完全隔离。总结：性能优化的前提是不牺牲安全隔离，我的方案在工程上是“共享系统前缀 + 私有用户后缀”。

**深入理解版参考答案：**
这个问题触及了推理框架的高级设计：如何在显存复用和租户隔离之间做精细平衡。

首先，连续的 KV cache 共享机制一般有两种实现方式：一是 vLLM 的 prefix caching 方式（基于 page 的 matching）；二是 SGLang 基于 RadixAttention 的方式。两者都面临同一个问题：如果多个 tenant 的请求具有完全相同的前缀 token（比如相同的系统 prompt），我可以让它们共享这一段的 KV cache，这样可以节省显存并减少重复计算。但共享显存页意味着在物理上，不同用户的中间计算结果是放在同一块物理显存上的。为了防止用户 A 通过某种 side-channel（比如共享页写后读冲突）获取用户 B 的私有推理信息，我必须做到两点：1）共享只发生在“标准化前缀”上——这些前缀必须是：完全固定、不包含用户标识、不包含任何可变 token 的内容。例如系统内置的角色设定、工具函数定义、few-shot 示例——这些模型参数级的 token 序列。2）一旦请求偏离共享前缀，哪怕只多一个 token，就必须走私有页路径。这要求 Trie 树在匹配时，准确返回这个请求的“共享前缀长度 L”——对于没有公共前缀的 token 序列，L 为 0，完全不共享。对于 L 个共享 token，后续 L+1 及之后的 token 必须写私有 page，不能再使用公共路径。

从工程实现角度，我会考虑这样一个层次结构：最底层是操作系统级 page 管理（PagedAttention 本质就是这个），它负责复用显存。在它之上，我封装一层“prefix sharing manager”，它维护一个树状结构，每个节点存储 token ID 和共享状态计数。当一个前缀节点同时被多个请求引用时，共享计数 > 1。当共享计数降到 0 时才能释放物理显存页。同时在调度层面：同一个 batch 内，不同请求可能经历了不同的“共享-私有”分界点，也就是说，请求 A 可能共享 L_a 个 token、然后私有；请求 B 共享 L_b 个 token。调度器需要灵活对齐：在计算 shared portion 时，L_a 和 L_b 的匹配前缀部分可以复用；在私有 portion，批次里的各请求长度不一致，我仍需要做 Continuous Batching 的 dynamic scheduling。

最后还要考虑一个生产问题：显存换出（swap）时的隔离。如果请求 A 的私有部分被 swap 到 CPU 内存，请求 B 不能因为 A 的 swap 导致自己的私有页被连带 swap。共享前缀和私有后缀必须分开管理。总结来说，KV cache 共享和安全隔离不是不可调和的矛盾：只要做到“共享点确定、私有页隔离、调度器按 token 位置分界”，就能在利用前缀缓存加速的同时，保证多租户安全。

**可能追问与简短回答：**
- **追问：这个前缀共享+L分界策略，在 Multi-Head Attention 中是否有额外复杂度？**
  回答：有的，因为不同 head 的 KV cache 在物理上是分开的，但共享前缀在 token 级别，意味着每个 head 的共享 prefix KV cache 页需要独立存储。我可以在 head 间独立管理：每个 head 维护一个共享页表，但共享深度对所有 head 是一致的。这样每个 head 的共享部分可以复用，不需要给每个 head 单独计算共享前缀长度。

- **追问：如果一个请求的私有部分因为长度不足而需要提前“截断”，共享前缀的缓存会受影响吗？**
  回答：不受影响。请求的私有部分生成结束或被截断时，只需释放该请求的私有 page 即可。共享前缀页的引用计数减 1 还是大于 0（因为还有其他请求在共享），所以不会被释放。这跟 Continuous Batching 的思想是顺应的：每个请求独立管理自己的生命周期，共享页通过引用计数完成回收。
