### 题目 1：LoRA 微调的原理是什么？秩 r 的选择会对模型表现产生什么影响？

**来源依据：**
- 资料标题：AgentGuide/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 需进一步验证。本题提取自该资料中“LoRA微调的原理是什么？秩 r 的选择会对模型表现产生什么影响？”问题。
- 资料标题：AI大模型八股面试题 ，覆盖Agent、RAG、LLM等面试题 - 技术博客 - 需进一步验证。

**可信度：** 中（来源于社区面经整理，需进一步验证）

**考察点：** PEFT 原理、LoRA 超参数调优、工程落地经验

**面试口头版回答：**
LoRA 是参数高效微调，核心思想是对预训练权重矩阵做低秩分解，冻结原始权重，只训练两个小的低秩矩阵 A 和 B，最终输出为 Wx + BAx。秩 r 决定了这个低秩矩阵的维度，也决定了可训练参数量和表达能力。r 太小（比如 1、2）模型表达能力不足，微调效果有限；r 太大（比如 128、256）虽然表达能力增强，但失去了参数高效的优势，甚至可能破坏预训练知识。通常在项目中我会先取 r=8 或 16 做快速实验，观察验证集精度变化，然后根据 Loss 曲线和下游任务复杂度动态调整。在训练 QLoRA 时我会同时监控 NF4+FP16 的精度影响，确保量化带来的噪声不会因为 r 太小而无法被低秩矩阵学习。工程上，我会配合 alpha 参数做缩放，一般 alpha 设为 r 的 2 倍，避免梯度爆炸。最后我会对比不同 r 值下的显存占用和推理速度，找到性价比最高的点。

**深入理解版回答：**
LoRA 的核心动机是解决全参数微调的成本问题。全参数微调需要更新整个模型的所有参数，以 7B 模型为例，全量微调需要保存梯度、优化器状态和参数副本，显存开销通常在 100GB+，远超单卡消费级 GPU 的能力。LoRA 的解决方案是：对于 Transformer 中的注意力矩阵（W_q, W_k, W_v, W_o），引入一个低秩分解 W' = W + BA，其中 B 和 A 的维度分别是 d x r 和 r x d。训练时 W 被冻结，只更新 B 和 A。r 的选择直接影响模型的行为。如果 r 取 1，相当于每一层只引入 d 个参数，表达空间极小，模型只能做非常细微的调整；如果 r 取 256，则引入了 256*d 个可训练参数，相当于在每层增加了一个小型 MLP，表达能力大幅提升，但同时显存开销也会增加。实际上，经验法则是对简单任务（分类、情感分析）取 r=8，对复杂任务（代码生成、角色扮演）取 r=16 或 32。在工程实践中，我还会关注 LoRA 的缩放因子 alpha，默认 alpha = 2r。如果梯度不稳定（Loss 剧烈震荡），我会增大 alpha 到 4r 或 8r，让 LoRA 的更新量相对较小，降低对预训练权重的干扰。如果 Loss 下降太慢，我会减小 alpha 到 1r 甚至 0.5r，让更新量变大加速收敛。在部署阶段，LoRA 的合并操作也很关键：我们不能在推理时每次都计算 BAx，而是提前把 B 和 A 的乘积合并到原始权重 W 中（merge 操作），这样不会增加推理延迟。需要注意的是，如果同时使用多个 LoRA 适配器（比如多租户场景），就不能做 merge 操作，而是要在推理时动态加载，此时 b 的维度会直接影响推理显存和速度。

**常见追问与回答：**

- 追问：为什么不直接把 W_q 和 W_o 都训练，而只选 W_q？答：注意力机制中 Q 和 V 的权重对任务适配最敏感，Q 影响查询向量的分布，V 影响输出向量的内容。训练 W_q 和 W_o 通常能取得最佳效果，训练 W_k 对效果提升有限但会增加显存。所以实践中通常只训练 W_q 和 W_o 的 LoRA。
- 追问：LoRA 微调后如何合并？如果不合并，推理速度会变慢多少？答：合并操作是 W_new = W_old + (B @ A) * alpha / r。如果不合并，推理时需要额外计算一次矩阵乘法 BAx，这部分计算量不大，但对延迟影响明显。实测在 7B 模型上，不合并的情况延迟增加约 10%-15%。
- 追问：什么时候应该用 LoRA 而不是全参数微调？答：算力限制（1-2 张 24GB 显卡）和快速迭代场景（需要几小时而不是几天）首选 LoRA。此外，多任务场景下，LoRA 可以在同一个基座上维护多个适配器，支持热切换，这是全参数微调难以做到的。

**项目联系：**
假设我在做一个面向企业客服场景的 SFT 微调项目，使用 Qwen-7B 作为基座。全参数微调需要 4 张 A100 并训练 3 天，我用 LoRA r=16 在单张 A100 上 4 小时就能完成一轮。更重要的是，我可以同时训练 3 个 LoRA 适配器，分别对应客服、销售、售后三个场景，部署时通过路由层动态切换，不重启模型。在迭代过程中，当发现客服场景的 Loss 不降时，我通过增大 alpha 加速收敛，同时将 r 从 16 提高到 32，最终上线 F1 提升了 3.5 个百分点。

**记忆关键词：** 低秩分解、秩 r 选择、alpha缩放、merge 操作、多适配器切换

### 题目 2：QLoRA 为什么选择 NF4 和 FP16 这组组合，而不是别的组合？NF4 的分布拟合逻辑是什么？

**来源依据：**
- 资料标题：AgentGuide/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 需进一步验证。本题提取自该资料案例9的面试问题：“QLoRA为什么选的是NF4和FP16这组组合？你能说说NF4的分布拟合逻辑吗”。

**可信度：** 中（来源于社区面经整理，需进一步验证）

**考察点：** QLoRA 量化原理、NF4 分布拟合、量化组合选择

**面试口头版回答：**
QLoRA 选择 NF4 作为量化格式是因为模型权重通常是正态分布的，NF4（Normal Float 4）专为适配正态分布而设计。普通的 INT4 是均匀量化，对于正态分布数据，中间密集的区域分配了过多量化级别，而两端稀疏的区域却分配了很少级别，导致信息损失。NF4 则采用非均匀量化，在数据分布密集的均值附近分配更精细的量化级别，在两端分布稀疏的区域分配较粗的级别，这样整体信息损失最小。NF4 和 FP16 的组合是权衡了精度和计算效率的结果。具体来说，QLoRA 使用 NF4 量化基础模型（权重），显存消耗大幅降低；但 LoRA 的适配器矩阵 B 和 A 使用 FP16，因为 LoRA 本身参数很少，提升其精度不影响显存瓶颈，却可以保证微调过程的数值稳定性。NF4+FP16 在保持接近全精度效果的同时，显存可减少近 4 倍，是目前性价比最高的选择。理论上可以用 INT4+FP16 或 NF4+INT8，但实验和论文（QLoRA 原论文 Table 5）表明 NF4+FP16 在大部分任务上效果最好，且训练稳定性高。

**深入理解版回答：**
QLoRA 的量化选择涉及三个关键设计。第一，为什么选择 4-bit 而不是 8-bit？4-bit 量化可以把 7B 模型的权重从 14GB（FP16）降至 3.5GB（INT4/NF4），使单卡 24GB 消费级 GPU 上能够训练 32B 模型。8-bit 虽然精度损失小，但显存节省只有约 2 倍，从经济性上看对低成本微调吸引力不足。第二，为什么选 NF4 而不是 INT4？NF4 的全称是 Normal Float 4，即正态浮点数。它的设计思路是：在 [−1, 1] 区间内分配 16 个离散量化级别，但分配密度呈正态分布。具体表示时，NF4 采用了“量化-反量化”两步法：先计算权重的均值和标准差，对其做归一化，然后用归一化后的值去查找 NF4 码本中的最近值，最后反归一化恢复。这个过程比 INT4 的线性量化多了一步归一化计算，但带来了更低的量化误差。论文实验显示，在相同比特数下，NF4 的困惑度（PPL）比 INT4 低约 0.5-1.0。第三，为什么 LoRA 适配器用 FP16？LoRA 适配器 B 和 A 的总参数量通常只有基础模型的 0.1%-1%，如果也量化到 4-bit，显存节省有限（大约几十 MB），但会引入 4-bit 的精度损失，最终合并时反而降低效果。因此保留 FP16 是最佳性价比。此外，QLoRA 还引入了“双重量化”（Double Quantization）技术，将量化常数（本来每个块一个 FP32 值）再量化到 8-bit，进一步节省显存。实战中，当我用 QLoRA r=64 训练 LLaMA-13B 时，单张 RTX 4090 即可完成，验证集 F1 相比全精度 LoRA 仅下降 0.3 个百分点。

**常见追问与回答：**

- 追问：能不能用 NF4 同时量化权重和激活值，来进一步压缩显存？答：激活值通常不是正态分布，不能用 NF4 直接套用。需要针对激活值的分布特性（如高斯分布或均匀分布）设计专门的量化策略，比如 SmoothQuant 就提出了一种确保激活值分布相对均衡的量化方案。
- 追问：QLoRA 中的双重量化是什么？减少了多少显存？答：双重量化是指对原始量化过程中的缩放因子（Scale）再做一次量化。假设每 64 个权重共享一个 FP32 缩放因子，这个因子本身占用 4 字节，如果将其量化到 8-bit，则存储缩小 4 倍。对于 7B 模型，双重量化可以额外节省约 0.5GB 显存。
- 追问：在你的项目中，使用 QLoRA 后模型效果下降通常出现在哪些场景？答：主要是两类场景：一是代码生成或数学推理任务，这类任务对数值精度敏感，量化引入的噪声会导致生成结果出错；二是需要细粒度实体识别的任务（如人名、术语），因为量化会模糊权重的区分能力。遇见这种情况我会换用 8-bit 量化或直接使用 LoRA 全精度。

**项目联系：**
我在一个知识库问答项目中需要微调 Qwen-7B 模型。受限于开发环境只有 2 张 RTX 4090，我选择 QLoRA NF4+FP16 方案，r=16，batch_size=4，显存占用约 14GB。训练完成后对比全精度 LoRA（需要 40GB 显存，只能在 A100 上跑）的效果，发现两个模型在 RAG 检索后的回答质量（准确率和忠实度）上几乎一致，差异不到 0.5%。这让我意识到对于 RAG 场景，QLoRA 足以满足要求，后续投入到生产环境的成本也大幅降低。

**记忆关键词：** QLoRA、NF4 正态浮点数、分布拟合、双重量化、NF4+FP16 组合

### 题目 3：微调时，训练数据是怎么构建的？如何保证样本的多样性和质量？

**来源依据：**
- 资料标题：AgentGuide/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 需进一步验证。本题提取自该资料案例1的一面题6：“微调时的训练数据是怎么构建的？如何保证样本多样性和质量？”
- 资料标题：字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析... - 面经或社媒 - 需进一步验证。

**可信度：** 中（来源于社区面经整理，需进一步验证）

**考察点：** SFT 数据工程、数据治理、质量控制

**面试口头版回答：**
SFT 数据的构建可以分为数据采集、清洗、格式化和质量检查四个步骤。第一步采集：从业务日志中收集真实用户对话，同时从外部高质资源（领域论文、行业报告）中整理问答对。第二步清洗：去掉 PII（个人身份信息）、去除重复、去除低质量回复（如拒答、错误回答）。第三步格式化：统一成 Instruction-Input-Output 格式，对于多轮场景按照 ChatML 标准标注。第四步质量检查：我通常采用“规则 + 模型”混合策略。规则层面检查最小文本长度、重复率、特殊字符比例等；模型层面使用强模型（如 GPT-4o）按照 5 维度（有用性、真实性、无害性、可读性、格式正确性）打分，低于阈值的丢弃。保证多样性的方法包括：对任务类型打标签（分类、抽取、改写、生成等），控制各类任务比例；使用聚类算法对用户 query 做 embedding 聚类，确保每类 query 都有样本；统计关键词覆盖度，确保领域术语被充分覆盖。最后我还会构造 hard negative（比如故意把反事实例子加进去），增强模型的鲁棒性。

**深入理解版回答：**
SFT 数据构造是整个微调工程中最重要的环节之一，因为“垃圾进垃圾出”这个铁律在 LLM 微调中尤为突出。一个高质量的数据集应该具备以下特性：**正确性**（回复正确且符合事实）、**多样性**（覆盖任务类型、领域、句式、难度）、**平衡性**（正负样本、来源渠道、场景分布）、**纯净性**（无噪声、无冗余、无矛盾）。在实际项目中，我先从业务日志中筛选出高赞样本（用户反馈为 helpful 的）作为种子集，这批数据量虽少但质量极高。然后我用种子集作为 few-shot 示例，用 GPT-4o 生成扩展数据。接着我要求人工标注员对扩展数据做 3 轮筛选：第一轮删除离线答案，第二轮删除不安全的，第三轮修正格式错误。整个过程我会监控几个指标：自动评分曲线（每批数据的平均质量分）、任务类型分布是否均衡（发现某一类偏多则裁剪，偏少则补充）、领域术语覆盖率（通过词典比对，发现缺失时补充相关 seed 样本）。多样性的工程实现上，我用的是 k-means + 最大最小距离采样法：对所有 query 做 embedding，聚类后从每个簇中按比例抽取样本，保证每个簇都有代表。对于生成类任务，我还会按输出长度分层（短回答、中回答、长回答），避免模型只擅长某种长度。我还会特别留意构造 hard negative 样本。比如对于 RAG 场景，我会构造一些“从看似的上下文却推导不出答案”的样本，训练模型学会拒答。这些 hard negative 对提升模型的安全性和边界意识非常有帮助。

**常见追问与回答：**

- 追问：如何自动化检测数据质量，避免引入噪声？答：可以用一个小模型（如 BERT-score）对生成的回答和原始背景文档做忠实度打分，低于 0.7 的丢弃。还可以用 k-fold 验签：将数据随机分成 5 份，每次用 4 份训练 1 份验证，观察 Loss 异常的样本（Loss 特别大的往往有问题），手动复核后删除或修正。
- 追问：如果你的数据分布与推理时实际用户分布不一致（如数据中对话偏长，但用户偏短），怎么办？答：在数据构造阶段就引入知识库之外的分布模拟。比如我日常收集用户日志，统计出实际对话的平均长度、句式复杂度、实体分布，然后调整数据构造比例，使构造数据的分布尽量拟合真实分布。上线后持续监控分布变化，发现漂移后增量补充新分布的数据。
- 追问：构造的数据样本量级大概需要多少？答：对于 SFT，经验法则是在 10000-50000 条左右效果最优。少于 5000 条时模型可能学不到什么新行为；超过 50000 条时边际收益递减，并且容易过拟合。数量和质量之间我优先质量，10000 条高质量数据通常比 50000 条噪声数据效果好。

**项目联系：**
在一个金融领域 Agent 项目中，我需要微调一个对话模型来回答产品条款问题。我先从客服日志中抽取了约 8000 条真实会话；然后用 GPT-4o 对每条会话做改写（去掉 PII、修正错误回复），产生 20000 条扩展数据；最后用规则 + GPT-4o 做质量过滤，最终保留 15000 条。然后构造了 3000 条 hard negative（比如用户问“这款产品保不保心脏病”，答案要回答“条款未明确该情况，建议咨询人工”而不是胡说）。上线后模型拒答准确率从初版的 78% 提升至 95%，用户反馈改善明显。

**记忆关键词：** 数据构造三步法、多样性控制、质量过滤、hard negative、分布拟合

### 题目 4：RAG 和微调（SFT）的核心区别是什么？能不能一起用？

**来源依据：**
- 资料标题：ai-agents-from-zero/AI智能体与大模型应用开发面试题库.md - GitHub - 面经或社媒 - 需进一步验证。本题提取自该资料中 Q1-1 题：“RAG 和微调（SFT）的核心区别是什么？能不能一起用？”

**可信度：** 中（来源于社区整理的面试题库，需进一步验证）

**考察点：** 技术选型、RAG vs 微调、混合方案设计

**面试口头版回答：**
RAG 和微调本质是解决不同问题的方案。RAG 解决的是“知识从哪里来”——通过动态检索外部知识库来提供事实性信息，本质是非参数记忆。它适合处理时效性强、更新频繁、需要可溯源的知识。微调解决的是“模型应该怎么表现”——通过修改模型参数来优化行为、风格、输出格式。它适合固定行为模式但不需要频繁更新知识的场景。两者可以一起用，而且很多企业项目就是这么落地的：用 RAG 提供最新的领域知识，用微调或结构化输出来保证回答的格式和风格。比如在客服项目中，产品政策靠 RAG 保持最新，回复的格式和字段输出靠微调或结构化输出保证稳定。我通常建议采用 RAG-First 策略：先跑通 RAG 验证业务可行性，如果发现模型在语感、格式、逻辑上不达标，再用微调打补丁。

**深入理解版回答：**
这个问题的核心是理解两种技术的本质差异。RAG 是不改模型参数的，它在运行时动态从外部向量库检索相关 chunk，然后与用户 query 一起构建 prompt，让模型基于这些信息生成答案。这意味着你可以随时更新知识库（只需要重新索引），模型本身不需要重新训练。这是 RAG 最大的优势：知识插拔的成本几乎是零。微调则需要收集标注数据，经过多轮训练，模型参数被永久修改。一旦上线后知识更新，必须重新训练模型。但微调也有不可替代的优势：它可以从根本上改变模型的行为模式，比如让模型学会输出特定格式的 JSON、使用特定语气、避免某些词汇。在实际项目中，我经常遇到决策困境。比如用户想要一个“能回答公司最新政策”的智能助手。我第一反应是上 RAG：只需梳理 FAQ、政策文档，做切分、embedding，一周内就能上线。上线后如果发现模型回答太啰嗦或格式不对（比如不输出结构化字段），我再用 LoRA 微调做一个“输出风格适配”的模型，跑完训练后替换掉原始 LLM，这样既保持了知识时效性，又优化了输出风格。这里的关键思路是：**RAG 负责“知识的厚度”，微调负责“行为的深度”**，两者互补而非替代。另外需要注意，在预算有限的情况下，我建议先验证 RAG，因为它的启动成本最低（不需要训练，只需要推理）。只有 RAG 无法满足要求时（比如模型总是不遵守格式、或需要特定领域的说话方式），才投入微调资源。

**常见追问与回答：**

- 追问：如果微调后的模型在处理 RAG 提供的检索内容时仍然表现不佳（比如忽略上下文、或自己脑补信息），怎么排查？答：先检查微调数据中是否包含了足够的“基于上下文回答”的样本。如果没有，需要补充。然后排查微调模型对新注入的检索上下文是否敏感，可以做一个消融实验：分别给微调模型注入相关与无关的检索上下文，观察回答变化。发现变化微弱的说明模型已经学成了参数记忆模式，需要降低学习率或者减少微调步数，保留更多预训练的上下文跟随能力。
- 追问：你觉得未来 RAG 和微调的边界会越来越模糊吗？答：是的。现在已经有研究（如 In-Context RALM）直接在微调阶段引入了检索增强信号，训练模型从上下文 token 中推断是否需要检索。未来很可能出现“端到端”的模型，既能动态检索，又能根据检索结果自适应调整输出行为。
- 追问：在哪些场景下你一定会优先选择微调而不是 RAG？答：对输出格式要求极其严格（如医疗诊断报告必须输出固定字段）、需要学习大量领域术语的发音模式（如专业语音合成）或生成内容的创意性远超事实性（如故事生成）。这些场景 RAG 无法提供稳定的行为控制。

**项目联系：**
我在一个企业法律咨询项目中的实践很典型：我先用 RAG 接入最新的法律条文和判例，回答准确率提升了 30%。但用户反馈回答太像在念法条，不够自然。我于是用 LoRA 微调了一个“律师助理风格”的模型，学习如何将法律语言转化为通俗易懂的解答。上线后用户满意度提升了 50%。这个案例完美体现了 RAG 和微调的互补关系。

**记忆关键词：** RAG 解决知识、微调解决行为、RAG-First、互补使用、知识厚度 vs 行为深度

### 题目 5：训练 LoRA 模型时，你是如何选择冻结层的？依据是什么？这和在 QLoRA 中量化哪些层有关联吗？

**来源依据：**
- 资料标题：AgentGuide/docs/04-interview/12-company-interview-cases.md - 面经或社媒 - 需进一步验证。本题提取自该资料案例1的一面题8：“训练 LoRA 模型时，你是如何选择冻结层的？依据是什么？”

**可信度：** 中（来源于社区面经整理，需进一步验证）

**考察点：** LoRA 工程细节、冻结层策略、量化与微调的关联

**面试口头版回答：**
在 LoRA 微调中，选择冻结层是指决定哪些参数参与梯度更新、哪些保持不动。通常我会选择 Transformer 中的注意力层（W_q, W_v, W_o 等）进行 LoRA 注入，而冻结所有归一化层、全连接层、embedding 层、LM Head 层。原因是：注意力层是模型中最敏感、最擅长迁移学习的部分——通过少量低秩更新就可以改变模型对上下文信息的关注方式。归一化层（LayerNorm/RMSNorm）非常脆弱，轻微更新就可能改变分布特性，导致训练不稳定，所以完全冻结。全连接层（FFN）包含了领域知识，更新它可能会导致模型遗忘预训练知识，所以通常冻结。Embedding 层如果更新，需要极大的显存（因为 embedding 矩阵非常大），而且改动它可能会影响词向量的泛化能力，所以也冻结。LM Head 层是输出层，更新它可能让模型无法合理预测下一个 token 的概率，通常不训练。在 QLoRA 中，量化的是基础模型的权重（包括注意力层、FFN 层等所有的权重参数），而注入的 LoRA 适配器（B, A）仍然是 FP16。这意味着：虽然基础模型权重被量化到了 4-bit，但由于 LoRA 适配器保持高精度，微调过程中损失的精度可以被 LoRA 的增量更新重新捕获。量化哪些层和 LoRA 注入哪些层不完全一样。量化作用于所有层以节省显存，而 LoRA 只注入需要适配的层。实践中我会让 LoRA 注入的层（如 W_q, W_o）在量化过程中也保留较高的优先级（比如使用 NF4 而不是更粗糙的格式），因为后续的更新正是要针对这些层发挥作用。

**深入理解版回答：**
这个问题的背后是理解 LoRA 的假设：预训练模型学到的知识可以迁移到新任务，只需要在关键位置上做“增量”轻量调整。因此选择的层应该是“对任务最敏感、对预训练知识影响最小”的层。我总结了一个四层选择原则：

1. **注意力层优于 FFN 层**：注意力层关注“哪些 token 之间相互影响”，是任务敏感的。FFN 层存储了大量的“事实性”知识，修改它容易导致知识遗忘。
2. **Query 和 Value 投影（W_q, W_v）通常优于 Key 投影（W_k）**：W_q 控制 query 的分布，W_v 控制输出内容，两者对最终输出影响最大。W_k 更多在 attention score 中起到“筛选器”作用，不直接影响输出语义。
3. **早期层 vs 晚期层**：早期层更多编码语法和基础模式，冻结；晚期层编码高级语义和任务知识，适合训练。因此我在 LoRA 中优先训练最后几层（靠近输出层）的注意力层。
4. **不训练所有注意力层**：即使训练 LoRA 比全参数微调轻量，训练所有层（12层/32层）的 LoRA 仍然会带来显著显存开销。实践上我会随机选择 1/3 到 1/2 的层做 LoRA 注入，效果接近全 LoRA 且显存节省明显。

在 QLoRA 的量化决策中，我有一个经验：**优先量化 FFN 层的权重**。因为 FFN 层参数量大（占 7B 模型的约 30%），但 NF4 量化对它们的精度影响较小（因为 FFN 权重分布更均匀）。注意力层的参数量较小，但我倾向于保留更高的精度（比如用 8-bit 或直接 NF4 但减少量化块的大小），因为注意力权重的微小变化对输出影响很大。

**常见追问与回答：**

- 追问：如果我把 FFN 层也加入 LoRA 训练，常见的效果提升或下降是多少？答：在大部分任务上，加入 FFN 层训练会带来约 1-3% 的微弱提升（比如分类准确率），但代价是显存增加约 2-3GB（7B 模型）。对于需要大量领域术语的微调任务（如医疗报告生成），加入 FFN 层训练可以获得明显增益（5-10%）。具体取舍取决于你对显存的容忍度和任务复杂度。
- 追问：在 QLoRA 中对所有层都用相同的量化 bit 数（4-bit）合适吗？答：不一定。可以考虑混合精度量化：对注意力层使用 4-bit NF4，对 FFN 层使用 3-bit 或 4-bit NF4。有研究表明，对注意力层保留稍高精度（4-bit）对效果影响小，而对 FFN 层使用 3-bit 节省显存更多。但这需要实验验证。
- 追问：实际项目中你是如何决定冻结层集合的？答：我会先做一个小实验：选取 1000 条数据，分别训练“只训练 W_q 和 W_o 的 LoRA”“训练所有注意力层的 LoRA”“训练注意力和 FFN 的 LoRA”三种配置，对比验证集 Loss 下降速度和最终准确率，选择收益最大的组合。通常第一种就足够了。

**项目联系：**
在微调一个医疗病历生成模型时，我选择了只训练最后 6 层（模型共 32 层）的 W_q 和 W_o，发现验证集 BLEU 已经达到 0.73，而训练所有 32 层的 LoRA（包括 FFN 的 LoRA）只提高到 0.74，但显存从 8GB 增加到 14GB，得不偿失。因此我的生产配置是“最后 1/3 层的 W_q + W_o”，在显存和效果之间取得平衡。

**记忆关键词：** 冻结层原则、注意力层优先、早期冻结晚期训练、QLoRA 量化粒度

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** 为什么 LoRA 只训练注意力层（W_q / W_v）而不训练全连接层（FFN）？
- **A：** FFN 层存储了大量事实性知识，更新容易导致遗忘预训练能力。注意力层是“语法-语义”的调节器，通过低秩更新即可调整模型对不同上下文的关注方式，因此更高效。此外，FFN 参数量大，训练它显存开销大但边际增益有限。
- **一句话记忆法：** 注意力调关注，FFN存事实；想动事实先过问，调关注用小LoRA。
- **复习标签：** #LoRA #冻结层 #PEFT

### 卡片 2
- **Q：** QLoRA 为什么用 NF4 量化权重，但 LoRA 适配器仍然保留 FP16？
- **A：** NF4 专为正态分布的权重设计，量化损失最小，显著降低显存。LoRA 适配器参数极少（占模型 0.1%-1%），保留 FP16 可以保证微调过程的数值稳定性，且对显存影响微乎其微。这种“低精度模型 + 高精度适配器”组合实现了性价比最大化。
- **一句话记忆法：** 基础参数压到4bit省显存，LoRA那点参数用16bit保精度。
- **复习标签：** #QLoRA #NF4 #量化

### 卡片 3
- **Q：** SFT 数据构造中如何保证样本多样性？
- **A：** 从三个维度控制：任务类型（分类/抽取/生成等）均衡覆盖；query embedding 聚类后按比例采样，确保每类语义都有代表；输出长度分层采样，避免模型只擅长单一长度。同时利用硬负样本（hard negative）训练模型学会拒答。
- **一句话记忆法：** 任务类、语义簇、长度层——三招均衡敲齐所有角落。
- **复习标签：** #SFT #数据构造 #多样性

### 卡片 4
- **Q：** LoRA 微调过拟合时，有什么缓解方案？
- **A：** 降低 r（例如从 32 降到 8）、增大 alpha（从 2r 增大到 4r 或 8r）以降低更新幅度；降低学习率（如 1e-5 调低到 2e-6）；增加数据量或数据增强（标记反转、同义词替换）；减少训练层数（只训练 Q 和 O，不训练 FFN）；增加 dropout（LoRA 自带 0.1-0.5 的 dropout）。
- **一句话记忆法：** LoRA过拟？降低r、增大alpha、降学习率、增dropout——四步绕过去。
- **复习标签：** #LoRA #过拟合 #PEFT

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** LoRA 微调过程中，为什么会出现 Loss 不降反升的现象？请从梯度更新、超参数选择、数据质量三个角度分别给出可能原因和对应解决方案。

**面试口头版参考答案：**
Loss 不降反升在 LoRA 中比较常见。最可能的原因是学习率过大，导致低秩矩阵 B 和 A 的更新幅度太大，破坏了预训练权重的分布。我会先检查学习率，通常 LoRA 的合理学习率在 1e-5 到 5e-5 之间，如果超过 1e-4 很容易出现 Loss 飙升。第二个原因是 r 和 alpha 的比值不合适。如果 alpha 远大于 r（比如 alpha = 4r），更新量会被放大，和预训练权重冲突。我会确保 alpha 与 r 保持 2:1 的比例。第三个原因可能是数据中存在大量噪声或矛盾样本。当不同样本对模型的要求相互冲突时，Loss 无法下降。我会先检查数据，用 k-fold 验签找出异常样本（Loss 特别大的样本），手动复核后剔除。如果以上都没问题，考虑是否 LoRA 注入的层数过多，导致模型在关键层上“踩了刹车”，可以尝试只注入更少的层（比如只注入 W_q，不注入 W_o）。

**深入理解版参考答案：**
Loss 不降反升是 LoRA 微调中最棘手的工程问题之一。需要从根源上理解 LoRA 的工作机制：LoRA 的更新是“增量式”的——W_new = W_old + (B @ A) * alpha/r。如果这个增量与 W_old 的符号和幅度不一致，就会导致模型输出混乱，进而表现为 Loss 上升。我从三个角度给出系统排查和解决方案。

**梯度角度**：当学习率过大时，LoRA 的 B 和 A 会快速偏离初始状态（通常初始化为 0 和随机），导致 BA 不再是小量，与 W_old 相加后输出异常。解决方案是梯度裁剪（clip_grad_norm=0.5），同时使用带 warm-up 的学习率调度器，让模型在前几百步以小学习率先探索稳定的低秩子空间。此外，检查 LoRA 的缩放参数 alpha 是否和 r 一致。我通常采用经验公式 alpha = 2r，当 Loss 不降时先将 alpha 设为 1r 或 0.5r，让更新量变小。如果仍然不行，同时降低 r（从 16 降到 8）并降低学习率（从 3e-5 降到 1e-5）。

**超参数角度**：除了学习率和 alpha，batch size 也是一个关键因素。batch size 太大（如 64）会导致梯度方向趋于平均，但 LoRA 是低秩更新，对平均梯度不敏感，容易出现 Loss 不降的情况。我通常用 4 或 8 作为默认 batch size，如果 Loss 不降，我试过减小 batch size 到 2 或 1，通常能激活模型学习。对于 LoRA 训练层数，过多的 LoRA 注入会导致梯度更新在多层间相互干扰。建议只选择 1/3 到 1/2 的层做 LoRA 注入，优先选择靠近输出层的层，因为修复这些层的知识对下游任务最重要。我常用的配置是：对 32 层模型，只对最后 10 层的 W_q 和 W_o 做 LoRA 注入。

**数据角度**：如果训练数据中存在大量不一致的回复（比如用户问“这个方法可行吗”，有些样本回答“可行”，有些样本回答“不可行”），模型无法从中学习统一的模式，Loss 自然不降。解决方案是用一个快速质量过滤器，用小模型（如 BERT-score）计算每个样本的 Loss 值，删除 Loss 值特别高（超过均值 3 倍标准差）的样本。同时检查样本的标签分布是否严重不均衡，比如回答类别 A 占 90%、类别 B 占 10%，模型会倾向于预测 A 而忽略 B，导致 B 类样本的 Loss 一直很高。采用 Focal Loss 加权重或在数据集中过采样少数类可以缓解。

**可能追问与回答：**

- **追问：** 如果你减小了 batch size 但 Loss 仍然不降，说明什么？答：说明问题不是梯度稀疏性，而更可能是数据质量或超参数组合。我会转向检查数据中是否有矛盾样本，以及是否 r 过大导致 LoRA 更新幅度过大。
- **追问：** LoRA 的 Loss 不降与全参数微调的 Loss 不降，原因和解决方案有什么不同？答：全参数微调时，Loss 不降通常是因为学习率过大或梯度爆炸，可以用梯度裁剪轻松解决。LoRA 则多了一个“低秩矩阵表达能力不足”的问题：如果 r 太小，模型即使学习率合适也无法降低 Loss。因此 LoRA 的解决方案要同时考虑 r 和 alpha，而不仅仅是学习率。
