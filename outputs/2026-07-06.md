# 今日大模型实习面试日报

> 日期：2026-07-06
> 今日重点：
> - RAG 分块（Chunking）策略不是越多越好，小块召回准但上下文不足，大块上下文足但噪声多——工程上要按文档类型动态选择。

## 1. 今日面试真题精讲

### 题目 1：在 RAG 系统中，你会如何设计分块（Chunking）策略？分块大小和重叠率如何影响检索效果？请给出工程选型建议。

**来源依据：**
- 大模型RAG面试题详解，检索增强生成的原理与实现 - https://blog.csdn.net/star_nwe/article/details/153679906 - 技术博客 - 中可信度，CSDN 博客，需进一步验证
- 大模型RAG难点、痛点、常见面试题 - https://www.nowcoder.com/feed/main/detail/c4acb29abe13485d97495cdcdce0ebef - 面经或社媒 - 低可信度，牛客网帖子，需进一步验证

**可信度：** 中

**考察点：** RAG 索引流程、分块策略工程选型、召回效果权衡

**面试口头版回答：**
分块策略直接影响召回质量。我的原则是：**先按文档类型决定分块方法**，再通过实验调优。对于结构化文档（如 Markdown、HTML），我会优先用**语义分割**，按标题、段落划分，保留自然边界，这样每个块的内容内聚性强。对于纯文本（如 PDF 论文），我会用**固定大小+滑动重叠**，比如 512 token 块加 128 token 重叠率。重叠能避免关键信息被切在边界上而丢失。分块大小要权衡：太小（<256 token）会导致上下文不足，模型缺乏背景；太大（>1024 token）会使检索噪声增加，而且 LLM 的“Lost in the Middle”效应会让中间信息被忽略。工程上我会先做离线评测：对不同 chunk 策略计算召回率@k，观察哪个策略在验证集上命中正确答案的比例最高。

**深入理解版回答：**
RAG 的分块本质是解决“检索粒度和上下文完整性”的矛盾。常见策略有三种：固定大小分块、递归分割（按分隔符递归切）、语义分割（利用 LLM 或 Embedding 模型识别语义边界）。固定大小实现简单但可能切断句子；递归分割适合有自然分隔符的文本（如段落、句子）；语义分割质量最高但开销大，常用于高质量文档库。重叠率的作用是**边界补偿**——如果一个关键词刚好在块末尾，下一个块的开头因重叠能再包含它，提高召回概率。但重叠率太高会增加索引存储量和检索延迟，通常 10%-20% 即可。工程中还有一个坑：**分块后必须保持元数据**（如标题、页号、时间戳），这样后续过滤和溯源才可行。我自己在项目中采用“多粒度分块”：先按段落粗分，对特别长的段落再用句子切割，最终块统一控制在 512 token 内，重叠 64 token。离线实验表明，相比固定 256 token 无重叠，召回率提升约 8%。调优时还可以结合**embedding 模型对长文本的退化现象**（大部分 embedding 模型在 512 token 外效果下降），所以分块大小不应超过 embedding 模型的最大输入。

**常见追问与回答：**
- **追问：你怎么评估分块策略的好坏？** 答：我会构建一个评估集，包含问题和正确答案所在原文片段。对每种分块方案，检索 top-5，计算“答案所在的块是否被召回”作为召回率。同时对比 chunk 数量对延迟的影响，选择召回率-延迟的帕累托最优。
- **追问：如果文档是图片或表格怎么办？** 答：图片需要 OCR 或专门的多模态模型转文字（如 Llama 3.2-Vision），表格需要保留结构化信息（如 Markdown 格式），再按文本分块。不能直接塞图片向量，因为 embedding 模型不是多模态的。

**项目联系：**
我做过一个企业知识库问答系统，文档包括 Word、PDF、Markdown。我实现了“文档解析五步法：读、洗、拆、标、存”。在“拆”阶段，我用 LangChain 的 RecursiveCharacterTextSplitter 按段落分隔，块大小设为 500 字符，重叠 50 字符。并且对每个块自动提取章节标题作为元数据（metadata），这样在后面 rerank 时可以根据章节过滤。

**记忆关键词：** #分块策略 #语义分割 #固定大小 #重叠率 #召回率 #元数据

### 题目 2：RAG 中的 Rerank（重排序）是什么？为什么需要它？工程上如何实现？

**来源依据：**
- RAG Interview Questions and Answers - https://www.interviewbit.com/llm-interview-questions-answers - 技术博客 - 中可信度，InterviewBit 面试题库，需进一步验证
- 大模型RAG面试题详解 - https://blog.csdn.net/star_nwe/article/details/153679906 - 技术博客 - 中可信度，需进一步验证

**可信度：** 中

**考察点：** RAG 检索后处理、rerank 原理、Cross-Encoder 与 Bi-Encoder 区别

**面试口头版回答：**
Rerank 是 RAG 检索后的精排阶段。原因在于：第一阶段检索（如向量检索）通常用 Bi-Encoder，它把 query 和 chunk 分别编码成向量，再算余弦相似度。但 Bi-Encoder 编码时是独立编码的，没有交叉交互，所以排序不够准。Rerank 用 Cross-Encoder，把 query 和 chunk 拼接后一起输入模型，直接输出一个相关性分数，精度更高。工程上，我会先做 Hybrid Search（向量+BM25）召回 top-50，然后用 Cross-Encoder（如 BGE-Reranker-v2）精排，再取 top-5 送入 LLM。Rerank 的代价是推理开销大，因为每个 pair 都要过模型，所以我只在候选池小（~100）时用，同时可以采用**异步批量 rerank** 来降低延迟。

**深入理解版回答：**
Rerank 的核心价值是解决 Bi-Encoder 的语义天花板。Bi-Encoder 把 query 和 chunk 投影到同一语义空间，但相似度计算是内积，缺少 query 对 chunk 的细粒度匹配（例如“苹果公司”和“水果苹果”的区分）。Cross-Encoder 通过交叉注意机制，可以捕捉 token 级别的交互信号，因此排序精度高 10-20%。工程实现通常有几步：1）从向量库或 BM25 拿到候选 list（如 top-100）；2）将每个 (query, chunk) 对用 tokenizer 拼接，截断到模型最大长度（如 512）；3）批量通过 rerank 模型，取出 score；4）按 score 排序，取 top-k。注意点：rerank 模型的输入长度限制决定了 chunk 不能太长，否则需要截断。另外，rerank 和检索阶段的 embedding 模型可以是不同系列的（比如 embedding 用 BGE-small，rerank 用 BGE-reranker-large）。在项目里，我还遇到过 rerank 模型对英文效果好，但对中文效果一般，需要单独微调。为了优化延迟，我采用“两级 rerank”：先用一个轻量级 BGE-reranker-base 粗排 top-100 到 top-20，再用重量的 BGE-reranker-large 精排 top-20 到 top-5。总体上 latency 增加约 200ms，但评测中 HitRate@5 从 75% 提升到 88%。

**常见追问与回答：**
- **追问：如果资源有限，能否不用 rerank？** 答：可以，但要付出代价。当 recall 已经很高（>90%）且 LLM 上下文窗口很大时，可以直接压 top-10 进 prompt 让 LLM 自己排序。但 LLM 自己排序同样受“Lost in the Middle”影响，效果不如显式 rerank。所以我会建议至少用一次轻量 rerank，如使用 Cohere Rerank 3 API。
- **追问：Rerank 模型和 Embedding 模型能共享吗？** 答：不能直接共享，因为结构不同。但可以同源：比如 BGE 系列同时提供 embedding 和 rerank 版本。迁移学习上，可以用对比学习先预训练 embedding，再用交叉熵 fine-tune rerank。

**项目联系：**
在我的知识库问答项目中，第一阶段用 `sentence-transformers/all-MiniLM-L6-v2` 做向量召回 top-50，第二阶段用 `BAAI/bge-reranker-v2-m3` 重排序。我测试过不加 rerank 时 LLM 经常给出错误的“主观排序”，加了 rerank 后答案引用文档的正确率提高了 15%。我把 rerank 步骤封装成独立的微服务，用 FastAPI + 异步，与主 RAG 服务解耦，避免阻塞。

**记忆关键词：** #Rerank #Cross-Encoder #Bi-Encoder #精排 #延迟优化 #两级排序

### 题目 3：RAG 中如何控制幻觉并确保答案的引用忠实度？请设计一个工程方案。

**来源依据：**
- 大模型RAG面试题详解 - https://blog.csdn.net/star_nwe/article/details/153679906 - 技术博客 - 中可信度，需进一步验证
- AI Agent常考面试题汇总- LoRA篇 - https://www.nowcoder.com/discuss/876512639952683008 - 面经或社媒 - 低可信度，牛客网专栏，需进一步验证

**可信度：** 低

**考察点：** 幻觉控制、引用标注、忠实度评测、prompt 工程

**面试口头版回答：**
RAG 中的幻觉本质是 LLM 在生成时脱离了检索到的证据。我的工程方案从三个层面控制：**Prompt 层面**、**后处理层面**、**评测层面**。Prompt 层面：在系统提示中明确要求 LLM 只基于提供的上下文回答，并**逐句标注引用编号**，例如“根据文档[1]提到，…”。同时设置低 temperature（0.1-0.3）减少自由发挥。后处理层面：解析 LLM 输出的引用标记，检查每个引用是否真的对应被检索到的 chunk，如果发现引用不存在或与内容矛盾，则拒绝输出或降级。还有 **RAG-Fusion**：当 LLM 输出包含无引用断言时，自动回退到检索补充上下文。评测层面：用 **Llama-3-as-judge** 或专家规则检查“答案中每个事实性断言是否能在给定上下文里找到”，计算 **忠实度分数**。我实际用过 LangChain 的 `create_citation` 链，配合正则提取引用，效果不错。

**深入理解版回答：**
幻觉控制需要贯穿全链路。首先，RAG 的**检索质量**是基础——如果检索到的 chunk 本身就不相关，LLM 再忠实也没用。所以先要保证高召回和高相关性。其次是**生成阶段的约束**。除了 Prompt 约束，还可以用 **Structured Output** 强制 LLM 输出 JSON，包含 `answer` 和 `citations` 字段，这样后续可以程序化校验。第三种方法是 **Self-RAG**：让 LLM 在生成每个句子时先判断是否需要检索（即“需检索则检索”），并在输出中标注“检索后生成”和“仅凭内部知识”两种模式，这样能区分哪些内容有外部证据。工程实施中我会设计一个 **Citation Validator** 模块：对每个标注的 chunk ID，计算 LLM 输出片段与该 chunk 文本的语义相似度（如 cosine similarity > 0.8），若低于阈值则标记为疑似幻觉。同时我还加入了 **否定检测**：如果 LLM 输出中包含“根据文档”但实际 chunk 中并未提及相关内容，则提示。离线评测时，我会构建一个 **Factuality Benchmark**：对每个问答对，人工标注出正确答案中每句的引用，然后用忠实度指标（如 AlignScore、FEVER）自动打分。线上监控则跟踪 **无引用回答比例** 和 **用户反馈中的“不准确”标记**。

**常见追问与回答：**
- **追问：如果 LLM 参照了多个文档，但其中一个文档有误，该怎么处理？** 答：需要在检索阶段做**权威性评分**，对低可信度来源的 chunk 降权；同时在引用标注中区分来源。后处理时可以对比多个来源对同一事实的描述，用多数投票或 LLM-as-judge 决定。
- **追问：强制引用会增加 token 消耗，怎么平衡？** 答：确实会，尤其是长答案。可以只在关键断言处要求引用，而非每句。或者用**重要性评分**（如句子的信息密度）决定是否加引用。

**项目联系：**
我在做一个医疗问答 RAG 系统时，特别重视引用忠实度。因为患者可能依据答案决策。我在 prompt 里加入了“所有回答必须附上原文出处编号，若无法找到支撑则回答‘信息不足’”。后处理中我写了个 Python 函数，用正则提取 `[数字]`，然后去向量库查对应 chunk 的原文，用余弦相似度验证。如果相似度低于 0.75，就触发二次检索。上线后，人工抽检 200 条，忠实度从 82% 提升到 93%。

**记忆关键词：** #幻觉控制 #引用标注 #Citation Validator #Self-RAG #忠实度

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** RAG 中分块大小如何选择？为什么一般不超过 Embedding 模型的最大输入长度？
- **A：** 分块大小通常取 Embedding 模型 max length 的 80%-100%（如 512 token）。因为超过该长度的文本在编码时会被截断或 padding，导致向量质量下降，检索精度变差。同时大小要与 LLM 上下文窗口匹配。具体选择需通过离线实验：对不同 chunk_size 计算召回率@k，选择召回率-延迟最优的尺寸。
- **一句话记忆法：** 分块就像切牛排——太大嚼不动（检索不准），太小没味道（上下文不足）。
- **复习标签：** #RAG #分块 #嵌入模型

### 卡片 2
- **Q：** Rerank 和 Embedding 检索的本质区别是什么？
- **A：** Embedding 检索是 Bi-Encoder 架构，将 query 和文档分别编码为向量，再计算相似度，速度很快但缺乏细粒度交互。Rerank 是 Cross-Encoder 架构，将 query 和文档拼接后一起编码，可以捕捉词级匹配，精度更高但速度慢。工程上通常先用低成本检索缩小候选池（如 top-100），再用 rerank 精排。
- **一句话记忆法：** Embedding 是“大网捞鱼”，Rerank 是“近看清鱼”。
- **复习标签：** #RAG #Rerank #Bi-Encoder #Cross-Encoder

### 卡片 3
- **Q：** RAG 中如何做引用标注（Citation）？
- **A：** 在 Prompt 中要求 LLM 输出时给每个事实性陈述标注对应的文档编号，例如“[1]”或“(来源：doc_3)”。后处理用正则提取引用标记，并验证该 chunk 中是否确实包含该事实。如果无法验证，可以拒答或提示。常用方案包括 LangChain 的 create_citation 链，或强制 Structured Output 输出包含 citations 字段的 JSON。
- **一句话记忆法：** 引用是 RAG 的“防伪标签”，没有引用的回答就是缺失验证。
- **复习标签：** #RAG #幻觉控制 #引用 #忠实度

### 卡片 4
- **Q：** 如何评估 RAG 系统中生成答案的忠实度（Faithfulness）？
- **A：** 常用指标是检查答案中的每个事实性断言是否能在提供的检索上下文中找到支持。具体方法有：1) 基于规则的 NLI（如 ANLI 模型）判断答句话与上下文是否蕴含；2) 用 LLM-as-judge（如 GPT-4 或 Llama-3）逐句打分；3) 参考 FEVER、TruthfulQA 等基准的忠实度定义。工程上我会构建一个验证集，人工标注每句的黄金引用，然后计算 HitRate 和准确率。
- **一句话记忆法：** 忠实度就是“答案和证据对质，看谁在撒谎”。
- **复习标签：** #RAG #评测 #忠实度 #LLM-as-judge

### 卡片 5
- **Q：** 什么是 Hybrid Search？为什么 RAG 中要结合向量检索和关键词检索？
- **A：** Hybrid Search 融合了语义检索（向量）和精确匹配（BM25/keyword）。向量检索擅长捕捉同义词和近义表达，但对低频专有名词、代码片段、精确 ID 等不敏感；BM25 恰好能处理精确匹配。两者互补可以提高整体召回率。融合方式常用 RRF（Reciprocal Rank Fusion）或加权评分。项目经验显示，Hybrid Search 相比纯向量检索在内部知识库上召回率提升 10-15%。
- **一句话记忆法：** 向量负责“懂你意思”，BM25 负责“找到原词”。
- **复习标签：** #RAG #Hybrid Search #BM25 #RRF

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 在 RAG 中，当检索到的上下文长度超过 LLM 的上下文窗口时，你如何处理？请从“位置偏差”和“信息丢失”两个角度分析，并给出工程方案。

**面试口头版参考答案：**
当上下文过长时，LLM 会出现“Lost in the Middle”问题——只记住开头和结尾，中间信息被忽略。我的处理方案是：1) **先压缩后输入**：用摘要模型（如 GPT-4o mini）将多个 chunk 合并成更短的摘要，或者让 LLM 只回答一个具体问题而非总结所有内容；2) **分段询问 + 合并答案**：把 chunks 分组，分别提问，再让 LLM 整合结果；3) **滑动窗口+位置重排**：用 rerank 把最重要的 chunk 放在最前面，确保开头是高价值信息。另外可以限制检索数量不超过 5 个 chunk，并截断每个 chunk 到 512 token，总长度控制在 3000 token 内。

**深入理解版参考答案：**
这个问题涉及两个核心挑战：**位置偏差**和**信息丢失**。位置偏差来自 Transformer 的 attention 机制——理论上 attention 可以关注所有位置，但实际训练中模型倾向于聚焦在序列两端（因为训练数据中关键信息常在开头或结尾）。信息丢失则是因为长上下文中大量噪声稀释了重要信号。我的工程方案从三个阶段解决：

1. **检索阶段**：通过**重要性评分**对 chunks 排序，最相关的排首位。同时使用**滑动窗口**技术，在拼接前对每个 chunk 截取最相关的部分（如用 query 和 chunk 的语义相似度截断首尾）。

2. **预处理阶段**：实施**上下文压缩**，例如使用 LLM 对每个 chunk 生成 2-3 句话的摘要，压缩率可达 70%。我实践过用 Llama-3-8B 做压缩，效果不错，代价是增加一次 LLM 调用延迟。也可以使用专门的上下文压缩模型如 LLMLingua。

3. **生成阶段**：在 Prompt 中显式要求模型“请先阅读所有上下文，重点在开头和结尾的信息”。或者使用**分段生成**：将上下文拆成多个子段落，对每段提出同一个问题，收集多个答案后再用投票或融合生成最终答案。我曾在项目中用 LangGraph 实现了一个“并行 QA + 合并”的节点工作流，虽然延迟增加但忠实度从 82% 提升到 91%。

**可能追问与简短回答：**
- **追问：如果用户问题需要综合分析多个 chunk，压缩可能会丢失跨 chunk 的关系，你怎么处理？** 答：这种情况下不适合压缩。我会保留所有 chunk 并采用**结构化 Prompt**，让 LLM 先提取每块的关键事实，再要求它比较和综合。或者使用**窗口回溯法**：先分段推理得到中间结论，再汇总到最终答案。
- **追问：你有没有测量过最优的上文长度？** 答：我做过实验，在 8k 窗口的 LLM 上，超过 4k token 时忠实度开始下降，超过 6k 时显著下降。所以工程上我会尽量控制上下文在窗口的 60% 以内，如果超过则用压缩或分段。
