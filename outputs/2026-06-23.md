### 题目 1：Agent 的长短期记忆系统如何设计？记忆的提取、压缩、冲突更新具体怎么做？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证（面经转载，细节较真实）
- AI Agent常考面试题汇总- LoRA篇 - https://www.nowcoder.com/discuss/876512639952683008 - 面经或社媒 - 需进一步验证

**可信度：** 中

**考察点：** Agent 记忆模块设计、长短期记忆、状态管理、冲突解决

**面试口头版回答：**

我把记忆拆成短期和长期两层，三个动作贯穿其中。提取：每轮对话结束后，用一个轻量抽取 prompt 把值得长期记的信息抽成结构化条目，包括类型、内容、来源轮次、时间戳，通过规则粗筛把寒暄丢掉。压缩：短期存最近几轮原文保证指代还原；长期只存结构化条目，同主题条目定期摘要合并，控制记忆规模。冲突更新：当新信息和旧记忆矛盾时，不直接删除旧条目，而是按时间戳 + 来源置信度仲裁——用户显式说“改成只看华南”就把旧条目标记失效；拿不准就两条都留，检索时优先取更新的。这样既不丢历史，也不会拿过期信息答题。

**深入理解版回答：**

Agent 记忆设计是面试中最容易被深挖的点，因为它直接决定了多轮对话的质量。

首先是分层设计。短期记忆存最近 N 轮对话原文，目的是解决指代消解和对话连贯性，比如用户说“那它呢”，需要短期的 context 才能还原。长期记忆存结构化条目，字段包括类型（用户画像、硬约束、关键事实、待办）、内容摘要、来源轮次、时间戳。

提取阶段，每轮结束后调用一个轻量 LLM 抽取结构化条目，同时用规则粗筛：对话轮次太短、纯寒暄（“好的”“谢谢”）直接丢弃。这里有一个关键点：不是所有对话信息都值得存，需要做“值不值得存”的判断，通常用小模型打标签。

压缩阶段有两个层面。短期压缩是滑动窗口，保留最近 3-5 轮原文。长期压缩是主题合并：同一主题的条目会在积累到一定数量后触发摘要合并，比如用户三次提到偏好，合并成一条最新画像。但要注意，对硬约束类信息（“只看华东”），我单独标记不参与合并，避免被平均掉。

冲突更新是最高频的追问点。核心逻辑是“时间戳优先 + 置信度仲裁”。时间戳是硬规则：新的优先。但冲突不能只靠时间，因为有些可能是临时需求。我引入置信度概念：用户显式说的置信度最高；模型从对话推断的次之；模糊的猜测最低。当新信息与旧信息语义冲突时，如果能明确是更新（显式说“改成”），就把旧条目标失效、写入新条目。如果语义模糊（比如类似但不同表述），两条都保留，检索时按“相关性 + 时近性”加权排序，优先取较新且置信度高的。

记忆检索也有讲究：我不会把长期记忆全量拼进上下文，而是先向量化条目，按相关性召回 Top-K，再用时近性重排，最终拼接 3-5 条最相关条目 + 短期 2-3 轮原文。硬约束例外，只要生效就每轮都带。

工程实现上的常见坑：存储选型。短期记忆通常用 Redis 或内存队列，长期记忆推荐向量数据库（如 Milvus/Pinecone）或关系型数据库存结构化字段。冲突仲裁的时效性：如果要在毫秒级完成判断，纯模型调用成本太高，建议用规则兜底 + 模型辅助的混合策略。

**常见追问与回答：**

**追问1：记忆压缩时，摘要合并会不会丢失关键细节？**
答：有这个风险。我只合并同主题且明显冗余的条目，合并时保留“最新值 + 关键历史变更”，不是揉成一句。硬约束单独标记，不参与摘要合并。另外，压缩策略需要上线前做评测对比，看合并前后在评测集上的答案正确率是否有下降。

**追问2：短期记忆的长度如何确定？窗口过大过小各有什么问题？**
答：窗口过小会导致指代消解失败，用户说“那它呢”找不到指代对象；窗口过大浪费 token，还可能引入噪声。我一般设置在 3-5 轮，关键约束例外。上线前会用评测集对比不同窗口大小下的正确率，找到曲线拐点。

**追问3：长期记忆的向量检索召回，你怎么保证只召回最相关的？**
答：关键在于向量化质量。每个记忆条目要有一个好的 embedding 表示，我会用专门的 embedding 模型（如 bge-large）来编码，并且定期对记忆条目做重嵌入，因为旧条目的语义表征可能随着对话更新变得不准确。

**项目联系：**
在做企业内部资料智能问答 Agent 时，一期上线后最大的问题是用户说过“只看华东区的数据”，下轮换个问法模型又把全国数据混进来。这就是典型记忆丢失问题。我通过构建评测集（几百条真实多轮对话），对比“带记忆 + 短期 + 长期”三种配置的答案正确率，发现完整的两层记忆设计能提升正确率约 15-20%。

**记忆关键词：** 短期记忆 / 长期记忆 / 结构化条目 / 摘要合并 / 冲突仲裁 / 时间戳置信度

### 题目 2：在 Agent 系统中，如何设计高吞吐、低延迟的工具调用（Tool Calling）链路？请从并发控制、超时重试、兜底策略三个维度展开。

**来源依据：**
- 字节跳动大模型实习面经 - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证（面经中提到了 asyncio、限流、超时重试）
- 小林面试笔记 - 大模型工程面试题 - https://xiaolinnote.com/ai - 技术博客 - 中（个人博客，但内容系统）
- AI Agent常考面试题汇总- LoRA篇 - https://www.nowcoder.com/discuss/876512639952683008 - 面经或社媒 - 需进一步验证

**可信度：** 中

**考察点：** Tool Calling 链路设计、并发控制、异常处理、工程优化

**面试口头版回答：**

高吞吐工具调用链路的关键是“控制并发 + 兜底恢复”。并发控制层面，我用 asyncio 异步编排工具调用，同时用信号量实现限流，比如同一 agent 实例内的并发工具调用上限设为 5，避免打爆下游 API。超时重试层面，每个工具调用都设置两阶段超时：连接超时 3 秒、读取超时 10 秒，超时后最多重试 2 次，使用指数退避。兜底策略层面，如果某个工具连续失败，就把这个工具标记为“不可用”，切换到一个简化的 fallback 逻辑，比如返回一个固定提示“该功能暂时不可用，请稍后再试”，而不是让整个 agent 卡死。

**深入理解版回答：**

这个问题的本质是：Agent 要同时调用多个工具（比如同时查天气、查航班、查日历），每个工具调用的延迟和稳定性都不一样，如何让整个调用链不阻塞、不崩溃？

第一层：异步编排。用 Python asyncio 做核心框架，把并发的工具调用设计成协程任务。关键不是简单用 asyncio.gather 完事，而是要设计一个工具调用调度器，它负责注册所有可调用的工具、管理调用状态、处理超时和重试。

第二层：限流与并发控制。同一 agent 实例的并发调用数必须限制，否则 LLM 可能一次性请求 10 个工具调用，瞬间打爆下游服务。我用信号量控制最大并发数（比如 5），并在调度器中内置一个任务队列，超过限制的任务排队等待。此外，还需要监控每个工具的负载，避免同一工具被高频调用。

第三层：超时和重试。两段式超时设计：连接超时（TCP 握手）设置 3 秒，读取超时（等待响应）设置 10 秒。超时后触发重试，最大重试次数 2 次，间隔使用指数退避（1s、2s、4s）。但重试不能无脑：对于幂等的工具（如查询类），可以直接重试；对于非幂等的工具（如下单、发送消息），必须先检查是否已执行，避免重复操作。

第四层：降级与熔断。如果某个工具连续失败超过阈值（比如 5 分钟内失败 5 次），触发熔断：标记该工具为不可用，后续调用直接返回 fallback 响应。这是一个局部的熔断，不影响其他工具。同时，熔断后要有一个半开机制（比如 30 秒后尝试调用一次），如果能成功就恢复。

第五层：状态管理。每次工具调用的输入、输出、状态、延迟都要记录下来，形成 trace。这不仅用于调试，还能在发生错误恢复时，重建上下文。比如一个工具调用失败后，agent 可以基于 trace 决定是重试还是跳过。

常见坑：不是所有工具都适合异步调用。比如一些需要保证执行顺序的工具（先查库存、再下单），不能简单并发。我会在工具注册时声明其依赖关系，调度器根据依赖图做拓扑排序，决定并发还是串行。

**常见追问与回答：**

**追问1：N+1 次调用的性能瓶颈怎么解决？比如 agent 需要先调一个列表接口获取 IDs，再对每个 ID 调详情接口。**
答：这是经典问题。方案是批量接口合并：如果下游 API 支持批量查询，直接传 ID 列表；如果不支持，可以用缓存 + 预加载。不过最根本的是在系统设计层面，限制工具调用的链条深度，或者引入缓存层。

**追问2：LLM 生成的函数调用参数格式错误怎么办？**
答：我会在工具调用调度器上加一层参数校验：每个工具的 schema 是预定义的（类似 OpenAI function calling 的 JSON Schema），LLM 返回的参数先过校验，格式不对就发回给 LLM 要求重新生成，重试 2 次后仍然失败则返回“参数格式异常”。

**项目联系：**
在我的 Agent 项目中，工具调用链路是整个系统的瓶颈，因为之前是串行调用，用户等 10 多秒才能得到结果。改成 asyncio 并发 + 带超时重试后，P95 延迟从 12 秒降到 4 秒。同时，熔断机制救了线上一次故障：某个天气 API 挂了，系统直接降级返回“天气查询暂时不可用”，而不是整个对话卡死。

**记忆关键词：** 异步编排 / 限流 / 超时重试 / 指数退避 / 熔断降级 / trace

### 题目 3：ReAct、Plan-and-Execute、Reflection 三种 Agent 设计范式有什么区别？实际项目中该如何选型？

**来源依据：**
- 小林面试笔记 - Agent 面试题 - https://xiaolinnote.com/ai - 技术博客 - 中（个人博客，但内容系统）
- AI Agent常考面试题汇总- LoRA篇 - https://www.nowcoder.com/discuss/876512639952683008 - 面经或社媒 - 需进一步验证
- datawhale/hello-agents - 面试问题总结 - https://github.com/datawhalechina/hello-agents/blob/main/Extra-Chapter/Extra01-%E9%9D%A2%E8%AF%95%E9%97%AE%E9%A2%98%E6%80%BB%E7%BB%93.md - 技术博客 - 中（开源项目，内容较完整）

**可信度：** 中

**考察点：** Agent 设计范式、推理模式、选型判断、场景分析

**面试口头版回答：**

这三种范式本质是“思考与行动”的不同编排方式。ReAct 是“想一步、做一步”，把推理和行动交错进行，每一步先想我要做什么、再调用工具、然后观察结果、继续想，适合动态交互场景，比如客服需要逐步确认信息。Plan-and-Execute 是“先想完再执行”，先制定完整计划，然后一步执行完，适合计划性强的任务，比如预定行程。Reflection 是“边做边反思”，每次行动后检查结果是否正确，发现错误就纠正，适合高精度场景，比如代码生成。选型时，如果任务复杂但稳定，用 Plan-and-Execute；如果任务动态变化大，用 ReAct；如果对错误零容忍，用 Reflection 或组合使用。

**深入理解版回答：**

首先要理解这三者的核心区别：

ReAct（Reason + Act）源于 Yao et al. 的论文，核心是“思考-行动-观察”循环。每一步：LLM 输出一个思考（Thought），决定要做什么，然后执行一个工具调用（Action），观察结果（Observation），再继续思考。这个循环一直持续到完成任务或达到最大步数。优点是灵活，能处理动态变化的局面；缺点是 token 消耗大，而且可能陷入“思考→调用→观察→思考”的死循环。

Plan-and-Execute 是先让 LLM 制定一个完整的多步计划（Plan），包括每一步做什么、调用什么工具、期望输出什么。然后调度器按计划依次执行。如果某一步失败，可以回退到计划层面调整。优点是效率高（不需要每步都让 LLM 思考）、token 省、可预测。缺点是灵活性差，计划外的变化处理不了。适合任务明确、步骤固定的场景。

Reflection 是让 Agent 在执行过程中或执行后，对结果进行自我评估和纠正。具体实现有几种：一种是“带反省的 ReAct”，即 ReAct 循环中加入一个“Reflect”步骤，检查前面的行动是否有效，无效则修正。另一种是事后反思：完成任务后，让 LLM 分析哪一步做得不好，为下一次类似任务提供经验。优点是精度高，适合代码生成、数学推理等对正确性敏感的场景。缺点是增加了延迟和 token 开销。

实际项目中的选型决策：

- 如果任务是开放式的、用户意图多变（如智能客服），选 ReAct，因为需要动态调整。
- 如果任务是流程化的、步骤固定（如数据报表生成），选 Plan-and-Execute，更高效可控。
- 如果任务是高精度的、错误代价大（如代码审查、金融交易），选 Reflection，加入自我纠错。

此外，实际工程中经常混合使用：外层用 Plan-and-Execute，内层用 ReAct 处理每一小步。这种混合架构能兼顾效率和灵活性。

常见坑：不要过度设计。简单任务不需要 Reflection，浪费 token。Plan-and-Execute 的 Plan 可能不准确，需要加入重新规划机制。

**常见追问与回答：**

**追问1：Plan-and-Execute 中，如果执行过程中发现计划错了，怎么处理？**
答：需要加入“重新规划”机制。可以在每次执行完一步后，让 LLM 检查当前进度是否符合原计划（这一步类似 Reflection），如果偏离严重，触发重规划——让 LLM 基于当前状态重新生成剩余步骤的计划。这其实就是 Plan-and-Execute + Reflection 的组合。

**追问2：ReAct 循环的 Token 开销太大，怎么优化？**
答：几种优化方式：1) 压缩每次的 Observation，只保留关键信息；2) 限制最大步数；3) 对于 ReAct 中“思考”步骤的 prompt，可以固定格式，减少 LLM 输出的冗余内容。更激进的做法是“先计划、后执行”：先让 LLM 生成一个简短的 ReAct 风格的思考链，然后一次性执行。

**追问3：Reflection 的具体实现方式有哪些？**
答：主流有两种。一种是在 ReAct 循环中加入 Reflection 步骤：每次 Action 后，让 LLM 检查结果是否正确，不正确则修正。另一种是事后反思：整个任务完成后，让 LLM 回顾过程、找出错误、总结经验，写入记忆，下次遇到类似任务可以直接参考。

**项目联系：**
我在做智能客服 Agent 时，最开始用纯 ReAct，发现很多简单问题也要走好几步，token 浪费大。后来改成 Plan-and-Execute 作为外层——先让 LLM 制定一个“回答计划”（需要查哪些知识库、是否需要查用户历史），然后按计划并行检索，最后生成回答。复杂问题（比如需要多轮确认的）才走 ReAct。这个混合架构让 P50 延迟降低了约 40%。

**记忆关键词：** ReAct / Plan-and-Execute / Reflection / 思考行动循环 / 自我纠错 / 选型决策

**可转化面试题：**

**Q1：Agent 长短期记忆系统中，如果新信息和旧记忆冲突，你怎么判断哪个是对的？**
**Q2：记忆检索时，“相关性”和“时近性”怎么平衡？分别注入多少条进上下文？**

**对应参考答案：**

**A1：** 核心是用“时间戳 + 置信度”仲裁。时间戳是硬规则：更新的优先。但冲突不能只看时间，因为可能存在临时需求。我按置信度分档：用户显式说的（比如“改成只看华东”）置信度最高；模型从对话中推断的（比如“用户似乎偏好华东”）次之；模糊猜测最低。新信息进来时先判断语义冲突，如果有冲突，再结合置信度：置信度高的新信息覆盖旧的；如果不确定，两条都保留，检索时优先取较新且置信度高的那条。这样既不会丢失历史，也不会拿过期信息答题。**Q2：** 我不把长期记忆全量拼进上下文。做法是：先向量化记忆条目，检索时按相关性召回当前 query 相关的 Top-K，再用时近性做加权重排——同样相关的，更新的优先。注入量通常卡在 3-5 条，外加短期记忆最近 2-3 轮原文。记忆塞太多一样会稀释问题、引噪声。但硬约束（比如“只看华东”）是例外，只要还生效就每轮都带。

### 卡片 1：

**Q：Agent 系统中，短期记忆和长期记忆各自的存储方案是什么？各自适合用什么数据库？**
**A：** 短期记忆存最近几轮对话原文，要求低延迟读写，通常用内存缓存（如 Redis）或本地内存队列。长期记忆存结构化条目（类型/内容/来源轮次/时间戳），需要做向量检索和结构化查询，推荐向量数据库（Milvus / Pinecone）搭配关系型数据库（PostgreSQL）存储条目元数据。硬约束类条目可以单独用 KV 存储实时查询。

### 卡片 2：

**Q：ReAct 范式的基本流程是什么？每一步做什么？**
**A：** ReAct 是“Reason + Act”的循环。每步流程：1) Thought：LLM 思考当前状态，决定下一步做什么；2) Action：LLM 输出一个工具调用指令（如 function name + 参数）；3) Observation：执行工具调用，返回结果给 LLM；4) 重复以上循环直到完成任务或达到最大步数。关键特点：思考与行动交替进行，每一步都有观察反馈。
**一句话记忆法：** 想一步、做一步、看一眼，再接着想。
**复习标签：** #Agent #ReAct #设计范式

---

### 卡片 3：

**Q：Agent 的工具调用中，如何处理 N+1 查询性能瓶颈？**
**A：** N+1 问题指先调一个列表 API 获取 IDs，再对每个 ID 调详情 API。三种方案：1) 批量合并：如果下游 API 支持批量查询，直接传 ID 列表；2) 缓存 + 预加载：预先把所有 ID 的详情数据缓存下来；3) 限制链路深度：限制 Agent 的工具调用层数，或者在架构层引入 GraphQL / BFF 层一次性聚合数据。最根本方案是在系统设计上避免 N+1 模式。
**一句话记忆法：** N+1 慢在串行，批量合并或缓存来解决。
**复习标签：** #Agent #Tool Calling #性能优化

### 卡片 4：

**Q：LLM（如 GPT）的 Function Calling 能力是怎么训练出来的？**
**A：** 核心思路是“指令微调 + 数据合成”。具体步骤：1) 构造包含函数调用场景的训练数据（用户 query + 函数 schema + 需要调用的函数名+参数）；2) 数据中混合“调用”和“不调用”的场景，训练模型判断何时需要调用；3) 训练数据中函数 schema 和调用参数要覆盖多种合法和非法形式；4) 关键技巧是在训练过程中加入“不调用正确”的负样本，避免模型无脑调用。现在很多模型（如 GPT-4、Claude）通过大量这类数据微调获得稳定的 function calling 能力。
**一句话记忆法：** 大量调不调的正反例，喂模型学会选。
**复习标签：** #Tool Calling #Function Calling #训练

### 卡片 5：

**Q：Agent 设计中的 Reflection（反思）机制具体怎么实现？**
**A：** Reflection 是让 Agent 对自身行为进行自我评估和纠错。有两种主流实现：1) 在线反思：在 ReAct 循环中插入一个 Reflect 步骤，每次 Action 后让 LLM 检查结果是否正确，若不正确则修正 ction 参数或者直接纠正输出；2) 事后反思：任务完成后让 LLM 回顾整个过程，分析哪一步出问题，记录到记忆库中，下次遇到类似任务时直接参考经验。实现时需要设计好 prompt，要求 LLM 用结构化格式输出反思结果（错误类型、正确做法、经验总结）。
**一句话记忆法：** 边做边纠偏是在线，做完复盘存经验是离线。
**复习标签：** #Agent #Reflection #自我纠错

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** Plan-and-Execute 范式中，如果 LLM 生成的初始计划发生错误（比如遗漏了关键步骤或者步骤顺序错误），如何在执行阶段发现并纠正？请从“计划验证”和“动态重规划”两个角度展开。

**面试口头版参考答案：**

Plan-and-Execute 的弱点就是计划可能出错。我的做法是两阶段：首先在做完计划后，加入一个计划验证步骤：让 LLM 检查计划是否合理、是否有遗漏、顺序是否正确，发现错误就立刻修正。其次在执行阶段，每一步执行完都把结果和预期对比，如果某一步的结果和计划预期明显不符，就触发动态重规划：让 LLM 基于已完成的步骤和当前状态，重新生成剩余计划。这样既能保留 Plan-and-Execute 的高效优势，又不至于因为初始计划错误而全盘失败。

**深入理解版参考答案：**

Plan-and-Execute 的核心问题是“计划可能不完美”，尤其是在任务复杂、信息不完整的场景下。解决思路是将“静态计划”变成“动态调整”。

第一阶段：计划验证。LLM 生成计划后，不直接执行，而是用一个验证器对计划进行校验。验证器可以是规则集（比如检查步骤格式、工具调用合法性）、也可以写另一个 LLM 来判断。验证的内容包括：步骤是否覆盖了所有需要的信息、步骤依赖关系是否正确、是否有冗余步骤。发现错误后，让 LLM 修正计划，反复直到验证通过或达到最大尝试次数。

第二阶段：执行时的动态重规划。在执行过程中，每次执行完一步，收集客观结果（Observation），将结果与计划的预期输出对比。如果发现“预期错误”（比如计划说查 A 表，但结果发现 A 表没有需要的数据），就触发重规划。重规划的核心是：不是废弃原计划重头来，而是保留已完成步骤的上下文，让 LLM 基于当前状态生成剩余步骤。这样可以避免从头计算，节约 token。

工程实现上，有一个关键设计：计划和执行之间需要一个“状态管理”层，记录每一步的输入、输出、状态、预期。这样当重规划触发时，可以快速重建当前上下文，而不是重算。

常见坑：重规划可能被无限触发，需要设置最大重规划次数。另外，重规划依赖 LLM 的推理能力，如果 plan 的方向本来就是错的多次重规划也解决不了，需要加入“上报失败”的兜底。

在项目中，我实初始 Phase 的 Plan-and-Execute 成功率约 70%。加入计划验证后提升到 85%，再加入执行态重规划后提升到 95%。但代价是增加了约 30% 的 token 开销，不过相比完全走 ReAct 还是节省了很多。

**常见追问与回答：**

**追问1：重规划怎么避免成本爆炸？一次重规划可能产生多轮 LLM 调用。**
答：是的，所以需要限制重规划的最大次数（通常设 2-3 次），并且每次重规划要求 LLM 输出简洁的计划，不重复已完成的步骤。另外，如果两次重规划后的计划一致但还是失败，说明该任务当前无法执行，直接返回兜底响应，避免 token 浪费。

**追问2：计划验证这一步用什么模型？用大模型会不会浪费？**
答：计划验证不需要特别强的模型。我用的是一个中等规模的模型（7B 级别），配合固定格式的 prompt 做结构化验证。规则验证用了一个轻量的 schema checker，纯代码实现，没有 LLM 调用的成本。真正需要模型推理的是“语义验证”——比如判断计划是否遗漏了关键步骤。这种场景用小模型成本可以接受。
