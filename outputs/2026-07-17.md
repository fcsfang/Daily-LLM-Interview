### 题目 1：SGLang 的 RadixAttention 与 vLLM 的 PagedAttention 有什么区别？实际项目中你会怎么选？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证

**可信度：**
中（来自面经整理，非官方文档）

**考察点：**
推理引擎选型、注意力机制优化、显存管理、延迟优化

**面试口头版回答：**
PagedAttention 是 vLLM 的核心创新，它把 KV cache 按固定大小的 block 管理，解决了显存碎片和浪费问题，让批处理更高效。RadixAttention 是 SGLang 的方案，它在 PagedAttention 基础上加了 radix tree 结构来复用公共前缀的 KV cache，比如多轮对话的前缀或系统 Prompt。选型上，如果服务是多轮对话或重复前缀场景多，我会优先选 SGLang；如果场景是短查询、高吞吐批处理，vLLM 更成熟稳定。

**深入理解版回答：**
PagedAttention 借鉴了操作系统分页管理的思想。在标准 Transformer 推理中，每个序列的 KV cache 是连续分配的，但不同序列长度差异大，导致大量碎片。vLLM 把显存分成固定大小的 block，每个序列只需要保存 block table，像 OS 的页表一样。这让它支持大 batch size，吞吐高。RadixAttention 更进一步：它把所有序列的 KV cache 组织成 radix tree，同一前缀的 cache 可以在不同请求间共享。比如“请根据以下资料回答问题：[长系统Prompt]”，所有请求的前半部分 token 相同，RadixAttention 直接复用 cache，不需要重新计算。而 vLLM 即使前缀相同，block 也不能跨请求共享。但 RadixAttention 的树结构维护和 eviction 策略更复杂，在非前缀场景优势不明显。所以项目选型时，我会先评估请求中公共前缀的比例。如果超过 30%，SGLang 的延迟优化明显；反之 vLLM 更通用。

**常见追问与回答：**
- 追问：两种方案在多轮对话场景下，哪个显存更省？
  回答：RadixAttention 能跨轮次复用历史前缀，显存节省约 20-40%。但如果对话轮次深度大，历史太长，前缀树会膨胀，管理成本上升。
- 追问：你们项目最终选了哪个？为什么？
  回答：我们做的是企业内部知识库问答，用户请求的 system prompt 很长且固定，所以我选了 SGLang。配合前缀缓存，首 token 延迟降低了约 30%。

**项目联系：**
在我做的 RAG + Agent 项目中，系统 Prompt 包含工具定义和几十个指令，长度约 2000 tokens。使用 SGLang 后，因为所有请求共享前缀，显存占用和首 token 延迟都明显下降。部署时我还在 SGLang 配置中打开了 `--enable-prefix-caching` 和显存预留参数。

**记忆关键词：**

### 题目 2：你们是怎么优化推理速度的？聊一聊量化、投机解码、前缀复用这几个手段的协同关系。

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证

**可信度：**
中（来自面经整理）

**考察点：**
推理优化全景、技术协同、工程落地顺序

**面试口头版回答：**
这三个手段分别解决不同瓶颈。量化降低单次计算的显存和数据搬运，是基础手段。投机解码通过小模型“猜词”，大模型只需要验证，减少了大模型 decode 的步数。前缀复用减少 KV cache 的重复计算，尤其适合多轮对话。协同关系是：量化让整体显存更宽松，为投机解码和更大 batch 腾出空间；投机解码降低延迟；前缀复用直接减少计算量。我一般先做量化，再上投机解码，最后加前缀缓存，每一步都要重新测吞吐和延迟。

**深入理解版回答：**
推理优化需要按瓶颈分类。量化（如 INT4）对 memory bandwidth bound 的 decode 阶段效果最好，因为此时 GPU 主要时间花在把权重从显存搬到计算单元。单次量化可以让 token 生成速度提升 2-3 倍。投机解码（speculative decoding）解决的是模型自身推理速度慢的问题：用一个轻量 draft 模型快速生成 K 个候选 token，然后主模型一次并行验证这 K 个 token 是正确的还是错误的。如果正确率 80%，平均每步实际生成 0.8K + 1 个 token。而前缀复用（prefix caching）减少的是预填充（prefill）阶段的计算量，避免重复计算系统 Prompt 和历史对话的 KV cache。三个技术组合时，量化先确保基础速度，投机解码在 decode 阶段进一步提速，前缀复用消除 prefill 阶段的冗余。但要注意：投机解码的 draft 模型需要和主模型对齐，否则 reject 率高反而变慢；前缀复用需要前缀一致，不适合无系统 Prompt 的场景。所以我在项目里是分阶段上线的：先量化到 INT4 测试，再挂载 draft 模型（如用同系列的 0.5B 模型），最后在响应体里加前缀 cache 标识。

**常见追问与回答：**
- 追问：投机解码怎么保证 draft 模型的正确率？
  回答：常用方法是用主模型做蒸馏训练 draft 模型，或者直接用小版本的 same family 模型。
- 追问：你们量化用了 INT4，准确率掉了几个点？怎么补偿的？
  回答：评测下来在 RAG 场景下丢掉不到 1% 的准确率。通过增加一个 rerank 阶段来补偿，保证 top-K 返回质量不下降。
- 追问：三个手段一起上，延迟能降多少？
  回答：在我们的场景里，首 token 延迟从 1.2s 降到 0.4s，平均每个 token 生成从 40ms 降到 18ms。但这是理想情况，取决于 draft 成功率（我们测得约 75%）。

**项目联系：**
在我部署 Agent 服务时，主要瓶颈是响应慢，尤其是 multi-step tool calling。我先用 AutoGPTQ 量化到 INT4，显存从 24GB 降到 8GB，然后挂了一个 0.5B 的聊天模型做 draft，最后在 SGLang 上开了前缀缓存。上线后 P99 延迟从 6s 降到 2s。

**记忆关键词：**
#量化 #投机解码 #前缀复用 #协同优化 #延迟

### 题目 3：请解释一下 KV cache 是什么？它的显存占多少？优化手段有哪些？

**来源依据：**
- LLM Interview Questions and Answers (2026) - https://www.interviewbit.com/llm-interview-questions-answers - 技术博客 - 中
- LLM面试精选 - https://developer.volcengine.com/articles/7446997500902572070 - 技术博客 - 中

**可信度：**
中（技术博客汇总，非官方文档）

**考察点：**
Transformer 推理机制、显存模型、工程优化

**面试口头版回答：**
KV cache 是在自回归生成时，把每层 self-attention 中计算过的 key 和 value 矩阵缓存下来，避免每个 token 都重新计算前面的 attention。它主要占用显存，计算公式近似：2 * batch_size * seq_len * num_layers * hidden_dim * 2（字节数，以 FP16 为例）。比如 7B 模型、batch=1、seq_len=4096，单轮对话 KV cache 约占用 2GB。优化手段包括：GQA/MQA 减少 KV heads 数、PagedAttention 消除碎片、KV cache 量化到 INT8、以及通过 streaming LLM 或 window attention 限制缓存长度。

**深入理解版回答：**
KV cache 是推理时最关键的显存瓶颈。标准计算：以 LLaMA 7B（32 layers, hidden 4096）为例，每个 token 每层需要存储 2 * 4096 * 1（key + value），共 32 * 2 * 4096 = 262K 个 float16 元素，约 0.5 MB。当序列长度 4096 并 batch 为 32 时，KV cache 需 0.5 MB * 4096 * 32 ≈ 64 GB，远超模型权重（约 14GB）。所以 KV cache 是决定最大批处理的关键。大 batch 能提高吞吐，但 KV cache 会限制 max_batch_size。优化重点：一是架构层面用 GQA（如 LLaMA 2-70B 用 8 个 KV heads）直接减少 KV cache 到 1/8；二是工程层面用 PagedAttention 或 flash decoding 减少实际分配量；三是结合量化把 KV cache 用 INT8 存，显存减半。我在项目中采用了 GQA 模型 + PagedAttention + KV cache INT8 量化的组合，使同样显存下 max_batch_size 从 4 提升到 16。

**常见追问与回答：**
- 追问：KV cache 为什么用 FP16 而不是 INT8？
  回答：主要是精度。KV cache 对生成质量敏感，直接 INT8 可能影响。但可以通过 per-token 量化或 smoothquant 来做。
- 追问：流式输出时 KV cache 如何管理？
  回答：保持连续长度增加。PagedAttention 可以动态 allocate 新 block，流式场景天然支持。

**项目联系：**
在部署 RAG 服务时，我们需要支持 50+ 并发请求，每个请求平均 2000 tokens。如果不优化 KV cache，显存根本撑不住。我用 vLLM 部署，它通过 PagedAttention 做显存复用，才使得 8 张 A100 能跑起 50 个并发 batch。

**记忆关键词：**
#KVcache #显存计算 #GQA #PagedAttention #吞吐

### 题目 4：大模型推理时，批处理（batching）是怎么影响延迟和吞吐的？你们在实践中怎么调 batch size？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你轻松拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- LLM Interview Questions and Answers (2026) - https://www.interviewbit.com/llm-interview-questions-answers - 技术博客 - 中

**可信度：**
中（多次出现的高频问题）

**考察点：**
工程集群优化、吞吐 vs 延迟 tradeoff、动态批处理

**面试口头版回答：**
批处理决策本质是在吞吐和延迟之间做 tradeoff。小 batch 延迟低但吞吐低，大 batch 吞吐高但首 token 延迟增加，因为需要等最慢的那个请求。实践中我常用动态批处理（continuous batching），它不等全部请求完成才释放 batch，而是先完成的请求立即释放，新请求立即插入。调优时我通过压测找“拐点”：逐步增大 batch，看延迟何时超出 SLA（如 2s），就固定在那个 batch 大小。

**深入理解版回答：**
批处理的核心矛盾：Pre-fill 阶段是计算密集型，批处理能摊薄 GPU 计算；Decode 阶段是 memory bandwidth 密集型，批处理后所有请求的 KV cache 加载会增大带宽压力。所以连续批处理（如 vLLM 的实现）对 decode 阶段更有优势。连续批处理的核心是：每个请求独立管理，GPU 时间片轮转；当一个请求生成结束，立即从 pending 队列中取出新请求，让 GPU 一直被利用，不像传统的静态批处理必须等所有请求结束。调优步骤：1. 先用单请求测基准延迟和显存；2. 用压测（如 locust）逐步升高并发，记录 P50/P99 延迟；3. 找到“延迟拐点”——即延迟开始非线性飙升的点，这通常是显存或带宽瓶颈；4. 最后取该 batch 值作为线上限值。我实践中发现，对于 LLaMA 7B，batch size 从 8 升到 16，吞吐增加约 40%，但 P99 延迟增加 1.5 倍，所以最终保持在 12。

**常见追问与回答：**
- 追问：连续批处理带来的问题是什么？
  回答：实现复杂度高，需要动态调度；并且部分请求可能被“饿死”（一直 pending 的短请求要等长请求结束）。
- 追问：如何避免长请求阻塞短请求？
  回答：可以用时间片切分或 pre-emption，如 Suspend & Resume 机制。

**项目联系：**
在 RAG 服务中，不同查询的输入长度差异很大（从 50 tokens 到 2000 tokens）。我通过 vLLM 的“最长优先”调度策略减少长请求阻塞。同时设置了 `max_model_len` 为 8192，避免单个请求撑爆显存。

**记忆关键词：**
#连续批处理 #吞吐延迟 #动态调度 #压测拐点

### 题目 5：你在项目里怎么测模型推理的延迟和吞吐？用什么工具？怎么看服务的稳定性？

**来源依据：**
- 字节跳动大模型实习面经：从Agent记忆到RAG优化，13道真题解析助你拿下Offer！ - https://gitcode.csdn.net/6a2ccdca10ee7a33f27c07bf.html - 面经或社媒 - 需进一步验证
- 大厂RAG面试题：24个RAG八股文 - https://www.cnblogs.com/crazymakercircle/p/18980652 - 技术博客 - 中

**可信度：**
中（综合多家资料）

**考察点：**
压测实践、监控指标、服务稳定性

**面试口头版回答：**
我主要用三套手段。第一，离线压测：使用 locust 或自己写的压力脚本，模拟线上请求分布，测 P50、P99 延迟和 QPS。第二，在线监控：用 Prometheus + Grafana 监控 GPU 利用率、显存占用、请求排队队列长度。第三，稳定性测试：跑 1 小时高压场景，看延迟分布是否稳定，有没有 OOM 或显存泄露。稳定性核心指标是不超过 SLA 的“失败率”，我一般要求 P99 延迟 < 3s 且错误率 < 0.1%。

**深入理解版回答：**
测延迟和吞吐要区分预填充和解码阶段的指标。预填充阶段测“首 token 延迟”（TTFT），解码阶段测“每个 token 时间”（TPOT）。吞吐指标有两个：QPS（每秒请求数）和 Tokens Per Second。工具方面，我常用 vLLM 自带的 `benchmark_serving.py` 脚本，它可以模拟真实的请求分布，并输出 P50/P90/P99 统计。更细粒度的方法是：加 tracing（如 OpenTelemetry）记录每个请求的各个阶段耗时，比如 retriever、reranker、LLM generation。这样才能定位瓶颈。服务稳定性检查：1. 长时间压测时监控 GPU 显存是否线性增长（有泄露）；2. 看请求排队队列长度，如果队列持续增长说明容量不够；3. 设置熔断（circuit breaker）——当失败率超阈值或 P99 超过 5s，自动拒绝部分请求，保护集群不雪崩。我在项目里还做了一个轻量级的“健康检查”，定期发一个已知 short query，如果延迟超 10s，就标记该副本不健康并重启。

**常见追问与回答：**
- 追问：TTFT 和 TPOT 哪个对用户体验影响大？
  回答：对聊天类，TTFT 更重要（用户不想等第一个字）；对摘要类，TPOT 更重要（用户等着读完）。
- 追问：你们线上 P99 延迟不稳定，可能是什么原因？
  回答：常见原因是显存不够时触发显存换入换出，或者请求的输入长度差异过大导致 batch 不均。

**项目联系：**
在项目上线前，我专门搭建了一套压测环境，通过 SGLang 的 `--log-requests` 参数获取每个请求的详细耗时日志，然后用 pandas 分析各阶段耗时占比，发现 reranker 的延迟占了 40%，最后通过把 reranker 也部署到 GPU 并加 batch，才降低了 P99。

**记忆关键词：**
#延迟 #吞吐 #压测 #TTFT #监控

  - 1. MHA、GQA、MQA 的核心区别是什么？推理时显存差异有多大？
  - 2. FlashAttention 的原理是什么？为什么要用它？

**对应参考答案：**

**问题 1：MHA、GQA、MQA 的核心区别？推理时显存差异？**
MHA 是每层每个 head 都有独立的 K、V、Q 映射，KV cache 最大；GQA 是多 query heads 共享少数 KV heads（如 LLaMA 2-70B 是 8 个 KV heads）；MQA 是所有 query heads 共享一个 KV head。显存方面，假设 head_dim=128，heads=32，seq_len=4096，batch=1，MHA 的 KV cache 约 2 * 32 * 128 * 4096 * 2 字节 ≈ 64MB；GQA 如果 KV heads=8，则降为 16MB；MQA 降为 2MB。所以 GQA/MQA 能显著提升 max_batch_size。

**问题 2：FlashAttention 的原理？为什么要用它？**
FlashAttention 通过 tiling 和 online softmax 避免把整个 N×N 注意力矩阵写回 HBM（高频内存），而是在 SRAM（高速缓存）中分段计算。这样减少了显存带宽开销，使模型在长序列训练和推理时更快，显存占用也从 O(N²) 降到 O(N)。因为它减少了 HBM 读写，实际训练加速约 2-4 倍。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** KV cache 占用显存的计算公式是什么？以 LLaMA 7B 为例，单请求 4096 tokens 大约需要多少显存？
- **A：** 公式：2 * batch_size * seq_len * num_layers * hidden_dim * 2 字节（FP16）。LLaMA 7B 有 32 层、hidden=4096，单请求 4096 tokens 约需 2 * 1 * 4096 * 32 * 4096 * 2 ≈ 2 GB。这是推理时的主要显存开销。
- **一句话记忆法：** KV cache 像栈——但每次 push 一个 token 都要占一层显存；层数（layers * hidden）决定“每层多厚”，长度决定“多高”。
- **复习标签：** #模型部署 #KVcache #显存

### 卡片 2
- **Q：** vLLM 的 PagedAttention 怎么解决显存碎片问题？
- **A：** 像操作系统分页管理一样，把显存切成固定大小的 block。每个序列只保存 block table，不同序列之间可以共享 block，减少了内部碎片和外部碎片。这使得支持更大 batch size，提高吞吐。
- **一句话记忆法：** 以前是“包车位”，现在变成了“共享车架”，更省地儿、更灵活。
- **复习标签：** #模型部署 #PagedAttention #显存优化

### 卡片 3
- **Q：** SGLang 的 RadixAttention 相比 PagedAttention 有什么额外优势？
- **A：** 它用 radix tree 组织 KV cache，允许不同请求共享公共前缀的 cache（如系统 Prompt、多轮对话的前几轮）。在系统 Prompt 很长或多轮对话场景中，可以大量复用，降低首 token 延迟 30-50%。
- **一句话记忆法：** “公共前缀共享”，就像共享单车的首 30 分钟免费，后面才各付各的。
- **复习标签：** #模型部署 #RadixAttention #前缀复用

### 卡片 4
- **Q：** 推测性解码（Speculative Decoding）的原理是什么？能提升多少速度？
- **A：** 用一个小模型（draft model）快速产生 K 个候选 token，大模型并行验证。如果候选正确率 80%，每步实际生成约 1 个 token 的时间可以生成 1+K*0.8 个 token。速度提升通常可达 2-3 倍，但不适合长尾词汇多的任务。
- **一句话记忆法：** 小模型打草稿，大模型抄作业，抄对的就省去了重新想的时间。
- **复习标签：** #模型部署 #投机解码 #提速

### 卡片 5
- **Q：** 量化（INT4/INT8）为什么能提速？量化的副作用是什么？
- **A：** 量化把权重从 FP16 变成更小的格式（INT4 是 4 比特），减少显存占用和数据搬移带宽，让 GPU 在 memory bound 的 decode 阶段更快。副作用是精度损失，可能在复杂推理任务上降低 1-5% 的准确率，需结合下游评测决定是否接受。
- **一句话记忆法：** 减肥减的是显存，但可能稍微丢点智商，得看值不值得。
- **复习标签：** #模型部署 #量化 #提效 #精度损失

## 4. 今日面试专项训练

### 4.1 大模型理论追问题：FlashAttention 的原理

**问题：**
请解释 FlashAttention 的核心原理。它是如何在不改变模型参数的前提下，实现注意力计算的加速和显存降低的？

**面试口头版参考答案：**
FlashAttention 的核心是“分块计算”+“在线 softmax”。传统 attention 会计算一个很大的 N×N 注意力矩阵并写回 HBM（高频显存），导致带宽瓶颈。FlashAttention 把 Q、K、V 分成小块（tiling），每次只加载小块到 GPU 的 SRAM（高速缓存）中计算。同时用 online softmax 技术，在不知道全局最大值的情况下，增量式地算出最终结果。这样避免了大量 HBM 读写，训练和推理都更快，显存也从 O(N²) 降到 O(N)。

**深入理解版参考答案：**
标准 attention 计算流程：先算 S = QK^T（产生 N×N 矩阵，写回 HBM），再对每行做 softmax（又需要读取 S），然后加权 V。这个过程中 HBM 读写量是 O(N² + N² + N×d)。FlashAttention 的思路是打破这种“先算出完整矩阵”的做法。它用 tiling（分块）：把 Q 分成多个块，K、V 也分块。只在 SRAM 里加载当前块，并在块内部计算局部 softmax，维护一个“在线归一化统计量”来近似全局 softmax。这个过程是迭代的：每次处理一个块，更新状态，最终得到正确的输出。核心贡献是两篇论文（FlashAttention 和 FlashAttention-2），提出了“IO-aware”的算法，即算法设计时要显式考虑数据在哪里存的。效果是训练 LLaMA 7B 快 2-4 倍，且 N=16K 时显存从 O(256M) 降到 O(16K)。注意事项：FlashAttention 只改变计算顺序，不改变模型参数或结果精度（数值上等价）。框架支持方面，PyTorch 2.0 原生集成，vLLM 和 SGLang 也都内置了 fuse 版本。

**2 个可能追问与简短回答：**

- **追问 1：FlashAttention 对推理的影响大吗，还是只对训练有效？**
  - 回答：对推理也有效。尤其是 decode 阶段，虽然 decode 是 memory bound，但预填充（prefill）阶段也有用。FlashAttention-2 对 decode 阶段做了优化，使用更少的 wave（GPU 线程束），提高利用率。
- **追问 2：如果我用 FlashAttention，还需要手动配置 tiling 的尺寸吗？**
  - 回答：不需要，库做了最优化。但可以通过设置 `attn_implementation="flash_attention_2"`（Hugging Face）来启用。不同 GPU（A100 vs H100）的 SRAM 大小不同，库内部会根据硬件自动选择最优 block size。
