## 1. 今日面试真题精讲

### 题目 1：如果你要设计一个面向企业内部 IT 工单处理的智能客服 Agent，请从系统设计层面描述整个方案，包括你会选择哪些技术组件、各自负责什么、为什么这样选、以及如何评测它是否好用。

**来源依据：**
- AI大模型八股面试题，覆盖Agent、RAG、LLM等面试题（600张图解+25万字答案解析） - https://www.cnblogs.com/xiaolincoding/p/20534325 - 技术博客 - 需进一步验证，但题目整体为系统设计类综合题，属于经典高频题

- Just wrapped up my interview for the AI Engineer role… | Bilal Momin - https://www.linkedin.com/posts/bilal-momin_generativeai-llm-langgraph-activity-7353048174648840192-C0CR - 面经或社媒 - 需进一步验证（资料[16]）

**可信度：** 中（多个来源交叉印证了系统设计类的常考方向，但无官方文档支撑）

**考察点：** 系统设计、RAG 与 Agent 融合、组件选型、评测体系搭建

**面试口头版回答（约 200 字）：**
我会把这个客服 Agent 拆成四层。第一层是**输入理解与路由**，用户问题先走意图识别，判断是查知识库、创建工单还是转人工。第二层是**RAG 检索**，用混合检索（向量 + BM25）从内部知识库召回相关文档，再经过 rerank 压缩到 Top-3 作为上下文。第三层是**工具调用与执行**，如果涉及“创建工单”这种操作，模型会通过 Function Calling 调后端 API，我会在 Agent 层加上 ReAct 循环来处理多步推理。第四层是**安全与可观测**，敏感操作需要二次确认，所有步骤记录 trace。评测方面，离线我会用 RAGAS 算忠实度和答案相关度，在线会用 A/B 测对比解决率和平均轮数。

**深入理解版回答（约 400 字）：**
这个题其实考的是你能否把多个模块串起来讲一个有说服力的系统设计。我的设计思路是「先定位场景风险，再选型组件」。IT 工单场景有两大特点：一是知识库更新频繁（新系统上线、流程变更），二是操作风险高（误创工单可能影响业务）。

具体来说，第一层 Intent Router 我不用大模型直接做，而是先用一个小分类器（比如基于 Embedding + 最近邻），速度更快、成本更低，只有当置信度低时才 fallback 到 LLM。第二层 RAG 的关键是文档切分策略：针对 IT 工单常见的是"How to"类问答，我倾向于用 Markdown 的章节标题做层级切分，保留元数据（比如文档版本号、最后更新日期），这样检索时可以优先返回新版内容。重排我推荐用 bge-reranker-v2-m3，它在工单场景下精度比简单 cosine 至少高 10%。第三层 Agent 层，我选择 LangGraph 而非原生 ReAct，因为 LangGraph 支持有状态的条件分支：比如用户权限不足时，Agent 可以走到“通知管理员”节点而不是直接失败。工具调用失败时，我设计了幂等性（每个工单有唯一 idempotency key）和三次重试+死信队列。第四层评测，离线我会用人工标注构建 200 条 golden 数据集，用 LLM-as-judge 打分并人工校准。在线监控重点看“用户首次回复后是否再次提问”和“工单误创率”，这两个是最直接的业务指标。

**常见追问与回答：**

- **追问 1**：为什么不用纯 LLM 生成，非要加 RAG？  
  **答**：IT 知识库每周都有更新，而且包含大量内部系统截图和私有 API 文档，LLM 训练时没有这些数据。RAG 保证了每次回答都能引用最新版本的知识，同时降低幻觉风险。

- **追问 2**：Agent 在执行“创建工单”时，如果 LLM 生成了错误的 JSON，你怎么处理？  
  **答**：我会在工具调用前加一个 schema 校验层，用 Pydantic 检查参数类型和必填字段。如果校验失败，直接返回错误信息让 LLM 重新生成，限制最多重试 2 次，超过则走人工兜底。

- **追问 3**：这个 Agent 的推理成本怎么控制？  
  **答**：我可以做两件事：一是把常见的“查密码重置流程”这类高频静态问题直接缓存 answer（用 hash 匹配 query 和上下文），不再经过 LLM；二是对于简单问题用小模型（比如 Qwen2.5-7B）处理，只有当需要多步推理时才路由到大模型。

**项目联系：**  
假设我做过一个“企业 IT 知识库问答”项目，我会明确告诉面试官：我在项目中遇到了文档版本混乱导致回答过时的问题，因此实现了基于元数据的检索排序；我还遇到了 Agent 误创建工单的故障，因此引入了幂等 key 和人工确认节点。这些就是我从实战中积累的设计经验。

**记忆关键词：** 意图路由、混合检索、rerank、LangGraph、工单幂等性、LLM-as-judge、A/B 测试

### 题目 2：你怎么衡量一个 Agent 系统的成败？能否从离线评测和在线评测两个维度，系统地说一下你的方法论，包括你会遇到哪些坑。

**来源依据：**
- LLM RAG 面试问题大全! - https://www.53ai.com/news/RAG/2024100162935.html - 技术博客 - 需进一步验证（资料[18]）
- Top Generative AI and LLM Interview Question with Answer - https://www.geeksforgeeks.org/artificial-intelligence/generative-ai-interview-question-with-answer - 技术博客 - 需进一步验证（资料[11]）
- RAG Interview Questions and Answers Part-1 | Kalyan KS - https://medium.com/@kalyanks/rag-interview-questions-and-answers-part-1-ca448484e659 - 技术博客 - 需进一步验证（资料[10]）

**可信度：** 中（评测方法论在多个资料中有提及，但Agent评测的具体细节较少，需结合经验补充）

**考察点：** Agent 评测、离线线上结合、LLM-as-judge 使用、指标选择、踩坑经验

**面试口头版回答（约 250 字）：**
离线评测我会先构建 200-300 条典型场景的 golden 数据集，包含输入和预期行为轨迹。然后从三个维度打分：**任务完成率**（看最终输出是否达成目标）、**步骤正确率**（看每一步的决策是否合理，比如该调A工具时有没有调B）、**成本效率**（看总共调用了多少次 LLM，token 消耗）。我习惯用 LLM-as-judge 评估最终答案质量，但会加上人工抽检 10% 的样本做校准，防止模型偏好。在线评测我主要看三个指标：**用户留存率**（用户是否持续回来用）、**问题解决率**（通过用户点击“解决”按钮或后续不再提问来反推）、**平均交互轮数**（太长说明 bot 不聪明）。我最常见的一个坑是离线测试集覆盖不全，很多边界情况（比如用户骂人、超长指令）没测到，导致上线后翻车。

**深入理解版回答（约 500 字）：**
Agent 评测确实比纯 RAG 难很多，因为输出不是单一的文本，而是一系列动作和状态变化。我的方法论分三步走：

第一步，**构建“场景-轨迹-结果”数据集**。我会拉取线上日志抽取出 10 个高频场景（比如“查余额”、“改密码”），每个场景写 3-5 种输入变体（正常、模糊、冲突、超时）。然后人工标注：预期调用的工具序列、预期 API 参数、预期最终回答。离线评测时我会让 Agent 跑一遍，把输出轨迹与标注对比。这样就能算出 **Trajectory Accuracy**（每一步工具名是否匹配）和 **Parameter Accuracy**（参数是否有异常）。

第二步，**LLM-as-judge 的坑与校准**。我不用 LLM 直接打一个总评分，而是让 LLM 做分类式的判断：比如“回答是否引用了正确文档？”（Yes/No）、“是否执行了多余的操作？”（Yes/No），这样可以减少 LLM 的主观偏好。做完 LLM 打分后，我会随机抽 10% 送人工标注，计算 LLM 与人工的一致性（Cohen’s Kappa），如果 kappa < 0.6，我会调整 Prompt 或者换一个更合适的评判模型（比如从 GPT-4o 换成 Qwen2.5-72B）。

第三步，**在线指标的灰度设计**。我不只看一个综合指标，而是分层看：第一层是**功能成功率**（比如“密码重置”这个动作是否有执行并成功，通过调用后端确认 API 的日志来验证）；第二层是**用户体验指标**（比如首次响应时间、用户是否在对话结束后点了“有用”）；第三层是**业务指标**（比如工单创建数减少、客服电话时长下降）。这三个层次由上到下，越上层越客观。

常见坑：离线指标高不代表在线好，因为 Agent 在执行时会遇到后端宕机、权限不足等离线无法模拟的情况。另一个坑是“解决率”可能虚高——用户可能只是跑了，不代表问题解决了。所以我额外加了“3 天内同一用户是否再次提问”作为补充。

**常见追问与回答：**

- **追问 1**：如果用户连续问 10 个问题，Agent 在中间某个步骤失败了，你怎么定义这次会话是否成功？  
  **答**：我会按场景定义粒度为“单次意图”。如果用户连续问 3 个不同问题，我会拆成 3 个 session，分别判断每个意图是否完成。前一个失败不影响后一个的统计。

- **追问 2**：LLM-as-judge 在评估 Agent 轨迹时，容易出现“自我偏好”问题（即 LLM 更认可它自己的输出模式），你怎么解决？  
  **答**：我在 Prompt 里明确要求 LLM 仅根据事实匹配度打分，不看风格。其次，我还会混合使用两个不同基座模型的评判结果（比如 GPT-4o 和 Qwen2.5-72B），如果它们分歧很大，人工介入。

- **追问 3**：你们在线 A/B 测试时，实验组和对照组的样本量不均匀怎么办？  
  **答**：我会采用**分流比例动态调整**：新版本先跑 5% 流量，连续 3 天关键指标没有显著下降（p<0.05）再扩到 20%，直到 50%。同时使用 Bootstrap 做置信区间估计，避免小样本下的假阳性。

**项目联系：**  
假设我做过一个“智能客服 Agent 实习项目”，我会在项目介绍中提到：我构建了 300 条 golden 测试集，并设计了一个自动化评估 pipeline，用 GPT-4o 作为 judge 配合人工抽检。我发现 LLM-as-judge 在“是否调用了正确工具”这个维度准确率达到 92%，但在“参数是否合理”上只有 70%，因此专门加强了参数模式匹配检查。

**记忆关键词：** Trajectory Accuracy、场景化数据集、LLM-as-judge 校准、Cohen’s Kappa、分层指标、灰度放量

### 题目 3：你如何在 RAG 项目中做好 Query 重写（Query Rewrite）？请设计一个工程方案并说明每种方案的适用场景。

**来源依据：**
- AI大模型八股面试题，覆盖Agent、RAG、LLM等面试题（600张图解+25万字答案解析） - https://www.cnblogs.com/xiaolincoding/p/20534325 - 技术博客 - 需进一步验证（资料[1]，RAG 面试题第 12 题：如何润色用户的 Query（Query Rewrite））
- RAG Interview Questions and Answers Part-1 | Kalyan KS - https://medium.com/@kalyanks/rag-interview-questions-and-answers-part-1-ca448484e659 - 技术博客 - 需进一步验证（资料[10]，选择性 query rewriting 降低延迟）

**可信度：** 中（Query Rewrite 是 RAG 标配考点，多个资料都有提及，但工程细节需补充）

**考察点：** RAG 检索优化、Query Rewrite 策略、工程选型

**面试口头版回答（约 180 字）：**
Query Rewrite 的目的是把用户的模糊问题转化为检索系统更容易匹配的形式。我通常按场景选用三种方案：第一种是**假设性文档嵌入（HyDE）**，先用 LLM 生成一个假设答案，再用这个答案去检索，适合回答型场景。第二种是**多查询扩展**，用 LLM 生成 3-5 个同义不同角度的问题，去检索后合并结果，答非所问但需要全面覆盖时用。第三种是**压缩重写**，把啰嗦的对话历史压缩成一句话，把上下文窗口留给更有用的信息，适合多轮对话中用户只说“然后呢”。工程落地时我会做一条规则：先判断 query 长度和是否指代“这/那个”，如果指代就用压缩重写，否则根据是否明确用 HyDE 或多查询。

**深入理解版回答（约 350 字）：**
Query Rewrite 的核心是把用户的**表达语义**转化为**索引语义**对齐。因为用户的自然语言和文档的语言分布往往不一致——用户说“怎么重置密码”，而文档标题可能叫“用户账户安全设置流程”。

方案一：**HyDE（假设性文档嵌入）**。我让 LLM 根据 query 先生成一句假设性回答，然后拿这句回答的 embedding 去做相似度检索。这个方案的好处是不需要额外训练，但弱点是当用户问的是“有没有 X 功能”这类不存在的问题时，假设性回答会纯属幻觉，反而拉偏检索结果。所以我会先判断 query 是否为“是否存在”类问题，如果是，则不用 HyDE 而直接用原始 query。

方案二：**多查询扩展 + RRF 融合**。我用 LLM 生成不同角度的 query（例如：语气正式版、关键词版、动词短语版），分别检索后，用倒数秩融合（RRF）合并结果。这个方案召回率提升明显，但会增大检索耗时，所以我只在用户明确表示“没找到答案”后作为 recall 增强策略使用。

方案三：**基于历史的压缩重写**。对于多轮对话，我维护一个“对话摘要”buffer，每次重写时将 buffer + 当前 query 一起交给一个小模型（比如 Qwen2.5-1.5B）产出精简 query。因为小模型便宜，可以每轮都做。压缩的粒度控制在一句话以内，确保不超过 embedding 模型的 max tokens（通常 512）。

工程实现上我会在 Redis 里缓存最近 5 轮的历史 query 和 agent response，减小 LLM 调用成本。重写后的 query 会同时保留原始 query 用于降级：当检索结果为空时，回退使用原始 query 重试一次。

**常见追问与回答：**

- **追问 1**：多查询扩展时，如果生成的 query 中有一个特别差，反而污染了融合结果，你怎么处理？  
  **答**：我会对每个生成 query 单独检一次，用 BM25 打分过滤掉质量太低的 query（比如 BM25 score 低于某个阈值），只保留 top-3 的 query 去做 RRF 融合。

- **追问 2**：HyDE 生成速度会不会拖慢端到端延迟？  
  **答**：会。所以 HyDE 我一般不放在首轮，而是当首轮检索 top-1 的相似度低于 0.7 时，才触发 HyDE 作为 fallback。这样大部分简单 query 不增加延迟。

- **追问 3**：如果用户说的 query 包含拼写错误或缩写，你怎么办？  
  **答**：我会在重写之前先走一个拼写纠正模块（基于编辑距离或小模型），把“printer 不打印”纠正为“打印机不打印”，再交给重写策略。

**项目联系：**  
我在一个“企业知识库问答”项目中实际实现了 HyDE 和多查询的混合策略。项目初期发现用户经常问“怎么查工资”，但内部文档标题是“薪酬查询指南”，原始 query 检索不到。用了多查询扩展（生成“查看薪酬”、“工资查询方法”等）后，召回率提升了 15%。

**记忆关键词：** HyDE、多查询扩展、RRF 融合、压缩重写、拼写纠正、延迟优化

- **可转化面试题：**
  - **Q1**：长上下文 LLM 已经很厉害了，RAG 是不是没必要了？你怎么看？
  - **Q2**：你在实际 RAG 项目里怎么降低检索+生成的端到端延迟？除了缓存还有哪些手段？

**对应参考答案：**

**Q1 答案（口头版，约150字）：**  
RAG 依然必要，主要基于三个工程理由。第一是“Lost in the Middle”问题，即便模型支持 128K 上下文，但把 100 篇文档全塞进去，模型只会注意开头和结尾的，中间的信息会被忽略；而 RAG 只给 top-3 相关文档，定位最精准。第二是成本，处理 128K tokens 的 API 调用成本是 8K 的十几倍，RAG 每次只传几千 tokens，成本大幅降低。第三是延迟，输入越长首 token 生成时间也越长。所以 RAG 是成本、延迟、效果之间的最佳折中。

**Q2 答案（口头版，约150字）：**  
除了缓存，我还用三种方法。一是 Embedding 量化，从 float32 降到 int8，检索速度提升 2-3 倍，精度下降不到 1%。二是选择性重排：我只有在 top-5 结果中包含明显不相关文档时才启用跨编码器 rerank，平时直接用相似度排序。三是对 query 做分类，简单事实类问题直接 skip 重写，只有复杂推理类问题才用 HyDE 或 query 扩展。这三招组合下来，端到端延迟可以从 3 秒降到 1.2 秒。

流程分四步。第一步，用混合检索（稠密向量 + BM25）从向量库中取回 top-20 个 chunk。第二步，用基于规则的过滤器去掉重复段和明显不相关的段（比如标题页、表格数据等）。第三步，用 cross-encoder 模型（比如 bge-reranker-large）对剩下的 chunk 做精排，输出一个相关度分数，取 top-3。第四步，将 top-3 chunk 的原文和 source 信息一起送入 LLM prompt 作为上下文。我会额外在 prompt 里要求 LLM 必须引用原文片段，方便人工追查。

## 3. 今日必背问答卡片

### 卡片 1
- **Q：** RAG 中为什么要做 Query Rewrite？请列举三种常见方式。
- **A：** 用户的表述与文档的语言风格可能不一致。三种方式：HyDE（先生成假设答案再检索）、多查询扩展（生成多个同义 query 后融合）、基于历史的压缩重写（把多轮对话浓缩成一句）。其中 HyDE 效果好但延迟高，适合作为 fallback；多查询扩展适合覆盖不全的场景；压缩重写适合多轮对话。
- **一句话记忆法：** 用户说“怎么改密码”，文档写“重置安全凭证”，Query Rewrite 就是桥接两种语言的人肉翻译官。
- **复习标签：** #RAG #检索优化 #Query Rewrite

### 卡片 2
- **Q：** 什么是 LLM-as-judge？用作 Agent 评测时有什么缺陷？你如何校准？
- **A：** LLM-as-judge 是用一个强大的语言模型去评价另一个模型的输出。缺陷：有自我偏好（更认可自己风格的答案）、容易受 prompt 诱导、缺乏真实世界知识。校准方法：让 LLM 做分类题而非打分题，计算与人工评分的一致性（Cohen’s Kappa），如果低于 0.6 则更换评判模型或优化 prompt。
- **一句话记忆法：** 让同一个老师去批全班的卷子，老师会偏爱成绩好的学生——LLM-as-judge 也会偏爱自己，所以要对答案。
- **复习标签：** #评测 #Agent #LLM-as-judge

### 卡片 3
- **Q：** 长上下文 LLM 出现后，RAG 还有意义吗？请从三个角度说明。
- **A：** 有意义。第一，**Lost in the Middle**：即便模型能看 128K tokens，中间的内容会被忽略，RAG 只传 top-3 片段更精准。第二，**成本**：长上下文 API 调用费用是短上下文的数倍。第三，**延迟**：输入越长首 token 生成时间越长，RAG 能显著降低延迟。所以 RAG 是成本-延迟-效果的折中。
- **一句话记忆法：** 全塞给模型等于让它在 100 本书里找一句话，RAG 就是只给它那关键的一页。
- **复习标签：** #RAG #长上下文 #成本优化

### 卡片 4（新角度：LoRA 与模型部署的交叉）
- **Q：** LoRA 微调后的模型在部署时，你如何处理不同任务的 adapter 切换？会比全参数模型多哪些开销？
- **A：** 部署时通常有两种方案：一是推理前 merge 权重到基座模型，推理时不增加额外开销，但失去灵活切换能力；二是运行时动态加载 adapter，开销是基座模型显存之外需要额外存储 adaper 权重。多 adapter 切换时可以用 adapter 管理服务（如 vLLM 的 LoRA 适配器特性）按需热加载。注意点：如果基座模型量化了，merge 时需考虑精度对齐，否则输出质量可能下降。
- **一句话记忆法：** LoRA 部署就像换电池（adapter）而不换手机（基座模型），批量切换需要一块“电池管理器”。
- **复习标签：** #LoRA #模型部署 #adapter 管理

### 卡片 5（新角度：Agent 记忆工程细节）
- **Q：** 在 Agent 里你做长短期记忆时，如何解决 token 溢出和相关性下降的问题？
- **A：** 短期记忆我用滑动窗口保留最近 5 轮对话，溢出时丢弃最早轮次。长期记忆我用向量数据库存储历史关键信息（如用户偏好、已完成任务摘要），通过检索相关性后注入上下文。关键设计：需要对长期记忆做压缩和去重（例如使用 LLM 对同一主题的多条信息摘要合并）。同时我设置 retrieval threshold，只有当相似度 > 0.6 时才会注入，避免噪声干扰。
- **一句话记忆法：** 短期记忆是桌面上的便利贴，长期记忆是书柜里的文件夹，塞不下时先删最旧的。
- **复习标签：** #Agent #记忆管理

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 在长上下文 LLM 时代，RAG 系统还需要做“早期检索过滤”吗？或者说，既然模型能看 128K 了，我们直接检索 top-100 个 chunk 全塞进去不就好了？请从工程实践角度分析这样做的问题，并给出你的设计方案。

**面试口头版参考答案（约 200 字）：**
不推荐全塞进去。主要有三个问题：第一是**精度问题**，也就是 Lost in the Middle，模型只关注头尾，中间 50 个 chunk 几乎失效。第二是**成本问题**，128K 输入的 API 费用大约是 8K 输入的 16 倍，而且首 token 延迟会从几百毫秒增加到几秒。第三是**检索噪声问题**，top-100 里至少有 30% 是不相关或弱相关的文档，这些噪声反而会误导模型生成错误答案。我的做法是：先用轻量级检索（BM25 + 稠密向量混合）快速过滤掉无关文档，再用 cross-encoder 重排选出 top-3 到 top-5 个高相关 chunk，只把这些送给 LLM。这样既能保证精度，又能控制成本和延迟。

**深入理解版参考答案（约 450 字）：**
这个问题考察的是对 RAG 系统全链路开销的理解，不只是效果问题。我们先拆解一下“全塞进去”的代价：

1. **注意力稀疏性**：即使注意力机制理论上能看所有位置，但实验表明（参考 Liu et al. 2024 的 "Lost in the Middle" 论文），当相关文档位于中间位置时，准确率会从 80% 骤降到 40%。这是因为模型训练时句子序列长度有限，对于极长上下文，注意力权重会分布不均。所以全塞进去反而是“坏”的做法。

2. **经济成本**：以 GPT-4o 为例，输入每 1K tokens 约 $0.005，128K tokens 一次推理约 $0.64，而选择 top-3 chunks（假设每个 1K tokens）+ query 本身，总共不到 5K tokens，成本 $0.025。差距约 25 倍。如果每天 10 万次请求，成本从 $2500 降到 $250。

3. **延迟放大**：首 token 生成时间（TTFT）与输入长度近似线性增长。128K 输入对应的 TTFT 可能达到 10 秒级别，而 5K 输入不到 1 秒。这是无法接受的用户体验。

所以我的方案是**分级漏斗**：
- 第一层：粗滤。使用 BM25 + 向量检索取 top-50，计算速度快（几十毫秒），利用关键词语义双通道保证召回。
- 第二层：精滤。用 cross-encoder reranker（如 bge-reranker-v2-m3）对 top-50 排序，取 top-3。reranker 虽然单次慢（约 20ms 一个 pair），但只处理 50 对，总耗时约 1 秒，可控。
- 第三层：生成。将 top-3 chunks 按相关度降序排列后注入 LLM。

特殊情况：如果用户的 query 特别复杂（比如“把过去三年的财报总结一下”），确实需要多文档覆盖时，我会在 prompt 里加一个“请汇总以下多个段落的观点，逐个引用原文”，并把 top-10 个 chunk 传进去，但此时我会用一个专门做总结的 prompt 模板，并要求 LLM 先罗列每段的来源，再做汇总——这样可以强迫模型关注所有输入。

**可能追问与回答：**

- **追问 1**：如果检索的 top-3 都不相关（比如用户问了个系统里没有的问题），你怎么处理？
  **答**：我会在生成阶段加入“置信度校验”。在 prompt 里要求 LLM：“如果上下文不足以回答问题，请直接回答‘当前知识库未收录相关信息’。”同时我会在响应前检查 output embedding 与 context 的平均相似度，低于阈值则触发无答案处理。

- **追问 2**：你提到用 cross-encoder reranker，但如果请求非常频繁（如 100 QPS），reranker 能扛住吗？
  **答**：我会引入**选择性重排**策略：对于简单 query（特征：长度短、包含高频关键词），跳过 reranker 直接用向量检索的 top-3。只有复杂 query 才过 reranker。同时，我会把 reranker 部署在 GPU 上（比如 NVIDIA T4 就够），batch 处理多个查询的 rerank 请求，单卡可以支撑 50 QPS。如果超过瓶颈，就用模型量化（int8）和分片部署，以及前置语义缓存降低重复查询。
