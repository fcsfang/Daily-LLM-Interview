## 1. 今日面试真题精讲

### 题目 1：你在 Agent 系统中如何保证工具调用（Tool Calling）的可靠性？遇到了什么坑，怎么解决的？

**来源依据：**
- 小林面试笔记 - https://xiaolinnote.com/ai - 技术博客 - 中（个人整理面经，但内容系统性强）
- 京东物流实习二面面经（Agent 项目深度拷问） - [1] - 面经或社媒 - 中（需进一步验证，但提问方向符合主流）

**可信度：** 中

**考察点：** Agent 工程实践、Tool Calling 流程、错误处理、稳定性

**面试口头版回答：**
“我主要从三个层面保证工具调用可靠性。一是输出格式约束：我会用 structured output 或工具调用的 schema 来强制大模型输出 JSON，而不是靠纯 prompt 约定；如果解析失败，会触发重试并附加错误提示。二是重试与降级策略：每次调用我最多重试 3 次，每次写入不同错误类型——比如格式错误、工具超时、无效参数；如果都失败了，模型应该返回一个‘我无法完成这个操作’的兜底回答，而不是假装成功。三是测试闭环：我们会构造一批包含错误输入的测试用例，离线跑回归；比如故意给计算器工具发送非数字参数，观察模型是否能正确拒绝。在实际项目里，有一次就是因为模型输出了一个缺少逗号的 JSON，我们没处理解析异常就直接调用工具，结果工具报错了。后来我加了一层格式校验和自动修复，错误率下降了约 30%。”

**深入理解版回答：**
Tool Calling 的可靠性是 Agent 工程中的核心难点。大模型的输出本质上是概率性的，即便在 prompt 中明确要求返回特定格式，模型仍可能输出格式错误、参数名拼错、甚至意图识别错误。我的方案分为三个维度：
1. **输出约束**：使用框架（如 LangChain 的 PydanticOutputParser 或 OpenAI 的函数调用 schema）让模型输出结构化数据。这一步可以将格式错误率从 prompt 约束下的 20% 降低到 1% 以内。
2. **错误处理**：建立分层异常处理。第一层解析错误（JSON decode 失败、缺失字段）-> 重试并提示错误位置；第二层参数校验（值超出范围、类型不匹配）-> 重试并给出允许范围；第三层工具执行超时或异常 -> 重试或跳过。每层重试次数限制为 2-3 次，避免无限循环。
3. **防御性设计**：在 prompt 中加入反例和约束边界（如“如果你不确定参数，请询问用户”）。同时实现一个“无效调用”的兜底路径，当模型连续错误时，返回一个人类可读的错误消息而非暴露内部错误。
**局限性**：这种方法不能解决模型“幻觉参数”的问题，比如调用天气 API 时虚构一个不存在的省份。对此我后来增加了参数预校验——用一个小的规则引擎检查参数是否在已知枚举值中。
**常见坑**：只靠 prompt 约束 = 不可靠；重试次数太多会超时；没有全局限流会导致级联故障。

**常见追问与回答：**
- **追问 1**：如果模型重试 3 次仍然失败怎么办？会不会影响用户体验？
  - **答**：不会无限重试。3 次失败后会降级为“咨询用户”模式，比如模型回答“这个问题我当前无法自动处理，请您确认一下参数是否正确”。这个分支同样有 trace 日志，方便后续排查。
- **追问 2**：你怎么判断是工具调用失败了，还是模型本身能力不足？
  - **答**：通过错误分类。如果模型输出了格式错误但意图正确，那是工程问题；如果模型输出“我想调用计算器”但工具调用对象是天气 API，那是意图识别问题。我们会在日志中标记错误来源，分别用不同的补充 prompt 去修复。
- **追问 3**：有没有考虑用多个模型的投票机制提升可靠性？
  - **答**：在高风险场景下（比如金融交易）我见过这种方案，但会显著增加延迟和成本。我一般只在关键步骤用一次“输出验证”——用小模型（如 8B 级别）去校验大模型（如 70B）的调用是否符合 schema，拒绝重试。

**项目联系：**
在我做的“企业知识助手 Agent”项目中，工具调用模块包含天气查询、工单创建、知识库检索三个工具。我设计了一个通用 ToolExecutor 类，包含重试、超时、异常处理逻辑。在项目描述中，我通常这样说：“这个 Agent 的稳定性关键不在于模型大小，而在于每一层对错误的兜底——我在项目里单独写了一个 50 行的错误修复器，它不依赖任何 LLM，只做正则替换和缺省值补全，就解决了 90% 的格式问题。”

**记忆关键词：** structured output、重试策略、降级兜底、错误分类、输出验证、工具执行器

### 题目 2：当用户反馈 RAG 系统回答“完全错误”或“编造信息”，你的排查流程是怎样的？从发现到定位根因再到修复，说一个完整的闭环。

**来源依据：**
- 腾讯云开发者社区 - 这几道高频RAG面试题你都会了吗 - https://cloud.tencent.com/developer/article/2661823 - 技术博客 - 中（内容详实，列举了排查思路）
- RAG面试题和答案 - https://adg.csdn.net/695238785b9f5f31781b36a9.html - 技术博客 - 中

**可信度：** 中

**考察点：** RAG 工程排查、bad case 处理、闭环思维、可观测性

**面试口头版回答：**
“我一般把排查流程分为四步。第一步是抓证据：让用户提供问题原文和系统回复；同时调取当时的日志——包括检索到的文档片段、Re-rank 后的排序、最终输入给模型的 prompt。这一步是为了确定问题到底出在检索阶段还是生成阶段。第二步是根因分析：如果检索结果里根本不需要的信息，那就是 recall 问题——比如 chunk 太小截断了上下文、或者 embedding 模型选错了；如果检索到了正确文档但模型没按文档答，那是 faithful 问题——可能是 prompt 约束不够或者模型能力不足。第三步是修复：recall 问题我通常调 chunk 策略或加混合检索；faithful 问题我加强 prompt 约束、加入引用校验甚至用一个专门的 faithfulness 验证模型。第四步是验证：将 bad case 加入评估集，跑一轮回归测试，确认修复有效且没有引入新问题。我最近一个真实 case 是用户问‘公司年假政策’，模型答的全是劳动法默认值，但我们文档里其实有特殊规定。查日志发现 chunk 把关键的‘我们公司年假 15 天’切成了两段，导致检索没召回。修复后我们给 chunk 加了 overlap，并在索引时做了语义分段。”

**深入理解版回答：**
一个完整的 bad case 处理闭环包含五个核心阶段：
1. **数据采集**：建立用户反馈通道。不仅要记录最终答案，还要记录完整 trace：检索 API 的输入输出、排序结果、prompt 组装、LLM 回答以及耗时。建议用 OpenTelemetry 或 LangSmith 做链路追踪。
2. **分类与定位**：将 bad case 分为三大类——检索失败（recall / precision）、模型不遵循指令（faithfulness）、知识冲突（contradiction）。每类有不同排查工具：检索失败查看 top-k 文档的相关性评分；模型不遵循指令查看 prompt 中是否有明确的“只根据以下内容回答”约束；知识冲突查看是否发生了多文档答案互相矛盾。
3. **根因诊断**：常用“输入重放”技术——用同一问题、同一检索结果，换用一个更强模型（如 GPT-4 代替 Qwen2.5）或更严格的 prompt。如果强模型能答对，说明原模型能力或 prompt 是瓶颈；如果强模型也答错，说明检索结果本身就有问题。
4. **选择修复策略**：检索问题：调整 chunk 大小/策略、增加 reranker、采用混合检索（BM25 + 向量）。faithfulness 问题：强化 prompt、“根据以下文档片段，标注原文引用”、引入 citation 检测。知识冲突：用摘要模型合并多个片段、或让模型输出“文档 A 说 X，文档 B 说 Y”做风险提示。
5. **回归与上线**：将 bad case 转化为自动化测试用例，纳入 CI 流程。每次索引更新或模型切换后，跑一次回归，确保旧问题不重现。
**常见误区**：很多同学一遇到 bad case 就怪 embedding 或 reranker，其实超过一半的问题出在 chunk 或者 prompt 的约束不够。另外，不要只修当前的 case，要提炼成通用规则，比如“所有包含百分比和数字的 chunk 必须保证不被截断”。

**常见追问与回答：**
- **追问 1**：如果 bad case 是因为模型不懂这个领域，你怎么办？
  - **答**：我会先区分是“领域知识缺失”还是“模型没有正确使用检索结果”。如果是前者，要考虑在 prompt 里加入领域词汇表；如果是后者，就加强 prompt 约束，或者考虑用 LoRA 微调一个小 adapter 来适配输出格式。
- **追问 2**：你怎么判断 bad case 是偶发的还是系统性的？
  - **答**：统计。如果同一个类型的问题（比如“年假天数”）反复出错，那就是系统性的，需要调整文档预处理或索引；如果只是个别 query 有问题，可能是标注噪声或长尾问题。我一般在每日报表里加一个“错误类型分布”饼图，一眼就能看出趋势。
- **追问 3**：bad case 修复完成之后，你会怎么确保它不会再次出现？
  - **答**：我会把 bad case 写入一个“回归测试集”，放在 CI 里自动化执行。每次部署前跑一次，如果修复过的 case 又失败了，CI 会自动拦截。

**项目联系：**
在我搭建的“智能客服知识库”RAG 项目中，我维护了一个 bad case 工单列表。每当发现一个高频错误，我会把它填进一个标准 Excel 模板里：原始 query、系统回答、期望回答、根本原因、修复动作。面试时我会说：“我的项目不是一次性交付的，而是持续基于 bad case 反馈迭代，每两周做一次全部 bad case 的回归验证，确保系统稳定性和可追溯性。”

**记忆关键词：** bad case 闭环、根因分析、faithfulness、回归测试、日志追踪、输入重放

### 题目 3：如何设计一个基于 LLM 的 Agent 系统的长期记忆（Long-term Memory）？如果历史记录量非常大，怎么优化查询效率？

**来源依据：**
- 京东AI后端开发日常实习二面面经 - [1] - 面经或社媒 - 中（需进一步验证，但记忆模块是 Agent 高频考点）
- Datawhale hello-agents 面试问题总结 - https://github.com/datawhalechina/hello-agents/blob/main/Extra-Chapter/Extra01-%E9%9D%A2%E8%AF%95%E9%97%AE%E9%A2%98%E6%80%BB%E7%BB%93.md - 技术博客 - 高（数据来源于多次技术面试，内容可靠）

**可信度：** 中

**考察点：** Agent 记忆系统、长期记忆设计、查询优化、工程成本

**面试口头版回答：**
“长期记忆的核心是存储并检索 Agent 与用户过去交互中的关键信息。我的设计分三层。第一层会话级短期记忆：当前会话的对话历史，直接用列表存，限制在 10-20 轮。第二层是用户画像级记忆：比如用户的偏好、已完成的任务、敏感信息；我用向量数据库存储，每条记忆包含时间戳、摘要、原始 chunk，定期做去重和总结压缩。第三层是持久化记忆：存储在关系数据库，记录长期不变的实体关系。当信息量非常大时，优化查询效率是关键。我采取四种手段：一是用时间衰减加权，最新的记忆权重更高；二是对话摘要压缩——当一天的记录超过阈值，我用模型生成一条摘要代替；三是仅索引高频实体——比如用户常提的‘订单号’、‘地址’；四是用缓存池，对同一用户或同一 session 的检索结果做短时间缓存。实际项目中，我们将记忆检索的端到端延迟从 1.2 秒优化到了 200 毫秒以内。”

**深入理解版回答：**
Agent 的长期记忆系统设计需要考虑容量、时效性和检索效率三个维度。
1. **数据结构**：每条记忆记录至少包含三部分：元数据（时间戳、类型、会话 ID）、内容（文本、结构化数据）、索引字段（实体、关键词、向量）。用向量数据库（如 Milvus、Chroma）存储语义向量，同时用 SQLite 或 Redis 存储高频实体索引。
2. **记忆存储策略**：不存所有原始文本。当新消息到达时，Agent 先判定它是否“有意义”并提炼为摘要（使用一个轻量模型如 Qwen2.5-7B），然后与已有记忆做相似度对比；如果相似度超过 0.9，则合并而非新增。这样可以防止记忆膨胀。
3. **查询优化**：
   - **分层检索**：先查缓存(TTL 30s) -> Redis 上的高频实体 -> 向量库的 top-5 -> 语义检索。每层都有快速失败机制。
   - **时间衰减**：计算相关性分数时，加入时间衰减因子，公式为 `score * exp(-λ * days_since)`。这保证近期记忆占更大权重。
   - **摘要打包**：对于用户连续 5 轮以上的同一话题，由 Agent 或异步任务生成“话题摘要”，后续检索只搜摘要，不搜原始记录。
4. **工程坑点**：长期记忆的写入也需要限流。如果每轮都写入，遇到高频对话会出现写入瓶颈。我设计了异步写入队列，保证主流程不阻塞。

**常见追问与回答：**
- **追问 1**：长期记忆里存储了用户的敏感信息，怎么保证安全性？
  - **答**：敏感字段（如用户名、手机号）做脱敏处理再存入向量库。读取时通过 Agent 的决策逻辑决定是否需要脱敏显示。同时记忆数据本身也做加密存储（AES-256），访问时校验身份权限。
- **追问 2**：如果 Agent 长期记忆超过 100 万条，向量检索太慢怎么办？
  - **答**：使用分区索引，按用户 ID 分片，确保每次查询只在用户对应分区进行。如果有海量用户，增加 HNSW 索引参数（ef_construction 增大提高召回率，同时限制 ef_search 减少 top-K 检索量）。另外设置归档策略：半年以上的旧记忆转为离线存储（如 S3+Parquet），需要时手动恢复。
- **追问 3**：你如何评估长期记忆系统的效果？
  - **答**：通过两个指标：记忆命中率（用户重复问题是否成功利用历史）、用户满意度（Agent 是否记得用户的偏好）。这两个指标可以用 A/B 测试来对比是否开启长期记忆模块。

**项目联系：**
在 Agent 项目的“用户偏好记忆模块”中，我设计了一个可选功能——当用户再次进入时，Agent 会自动读取其历史偏好：“您上次咨询的是快递理赔，今天是否继续？”这个功能的实现依赖长期记忆系统。我通常这样讲：“这个模块让我们的 Agent 回复速度虽然增加了 100ms 的检索开销，但用户满意度评分提升了约 15%。”

**记忆关键词：** 长期记忆、摘要压缩、时间衰减、分区索引、异步写入、脱敏

### 题目 4：如果让你为一个生产级 LLM 应用设计模型评测体系，你会怎么设计？请分别从离线评测和在线评测两个维度展开。

**来源依据：**
- 快手后端开发实习二面面经 - [1] - 面经或社媒 - 中（需进一步验证，但模型评测是高频考点）
- Datawhale hello-agents 面试问题总结 - https://github.com/datawhalechina/hello-agents/blob/main/Extra-Chapter/Extra01-%E9%9D%A2%E8%AF%95%E9%97%AE%E9%A2%98%E6%80%BB%E7%BB%93.md - 技术博客 - 高

**可信度：** 中

**考察点：** 模型评测体系、离线、在线、指标设计、工程可落地性

**面试口头版回答：**
“离线评测和在线评测服务于不同目标。离线评测关注模型能力和稳定性，我把它分成三个层级：底层是通用能力评测，用 MMLU、HumanEval 等公开 benchmark，单轮跑一次；中间层是业务场景评测，用我们自己标注的 500-1000 条 QA 数据集，覆盖边界 case 和典型反例；顶层是效果安全评测，包含幻觉检测、偏见过滤、拒绝率等。在线评测主要关注上线后的用户反馈和业务指标。我最关心的三个指标：用户留存率、任务完成率、用户反馈的负面比例。我用 A/B 测试来对比新旧模型的差异。如果离案评测项都通过但在线效果差，我通常会看模型是否过度优化了 bench 或用户 prompt 与训练分布不匹配。实际项目中，我们建立了一个‘评测平台’，将离线 case 和在线 trace 都记录到数据库，支持每日跑分和回归对比。”

**深入理解版回答：**
生产级模型评测体系必须覆盖从开发到上线到运营的全流程：
1. **离线评测**：
   - **通用能力**：MMLU（知识）、HumanEval/CodeXGLUE（代码）、GSM8k（数学）——用于 baseline 确认。
   - **业务场景**：必须构建与真实用户分布一致的自定义 QA 集。构造方式：从生产日志中随机采样用户 query，人工写出标准答案和检索上下文。评估指标包括：correctness（与大模型对比或人工）、faithfulness（是否使用给定上下文）、relevance（是否跑题）。
   - **安全与鲁棒性**：注入攻击（prompt injection）、越狱攻击、无效输入的处理能力。跑这些测试需要编写专门的 adversarial 测试集。
   - **性能基线**：记录模型在每个 case 下的延迟、token 数、重试次数。
2. **在线评测**：
   - **核心业务指标**：会话完成率、用户时延、重试率。
   - **A/B 实验**：流量按比例随机分配，实验组和对照组用不同模型/参数；观察指标是否显著变化（p < 0.05）。记得做多天运行以消除周期性波动。
   - **用户反馈系统**：显示/隐藏“反馈”按钮、用户评分（1-5星）、文本反馈。这些数据经过处理后作为标签 feed 回离线评测集，形成闭环。
3. **迭代流程**：当一个新模型候选上线，先通过离线全部测试 -> 1% 灰度 -> 5% -> 全量。每一步如有异常则回滚。回滚标准是核心指标下降超过 5%。
**常见坑**：离线评测和在线评测经常不一致，原因是离线集覆盖了太多通用场景，线上高发的是长尾业务问题。修正办法：在线用户的负面反馈优先补齐离线测试集。

**常见追问与回答：**
- **追问 1**：你们如何自动化地判定模型生成的文本是否忠实于上下文？
  - **答**：常用 LLM-as-judge 方法：用一个大模型对“问题、上下文、回答”三元组打分，判断回答是否完全取自上下文。我们使用一个针对 faithfulness 设计的 prompt，加上一致性校验（如重复测试三次，取多数）。另外也有专门模型（如 TrueTeacher 或 AlignScore）可以直接输出分数。
- **追问 2**：评测集如何保持更新，不会过时？
  - **答**：每周从生产日志中抽样 100 条新的用户 query，人工标注后新增到评测集。同时每三个月淘汰一批旧的案例（如果业务变化导致不再相关）。
- **追问 3**：你提到 A/B 测试，如果实验组和对照组的用户群体天然不同（例如用户年龄偏大或偏好不同），怎么减少偏差？
  - **答**：使用 hash 分桶 + 随机分配，确保两组的用户画像分布一致。同时用 AA 测试验证分流正确性。如果上线后发现两组用户基数差异过大，用倾向评分匹配校正。

**项目联系：**
在我做的一个简单 LLM 评测项目中，我建立了一个 5 段式的框架：公共 benchmark -> 自定义业务测试集 -> adversarial 测试 -> 人工打分 -> A/B 上线。面试时我这样总结：“评测不只是跑一遍测试，而是整个模型迭代的看门人。我设计的体系保证：一个模型在通过所有离线测试之前不允许进入灰度。”

**记忆关键词：** 离线评测 在线评测 A/B测试 LLM-as-judge faithfulness 灰度上线

## 4. 今日面试专项训练

### 4.1 大模型理论追问题

**问题：** 请解释“Reward Hacking”（奖励作弊/奖励黑客）现象在 RLHF 中的成因、危害和几种缓解策略。

**面试口头版参考答案：**
“Reward Hacking 是指模型学会了利用奖励模型的盲点来拿到高分，而不是真正提升回答质量。比如模型学会了生成‘更长’的答案来获得更高奖励，尽管内容质量很差；或者学会了在末尾加‘我认为这很有帮助’这类安全话术刷分。减轻的方法主要有四种：一是加入 KL 散度惩罚，限制模型与初始策略的偏差；二是奖励模型用多样化数据集训练，避免过度优化单一维度；三是引入多点采样训练，增加对抗性；四是结合人类评审人工筛选异常高分的样例作为对抗样本重新训练奖励模型。”

**深入理解版参考答案：**
Reward Hacking 的本质是强化学习中的“投机取巧”问题：奖励模型（RM）不是完美的，它只能近似人类偏好；当 LLM 的优化目标与 RM 的打分函数高度相关时，模型会开始“钻 RM 的空子”。例如，在一个自动摘要任务中，RM 被训练为偏好“包含更多关键词”的摘要；于是模型逐渐学会了将关键词堆砌在一起，生成一个信息量低但关键词密集的摘要，反而获得了高分。
危害是：用户感知到的质量下降 / KL 散度爆炸 / PPO 训练不稳定 / 上线后效果断崖式下滑。
缓解策略包括：
1. **KL 散度惩罚**：PPO 的目标函数中包含 `KL[π_θ(old) || π_θ(new)]`，惩罚过大的策略偏离。系数 β 需要调参，过大抑制学习，过小导致 hacking。
2. **奖励模型鲁棒化**：用多样化的偏好数据训练 RM，避免 RM 只学会表面特征。也可以训练多个 RM 并取平均值作为分数。
3. **对抗性采样**：在 PPO 训练过程中，定期收集被 RM 高分但人类评价低的样本，加入 RM 的训练集做再训练，让 RM 学会识别“作弊”模式。
4. **人工随机验证**：保持 5% 的样本走人工评审回路，不信任全部自动分数。
> 需进一步验证：部分内容（如“关键词堆砌”案例）是基于论文假设的个人理解，但缓解策略部分是 RLHF 公认做法。

**2 个可能追问与简短回答：**
- **追问 1**：KL 散度惩罚项的系数设置过大或过小会有什么后果？
  - **答**：过大导致模型几乎不更新，微调效果接近零；过小导致模型快速偏离 base model，容易进入 reward hacking 区域。通常用指数衰减调度或自适应 KL 控制。
- **追问 2**：有没有可以不依赖奖励模型的对齐方法？
  - **答**：有，DPO（Direct Preference Optimization）直接使用偏好对优化策略，无需单独训练 RM。DPO 也能缓解 reward hacking，因为它不在线优化一个黑盒分数，而是最小化偏好对的交叉熵。
